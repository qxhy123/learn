# 前言：如何阅读这门 Linux 文件系统教程

如果你已经会用 Linux，你很可能早就“在使用文件系统”了：

- 你每天在路径之间切换
- 你会创建、复制、删除、移动文件
- 你知道权限不对时会报错
- 你也许听说过 inode、ext4、挂载、page cache、overlayfs

但“会用”与“理解”之间，往往隔着一整套系统抽象。

很多困惑都来自同一个问题：

> 我们平时操作的“文件”，在内核和磁盘里到底对应什么？

本教程会把这个问题拆成五层：

1. **用户态名字空间**：路径、目录、工作目录、权限、链接
2. **内核对象层**：dentry、inode、file、fd、mount、superblock
3. **缓存与持久化层**：page cache、writeback、journal、`fsync`
4. **具体文件系统层**：ext4、xfs、btrfs、zfs、overlayfs、tmpfs、NFS
5. **工程语义层**：可见性、原子性、持久性、隔离性、性能与恢复

只要你能同时看见这五层，很多平时显得“古怪”的现象都会突然变得合理。

## 如何读得更有效

### 1. 把“名字”和“对象”分开

文件名不是文件内容，目录也不是文件内容的“容器”本身。目录首先是一个“名字到对象编号”的映射层；对象本体通常由 inode 描述，数据再由块或 extent 组织。只要这一层拆开，你会突然理解很多现象。

### 2. 看到异常时，先定位是哪一层的问题

例如：

- `Permission denied` 多半与权限、挂载参数、ACL 或 namespace 相关
- `No space left on device` 不一定是磁盘满了，也可能是 inode 用尽
- 文件删除后空间不释放，常常与“仍有进程持有打开文件”有关
- 容器里看不到宿主机某目录，可能是 mount namespace、propagation 或 bind mount 的结果
- `write()` 成功后断电仍丢数据，多半是缓存、writeback、日志模式与目录持久化语义没分清

### 3. 不要把“快”与“安全”混为一谈

page cache、批量回写、journal、`fsync`、direct I/O、mmap 各自优化的是不同目标。理解文件系统时，必须同时看性能、持久化和崩溃恢复语义。

### 4. 每遇到一个机制，都问它到底保证什么

例如：

- `rename` 更偏向目录项切换的原子性，不天然等于“数据一定已经安全写到磁盘”
- `fsync(file)` 关注文件数据与相关元数据，不自动等于父目录项也已经持久化
- NFS 的 close-to-open 是可接受的工程折中，不是“和本地 ext4 一模一样”
- overlayfs 的 copy-up 和 whiteout 是联合视图的语义工具，不是“复制一整棵树”

### 5. 用命令观察真实系统

本教程会反复建议使用如下命令做观察：

```bash
pwd
ls -li
stat <path>
df -h
df -i
du -sh <dir>
findmnt
mount | head
lsblk
lsof +L1
grep -E 'Dirty|Writeback' /proc/meminfo
```

这些命令大多是只读的，足以帮助你建立直觉。

## 阅读顺序建议

- 如果你是初学者：先读第1-4章，再读第9章和第11章
- 如果你已经会用 Linux：可从第5章、第6章切入，快速补上 fd、path walk 和缓存语义
- 如果你做容器或存储：重点看第9章、第10章、第15章、第16章、第17章
- 如果你做性能排障：重点看第11章、第12章、第13章、第14章、第18章
- 如果你最关心“崩溃后到底会怎样”：重点看第8章、第10章、第12章

## 什么叫“真正学会”

当你能解释下面这些问题时，说明你已经从“会用命令”走向了“理解系统”：

- 路径查找为什么需要 dentry 和 inode 两层？
- 为什么 `unlink` 之后文件对象可能还活着？
- 为什么 `rename` 常被用于可靠写入，但还要讨论目录 `fsync`？
- 为什么 page cache 会让“性能测试结果”看起来比设备本身更快？
- 为什么 overlayfs 适合容器镜像分层？
- 为什么网络文件系统很难完整复制本地 POSIX 语义？

## 本教程不追求什么

本教程不追求把某个文件系统源码逐函数讲完，也不追求覆盖全部 Linux 存储栈。它更重视：

- 建立稳定的系统地图
- 解释常见工程现象背后的根因
- 帮你知道下一步应该深入哪里

欢迎开始阅读。建议你边读边观察自己的系统，而不是只把这些概念当作名词表来背。
