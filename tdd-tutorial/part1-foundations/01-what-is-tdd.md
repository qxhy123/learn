# 第一章：TDD 是什么？

## 1.1 定义

**测试驱动开发（Test-Driven Development，TDD）** 是一种软件开发方法论：**在编写生产代码之前，先编写测试代码**。

这不仅仅是"先写测试"，它是一种设计工具——测试迫使你在实现之前思清楚接口和行为。

---

## 1.2 Red → Green → Refactor 循环

TDD 的核心是一个三步循环，通常称为 **RGR 循环**：

```
    ┌─────────────────────────────────────────────┐
    │                                             │
    │   RED         GREEN        REFACTOR         │
    │  (失败)  →   (通过)   →   (重构)            │
    │    ↑                         │              │
    │    └─────────────────────────┘              │
    │                                             │
    └─────────────────────────────────────────────┘
```

### Red（红）：写一个失败的测试
- 描述你期望的行为
- 测试此时必须失败（否则说明测试没有价值）
- 确认失败原因是"功能未实现"，而非"测试写错了"

### Green（绿）：写最少的代码让测试通过
- **不要**写超出测试需要的代码
- 哪怕实现看起来很丑，先让测试通过
- "让测试通过"是唯一目标

### Refactor（重构）：改善代码，保持测试绿色
- 消除重复、改善命名、提升结构
- **每次重构后必须运行所有测试**
- 如果测试变红，立即回滚

---

## 1.3 为什么 TDD 有价值？

### 设计视角
```
传统开发：实现 → 设计接口（被迫）
TDD 开发：  设计接口（主动）→ 实现
```

先写测试迫使你站在**调用者**的角度设计 API。结果通常：
- 接口更简洁
- 依赖更清晰
- 耦合更低

### 安全网效应
每一个测试都是未来重构的安全网。当代码库有 500 个测试时，你可以**放心重构**，因为任何行为回归都会立刻暴露。

### 文档效应
```python
def test_empty_cart_has_zero_total(self):
    cart = ShoppingCart()
    self.assertEqual(cart.total(), 0)

def test_adding_item_increases_total(self):
    cart = ShoppingCart()
    cart.add_item(Item("apple", price=5.0))
    self.assertEqual(cart.total(), 5.0)
```

这两个测试比任何注释都更精确地描述了 `ShoppingCart` 的行为。

## 1.4 TDD 的常见误解

| 误解 | 真相 |
|------|------|
| "TDD 会让开发变慢" | 短期可能慢，长期节省大量调试时间 |
| "TDD 要求 100% 覆盖率" | TDD 关注行为，不是覆盖率数字 |
| "有了 TDD 就不需要 QA" | TDD 测试开发者的假设，QA 探索用户场景 |
| "TDD 只适合简单代码" | 复杂代码更需要 TDD 来管理复杂性 |
| "先写代码再补测试也一样" | 补写测试会被现有实现绑架，失去设计驱动的价值 |

---

## 1.5 TDD vs 其他方法

```
BDD（行为驱动开发）
  └── 是 TDD 的扩展，关注业务行为描述
  └── 常用 Given/When/Then 语法
  └── unittest 也可以实践 BDD 风格

ATDD（验收测试驱动开发）
  └── 从用户故事出发写测试
  └── 通常需要产品、开发、测试三方协作

测试后置（Test-Last）
  └── 先实现再写测试
  └── 测试往往验证实现，而非行为
```

---

## 1.6 FIRST 原则

好的测试应满足 FIRST 原则：

| 字母 | 含义 | 要点 |
|------|------|------|
| **F**ast | 快 | 单个测试 < 100ms，测试套件 < 10s |
| **I**ndependent | 独立 | 测试之间不相互依赖，可任意顺序运行 |
| **R**epeatable | 可重复 | 在任何环境运行结果相同 |
| **S**elf-validating | 自验证 | 测试本身判断通过/失败，无需人工检查 |
| **T**imely | 及时 | 在生产代码之前或同时编写 |

---

## 1.7 本章小结

- TDD 是"先测试后实现"的开发循环
- 核心：Red → Green → Refactor
- 价值：设计工具 + 安全网 + 活文档
