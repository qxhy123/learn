# 附录 C：练习参考答案

本附录为全部 24 章的练习题提供简要答案提示。这些提示旨在指引思考方向，而非给出完整答案——深入理解需要你自己动手实践。

> **使用建议**：先独立完成练习，遇到困难时再参考本附录。直接看答案会大幅降低学习效果。

---

## 第一部分：基础概念

### 第1章：Harness Engineering 的起源与动机

**练习 1.1**：列举 3 个因缺乏 Harness 导致的 AI 系统生产事故。
> 提示：从 Prompt 注入攻击（如 Bing Chat 早期 Jailbreak）、模型幻觉导致的错误信息传播、以及因缺乏监控而未及时发现的性能退化三个方向思考。

**练习 1.2**：对比传统软件系统和 AI 系统在"从 Demo 到生产"过程中面临的不同挑战。
> 提示：关键区别在于非确定性。传统软件的 Bug 是可复现的，而 AI 系统的问题可能是概率性的。这导致测试策略、回归策略和监控策略都需要根本性的改变。

**练习 1.3**：用你自己的项目经验，描述一个"如果有 Harness 就能避免"的问题。
> 提示：思考你遇到过的 LLM 输出质量问题、Prompt 调整后的意外退化、或者生产环境中难以排查的异常行为。

### 第2章：核心心智模型：Agent = Model + Harness

**练习 2.1**：选择一个开源 Agent 项目，分析其 Harness 成分占代码总量的比例。
> 提示：以 AutoGPT 或 Open Interpreter 为例，统计与模型调用直接相关的代码 vs 提示词管理、错误处理、输出解析、日志记录等 Harness 代码的比例。通常 Harness 代码占 60-80%。

**练习 2.2**：如果模型能力提升 10 倍，Harness 能否简化？哪些部分能简化，哪些不能？
> 提示：评估和可观测性不会因为模型更强而消失——你仍然需要知道系统在做什么、做得好不好。但某些护栏（如格式纠正）和重试逻辑可能可以简化。

**练习 2.3**：画出 Harness 五层架构图（评估/测试/编排/可观测/安全），标注各层之间的数据流。
> 提示：注意评估层的输出会反馈到编排层（指导 Prompt 调整）；可观测性层横跨所有其他层；安全层既有输入端护栏也有输出端护栏。

### 第3章：Guides 与 Sensors：前馈控制与反馈控制

**练习 3.1**：为一个客服 Agent 分别设计 3 个 Guide 和 3 个 Sensor。
> 提示：Guide 示例——系统提示中的角色定义、Few-shot 示例、可用工具的定义与描述。Sensor 示例——情感分析检测器（检测用户不满）、幻觉检测器（交叉验证回答与知识库）、响应延迟监控。

**练习 3.2**：解释为什么只有 Guide 没有 Sensor（或反之）的 Harness 是不完整的。
> 提示：只有 Guide 相当于"开环控制"——你告诉系统该怎么做，但不知道它是否做到了。只有 Sensor 相当于"只有温度计没有空调"——你知道温度不对，但没有调节手段。

**练习 3.3**：设计一个 Guide-Sensor 闭环：当 Sensor 检测到问题时，自动调整 Guide。
> 提示：一个简单的闭环——Sensor 检测到输出过长，动态在 Guide 中注入"请简洁回答，不超过 100 字"的指令。更复杂的闭环可以根据评估分数动态选择不同的 Few-shot 示例。

### 第4章：Harness Engineering 与相邻学科

**练习 4.1**：Harness Engineering 从控制论中借鉴了哪些核心概念？
> 提示：前馈控制（Feedforward）、反馈控制（Feedback）、闭环系统（Closed-loop）、稳态与振荡（Steady State & Oscillation）。Guide 对应前馈控制器，Sensor 对应传感器和反馈回路。

**练习 4.2**：对比 Harness Engineering 和传统 MLOps 在关注点上的差异。
> 提示：MLOps 关注模型的训练、版本管理和部署；Harness Engineering 关注模型之上的控制层——评估、编排、护栏。两者互补而非替代。Harness Engineering 更关注"运行时行为控制"而非"训练流水线管理"。

**练习 4.3**：从可靠性工程的角度，AI 系统的"故障模式"与传统软件有何不同？
> 提示：传统软件的故障通常是二元的（崩溃或不崩溃），AI 系统的故障往往是渐进的（质量缓慢退化）和概率性的（有时对有时错）。这要求不同的故障检测和恢复策略。

### 第5章：Harness 工程师的技能图谱与工具链

**练习 5.1**：评估你自己在 Harness Engineering 四大技能域（评估设计/统计分析/编排架构/可观测性）中的强弱项。
> 提示：诚实地给每个技能域打 1-5 分，识别最薄弱的环节，制定针对性的学习计划。本教程的设计就是帮你补齐短板。

**练习 5.2**：搭建本教程的完整开发环境，确认所有依赖可以正常安装。
> 提示：至少需要 Python 3.11+、pip 安装 lm-eval/deepeval/langfuse SDK、一个 LLM API Key（OpenAI 或 Anthropic）。用 `python -c "import deepeval; print(deepeval.__version__)"` 验证。

## 第二部分：评估 Harness

### 第6章：评估 Harness 基础

**练习 6.1**：区分离线评估和在线评估各自适合的场景。
> 提示：离线评估适合模型选型、Prompt 迭代、回归测试——数据集固定、可重复。在线评估适合生产监控、A/B 测试——使用真实流量、反映实际表现。两者结合才是完整的评估策略。

**练习 6.2**：设计一个最小评估流水线的伪代码。
> 提示：核心组件只需四个——DataLoader（加载评估集）、Executor（运行推理）、Metric（计算指标）、Reporter（输出结果）。先把这四个组件的接口定义清楚，再实现具体逻辑。

**练习 6.3**：解释为什么"准确率"对 LLM 评估来说通常不够用。
> 提示：LLM 的输出是开放式的，同一个正确答案可以有无数种表达方式。精确匹配会低估真实质量。需要语义层面的评估指标（嵌入相似度、LLM-as-Judge 等）。

### 第7章：lm-evaluation-harness 架构剖析

**练习 7.1**：用 lm-eval-harness 运行一个标准基准测试，分析结果报告。
> 提示：从 `lm_eval --model hf --model_args pretrained=gpt2 --tasks hellaswag --num_fewshot 0` 开始。重点关注结果中的 `acc` 和 `acc_norm`，理解 normalized accuracy 的含义。

**练习 7.2**：阅读 lm-eval-harness 中一个 Task 的 YAML 配置，解释每个字段的作用。
> 提示：关注 `dataset_path`、`doc_to_text`、`doc_to_target`、`metric_list` 这几个核心字段。它们分别定义了数据来源、输入构造、目标构造和评估指标。

**练习 7.3**：描述 lm-eval-harness 中 Filter 机制的作用和典型用法。
> 提示：Filter 用于在计算指标前对模型输出进行后处理——例如提取 JSON 中的答案字段、去除多余空格、截取第一行等。这是连接"模型原始输出"和"可评估输出"的桥梁。

### 第8章：构建自定义评估 Harness

**练习 8.1**：为一个文本摘要任务设计评估指标体系。
> 提示：至少包括三个维度——信息覆盖度（摘要是否涵盖关键信息）、忠实度（摘要是否与原文一致、无幻觉）、流畅度（语言质量）。可以用 ROUGE 作为基线，LLM-as-Judge 作为主要指标。

**练习 8.2**：实现一个 LLM-as-Judge 评估器，并用人工标注数据校准。
> 提示：关键步骤——设计评分 Prompt（定义评分维度、分值含义、评分标准）→ 收集 50+ 人工标注样本 → 计算 LLM 评分与人工评分的相关系数 → 调整 Prompt 直到相关系数 > 0.8。

**练习 8.3**：解释为什么评估结果需要统计显著性检验。
> 提示：如果评估集只有 100 条，模型 A 的准确率是 82%，模型 B 是 80%，这个差异可能完全是随机噪声。用 bootstrap 或 paired t-test 来判断差异是否具有统计显著性。

### 第9章：RAG 管线评估 Harness

**练习 9.1**：解释 RAGAS 框架中 Faithfulness 和 Answer Relevancy 指标的区别。
> 提示：Faithfulness 检查生成的答案是否忠于检索到的上下文（不幻觉）；Answer Relevancy 检查答案是否与用户问题相关（不跑题）。一个可以忠实但不相关，也可以相关但不忠实。

**练习 9.2**：为什么 RAG 评估需要分别评估检索质量和生成质量？
> 提示：如果只做端到端评估，当结果不好时你不知道问题在哪——是检索没找到相关文档，还是找到了但生成器没用好？分离评估才能精确定位问题环节。

**练习 9.3**：设计一个检测 RAG 系统"知识过时"问题的评估方案。
> 提示：构建一个包含时间戳的评估集——旧知识和新知识各占一半。监控系统在新知识上的准确率趋势。当新知识准确率显著低于旧知识时，说明知识库需要更新。

### 第10章：Agent 系统评估 Harness

**练习 10.1**：Agent 评估与 LLM 评估的核心区别是什么？
> 提示：Agent 评估需要评估多步骤的序列行为（轨迹），而不仅仅是单次输入-输出。还需要评估工具调用的正确性、步骤效率、以及任务完成的最终结果。

**练习 10.2**：设计一个 Agent 轨迹评估的评分方案。
> 提示：评估维度——任务完成度（最终是否完成目标）、步骤效率（用了多少步、是否有无效步骤）、工具使用正确性（是否调用了正确的工具、参数是否合理）、错误恢复能力（遇到错误是否能自行修复）。

**练习 10.3**：SWE-bench 的 Harness 设计有哪些值得借鉴的地方？
> 提示：关键设计——用真实的 Git 仓库和测试用例作为评估环境（不是人造数据）、用测试通过率作为客观指标（不依赖主观评分）、环境隔离确保评估的可复现性。

## 第三部分：测试与 CI Harness

### 第11章：AI 系统的测试 Harness 模式

**练习 11.1**：为一个 LLM 调用链编写 Mock 测试，不依赖真实 API。
> 提示：用 `unittest.mock.patch` 替换 LLM API 调用，返回预设的响应。Mock 的粒度很重要——Mock 整个 LLM 调用可以测试下游逻辑，但无法测试 Prompt 质量。

**练习 11.2**：解释 AI 系统中"契约测试"的含义。
> 提示：契约测试检查的不是具体输出内容，而是输出的结构和约束——例如"返回的 JSON 必须包含 answer 字段"、"回答长度不超过 500 字"、"不包含 PII"。这些契约在模型更换后仍然成立。

**练习 11.3**：传统单元测试的"断言相等"在 AI 系统中为什么失效？用什么替代？
> 提示：因为 LLM 输出是非确定性的。替代方案——属性断言（输出是合法 JSON）、语义断言（嵌入相似度 > 阈值）、分布断言（多次运行的平均分 > 阈值）、LLM 断言（用 LLM 判断输出是否满足条件）。

### 第12章：测试驱动微调（TDF）Harness

**练习 12.1**：TDF 的核心理念与传统 TDD 有何异同？
> 提示：相同点——先写"测试"（评估集）再写"代码"（微调）。不同点——TDD 的测试是精确的（通过/失败），TDF 的评估是统计的（分数是否显著提升）；TDD 的迭代周期是秒级，TDF 的迭代周期是小时级。

**练习 12.2**：如何检测微调过程中的过拟合？
> 提示：维护一个独立的验证集（不参与微调）。在每个训练检查点上同时评估训练集和验证集。当训练集指标继续上升但验证集指标开始下降时，就是过拟合的信号。

**练习 12.3**：设计一个微调质量门禁的检查清单。
> 提示：至少包括——验证集指标不低于基线 X%、不出现灾难性遗忘（通用能力保持）、毒性指标不上升、推理延迟不超过容忍值、人工抽检通过率 > Y%。

### 第13章：CI/CD 集成与质量门禁

**练习 13.1**：设计一个 GitHub Actions 工作流，在 PR 中自动运行评估。
> 提示：核心步骤——checkout 代码 → 安装依赖 → 运行评估脚本 → 将结果与基线对比 → 用 `github-script` Action 在 PR 中发布评估报告评论。注意控制评估成本（采样而非全量）。

**练习 13.2**：如何在 CI 中平衡评估的全面性和成本？
> 提示：分层策略——PR 阶段只运行快速检查（100 条样本、低成本指标）；合并到主分支后运行中等规模评估（1000 条样本、包含 LLM-as-Judge）；发布前运行全量评估。

**练习 13.3**：评估结果如何版本化管理？
> 提示：将评估结果以 JSON 格式保存在 Git 中（或用 DVC/MLflow）。每次评估记录——模型版本、Prompt 版本、评估集版本、评估指标、时间戳。用这些数据绘制质量趋势图。

### 第14章：非确定性系统的回归测试

**练习 14.1**：为什么对同一输入运行 1 次 LLM 调用不足以做回归测试？
> 提示：1 次调用的结果受随机种子影响，可能恰好特别好或特别差。需要 N 次采样来估计输出质量的分布，用置信区间而非点估计来判断是否回归。

**练习 14.2**：实现一个基于 bootstrap 的置信区间计算。
> 提示：对 N 个评估分数进行有放回抽样 B 次（B=1000），每次计算均值，取 B 个均值的 2.5% 和 97.5% 分位数作为 95% 置信区间。当新版本的置信区间与旧版本不重叠时，认为有显著差异。

**练习 14.3**：设计一个 A/B 评估框架的核心接口。
> 提示：核心接口——`run_ab_evaluation(model_a, model_b, dataset, metrics, n_samples)` → 返回每个指标的效应量（Effect Size）、p 值和置信区间。支持 paired 和 unpaired 两种模式。

## 第四部分：编排 Harness

### 第15章：单 Agent Harness 设计

**练习 15.1**：从零实现一个单 Agent Harness 的最小骨架。
> 提示：核心组件——InputValidator（验证输入格式和安全性）→ PromptBuilder（构造 Prompt）→ LLMClient（调用模型）→ OutputParser（解析输出）→ RetryHandler（错误重试）。先定义接口，再实现具体逻辑。

**练习 15.2**：为 Agent Harness 设计一个状态机，处理正常流程和异常流程。
> 提示：状态至少包括——INIT → VALIDATING → EXECUTING → PARSING → VALIDATING_OUTPUT → DONE。异常状态——RETRYING、FALLBACK、FAILED。用显式状态机避免嵌套 if-else 的复杂性。

**练习 15.3**：讨论"胖 Harness"和"瘦 Harness"的设计权衡。
> 提示："胖 Harness"把更多逻辑放在 Harness 中（详细的输入处理、多层重试、复杂的输出后处理），模型的任务更简单。"瘦 Harness"更信任模型能力，Harness 只做最小控制。选择取决于模型能力和可靠性要求。

### 第16章：OpenAI Harness 蓝图

**练习 16.1**：用 OpenAI Agents SDK 实现一个支持 Handoff 的双 Agent 系统。
> 提示：Agent A（Triage Agent）负责理解用户意图，Agent B（Specialist Agent）负责执行具体任务。在 Agent A 的工具列表中定义 `transfer_to_specialist` 函数，触发 Handoff。

**练习 16.2**：分析 OpenAI 百万行代码实验中 Harness 的关键设计决策。
> 提示：关键决策——将评估嵌入开发循环（不是事后补丁）、用真实代码库而非人造任务做评估、Harness 代码量远超 Agent 逻辑本身、渐进式地增加 Harness 复杂度。

**练习 16.3**：对比 OpenAI Agents SDK 和 LangGraph 在 Harness 设计理念上的差异。
> 提示：Agents SDK 偏"声明式"——定义 Agent 的指令和工具，框架处理编排细节。LangGraph 偏"命令式"——你显式定义状态图的节点和边。前者上手更快，后者控制力更强。

### 第17章：多 Agent Harness 架构

**练习 17.1**：对比委托模式、路由模式和并行模式各自的适用场景。
> 提示：委托——上级 Agent 分解任务给下级（适合层次化任务）。路由——根据输入类型分发到不同 Agent（适合多领域问题）。并行——多个 Agent 同时处理同一输入（适合需要多视角的场景）。

**练习 17.2**：设计 Agent 间通信的消息格式。
> 提示：消息至少包括——sender_id、receiver_id、message_type（request/response/handoff/error）、payload（任务描述或结果）、context（共享上下文摘要）、trace_id（用于追踪）。

**练习 17.3**：多 Agent 系统中如何防止"无限循环"？
> 提示：三种机制——最大步骤数限制（硬上限）、循环检测（检测相同状态的重复出现）、Token 预算控制（总消耗超过阈值时强制终止）。在 Harness 层实现，而非依赖 Agent 自己判断。

### 第18章：长时运行 Agent Harness

**练习 18.1**：设计一个检查点机制的数据模型。
> 提示：检查点至少保存——task_id、step_index、agent_state（当前 Agent 的内部状态）、conversation_history（对话历史摘要）、intermediate_results（中间结果）、timestamp。支持序列化到 JSON 或数据库。

**练习 18.2**：如何在长时运行 Agent 中实现"人类介入"（Human-in-the-Loop）？
> 提示：在 Harness 中定义"暂停点"——当 Agent 的置信度低于阈值、或即将执行高风险操作时，暂停执行、持久化状态、通知人类审批。人类审批后恢复执行。

**练习 18.3**：长时运行 Agent 的超时策略应该如何设计？
> 提示：分层超时——单次 LLM 调用超时（30s）、单个步骤超时（5min）、整个任务超时（1h）。超时后不是直接失败，而是进入优雅降级——保存检查点、通知人类、返回部分结果。

### 第19章：RAG 生产系统 Harness

**练习 19.1**：设计一个支持混合检索的 RAG Harness 架构。
> 提示：查询 → 查询改写（Harness 组件）→ 并行检索（向量检索 + 关键词检索）→ 结果融合（RRF 或交叉编码器重排）→ 上下文组装 → 生成 → 输出验证（Harness 组件）。每个环节都需要 Harness 控制。

**练习 19.2**：RAG 系统中的缓存策略应该在 Harness 的哪一层实现？
> 提示：多层缓存——查询级缓存（语义相似的查询返回缓存结果）在路由层；检索结果缓存在检索层；LLM 响应缓存在生成层。每层缓存的失效策略不同。

**练习 19.3**：设计一个查询路由策略，区分"可以用 RAG 回答"和"需要直接生成"的问题。
> 提示：用一个轻量级分类器（或 LLM 判断），基于查询类型路由——事实性问题 → RAG 路径；开放性讨论 → 直接生成；代码生成 → 专用 Agent。分类器本身也需要评估 Harness。

## 第五部分：生产运维与前沿

### 第20章：可观测性 Harness

**练习 20.1**：为一个 Agent 系统设计 Trace 结构，包含哪些 Span？
> 提示：顶层 Span（用户请求）→ 路由 Span → 检索 Span（包含向量查询耗时、返回文档数）→ LLM 调用 Span（包含模型名、Token 数、延迟）→ 后处理 Span。每个 Span 记录输入/输出/延迟/错误。

**练习 20.2**：设计 3 个关键的 AI 系统监控告警规则。
> 提示：1）LLM 调用延迟 P99 > 10s 持续 5 分钟（性能退化）。2）评估分数的 7 天滚动均值下降 > 5%（质量漂移）。3）护栏拦截率突然上升 > 3 倍（可能遭受攻击或模型异常）。

**练习 20.3**：Token 成本追踪应该在 Harness 的哪个层面实现？
> 提示：在 LLM Client 的 Wrapper 层——每次调用记录模型名、输入 Token 数、输出 Token 数、单价，汇总到可观测性平台。支持按用户、按功能、按 Agent 维度聚合分析。

### 第21章：部署 Harness 与流水线

**练习 21.1**：设计一个 Harness 感知的金丝雀发布策略。
> 提示：不仅看传统指标（错误率、延迟），还要看 AI 质量指标——将新版本流量的 AI 评估分数与旧版本实时对比。如果新版本的评估分数显著下降，自动回滚。

**练习 21.2**：Model 和 Harness 的版本应该如何管理？
> 提示：分开版本化但关联——Model 版本（gpt-4-0613）+ Harness 版本（v2.3.1）= 系统版本。记录每个系统版本的评估结果。回滚时可以只回滚 Harness（如 Prompt 变更），不一定要连模型一起回滚。

**练习 21.3**：特性标志（Feature Flag）在 AI 系统中有哪些特殊用法？
> 提示：用特性标志控制——新 Prompt 模板的灰度、新模型的流量比例、新护栏规则的开关、新评估指标的启用。比代码部署更灵活，可以快速开关而不需要重新部署。

### 第22章：安全、护栏与合规 Harness

**练习 22.1**：设计一个三层护栏架构的配置文件。
> 提示：输入层——Prompt 注入检测（正则 + ML 模型）、输入长度限制、敏感词过滤。处理层——工具调用白名单、权限控制、速率限制。输出层——PII 检测与脱敏、有害内容过滤、幻觉标记。每层可独立启停。

**练习 22.2**：如何检测和防御间接 Prompt 注入？
> 提示：间接注入是通过检索到的文档或工具返回的结果注入恶意指令。防御——对检索结果进行安全扫描、将用户输入和系统指令用强分隔符隔离、用独立的 LLM 检查指令边界是否被突破。

**练习 22.3**：EU AI Act 对高风险 AI 系统的合规要求中，哪些可以用 Harness 自动化？
> 提示：可自动化的——数据质量检查（评估 Harness）、性能监控（可观测性 Harness）、日志审计（Trace 记录）、透明度报告生成（自动化报告）。不能完全自动化的——风险评估、人工监督机制的建立。

### 第23章：熵管理与持续改进

**练习 23.1**：解释 AI 系统的"熵增"隐喻——哪些因素导致系统质量随时间退化？
> 提示：模型 API 更新（底层模型行为变化）、数据分布漂移（用户使用模式变化）、知识过时（世界变了但知识库没更新）、Prompt 堆积（持续 patch 导致 Prompt 越来越长越来越脆弱）。

**练习 23.2**：设计一个检测 Prompt 漂移的方案。
> 提示：对 Prompt 的历次修改建立版本记录，每个版本关联评估结果。当新版 Prompt 的评估分数与版本 N-3 相比没有显著提升，但长度增加了 50%+，说明可能在做无效的 Patch——需要重构而非继续堆叠。

**练习 23.3**：设计一个持续评估闭环的架构。
> 提示：生产流量采样 → 自动评估 → 结果写入时序数据库 → 异常检测算法监控趋势 → 触发告警或自动响应（切换备用 Prompt、通知团队、触发重新校准流水线）。这个闭环本身也需要监控。

### 第24章：Harness Engineering 的未来

**练习 24.1**：设想一个"自适应 Harness"的工作机制。
> 提示：Harness 中嵌入一个 Meta-Agent，它监控所有 Sensor 的输出，当检测到质量下降时自动尝试调整 Guide（更换 Few-shot 示例、调整系统提示、切换模型）——但每次调整都经过 A/B 评估验证，而非盲目尝试。

**练习 24.2**：Agent 自我改进的闭环中，如何防止"自我退化"？
> 提示：关键——设置不可变的"底线评估集"作为安全网。Agent 可以自我优化，但任何变更都必须通过底线评估集的检查。如果底线评估分数下降，自动回滚。底线评估集本身由人类维护，不受自动化影响。

**练习 24.3**：未来 Harness Engineering 可能形成哪些行业标准？
> 提示：候选方向——Harness 配置的标准格式（类似 OpenAPI Spec）、评估指标的标准化定义、Agent 通信协议标准、安全护栏的合规检查清单、可观测性数据的标准 Schema（扩展 OpenTelemetry）。
