# 前言：如何使用本教程

## 教程设计理念

构建可靠的 AI Agent 系统不是"调通 API、拼好 Prompt"就完成的事。真正的工程挑战在于：如何让一个非确定性系统在生产环境中持续、稳定、安全地运行。这正是 Harness Engineering 要解决的问题。

本教程围绕这个核心目标展开，遵循以下设计原则：

1. **从零开始，但直指核心**：默认你没有 Harness Engineering 经验，但不会用大量篇幅重复 LLM 基础知识。教程从第一章就直接进入 Harness 的世界
2. **先建心智模型，再学具体工具**：每个主题先讲"为什么需要这层 Harness"、"不做会怎样"，再讲"怎么做"
3. **实验驱动**：关键概念通过可运行的 Python 代码和实际实验说明，而非纯理论推演
4. **贴近生产**：教程不止步于"玩具示例"，每章都讨论生产环境中的真实考量——成本、延迟、可维护性、团队协作
5. **中文讲透，术语保真**：用中文把概念和直觉讲清楚，但技术术语保留英文原文，避免翻译带来的歧义

## 为什么这套教程值得学

2026 年的 AI 工程领域有一个显著趋势：**模型能力在快速收敛，而工程差距在持续扩大**。

各家顶级模型（GPT-5、Claude 4、Gemini 2）在基准测试上的差距越来越小，但不同团队构建的 AI 系统在可靠性、可维护性和演进速度上的差距却越来越大。差距的根源不在模型，而在 Harness。

很多团队的现状是：

- Agent 系统能跑通 Happy Path，但面对边缘情况就崩溃
- 模型升级后，不知道质量是变好了还是变差了——因为没有系统化的评估
- 测试要么没有，要么只是"人工试几个 Case"
- 多 Agent 编排靠硬编码，改一个 Agent 全链路受影响
- 生产环境出了问题，排查全靠看日志猜——没有端到端的 Trace
- 安全护栏是事后补丁，不是设计时就内建的

本教程的目标不是让你"知道 Harness Engineering 是什么"，而是让你能够：

1. 为任何 AI 系统设计合适的 Harness 架构
2. 构建系统化的评估 Harness，用数据而非直觉做决策
3. 为非确定性系统建立可靠的测试和 CI 流水线
4. 设计可扩展的 Agent 编排 Harness
5. 部署完整的可观测性和安全 Harness
6. 持续演进 Harness，应对模型漂移和需求变化

## 章节结构说明

每章采用统一的结构，便于系统学习和快速查阅：

### 1. 学习目标

每章开头列出 5 个学习目标，帮助你在开始前明确重点。例如：

> - 理解评估 Harness 的核心组件及其职责
> - 掌握离线评估与在线评估的区别和适用场景
> - ...

### 2. 正文内容

正文一般包含 4-6 个部分，通常覆盖：

- **问题与动机**：这层 Harness 要解决什么问题？不做会怎样？
- **核心概念与架构**：关键概念定义、架构设计、设计权衡
- **代码示例与 API 说明**：可运行的 Python 代码和详细注释
- **常见误区与陷阱**：实际工程中容易犯的错误
- **工程实践建议**：在生产环境中的最佳实践

### 3. 本章小结

以 5 条要点的形式总结本章核心内容，适合快速回顾。

### 4. 动手实验

每章包含 1-2 个动手实验，提供分步指导。实验设计遵循：

- **最小化外部依赖**：尽量只需要 Python + 一个 LLM API Key
- **渐进式**：从简单到复杂，每步都有预期输出
- **可验证**：实验结果有明确的成功标准

### 5. 练习题

每章末尾 3-5 道练习题，分为三个层次：

- **理解题**：检验对核心概念的理解
- **应用题**：将本章知识应用到新场景
- **思考题**：开放性问题，鼓励深入思考和延伸

练习参考答案见[附录 C](./appendix/answers.md)。

## 代码约定

本教程的代码示例遵循以下约定：

```python
# 文件路径会标注在代码块上方
# src/harness/evaluator.py

from typing import Any
import asyncio

class Evaluator:
    """评估器基类 —— 所有评估 Harness 的核心组件"""

    async def evaluate(self, input: str, output: str) -> float:
        # 返回 0.0 ~ 1.0 的评分
        ...
```

- **Python 版本**：3.11+
- **类型注解**：所有函数签名都包含类型注解
- **异步优先**：涉及 I/O 的代码默认使用 async/await
- **注释语言**：代码注释以中文为主，变量名和函数名用英文
- **省略号**：`...` 表示此处省略了非关键代码，完整代码见对应实验仓库
- **终端命令**：以 `$` 开头表示 shell 命令

```bash
$ pip install lm-eval deepeval langfuse
$ python -m harness.evaluate --task my_task
```

## 术语约定

本教程中，英文技术术语保留原文，首次出现时给出中文说明。以下是核心术语对照表：

| 英文术语 | 中文说明 | 首次出现章节 |
|----------|----------|------------|
| Harness | 控制层 / 线束（包裹在模型外面的工程基础设施） | 第1章 |
| Agent | 智能体（Model + Harness 的组合体） | 第1章 |
| Guide | 前馈控制器（系统提示、工具定义、Few-shot 等） | 第3章 |
| Sensor | 反馈感知器（评估器、断言、监控等） | 第3章 |
| Evaluation Harness | 评估 Harness（系统化评估 AI 系统的基础设施） | 第6章 |
| Benchmark | 基准测试（标准化的评估数据集与指标） | 第6章 |
| LLM-as-Judge | LLM 评审员（用 LLM 评估另一个 LLM 的输出） | 第8章 |
| Test Harness | 测试 Harness（AI 系统的自动化测试基础设施） | 第11章 |
| Quality Gate | 质量门禁（CI/CD 中的自动化质量检查点） | 第13章 |
| Orchestration Harness | 编排 Harness（管理 Agent 协作和工作流的控制层） | 第15章 |
| Handoff | 任务交接（Agent 之间的控制权转移） | 第16章 |
| Guardrail | 护栏（防止 AI 系统产生有害输出的安全机制） | 第22章 |
| Observability | 可观测性（Trace + Metric + Log 三大支柱） | 第20章 |
| Drift | 漂移（模型或数据分布随时间的偏移） | 第23章 |
| Entropy Management | 熵管理（对抗 AI 系统不确定性累积的方法论） | 第23章 |
