# 第13章：特征、向量与缓存

> 进入检索、推荐、RAG 和在线特征场景后，平台管理的就不再只是模型权重，而是一整套“特征 / embedding / 索引 / 缓存”的状态系统。

## 学习目标

完成本章学习后，你将能够：

1. 区分特征、embedding、向量索引和缓存的职责
2. 理解训练与在线服务之间的数据一致性问题
3. 认识向量系统中的版本、更新和权限边界
4. 学会分析缓存收益与失效风险
5. 设计一个基础的向量与特征治理方案

## 正文内容

### 13.1 为什么这一层越来越重要

很多现代 AI 系统并不是“一个模型吃原始输入直接输出结果”这么简单。  
它们常常还会依赖：

- 在线特征
- embedding 向量
- 向量检索
- reranker 特征
- 多层缓存

这意味着，平台不仅要管理模型，还要管理围绕模型运行的一整套状态。

### 13.2 特征、embedding、向量索引分别是什么

#### 特征

通常是结构化或半结构化输入，可能来自：

- 用户画像
- 商品属性
- 实时行为统计
- 业务上下文

#### Embedding

是把文本、图片、文档等映射到向量空间后的表示。  
它通常由某个特定模型版本生成。

#### 向量索引

是为了高效近邻检索而组织的结构，可能包含：

- 向量本身
- 文档 ID
- 元数据过滤条件
- 分区与分片信息

它们之间的关系非常紧密：

> embedding 来自模型，索引依赖 embedding，缓存又依赖索引和访问模式。

### 13.3 一致性问题为什么难

现实中常见的错误不是“系统挂了”，而是“不一致”：

- 训练使用的是一套特征定义，线上使用的是另一套
- embedding 模型更新了，但索引没重建
- 文档已经删除，但向量索引还保留旧记录
- cache 命中了旧数据，导致线上结果滞后

这种问题最麻烦的地方在于：

- 服务表面看起来是正常的
- 指标可能也不立刻异常
- 但模型质量会逐步漂移

### 13.4 向量系统里的版本管理

一个向量索引至少应该绑定：

- embedding 模型版本
- 文档 / 数据版本
- chunk 规则版本
- 构建时间
- 构建任务 ID

例如：

```yaml
vector_index:
  name: docs-prod
  embedding_model: text-embed-v4
  source_dataset: kb-2026-04-10
  chunk_policy: title-paragraph-v2
  build_job: reindex-20260410-003
```

没有这些信息时，RAG 与检索系统的问题几乎无法可靠定位。

### 13.5 缓存在这里到底缓存什么

缓存不只是普通意义上的“把结果存一下”，在 AI 系统里常见的缓存包括：

- 在线特征缓存
- embedding 结果缓存
- 检索结果缓存
- 文档元数据缓存
- prefix cache / KV Cache

缓存的收益通常是：

- 更低延迟
- 更少重复计算
- 更低下游压力

代价通常是：

- 新鲜度下降
- 版本一致性变难
- 权限边界更复杂

### 13.6 权限问题为什么容易被忽略

在 RAG、检索、企业知识问答场景里，一个常见风险是：

- 用户无权访问某文档
- 但缓存或索引没有正确执行权限过滤
- 最终回答引用了本不该暴露的信息

这说明向量与缓存系统不是单纯性能组件，还承担了安全边界。

### 13.7 工程建议

- 特征定义必须版本化
- embedding 模型变化要和索引构建一起治理
- 缓存命中不能绕过权限检查
- 在线与训练的特征含义要保持一致
- 向量索引要像模型包一样具备元数据和生命周期状态

## 本章小结

| 对象 | 主要作用 | 常见风险 |
|------|----------|----------|
| 特征 | 提供结构化上下文 | 训练 / 线上定义不一致 |
| embedding | 提供向量表示 | 模型版本漂移 |
| 向量索引 | 提供高效检索 | 索引与数据版本脱节 |
| 缓存 | 降低延迟与重复计算 | 新鲜度与权限风险 |
