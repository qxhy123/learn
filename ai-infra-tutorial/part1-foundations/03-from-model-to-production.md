# 第3章：从模型实验到生产系统

> AI 系统最大的工程鸿沟，通常不在“模型写出来”这一刻，而在“模型要被稳定交付”这一刻。

## 学习目标

完成本章学习后，你将能够：

1. 理解 notebook、训练任务、模型制品、推理服务之间的演化关系
2. 知道从实验到生产通常会增加哪些基础设施层
3. 识别“能跑”和“可运营”之间的差别
4. 理解为什么模型交付需要评测、发布、回滚和观测闭环
5. 用系统视角描述一个 AI 项目的端到端链路

---

## 正文内容

### 3.1 从 notebook 到生产，真正增加了什么

一个最初的 AI 项目通常从几段实验代码开始：

- 读数据
- 训练模型
- 看评测指标
- 保存一个 checkpoint

这时系统很简单，但通常只适合：

- 单人使用
- 单机运行
- 低频实验
- 无严格上线要求

一旦需求变成“给团队共用、给真实用户服务、要可观测可回滚”，系统就会增加一整层能力：

- 训练任务提交与资源调度
- 模型制品管理
- 评测门禁
- 发布流程
- 推理服务
- 监控和告警
- 成本和权限治理

这就是 AI Infra 登场的地方。

### 3.2 一个端到端链路图

一个典型 AI 项目从实验到生产，常见链路如下：

```text
数据采集
  -> 数据清洗 / 切分
  -> 训练任务
  -> 评测任务
  -> 模型制品注册
  -> staging 部署
  -> 灰度发布
  -> 线上推理
  -> 监控 / 反馈 / 重新训练
```

这条链路有两个关键特点：

1. **它不是单向的**：线上反馈和数据漂移会反过来触发新的训练
2. **它不是只有一个版本**：多个模型版本、多个数据版本、多个服务实例会并存

也就是说，AI 系统不是一个脚本，而是一组持续运行的闭环。

### 3.3 “能跑”与“可运营”的差异

一个 notebook 里的模型能跑，不代表它已经是一个可运营系统。  
所谓“可运营”，至少意味着：

- 有明确输入和输出
- 有稳定运行环境
- 有版本与证据链
- 有错误处理与回滚路径
- 有观测面板和责任边界

这可以用一个简单对比理解：

| 阶段 | 关注点 | 典型形态 |
|------|--------|----------|
| 实验阶段 | 先看到效果 | notebook、脚本、临时 checkpoint |
| 工程阶段 | 可复现、可协作 | 训练 Job、配置、制品仓库 |
| 生产阶段 | 可服务、可回滚、可治理 | 发布流程、推理集群、监控告警 |

### 3.4 为什么模型交付要额外引入评测与发布

普通代码发布通常更关注功能和稳定性，而模型发布还要额外关注：

- 模型质量是否退化
- 评测集是否稳定
- 线上流量分布与离线数据是否一致
- 新版本是否引入更高时延或更高成本

这意味着，模型交付链路通常会比普通服务多出几步：

```text
训练完成 -> 离线评测 -> 模型注册 -> staging -> 灰度 -> 放量 -> 回滚保留
```

如果缺失其中任何一步，模型上线就更像一次裸奔。

### 3.5 为什么 AI 系统天然需要闭环

AI 系统不是“上线后就结束”的系统。因为：

- 数据会变化
- 业务会变化
- 用户行为会变化
- 基础模型和依赖也会变化

因此，AI 平台需要支持的不是一次性交付，而是持续闭环：

1. 观测线上表现
2. 判断是否退化
3. 分析退化来自系统还是模型
4. 决定是否重新训练、重建索引或回滚

这也是为什么 AI Infra 比“模型部署”更大，它包含了 **再训练、再评测、再发布** 的整个过程。

### 3.6 一个团队常见的演进路线

你可以把团队的 AI 工程能力粗略分成四个阶段：

| 阶段 | 特征 | 主要风险 |
|------|------|----------|
| 阶段 0 | notebook / 本地脚本 | 结果不可复现 |
| 阶段 1 | 训练脚本 + 手工部署 | 版本、环境、回滚混乱 |
| 阶段 2 | 训练平台 + 制品管理 + 推理服务 | 观测和治理不足 |
| 阶段 3 | 完整闭环平台 | 平台复杂度与成本控制成为重点 |

本教程的目标不是让每个团队都跳到“阶段 3”，而是让你知道自己现在在哪一阶段，以及下一步该补哪块能力。

## 本章小结

| 主题 | 结论 |
|------|------|
| 从实验到生产 | 增加的不是代码量，而是工作流、状态管理和治理能力 |
| 可运营系统 | 要求可复现、可发布、可回滚、可观测 |
| AI 交付特殊性 | 模型上线不仅是代码发布，还要经过评测与质量门禁 |
| 平台价值 | 把一次性流程变成持续闭环 |
