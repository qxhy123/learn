## 第1章：什么是 AI Infra

### 第1题：用你自己的话说明 AI Infra 与“训练一个模型”有什么区别

**思路**

- 这道题的关键不是复述定义，而是区分“一个模型任务”和“一整套持续运行系统”。
- 可以从目标、对象、时间尺度、失败处理、团队协作五个维度回答。

**参考答案**

“训练一个模型”更像是一个局部任务，它关注的是：

- 数据是否准备好了
- 训练代码能否跑通
- loss 是否下降
- 最终指标是否达标

而 `AI Infra` 关注的是围绕这个模型任务的一整条链路是否能长期、稳定、可治理地运行，包括：

- 数据如何进入和版本化
- 训练任务如何调度与恢复
- 模型制品如何注册、评测、发布和回滚
- 推理服务如何扩缩容、限流和观测
- 多团队如何共享 GPU、配额和预算

可以用一个简单对比说明：

| 维度 | 训练一个模型 | AI Infra |
|------|--------------|----------|
| 目标 | 得到一个效果可接受的模型 | 让模型从实验到生产形成稳定闭环 |
| 时间尺度 | 一次实验或一批实验 | 长期持续运行 |
| 关注对象 | 模型、数据、训练脚本 | 资源、任务、数据、制品、服务、治理 |
| 失败处理 | 人工重跑 | 自动恢复、回滚、限流、审计 |
| 协作方式 | 个人或小组实验 | 多团队共用平台与规则 |

所以更准确地说，训练模型是 AI 工作负载中的一个环节，而 AI Infra 是承载整条工作负载生命周期的系统能力。

**延伸提醒**

- 如果一个团队只能“训练出模型”，但无法稳定发布、回滚和治理，它还没有真正拥有 AI 基础设施。

### 第2题：画出一个包含业务层、模型层、平台层、资源层的系统分层图

**思路**

- 这道题考察的是分层意识。
- 重点不是画得多复杂，而是每一层有清晰职责，并且上下游关系明确。

**参考架构**

```text
业务层
  - 产品接口
  - 用户请求
  - SLA / 成本目标

模型层
  - 训练逻辑
  - 推理逻辑
  - 评测逻辑

平台层
  - 训练任务编排
  - 模型仓库
  - 发布与回滚
  - 监控、告警、审计
  - 队列、配额、权限

资源层
  - GPU / CPU
  - 内存 / 显存
  - 网络
  - 存储
  - 容器运行环境
```

**参考答案**

一张合格的分层图至少要表达两个关系：

1. **职责关系**：业务层关心用户价值，模型层关心效果，平台层关心工作流和治理，资源层关心容量和带宽。
2. **依赖关系**：上层依赖下层提供能力，下层并不知道上层业务细节。

如果进一步补充，可在每层边上标注典型问题：

- 业务层：接口超时、结果不稳定、成本超标
- 模型层：效果退化、评测不过、模型版本冲突
- 平台层：队列拥塞、回滚困难、工件找不到
- 资源层：GPU 不够、网络慢、存储吞吐低

### 第3题：如果一个推理服务超时变多，你会优先从哪三层开始排查？为什么？

**思路**

- 这道题不要求“唯一顺序”，而是要求给出合理排障顺序。
- 最好的回答不是“从最底层开始”，而是“从最有可能解释现象的层级开始”。

**参考答案**

我会优先看这三层：

1. **平台层**
2. **资源层**
3. **模型 / 依赖链路层**

原因如下：

#### 第一步：先看平台层

因为超时往往先体现在：

- 队列等待时间升高
- 某个版本刚刚灰度
- 路由策略变化
- 副本数不够或扩容失败

这些都是平台层直接可见的问题，而且能最快缩小范围。

#### 第二步：再看资源层

如果平台层显示请求在排队或副本忙不过来，就要检查：

- GPU 利用率和显存是否打满
- 节点是否有冷启动
- 网络是否有抖动
- 下游存储或索引是否变慢

#### 第三步：最后看模型 / 依赖链路层

如果平台和资源看起来都正常，就要怀疑：

- 模型版本变大了
- prompt / 上下文长度变长
- 下游向量库、重排器、特征服务变慢
- 某个模型输出长度失控

**一个实用排障顺序**

```text
先看 queue wait -> 再看 GPU / 显存 / 副本 -> 再拆 downstream / model
```

### 第4题：为什么说“开源组件拼起来”不等于“拥有 AI 平台”？

**思路**

- 这题考察“组件”和“平台能力”的区别。
- 关键在于：平台不仅是功能集合，更是规则、状态和治理的统一系统。

**参考答案**

开源组件只能提供局部能力，例如：

- Kubernetes 负责通用编排
- MLflow 负责实验记录或模型仓库
- vLLM 负责 LLM 推理
- Prometheus 负责监控指标

但真正的平台还必须解决这些“组件之间”的问题：

- 训练任务输出如何自动进入模型仓库
- 评测结果如何影响发布门禁
- 推理版本如何与模型版本、镜像版本、索引版本绑定
- 多团队如何共享配额、权限和预算
- 出事故后如何回滚、审计和复盘

可以说：

> 组件解决“某一段能做什么”，平台解决“整条链路如何稳定协同”。

所以如果只是把开源工具堆在一起，但没有统一的身份、状态、规则和流程，系统仍然会停留在“工具集合”，而不是“平台”。

## 第2章：算力、存储与网络

### 第1题：为什么说 AI 系统吞吐通常由最慢的一段决定？

**思路**

- 这题本质是木桶效应。
- 回答时要把“单点性能”和“端到端吞吐”区分开。

**参考答案**

AI 系统通常是一条串联链路，例如训练场景：

```text
存储读取 -> CPU 预处理 -> H2D 拷贝 -> GPU 计算 -> 梯度同步
```

在这种串联系统里，哪怕某一段非常快，只要其中一段明显更慢，整体吞吐仍会被这段限制。

粗略写法可以是：

$$
\text{effective throughput} \approx \min(\text{storage},\text{transfer},\text{compute},\text{network})
$$

举例：

- 如果 GPU 理论吞吐很强，但数据加载只能每秒供应 100 个样本，那么整体吞吐就不会超过 100。
- 如果单机算得很快，但多机梯度同步太慢，那么整体 step time 仍然会被通信拖长。

**结论**

所以端到端优化必须先找到最短板，而不是只优化最显眼的那一段。

### 第2题：对象存储、SSD、主存、显存分别适合放什么类型的数据？

**思路**

- 这题考察分层存储意识。
- 回答时要把“容量 / 延迟 / 带宽 / 用途”一起说清楚。

**参考答案**

| 层级 | 适合放什么 | 原因 |
|------|------------|------|
| 对象存储 | 原始数据、长期 checkpoint、模型仓库 | 容量大、成本低，但延迟高 |
| SSD / NVMe | 热点训练数据、本地缓存、临时中间结果 | 吞吐高，适合热路径缓存 |
| 主存 | dataloader 缓冲、预处理结果、运行中间态 | 延迟低，适合 CPU 侧快速访问 |
| 显存 | 模型权重、激活、batch 数据、KV Cache | 最接近 GPU 计算，带宽最高但最贵最稀缺 |

**一个合理分层示意**

```text
对象存储：存“长期和大”的东西
SSD：存“近期会重复用”的东西
主存：存“马上要处理”的东西
显存：存“此刻正在算”的东西
```

### 第3题：为什么网络既会影响训练吞吐，也会影响线上推理时延？

**思路**

- 关键是说明网络不仅存在于“分布式训练”。

**参考答案**

网络会影响训练吞吐，因为训练里有：

- 多机数据读取
- 分布式梯度同步
- 参数广播
- checkpoint 写远端存储

网络也会影响线上推理时延，因为推理链路通常不是单进程孤立运行，而会调用：

- 网关
- 特征服务
- 向量检索
- 重排器
- 下游安全过滤

所以网络在训练里更常体现为“吞吐下降或扩卡失效”，在推理里更常体现为“尾延迟恶化”。

### 第4题：请把一个训练 step 拆成至少 5 个时间段，并思考哪个最容易成为瓶颈

**思路**

- 这题要给出时间拆解和瓶颈判断方法。

**参考答案**

一个训练 step 可以拆成：

$$
t_{\text{step}} =
t_{\text{load}} +
t_{\text{preprocess}} +
t_{\text{h2d}} +
t_{\text{forward}} +
t_{\text{backward}} +
t_{\text{optim}} +
t_{\text{sync}}
$$

如果是单机，可以先忽略 `t_sync`；如果有 checkpoint 或日志，也可再加：

- `t_checkpoint`
- `t_logging`

**哪个最容易成为瓶颈？**

- 在早期系统里，常见是 `t_load` 或 `t_preprocess`
- 在大模型训练里，常见是 `t_h2d`、`t_backward` 或 `t_sync`
- 在多机训练里，`t_sync` 非常容易被低估

**排查建议**

先画这个时间分解，再用 profiler 或日志验证每一项占比，而不是凭感觉判断。

## 第3章：从模型实验到生产系统

### 第1题：为什么说“模型能跑起来”不等于“模型可以交付”？

**思路**

- 区分实验成功和交付成功。

**参考答案**

“模型能跑起来”只说明：

- 训练脚本可执行
- 有一组结果
- 在某个数据集或样例上看起来有效

但“模型可以交付”还意味着：

- 可复现：别人能用同样数据和配置重现结果
- 可发布：模型包、镜像、依赖和配置都能进入线上系统
- 可回滚：线上出问题时能快速切回上一版本
- 可观测：知道系统和模型是否退化
- 可治理：权限、成本、配额、审计都可控

所以交付成功比实验成功多了一整套平台和治理要求。

### 第2题：请画出一个从数据到训练、评测、发布、推理、反馈的完整链路

**参考架构**

```text
数据采集
  -> 清洗 / 去重 / 标注
  -> 数据版本
  -> 训练 Job
  -> checkpoint / 日志 / 指标
  -> 离线评测
  -> 模型注册
  -> staging 部署
  -> 灰度发布
  -> 在线推理
  -> 监控 / 用户反馈 / 人工审核
  -> 数据回流
  -> 下一轮训练
```

**参考答案**

这条链路最重要的不是节点数量，而是“每个节点都留下可追踪状态”：

- 数据有版本
- 训练有 job id
- 模型有 registry 版本
- 发布有灰度记录
- 线上有观测和反馈

只有这样，这条链才不是一次性的，而是真正的闭环系统。

### 第3题：与普通后端服务相比，模型发布多了哪些关键步骤？

**参考答案**

模型发布通常比普通代码发布至少多出这些步骤：

1. 离线评测
2. 回归集检查
3. 模型 / tokenizer / 配置一致性校验
4. 模型包注册
5. 质量灰度观察
6. 线上业务指标验证

也就是说，模型发布不只是“接口能不能通”，还要回答“结果是否仍然有业务价值”。

### 第4题：你的团队如果现在还处于“训练脚本 + 手工部署”阶段，最先应该补哪类平台能力？

**思路**

- 这题不是让你列全平台，而是选“第一块最值钱的能力”。

**参考答案**

优先补 **训练输出与模型发布之间的最小闭环**，也就是：

- 统一训练 Job 入口
- 统一保存 checkpoint / 配置 / 日志
- 简单模型注册表
- 简单离线评测门禁

原因是：

- 这是从“人肉交付”到“可重复交付”的第一步
- 比直接做复杂调度平台更快见效
- 没有这一步，后面的灰度、回滚和治理都无从谈起

**最小闭环示意**

```text
训练脚本
  -> 统一输出目录
  -> 评测结果
  -> 模型注册
  -> staging 服务
```

## 第4章：GPU 与加速器

### 第1题：为什么 GPU 选型不能只看峰值算力？

**参考答案**

因为峰值算力只说明“理论上最多能算多快”，不能说明：

- 模型能不能放下
- 数据能不能喂满
- 多卡能不能高效通信
- 线上冷启动和缓存是否合适

至少还要同时看：

- 显存容量
- 显存带宽
- 互联拓扑
- 驱动和软件栈兼容性
- 单位成本

所以选型应看“有效算力”，而不是只看“理论算力”。

### 第2题：训练和推理在硬件诉求上最大的差异是什么？

**参考答案**

训练更关注：

- 大显存
- 高带宽
- 多卡互联
- 长时间吞吐稳定

推理更关注：

- 单位成本吞吐
- 冷启动速度
- 显存复用
- 并发管理
- 尾延迟控制

简化理解：

> 训练更像“持续重负载生产”，推理更像“有波峰波谷的在线服务”。

### 第3题：用“算得动、放得下、喂得满、连得快”分析一个你熟悉的模型场景

**示例场景：13B LLM 在线推理**

**思路**

用四个问题拆解：

1. 算得动吗：单卡吞吐是否够
2. 放得下吗：权重 + KV Cache 是否能放进显存
3. 喂得满吗：请求量是否足以形成高效 batch
4. 连得快吗：如果多卡分片，互联是否足够

**参考答案**

- **算得动**：若请求量高且上下文中等，单卡 Tensor Core 吞吐通常足够支撑一部分流量
- **放得下**：13B 权重若用 FP16 很吃显存，再叠加 KV Cache，单卡可能放不下，需要量化或多卡
- **喂得满**：如果流量稀疏、请求长度差异大，实际利用率未必高
- **连得快**：如果模型分片到多卡，PCIe-only 环境可能导致 decode 受通信拖累

**结论**

这个场景下，最关键的不是单纯“算力高不高”，而是“显存和调度是否匹配”。

### 第4题：设备异构会如何增加平台治理成本？

**参考答案**

设备异构会带来至少四类治理成本：

1. 调度规则更复杂：不同任务要匹配不同卡型
2. 镜像兼容更复杂：驱动、runtime、库版本组合增多
3. 成本归因更复杂：不同卡型单价不同
4. 用户心智更复杂：团队要理解哪些队列适合哪些任务

如果没有明确标签、队列和定价策略，异构算力池会迅速变成“大家都抢最好卡”的混乱场面。

## 第5章：内存、互联与 IO

### 第1题：为什么很多 AI 系统优化要从“数据怎么走”开始，而不是从“算子怎么写”开始？

**参考答案**

因为在很多实际系统里，计算并不是第一瓶颈。  
更常见的问题是：

- 数据加载太慢
- 小文件太多
- H2D 拷贝过多
- 远端依赖过长
- checkpoint 回写拖慢训练

如果数据根本没能高效到达 GPU，再优化 kernel 往往没有明显收益。

### 第2题：PCIe、NVLink、RDMA 分别更影响哪些场景？

**参考答案**

| 技术 | 主要影响场景 |
|------|--------------|
| PCIe | CPU 和 GPU 之间的数据拷贝、设备接入 |
| NVLink | 同机多 GPU 训练、张量并行、大模型分片推理 |
| RDMA | 多机训练通信、低 CPU 参与的数据传输 |

可以这样记：

- PCIe：主机到设备
- NVLink：设备到设备（同机）
- RDMA：机器到机器

### 第3题：请解释为什么对象存储适合冷数据，但不一定适合直接承担训练热路径

**参考答案**

对象存储适合：

- 大容量
- 长期保存
- 成本敏感场景

但不适合直接承担训练热路径的原因包括：

- 单次访问延迟高
- 小文件和高频元数据访问开销大
- 带宽波动大
- 需要更多本地解码和预处理

因此更常见的做法是：

```text
对象存储做源头
  -> 本地 SSD 或文件系统做热缓存
  -> 主存和显存承担运行时热路径
```

### 第4题：写出一个训练 step 的时间拆解式，并说明你最想先优化哪一项

**参考答案**

$$
t_{\text{step}} =
t_{\text{load}} +
t_{\text{preprocess}} +
t_{\text{h2d}} +
t_{\text{forward}} +
t_{\text{backward}} +
t_{\text{optim}} +
t_{\text{sync}}
$$

如果是单机训练，我通常会优先优化 `t_load` / `t_preprocess`，因为：

- 它们最常被低估
- 优化收益往往最稳定
- 数据供给一旦稳定，后续算力优化才更有意义

如果是多机训练，我会优先关注 `t_sync`，因为扩卡收益是否成立往往取决于它。

## 第6章：CUDA、运行时与算子执行

### 第1题：为什么大量小 kernel 往往会降低整体效率？

**参考答案**

因为每个 kernel launch 都有固定开销。如果一个任务被拆成大量很小的 kernel，就会出现：

- launch 开销占比变高
- 设备执行被频繁切碎
- 内存访问难以形成高效模式
- 调度和同步点变多

所以“很多小 kernel”常常意味着：

> 设备大部分时间不在做高价值计算，而在做频繁切换和等待。

### 第2题：同步点为什么会破坏计算与拷贝重叠？

**参考答案**

如果本来可以并行进行：

- 下一批数据拷贝
- 当前批次计算

但在中间插入同步点，那么系统就会强制等待上一阶段完全结束后再继续。  
结果就是：

```text
原本：拷贝 || 计算
加入同步后：拷贝 -> 等待 -> 计算
```

也就是说，同步点会把本可以重叠的链路串行化。

### 第3题：在什么情况下应优先调用成熟库而不是自己拼很多算子？

**参考答案**

当问题属于高频通用算子时，应优先考虑成熟库，例如：

- GEMM
- 卷积
- 标准 attention
- 通信原语

原因是成熟库通常已经优化了：

- kernel 选择
- 内存布局
- 硬件适配
- 稳定性和边界处理

只有当：

- 现有库完全不支持
- 你的数据形状非常特殊
- 性能瓶颈已明确指向这一点

才值得投入自定义实现。

### 第4题：如果两个引擎跑同一模型速度差很多，你会先排查哪几层？

**参考答案**

我会按以下顺序排查：

1. 输入形状和 batch 策略是否一致
2. 精度和量化设置是否一致
3. 编译 / kernel 路径是否一致
4. 底层库和运行时版本是否一致
5. 设备、驱动、互联环境是否一致

因为很多“引擎快慢差异”，根本不是引擎本身，而是运行条件不一致。

## 第7章：单机训练系统

### 第1题：为什么说多机训练前必须先掌握单机训练基线？

**参考答案**

因为多机训练会叠加更多变量：

- 通信
- 调度
- 节点差异
- checkpoint 分片

如果单机基线都不清楚，你根本无法判断：

- 多机性能差是通信问题，还是单机本来就没优化好
- loss 不稳定是并行问题，还是单机配置问题

一个合格的单机基线至少包括：

- step time
- 样本吞吐
- 显存占用
- GPU 利用率
- 数据加载时间占比

### 第2题：写出一个训练 step 的时间拆解式，并说明哪一项最容易被忽视

**参考答案**

$$
t_{\text{step}} =
t_{\text{load}} +
t_{\text{preprocess}} +
t_{\text{h2d}} +
t_{\text{forward}} +
t_{\text{backward}} +
t_{\text{optim}}
$$

最容易被忽视的通常是：

- `t_load`
- `t_preprocess`
- `t_checkpoint`（如果存在）

因为很多人天然只盯前向和反向，但真实系统常常先被供给链拖慢。

### 第3题：如果 GPU 利用率周期性掉到很低，你会优先怀疑什么？

**参考答案**

优先怀疑：

1. 数据加载断续
2. CPU 预处理跟不上
3. H2D 拷贝过慢
4. checkpoint / eval / logging 周期性插入

这是典型的“供给链锯齿”症状，而不一定是 GPU 算子本身有问题。

### 第4题：显存预算里为什么不能只看模型参数大小？

**参考答案**

因为训练显存至少包括：

- 参数
- 梯度
- 优化器状态
- 激活
- 临时 buffer
- 碎片

很多模型能做推理但不能训练，原因就是训练时显存结构远比“只放权重”复杂。

## 第8章：数据并行

### 第1题：为什么数据并行扩展到多机后收益常常变差？

**参考答案**

因为扩卡带来的不是只有新增算力，还会引入新增通信。  
多机时，AllReduce 和同步开销会增加，而且网络拓扑也更复杂。

收益变差的常见原因：

- `t_allreduce` 占比上升
- per-device batch 太小
- 数据供给跟不上
- 节点异构导致慢节点拖全局

### 第2题：ring all-reduce 中，梯度量和设备数如何影响通信成本？

**参考答案**

一个常用近似式是：

$$
\text{bytes per rank} \approx 2 \times \frac{N-1}{N} \times G
$$

其中：

- `N` 是设备数
- `G` 是梯度总字节数

这意味着：

- 梯度越大，通信成本越大
- 设备越多，协调越复杂
- 如果网络跟不上，新增设备带来的收益会越来越小

### 第3题：扩展效率下降时，你会优先从哪三类因素排查？

**参考答案**

优先看三类：

1. **通信因素**：AllReduce 占比、网络拓扑、NCCL 异常
2. **供给因素**：数据加载、CPU 预处理、H2D 拷贝
3. **训练配置因素**：batch、学习率、梯度累积、异构节点

这是因为效率下降未必完全是网络问题，也可能是你把单机瓶颈一起扩到了多机。

### 第4题：为什么“增大 batch”既可能提升效率，也可能影响训练质量？

**参考答案**

增大 batch 的好处：

- 每卡计算量更大
- 相对同步开销更小
- 设备更容易被喂满

代价：

- 显存压力变大
- 收敛路径改变
- 需要重新调学习率和训练策略

所以增大 batch 是典型的“系统效率”和“优化行为”之间的权衡。

## 第9章：模型并行与流水并行

### 第1题：为什么数据并行无法解决“模型放不下单卡”的问题？

**参考答案**

因为数据并行默认每张卡都持有一份完整模型。  
它解决的是“吞吐扩展”，不是“单卡内存不够”。

如果模型本身就放不下单卡，再复制多份只会更糟。

### 第2题：请解释张量并行和流水并行分别在“切什么”

**参考答案**

- **张量并行**：切层内部的矩阵计算
- **流水并行**：按层段切模型

简单理解：

```text
张量并行：切单层内部
流水并行：切层与层之间
```

### 第3题：pipeline bubble 为什么会出现？如何缓解？

**参考答案**

bubble 来自流水线前后阶段不能同时满载：

- 开始时后面 stage 还没活干
- 结束时前面 stage 已经闲下来

缓解方式：

- 增加 micro-batch 数
- 减少 stage 失衡
- 避免切得过细

### 第4题：什么时候混合并行会成为必要选择？

**参考答案**

当你同时遇到这些问题时：

- 模型放不下单卡
- 数据并行扩展不够
- 单层矩阵很大
- 模型整体很深

这时通常需要组合：

- 数据并行
- 张量并行
- 流水并行
- 状态切分

也就是说，混合并行不是“高级玩法”，而是超大模型训练的现实需求。

## 第10章：内存优化、检查点与恢复

### 第1题：为什么训练阶段的显存占用远不止模型参数本身？

**参考答案**

因为训练时还需要：

- 梯度
- 优化器状态
- 激活
- 临时 buffer

一个粗略式子是：

$$
M_{\text{train}} \approx M_{\text{params}} + M_{\text{grads}} + M_{\text{optim}} + M_{\text{activations}} + M_{\text{temp}}
$$

这就是为什么训练比推理更容易 OOM。

### 第2题：激活重计算的收益和代价分别是什么？

**参考答案**

收益：

- 降低激活占用
- 支持更大 batch / 更长序列

代价：

- 反向更慢
- 实现与调试复杂度更高

本质上是：**用更多计算换更少显存**。

### 第3题：为什么只存模型权重通常不足以支持断点续训？

**参考答案**

因为断点续训还需要：

- 优化器状态
- scheduler 状态
- 当前 step / epoch
- 随机数状态

只存权重更像“用于推理的模型包”，不是完整训练状态。

### 第4题：设计一个适合长训练任务的 checkpoint 策略

**参考答案**

一个可行策略如下：

```yaml
checkpoint_policy:
  every_n_steps: 1000
  keep_recent: 3
  keep_daily: 1
  keep_best_by_eval: 2
  write_path: s3://train-artifacts/project-a/run-007/
  latest_pointer: enabled
```

**思路**

- 周期性保存，控制最大可丢失进度
- 保留少量最近版本，支持快速恢复
- 保留少量最优版本，支持评测与回滚
- 将 `latest` 指针和实际文件解耦，避免半写入状态污染

## 第11章：数据管道

### 第1题：为什么数据管道问题常常会被误判成模型问题？

**参考答案**

因为很多症状最终都体现在模型结果上：

- loss 波动
- 线上效果下降
- 训练吞吐下降

但根因可能其实是：

- 切分规则变了
- 清洗逻辑变了
- 回流断了
- 样本格式变了

所以数据管道问题经常伪装成“模型训练变差”。

### 第2题：训练和在线特征管道在目标上有什么根本差异？

**参考答案**

训练管道更看重：

- 吞吐
- 分片
- shuffle
- 版本稳定

在线特征管道更看重：

- 低延迟
- 一致性
- 权限控制
- 热点缓存

所以训练管道更像批处理系统，在线特征管道更像低延迟服务系统。

### 第3题：设计一个包含回流环节的数据流图

**参考架构**

```text
原始日志 / 文档
  -> 清洗 / 去重 / 脱敏
  -> 标注 / 样本化
  -> 训练集 / 验证集 / 测试集
  -> 训练任务
  -> 模型上线
  -> 用户请求 / 用户反馈 / 人工审核
  -> 回流样本池
  -> 下一轮训练
```

**参考答案**

这条数据流的关键不是画出来，而是标明：

- 哪一步产生版本
- 哪一步产生标签或审核结果
- 哪一步是回流入口

只有这样，回流才不会只是“日志堆积”，而是真正进入再训练闭环。

### 第4题：为什么“先让数据能用”不等于“数据管道已经设计好”？

**参考答案**

“能用”通常只说明：

- 当前任务能读
- 当前样本能训练

但“设计好”还意味着：

- 可追踪
- 可回流
- 可切分
- 可扩展
- 可审计

一个能临时跑通的脚本，不等于一个可持续的数据系统。

## 第12章：制品、模型与检查点管理

### 第1题：为什么 checkpoint 不适合直接当生产模型包？

**参考答案**

因为 checkpoint 的目标是恢复训练，它通常包含：

- 模型参数
- 优化器状态
- scheduler 状态
- 当前训练步数

而生产模型包更需要：

- 轻量、可加载的权重
- 推理配置
- tokenizer
- 版本元数据

直接把 checkpoint 用于线上通常会带来：

- 体积大
- 无用状态多
- 配置和部署边界不清

### 第2题：一个模型版本的最小元数据应该包含哪些字段？

**参考答案**

建议至少包含：

```yaml
model_name: reranker
model_version: 2026-04-11-rc1
dataset_version: support-v3
code_revision: abc1234
training_job_id: train-20260411-007
eval_report_id: eval-20260411-002
image: reranker-serving:2026-04-11-rc1
status: staging
```

这些字段能把“模型来自哪里”“能不能上线”“出了问题怎么回滚”都串起来。

### 第3题：为什么模型状态管理必须进入平台控制面？

**参考答案**

因为状态会直接影响：

- 谁能发布
- 哪个版本能被流量命中
- 哪个版本还能回滚
- 哪些工件可以清理

如果状态只存在于文档或群消息里，平台就没法自动执行门禁、回滚和审计。

### 第4题：举一个“文件都在但无法回滚”的失败案例

**参考答案**

一个典型案例：

- 老模型权重还在
- 但老 tokenizer 已被覆盖
- 老镜像已被清理
- 索引或 prompt 模板已经升级

结果是：

- 文件都没丢
- 但系统无法恢复为“旧的完整运行态”

**结论**

回滚依赖的是一组彼此兼容的制品，而不是单个文件。

## 第13章：特征、向量与缓存

### 第1题：为什么 embedding 模型升级后，通常不能只替换线上模型而不重建索引？

**参考答案**

因为向量索引中的向量就是由旧 embedding 模型生成的。  
如果直接换线上 embedding 模型而不重建索引，就会出现：

- query 向量和文档向量不在同一分布空间
- 检索相似度失真
- 召回结果质量下降

所以 embedding 模型版本和索引版本必须绑定治理。

### 第2题：在线特征缓存为什么既能提升性能，也可能制造一致性问题？

**参考答案**

收益：

- 减少重复计算和远端访问
- 降低延迟
- 降低下游服务压力

风险：

- 读到过期值
- 训练和线上特征定义不一致
- 某些用户画像更新滞后

所以缓存不仅是性能问题，也是正确性问题。

### 第3题：解释“缓存命中不能绕过权限边界”这句话

**参考答案**

意思是：

- 即使某个检索结果或文档已经缓存在系统里
- 系统仍然必须重新确认当前用户是否有权限访问它

否则就会出现：

- A 用户本不该看到某文档
- 但由于缓存命中，系统直接返回了结果

这在企业知识库、RAG 和搜索系统里是非常危险的越权风险。

### 第4题：请设计一个最小的向量索引元数据结构

**参考答案**

```yaml
vector_index:
  name: docs-prod
  version: 2026-04-11
  embedding_model: text-embed-v4
  source_dataset: kb-2026-04-11
  chunk_policy: title-paragraph-v2
  build_job: reindex-20260411-003
  status: active
```

这个结构至少解决四件事：

- 索引来自哪份数据
- 用什么 embedding 生成
- 用什么 chunk 规则切分
- 当前是否可被线上服务使用

## 第14章：在线推理架构

### 第1题：为什么在线推理架构不能只看模型执行时间？

**参考答案**

因为完整请求延迟通常是：

$$
t_{\text{request}} =
t_{\text{queue}} +
t_{\text{preprocess}} +
t_{\text{model}} +
t_{\text{downstream}} +
t_{\text{postprocess}}
$$

如果你只看 `t_model`，就会漏掉：

- 队列等待
- 检索 / 特征调用
- 冷启动
- 路由和后处理

这也是为什么很多线上超时，并不是模型 kernel 慢，而是系统链路慢。

### 第2题：网关、路由层和模型副本池分别解决什么问题？

**参考答案**

| 组件 | 主要职责 |
|------|----------|
| 网关 | 统一入口、鉴权、限流、基础日志 |
| 路由层 | 按模型版本、租户、上下文长度分流 |
| 模型副本池 | 真正加载模型并执行推理 |

如果少了路由层，很多灰度、版本切换和长短请求分流都很难做；如果少了副本池治理，扩容和缓存策略就很难稳定。

### 第3题：为什么尾延迟比平均延迟更能决定用户体验？

**参考答案**

用户感受到的通常是“偶发特别慢”而不是“平均还不错”。  
平均延迟掩盖了最差那批请求，而这些请求往往正是：

- 高峰期请求
- 长上下文请求
- 依赖链路异常请求

所以线上推理更应该盯：

- P95
- P99
- queue wait time

而不是只盯平均值。

### 第4题：如果一个 RAG 服务变慢，你会先从哪几段时间开始拆解？

**参考答案**

建议先拆成：

```text
queue
-> embedding
-> retrieval
-> rerank
-> prompt build
-> llm prefill
-> llm decode
-> postprocess
```

然后优先看：

1. `queue`：是不是副本不够或批处理策略变化
2. `retrieval / rerank`：是不是下游依赖慢
3. `prefill / decode`：是不是 prompt 变长、输出变长或 KV Cache 打满

这是 RAG 推理最常见的三段瓶颈。

## 第15章：批处理、调度与 KV Cache

### 第1题：为什么批处理策略会直接影响用户延迟？

**参考答案**

因为批处理的核心做法是“先等一会儿，再一起算”。  
所以总延迟通常可写成：

$$
t_{\text{request}} = t_{\text{wait for batch}} + t_{\text{batch compute}}
$$

批次越大，计算效率可能越高；但如果等待过久，用户体验会变差。  
因此批处理天然是吞吐和延迟之间的权衡器。

### 第2题：Prefill 和 Decode 为什么应该区别对待？

**参考答案**

因为它们的资源形态不同：

- Prefill 更像大块并行计算
- Decode 更像小步增量生成

Prefill 更容易吃算力和带宽；Decode 更容易受调度粒度和 KV Cache 影响。  
如果把它们完全当成一种负载处理，系统很容易在某一侧失衡。

### 第3题：KV Cache 为什么既是性能利器，也是显存压力来源？

**参考答案**

收益：

- 避免每步重算全部历史上下文
- 明显降低 decode 重复计算

代价：

- 显存占用随层数、上下文长度、并发数增长

粗略估算：

$$
M_{\text{kv}} \approx 2 \times L \times H \times T \times B \times \text{dtype\_bytes}
$$

所以 KV Cache 同时在做两件事：

- 降计算
- 升显存压力

### 第4题：Continuous batching 相比固定 batch 在工程上解决了什么问题？

**参考答案**

固定 batch 的问题是：

- 不同请求长度差异大
- 短请求要等长请求
- 已完成请求还占位置

Continuous batching 解决的是：

- 动态回收已完成请求槽位
- 动态接纳新请求
- 在 decode 阶段持续维持较高利用率

也就是说，它把“静态组批”升级成了“运行中调度”。

## 第16章：量化、编译与推理引擎

### 第1题：为什么量化收益常常和模型类型、硬件类型绑定？

**参考答案**

因为量化效果取决于三件事是否匹配：

1. 模型是否对精度敏感
2. 硬件是否对目标低精度格式有高效支持
3. 引擎是否能稳定执行该量化路径

所以相同量化策略在不同模型或硬件上，收益和风险差异会很大。

### 第2题：编译优化和量化分别更偏向解决什么问题？

**参考答案**

- **量化**：更偏向减少显存占用和带宽压力
- **编译优化**：更偏向减少执行开销、优化 kernel 和内存布局

量化主要改“表示”，编译主要改“执行计划”。

### 第3题：推理引擎和框架直接推理各自适合什么场景？

**参考答案**

| 方式 | 更适合什么 |
|------|------------|
| 框架直接推理 | 调试、实验、模型快速验证 |
| 专用推理引擎 | 高吞吐、批处理、缓存管理、多租户线上场景 |

如果你还处于快速迭代模型阶段，框架更灵活；如果你要长期托管线上流量，专用引擎通常更合适。

### 第4题：如果吞吐提高了但排障更困难，是否一定值得？为什么？

**参考答案**

不一定值得。  
要综合判断：

- 提升的吞吐值不值得额外复杂度
- 质量是否退化
- 灰度和回滚是否更难
- 排障时间是否显著增加

一个好的原则是：

> 线上优化必须同时通过“性能收益”和“可运维性”两道门。

## 第17章：多租户与成本治理

### 第1题：为什么推理平台一旦服务多个团队，就必须考虑多租户治理？

**参考答案**

因为这时平台不再只是技术系统，而变成资源分配系统。  
它必须回答：

- 谁优先
- 谁保底
- 谁能借闲置资源
- 谁为账单负责

没有多租户治理，就会演变成“谁更会抢资源，谁就赢”。

### 第2题：单位请求成本为什么不能只看模型大小？

**参考答案**

因为成本至少还受这些因素影响：

- 上下文长度
- 输出长度
- batch 利用率
- cache 命中率
- 副本空转时间
- 下游依赖调用成本

所以同一个模型，在不同流量形态下，单位请求成本可能差很多。

### 第3题：公平性和利用率为什么会冲突？

**参考答案**

如果极度追求利用率，平台倾向于：

- 尽量填满 GPU
- 长任务也不轻易中断

这会让高优先级短任务体验变差。  
如果极度追求公平，则可能：

- 保留很多配额不借出去
- 导致资源闲置

所以平台必须在两者之间做政策取舍，而不是追求某个单一最优值。

### 第4题：请为 3 类租户设计一套简单的 GPU 与优先级策略

**参考答案**

```yaml
tenant_policy:
  core_online:
    priority: high
    reserved_gpu: 8
    burst_gpu: 16
    preemptible: false
  internal_product:
    priority: medium
    reserved_gpu: 4
    burst_gpu: 8
    preemptible: false
  research_experiment:
    priority: low
    reserved_gpu: 0
    burst_gpu: 6
    preemptible: true
```

**思路**

- 核心在线业务要保底且不可抢占
- 一般产品业务可保底但可限制峰值
- 实验业务尽量利用空闲资源，并允许被抢占

## 第18章：容器与运行时

### 第1题：为什么训练镜像和推理镜像的优化目标不同？

**参考答案**

训练镜像更关注：

- 依赖完整
- 调试工具
- 数据处理工具链
- 编译 / 扩展能力

推理镜像更关注：

- 体积小
- 启动快
- 安全面更小
- 运行路径更稳定

所以两者目标不同，通常不适合长期复用同一个镜像。

### 第2题：容器能启动但 GPU 不可用时，你会从哪几层排查？

**参考答案**

建议从这几层排查：

1. 宿主机驱动是否正常
2. 容器运行时是否正确暴露 GPU
3. 设备插件是否正常注册资源
4. 镜像内 CUDA runtime / 框架版本是否匹配

可以用一个简单顺序记：

```text
驱动 -> runtime -> device plugin -> image / framework
```

### 第3题：请设计一个 AI 镜像的最小分层结构

**参考答案**

```text
base-os
  -> cuda-runtime
    -> framework-runtime
      -> team-common-libs
        -> project-code
```

**设计理由**

- 底层稳定层变化少
- 团队公共依赖可统一治理
- 项目代码更新不会重复构建全部底层

### 第4题：为什么 AI 镜像治理必须和宿主机驱动治理一起看？

**参考答案**

因为镜像里的 CUDA runtime 和框架最终仍然要跑在宿主机驱动之上。  
如果两边版本矩阵不兼容，就会出现：

- 镜像能起，但 GPU 不工作
- 性能异常
- 某些功能缺失

所以 AI 镜像治理不是纯镜像问题，而是“镜像 + 驱动 + 运行时”的组合治理。

## 第19章：Kubernetes for AI

### 第1题：为什么说 Kubernetes 是 AI 平台底座，而不是 AI 平台本身？

**参考答案**

Kubernetes 擅长解决：

- 容器编排
- 资源声明
- 服务发现
- 基本扩缩容

但它并不天然理解：

- 模型评测门禁
- 训练恢复语义
- KV Cache 管理
- 多租户 GPU 策略
- 模型质量和发布治理

所以它是底座，不是完整 AI 平台。

### 第2题：训练任务和推理服务分别更适合哪些 K8s 对象？

**参考答案**

- 训练任务：`Job`、`CronJob`、训练类 `CRD / Operator`
- 推理服务：`Deployment`、`StatefulSet`、专用 serving 控制器

原因是训练更偏一次性批任务，推理更偏长期在线服务。

### 第3题：GPU 为什么在 K8s 里不能被简单当成“另一个 CPU”？

**参考答案**

因为 GPU 是：

- 离散资源
- 型号差异大
- 拓扑差异大
- 显存强约束资源

而 CPU 更像通用可切分资源。  
GPU 调度常常需要额外语义，例如：

- 卡型选择
- 多卡拓扑
- gang scheduling

### 第4题：请写出一个需要额外调度语义的 AI 场景，说明为什么原生资源对象不够

**参考答案**

场景：**8 卡分布式训练任务必须一次性拿到同一节点池、同一拓扑上的 8 张卡。**

为什么原生对象不够：

- 原生资源请求只表达“我要 8 张 GPU”
- 但表达不了“这 8 张必须拓扑相近、必须同时到位、必须一起启动”

这就是为什么很多 AI 平台需要额外的调度器、CRD 或 gang scheduling 语义。

## 第20章：队列、配额与自动扩缩容

### 第1题：为什么 AI 任务比普通 Web 服务更依赖队列治理？

**参考答案**

因为 AI 任务往往：

- 更长
- 更吃 GPU
- 更依赖资源形状
- 更容易因优先级差异产生冲突

普通 Web 请求通常来了就处理，AI 任务更像“稀缺资源上的排产问题”。  
所以队列治理在 AI 平台里是核心体验层，不是附属功能。

### 第2题：如果推理服务冷启动很慢，自动扩容策略应如何调整？

**参考答案**

应做三类调整：

1. **提前扩容**：不要等 P99 已经爆炸才扩
2. **保留预热副本**：维持最小热池
3. **多指标触发**：结合 queue wait、pending requests、GPU 水位，而不是只看 CPU

**示意策略**

```yaml
scale_out_if:
  queue_wait_p95_ms: "> 150"
  pending_requests: "> 50"
  gpu_utilization: "> 70%"
min_replicas: 3
warm_pool: enabled
```

### 第3题：配额和优先级冲突时，平台应该如何解释决策？

**参考答案**

平台必须给出透明解释，例如：

- 你所在租户当前已用满保底配额
- 当前请求优先级低于核心线上队列
- 当前借用资源已被回收给高优先级业务

最好让这类信息直接体现在：

- 调度事件
- 队列看板
- 审计日志

否则平台会被用户理解成“黑箱”。

### 第4题：为什么训练场景下的 autoscaling 往往不如推理场景直接？

**参考答案**

因为训练任务常常要求：

- 固定卡数
- 固定拓扑
- 长时间连续运行

而新扩出来的节点不一定能马上满足当前作业的拓扑条件。  
所以训练更常见的是“集群容量扩展”，而不是推理那种基于请求的细粒度 HPA。

## 第21章：可观测性与容量规划

### 第1题：为什么 GPU 利用率高不一定代表用户体验好？

**参考答案**

因为高 GPU 利用率只说明设备很忙，不说明：

- 请求有没有在排队
- P99 延迟是否恶化
- 下游依赖是否变慢
- 模型输出是否仍然高质量

一个推理系统完全可能出现：

- GPU 90%+ 利用率
- 用户却感受到更慢和更多超时

所以必须结合队列、延迟和质量指标一起看。

### 第2题：设计一个推理服务的最小指标面板

**参考答案**

建议至少包含：

1. 请求量 / token 量
2. P50 / P95 / P99 延迟
3. queue wait time
4. GPU 利用率
5. 显存占用
6. active sequences / batch tokens
7. cache hit ratio
8. 单位请求成本或每千 token 成本
9. 关键模型质量指标

**最小 dashboard 逻辑**

```text
入口压力
  -> 队列
  -> 模型执行
  -> 成本
  -> 质量
```

### 第3题：容量规划时为什么要考虑故障冗余？

**参考答案**

因为现实中必须假设：

- 某些节点会挂
- 某些可用区会有故障
- 某些副本会被回滚或重建

如果容量规划只按“正常状态刚刚够用”来算，任何一次节点故障都会让系统进入超载。

因此容量规划应至少包含：

- 正常容量
- 峰值容量
- 故障冗余容量

### 第4题：如果平均流量稳定，但 P99 延迟突然恶化，你会优先看哪些指标关联？

**参考答案**

优先看：

1. queue wait time
2. 长请求 / 长上下文占比
3. 下游检索 / 特征依赖延迟
4. cache hit ratio
5. GPU 显存是否接近上限

因为 P99 恶化通常说明某一类边缘请求开始拖尾，而不是平均流量本身上升。

## 第22章：评测、发布与故障处理

### 第1题：为什么 AI 发布不能只看“接口没报错”？

**参考答案**

因为 AI 发布除了系统可用性，还必须看：

- 模型质量是否退化
- 是否误答、幻觉或引用错误
- 成本是否大幅上升
- 安全或权限边界是否被突破

接口 200 只能说明“服务返回了”，不能说明“返回得对、返回得值”。

### 第2题：请为一个 RAG 服务设计最小灰度发布流程

**思路**

- 这道题不是让你写“一个上线步骤列表”而已，而是要体现 RAG 系统比普通模型发布多了索引、检索质量和引用正确性的额外风险。
- 所以答案应同时覆盖离线验证、线上小流量验证、回滚准备和关键观测指标。

**参考答案**

**参考架构 / 流程**

```text
新模型 / 新索引
  -> 离线评测
  -> staging 验证
  -> 1% 灰度
  -> 观察系统指标 + 检索质量 + 用户反馈
  -> 10% 灰度
  -> 全量
  -> 保留旧版本回滚窗口
```

**关键检查点**

- 检索召回是否退化
- 引用是否正确
- P95 / P99 是否恶化
- 单请求成本是否异常上升

**参考答案**

一个最小可执行的 RAG 灰度发布流程应至少包含四段：

1. **离线验证**：先在固定问答集和检索集上比较新旧版本，确认召回、引用正确率和响应成本没有明显退化。
2. **staging 验证**：把“模型 + 索引 + prompt 配置”作为一组完整版本部署到预发布环境，验证依赖、权限和日志链路。
3. **小流量灰度**：先让极小比例真实流量进入新版本，重点观察系统指标、检索质量和用户反馈。
4. **逐步放量与可回滚保留**：在确认没有系统退化、质量退化和权限问题后逐步放量，同时保留旧版本一键切回路径。

这道题真正想强调的是：RAG 灰度对象从来不是“只有模型”，而是“模型 + embedding / 索引 + prompt / 配置 + 路由规则”的组合版本。

### 第3题：为什么 AI 系统的回滚往往不只是回滚模型文件？

**参考答案**

因为一个版本往往同时绑定：

- 模型权重
- tokenizer
- prompt 模板
- 索引版本
- 推理镜像
- 路由策略

如果只回滚模型文件，而其他对象仍是新版本，系统仍可能处于不一致状态。

### 第4题：举一个事故复盘最终应沉淀成平台能力的例子

**参考答案**

例子：某次模型灰度后发现引用错误率显著上升，原因是新索引版本与旧 embedding 模型不一致。

如果只是复盘写文档，问题还会再发生。  
更好的平台化沉淀应是：

- 发布前强制校验 embedding 模型版本与索引版本一致
- 不一致则禁止进入灰度流程

也就是说，好的复盘最终要把“人工经验”变成“系统规则”。

## 第23章：安全、隔离与治理

### 第1题：为什么 AI 平台的安全面比普通在线服务更宽？

**参考答案**

因为 AI 平台除了接口本身，还要保护：

- 训练数据
- 模型权重
- 向量索引
- 日志与反馈
- GPU 资源
- 镜像供应链
- prompt / 上下文数据

这比普通在线服务多了更多状态对象和更多可能泄露的内容。

### 第2题：RAG 系统为什么特别容易出现权限越界问题？

**参考答案**

因为 RAG 会把“检索到的文档内容”拼进模型上下文。  
如果检索或缓存阶段权限控制做错，就可能：

- 把用户无权访问的文档检索出来
- 进一步在回答中泄露给用户

所以 RAG 的权限边界不是单点 API 权限，而是：

检索权限 + 文档权限 + 缓存权限 + 输出过滤

### 第3题：请写出一个最小治理配置中必须包含的 4 类规则

**参考答案**

至少应包含：

1. **身份规则**：谁能提交任务 / 调模型 / 发版本
2. **资源规则**：谁最多能占多少 GPU / 队列 / 副本
3. **数据规则**：谁能访问哪些数据、索引、日志
4. **审计规则**：关键操作必须留痕

可以写成：

```yaml
governance:
  identity: required
  quota: enforced
  data_access: least_privilege
  audit_log: mandatory
```

### 第4题：举一个“治理停留在文档里”最终会失败的场景

**参考答案**

场景：

- 文档里写着“生产模型发布前必须过评测门禁”
- 但发布系统没有强制检查
- 某次紧急上线直接绕过评测
- 最终把有缺陷的模型推到线上

这就是典型的“规则存在，但系统不执行”。  
治理只有进入平台控制面，才真正有效。

## 第24章：构建一个 AI 平台

### 第1题：为一个 3 个团队共享 GPU 的平台画出高层架构图

**参考架构**

```text
                      ┌─────────────────────┐
                      │  开发者入口 / Portal │
                      └─────────┬───────────┘
                                │
                ┌───────────────┼────────────────┐
                │               │                │
        ┌───────▼───────┐ ┌────▼─────┐ ┌────────▼────────┐
        │ 训练任务提交层 │ │ 模型仓库 │ │ 发布 / 灰度控制层 │
        └───────┬───────┘ └────┬─────┘ └────────┬────────┘
                │               │                │
        ┌───────▼────────────────▼────────────────▼───────┐
        │            队列 / 配额 / 优先级调度层            │
        └───────┬────────────────┬────────────────┬───────┘
                │                │                │
         ┌──────▼──────┐  ┌──────▼──────┐  ┌──────▼──────┐
         │ Team A 任务  │  │ Team B 任务  │  │ Team C 任务  │
         └──────┬──────┘  └──────┬──────┘  └──────┬──────┘
                │                │                │
         ┌──────▼────────────────▼────────────────▼──────┐
         │             GPU / CPU / 存储 / 网络资源池      │
         └───────────────────────────────────────────────┘

旁路系统：
- 数据版本与特征系统
- 评测与门禁系统
- 监控 / 告警 / 成本归因 / 审计
```

**参考答案**

这个架构的关键不是“组件全”，而是四件事：

1. 三个团队共享同一资源池，但不直接抢卡，而是先进入统一调度层
2. 训练、发布和推理共享一套模型仓库和版本控制
3. 评测和灰度是发布链路中的强制环节
4. 监控、成本和审计始终作为旁路控制面存在

### 第2题：如果你的业务以在线推理为主，平台设计的优先级应如何调整？

**参考答案**

如果业务以在线推理为主，优先级应调整为：

1. 先保证在线 SLA
2. 再优化批处理和缓存
3. 再建设模型发布与灰度
4. 最后补训练平台复杂能力

原因是：

- 直接影响用户体验的是服务稳定性
- 线上队列、路由、限流、缓存决定了最直观的业务表现
- 训练平台虽重要，但若当前主要矛盾在在线服务，优先级应后移

### 第3题：试用本教程的框架分析一个你熟悉的 AI 系统

**示例：企业内部知识问答 RAG 系统**

**思路**

按照“资源 -> 链路 -> 平台 -> 治理”四层分析。

**参考答案**

#### 资源层

- GPU：承担 embedding 与 LLM 推理
- 存储：文档库、索引文件、日志
- 网络：API 网关、检索服务、模型服务之间调用

#### 链路层

```text
文档接入 -> 清洗 -> embedding -> 索引
用户请求 -> query embedding -> 检索 -> 重排 -> LLM -> 返回
```

#### 平台层

- 索引版本管理
- 模型版本与灰度
- 在线队列与限流
- 观测与成本归因

#### 治理层

- 文档权限边界
- 审计日志
- 发布门禁
- 成本与配额

**结论**

这个系统最容易出问题的地方通常不是 LLM 本身，而是：

- 索引版本错配
- 检索权限越界
- 队列与尾延迟恶化
- 成本失控

### 第4题：请为一个中型团队写出 AI 平台的三阶段建设路线图

**参考答案**

#### 阶段一：最小闭环（1-2 个季度）

目标：让训练 -> 评测 -> 模型注册 -> staging 发布跑通

重点建设：

- 统一训练 Job 入口
- 统一输出目录
- 简单模型 registry
- 最小评测门禁
- 基础监控

#### 阶段二：服务化与治理（2-3 个季度）

目标：让多模型、多团队共用推理和 GPU 资源

重点建设：

- 统一推理网关
- 队列、配额、优先级
- 灰度发布与快速回滚
- 成本归因和审计

#### 阶段三：平台成熟化（持续演进）

目标：让平台具备持续扩展和治理能力

重点建设：

- 多租户策略细化
- 自动容量规划与扩容
- 索引 / RAG / embedding 治理
- 事故复盘向平台规则沉淀

**阶段路线图示意**

```text
阶段一：先打通闭环
阶段二：再提升服务化和治理
阶段三：最后追求平台成熟与自动化
```
