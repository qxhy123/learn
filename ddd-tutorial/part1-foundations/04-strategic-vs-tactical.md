# 第04章：战略设计与战术设计的关系

## 学习目标

- 清晰区分战略设计和战术设计的职责
- 理解两者的协作关系和边界
- 掌握何时使用何种设计工具
- 避免常见的"有战术无战略"或"有战略无战术"的陷阱

---

## 4.1 两个视角看软件设计

DDD 的设计活动发生在两个不同的视角层次：

```
                     ← 视角层次 →
高层（鸟瞰）                              低层（显微镜）
    │                                          │
    ▼                                          ▼
战略设计                                    战术设计
Strategic Design                           Tactical Design
    │                                          │
    ├── 问：系统由哪些部分组成？              ├── 问：这个部分内部如何建模？
    ├── 问：各部分的职责边界在哪里？          ├── 问：如何表达业务规则？
    ├── 问：各部分如何协作交互？              ├── 问：如何保护业务不变量？
    └── 问：各部分的战略价值是什么？          └── 问：如何处理领域事件？
```

一个比喻：
- **战略设计**就像城市规划——决定哪里是住宅区、商业区、工业区，以及道路如何连接
- **战术设计**就像建筑设计——决定某栋建筑内部的结构、布局、功能分区

城市规划确定了建筑应该建在哪里、建多大；建筑设计则决定具体怎么建。

## 4.2 战略设计工具箱

战略设计处理"大图景"，主要工具有：

### 工具1：子域识别

```
目的：区分业务功能的战略价值
输出：核心域、支撑域、通用域的分类

    高竞争力差异
         ↑
         │    ★ 核心域（自研精做）
         │  
         │        ○ 支撑域（自研适度）
         │
         │              △ 通用域（购买/开源）
         └─────────────────────────────→
              低复杂度              高复杂度
```

### 工具2：限界上下文划分

```
目的：划定语义一致的边界，保护模型纯粹性
输出：各上下文的职责范围和名称

┌─────────┐   ┌─────────┐   ┌─────────┐
│ Context │   │ Context │   │ Context │
│    A    │   │    B    │   │    C    │
└─────────┘   └─────────┘   └─────────┘
```

### 工具3：上下文映射

```
目的：定义上下文之间的集成关系
输出：上下文地图（Context Map），标注集成模式

┌─────────┐  共享内核  ┌─────────┐
│  销售   │◄──────────►│  市场   │
└─────────┘            └─────────┘
     │ 防腐层                │
     ▼                      │
┌─────────┐  客户-供应商 ┌─────────┐
│  物流   │◄─────────────│  仓储   │
└─────────┘              └─────────┘
```

### 工具4：事件风暴（Event Storming）

```
目的：通过协作建模，快速发现业务事件、命令、策略
输出：业务流程可视化，上下文边界初稿

便利贴颜色约定：
🟠 橙色：领域事件（已发生的事）
🔵 蓝色：命令（触发事件的操作）
🟡 黄色：聚合（处理命令的领域对象）
🟣 紫色：策略（事件触发的后续动作）
🔴 红色：热点/问题（需要深入讨论）
```

## 4.3 战术设计工具箱

战术设计处理单个限界上下文内部的建模，主要工具有：

### 工具1：实体（Entity）

```python
# 有唯一标识，通过标识判断相等性，可以改变状态
class Order:
    def __init__(self, order_id: OrderId):
        self._id = order_id  # 核心：唯一标识
        ...
    
    def __eq__(self, other):
        return self._id == other._id  # 标识相等即对象相等
```

### 工具2：值对象（Value Object）

```python
# 无标识，通过属性值判断相等性，不可变
@dataclass(frozen=True)
class Money:
    amount: Decimal
    currency: str
    
    def __add__(self, other: "Money") -> "Money":
        if self.currency != other.currency:
            raise ValueError("货币不同不能相加")
        return Money(self.amount + other.amount, self.currency)
```

### 工具3：聚合（Aggregate）

```python
# 一致性边界，通过聚合根访问内部对象
class Order:  # 聚合根
    def add_item(self, product: Product, quantity: int) -> None:
        """只能通过聚合根方法修改聚合内的状态"""
        if self._status != OrderStatus.DRAFT:
            raise OrderException("只有草稿订单才能添加商品")
        item = OrderItem(product, quantity)  # 聚合内的实体
        self._items.append(item)
```

### 工具4：领域事件（Domain Event）

```python
# 领域中已经发生的事实，不可变
@dataclass(frozen=True)
class OrderPlaced:
    order_id: OrderId
    customer_id: CustomerId
    total_amount: Money
    occurred_at: datetime
```

### 工具5：仓储（Repository）

```python
# 聚合的持久化抽象，对上层隐藏存储细节
class OrderRepository(ABC):
    @abstractmethod
    def get(self, order_id: OrderId) -> Order: ...
    
    @abstractmethod
    def save(self, order: Order) -> None: ...
```

### 工具6：领域服务（Domain Service）

```python
# 不自然属于任何实体的领域逻辑
class PricingService:
    def calculate_order_price(
        self, order: Order, promotions: list[Promotion]
    ) -> Money:
        """计算订单价格，涉及多个聚合，放在领域服务中"""
        ...
```

## 4.4 两者的关系：战略指导战术

战略设计和战术设计不是两个独立的阶段，而是**相互影响、持续进行**的活动：

```
战略设计                          战术设计
    │                                 │
    │  1. 战略决定战术的范围           │
    │  ─────────────────────────────► │
    │  限界上下文 → 确定了在哪个上下文 │
    │  内使用战术模式                  │
    │                                 │
    │  2. 子域类型 → 影响战术投入度    │
    │  ─────────────────────────────► │
    │  核心域：全力应用战术模式         │
    │  通用域：可能只需简单CRUD         │
    │                                 │
    │  3. 战术发现 → 反馈战略认知      │
    │  ◄───────────────────────────── │
    │  建模过程中发现边界划分有问题，   │
    │  及时调整战略设计                │
```

### 具体例子：核心域 vs 通用域的战术差异

```python
# 核心域（推荐算法）：深度运用战术模式
class RecommendationEngine:
    """核心域——这里是竞争优势，值得精心建模"""
    
    def recommend_for_user(
        self, 
        user: UserProfile,
        context: BrowsingContext,
        strategy: RecommendationStrategy
    ) -> list[RecommendedItem]:
        # 复杂的领域规则，封装在领域模型中
        candidates = self._find_candidates(user, context)
        ranked = self._rank_by_strategy(candidates, user, strategy)
        filtered = self._apply_business_rules(ranked, user)
        return filtered

# 通用域（邮件发送）：直接用技术方案，不建领域模型
class EmailService:
    """通用域——不需要DDD，直接调用第三方"""
    
    def send(self, to: str, subject: str, body: str) -> None:
        # 直接调用SMTP或第三方API
        self._smtp_client.send_email(to, subject, body)
```

## 4.5 常见的设计陷阱

### 陷阱1：只有战术，没有战略

这是最常见的问题：团队学了实体、值对象、聚合，但没有做战略设计，结果：

```
问题现象：
- 一个限界上下文太大，包含了不相关的业务
- 同一个"User"在各处含义不同，到处有"上帝类"
- 模型边界模糊，服务间耦合严重

根本原因：
- 没有划分限界上下文
- 没有识别子域和核心域
- 没有建立统一语言

后果：
  Entity + ValueObject + 混乱边界 = 复杂的泥球
  ≠ DDD
```

### 陷阱2：只有战略，没有战术

相反的问题：画了漂亮的上下文地图，但内部代码仍然是贫血模型：

```python
# 有战略无战术的代码长这样：
# 限界上下文划分得很好，但内部...
class OrderService:  # 胖服务，所有逻辑在这里
    def place_order(self, dto):
        # 创建一个纯数据对象（贫血模型）
        order = Order()
        order.status = "pending"
        order.total = sum(item.price for item in dto.items)
        # 业务规则全部在服务层，领域对象只是数据容器
        if order.total > 10000:
            order.needs_approval = True
        db.save(order)
```

这种代码虽然有好的外部边界，但内部仍然是过程式编程。

### 陷阱3：战略设计一次完成，从不回顾

```
错误观念：先做两周战略设计，定好边界，然后专注战术

正确观念：战略设计是持续进行的，随业务发展持续演化

   第1个月：大致划分3个上下文
   第3个月：发现某个上下文太大，拆成2个
   第6个月：业务扩展，增加新上下文
   第12个月：合并两个关联紧密的小上下文
```

## 4.6 实践中的设计顺序

在一个新项目中，推荐的设计顺序：

```
Step 1: 领域探索（1-2周）
  ├── 与领域专家进行事件风暴
  ├── 识别业务流程中的关键事件
  └── 初步理解业务的核心逻辑

Step 2: 战略设计（1-2周）
  ├── 识别子域（核心/支撑/通用）
  ├── 划分限界上下文
  ├── 建立初版上下文映射
  └── 为各上下文建立统一语言词汇表

Step 3: 战术设计（持续进行）
  ├── 在每个限界上下文内部进行领域建模
  ├── 识别聚合、实体、值对象
  ├── 定义领域事件
  └── 设计仓储和服务接口

Step 4: 持续演化
  ├── 定期回顾战略设计
  ├── 根据新认识调整边界
  └── 深化战术设计
```

## 4.7 小团队的简化策略

DDD的完整实践对小团队可能过重，可以按以下方式简化：

```
团队规模 1-5人：
  ├── 战略设计：识别核心域，避免在通用域浪费时间
  └── 战术设计：实体+值对象+聚合，不一定需要仓储/事件

团队规模 5-20人：
  ├── 战略设计：完整的限界上下文划分和上下文映射
  └── 战术设计：全套战术模式，建立应用服务层

团队规模 20人以上：
  ├── 战略设计：完整实践 + 事件风暴 + 定期回顾
  └── 战术设计：全套战术模式 + CQRS + 领域事件总线
  └── 可考虑：微服务拆分（以限界上下文为边界）
```

## 本章小结

| 维度 | 战略设计 | 战术设计 |
|------|---------|---------|
| 关注点 | 系统边界和结构 | 单上下文内的模型 |
| 主要工具 | 子域、限界上下文、上下文映射 | 实体、值对象、聚合、领域事件 |
| 思维层次 | 鸟瞰全局 | 精细建模 |
| 产出 | 上下文地图、统一语言词汇表 | 领域模型代码 |
| 时间点 | 项目早期，持续维护 | 开发过程中，持续演化 |
| 关系 | 战略指导战术的范围和深度 | 战术反馈验证战略的合理性 |
