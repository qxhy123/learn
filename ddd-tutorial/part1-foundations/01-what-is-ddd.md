# 第01章：什么是领域驱动设计？

## 学习目标

- 理解软件复杂度的根本来源
- 明白DDD要解决什么问题
- 区分DDD与传统开发方式的核心差异
- 理解"以领域为中心"的设计哲学

---

## 1.1 软件开发的真正难题

许多开发者认为软件开发最难的部分是技术问题：如何选择框架、如何优化性能、如何处理并发。但Eric Evans在2003年的《领域驱动设计》中指出了一个更根本的问题：

> **软件最本质的复杂性，来自于问题域本身的复杂性——而非技术实现的复杂性。**

让我们通过一个例子来理解这句话。

### 例子：电商订单系统

想象你要开发一个电商的订单系统。从技术角度看，这不过是几张数据库表、几个CRUD接口。但当你深入业务时，你会发现：

- **订单**在不同状态下有不同的行为限制（待付款的订单可以取消，已发货的不行）
- **价格**涉及原价、折扣价、会员价、促销价，不同时间窗口规则不同
- **库存**需要区分实际库存、锁定库存、可用库存
- **退款**涉及部分退、全退、退货退款，且与支付方式相关
- **物流**状态与订单状态是两套独立却又关联的状态机

这些规则不是技术问题，而是**业务领域的固有复杂性**。如果开发者没有真正理解这些规则，写出来的代码迟早会出问题。

### 本质矛盾

```
业务专家（领域专家）    ←→    开发者（技术专家）
  懂业务，不懂技术              懂技术，不懂业务
  用自然语言描述问题            用代码实现解决方案
       ↓                              ↓
         ← 翻译鸿沟 →
    业务语言与代码脱节
    需求变化导致代码混乱
    软件越来越难以维护
```

DDD的核心目标，就是**弥合这道鸿沟**。

## 1.2 什么是领域驱动设计

**领域驱动设计（Domain-Driven Design，DDD）** 是一套软件开发方法论，其核心思想是：

> **把业务领域的知识放在软件设计的核心位置，让代码模型与业务模型保持深度一致。**

DDD不是一个框架，不是一套固定的技术栈，而是一种**思维方式和设计方法**。

### 三个核心主张

1. **聚焦核心领域（Focus on Core Domain）**  
   并非所有代码都同等重要。真正体现业务价值的"核心领域"值得投入最多精力，而通用功能（如权限、日志）可以选用现成方案。

2. **用模型探索领域（Explore Domain via Models）**  
   通过持续的建模活动，让开发者与领域专家共同构建对业务的深刻理解。模型不是一次性产出，而是随认知加深不断演化的工具。

3. **在统一语言中协作（Collaborate in Ubiquitous Language）**  
   开发者与领域专家使用同一套语言——这套语言既出现在需求讨论中，也直接体现在代码里。

## 1.3 DDD的两个层次

DDD分为两个层次，对应不同规模的设计问题：

```
┌─────────────────────────────────────────────────┐
│               战略设计 (Strategic Design)         │
│                                                  │
│  ┌──────────────┐    ┌──────────────┐           │
│  │  限界上下文   │    │   子域划分   │           │
│  └──────────────┘    └──────────────┘           │
│  ┌──────────────┐    ┌──────────────┐           │
│  │  上下文映射   │    │   统一语言   │           │
│  └──────────────┘    └──────────────┘           │
│                                                  │
│  解决：大系统如何划分边界？各子系统如何协作？     │
└─────────────────────────────────────────────────┘
             ↓  (战略指导战术)
┌─────────────────────────────────────────────────┐
│               战术设计 (Tactical Design)          │
│                                                  │
│  ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐  │
│  │ 实体 │ │值对象│ │ 聚合 │ │领域事│ │ 仓储 │  │
│  └──────┘ └──────┘ └──────┘ └──────┘ └──────┘  │
│                                                  │
│  解决：在单个上下文内，如何精确表达业务规则？     │
└─────────────────────────────────────────────────┘
```

战略设计回答"**在哪儿做**"，战术设计回答"**怎么做**"。

## 1.4 DDD与传统开发方式的对比

### 传统的数据驱动方式

```python
# 传统方式：先设计数据库表，代码围绕数据操作
class OrderDAO:
    def create_order(self, user_id, items, address):
        # 直接操作数据库
        db.execute("""
            INSERT INTO orders (user_id, status, total, created_at) 
            VALUES (?, 'pending', ?, NOW())
        """, user_id, total)
        
    def update_order_status(self, order_id, status):
        # 状态可以随意更新，没有业务规则保护
        db.execute("UPDATE orders SET status=? WHERE id=?", status, order_id)
```

**问题**：业务规则散落在各处，数据库表结构与业务概念脱节，随着需求增加，代码越来越难维护。

### DDD方式

```python
# DDD方式：先理解业务，代码表达业务语言
class Order:
    def __init__(self, order_id: OrderId, customer: Customer, items: list[OrderItem]):
        self._id = order_id
        self._customer = customer
        self._items = items
        self._status = OrderStatus.PENDING_PAYMENT
    
    def confirm_payment(self, payment: Payment) -> None:
        """确认支付——业务规则：只有待支付状态可以确认"""
        if self._status != OrderStatus.PENDING_PAYMENT:
            raise OrderException("只有待支付订单才能确认支付")
        self._status = OrderStatus.PAID
        self._record_event(PaymentConfirmed(self._id, payment))
    
    def cancel(self, reason: str) -> None:
        """取消订单——业务规则：已发货不可取消"""
        if self._status in (OrderStatus.SHIPPED, OrderStatus.DELIVERED):
            raise OrderException("已发货或已完成的订单不可取消")
        self._status = OrderStatus.CANCELLED
        self._record_event(OrderCancelled(self._id, reason))
```

**优势**：业务规则内聚在领域对象中，代码语言与业务语言一致，变更时只需修改对应的领域对象。

## 1.5 何时使用DDD？

DDD并不是万能药，它有其适用场景：

### 适合DDD的场景 ✅

| 特征 | 说明 |
|------|------|
| 复杂业务逻辑 | 涉及大量业务规则、状态流转、业务约束 |
| 长期维护项目 | 需要在变化中保持代码可维护性 |
| 领域专家可访问 | 能够与了解业务的人深度合作 |
| 核心竞争力系统 | 业务规则本身就是公司的核心价值 |

**典型案例**：金融交易系统、电商平台核心链路、ERP、医疗信息系统

### 不适合DDD的场景 ❌

| 特征 | 说明 |
|------|------|
| 简单CRUD应用 | 业务逻辑极简，主要是数据的增删改查 |
| 短期项目 | 投入大量设计成本不划算 |
| 纯技术工具 | 没有复杂业务领域（如工具库、基础设施） |
| 团队DDD经验不足 | 生搬硬套反而增加复杂度 |

> **DDD的成本**：前期需要大量的领域探索和建模，设计更复杂，学习曲线较陡。如果业务本身不复杂，这些成本无法回收。

## 1.6 DDD的历史与演化

```
2003年
  │  Eric Evans 出版《Domain-Driven Design》（蓝皮书）
  │  确立了DDD的核心概念体系
  ▼
2006-2010年
  │  社区开始实践，发现蓝皮书的战术模式难以落地
  │  Greg Young提出CQRS（2010年）
  │  Udi Dahan等人发展了Domain Events概念
  ▼
2013年
  │  Vaughn Vernon 出版《Implementing Domain-Driven Design》（红皮书）
  │  提供了更实用的实现指导
  ▼
2016-2018年
  │  微服务架构兴起，DDD的限界上下文与微服务边界高度契合
  │  Alberto Brandolini 推广 EventStorming 工作坊
  ▼
2018年至今
  │  DDD与微服务、事件驱动、云原生深度融合
  │  成为大型系统设计的主流方法论之一
```

## 1.7 一个建立直觉的比喻

把软件开发比作**翻译工作**：

- 业务领域是一本用"业务语言"写的书
- 代码是这本书的"代码语言"翻译版
- 传统开发是先按自己的理解翻译，再对照原文修改
- DDD是**先深入理解原文**，再翻译，且翻译结果要能被原文作者（领域专家）直接读懂

一个好的翻译，不仅内容准确，还要保留原著的结构和精神——这正是DDD的追求。

## 本章小结

| 核心要点 | 说明 |
|---------|------|
| 软件复杂度根源 | 问题域的业务复杂性，而非技术实现 |
| DDD核心目标 | 让代码模型与业务模型深度对齐 |
| 两个层次 | 战略设计（划边界）+ 战术设计（建模型） |
| 适用场景 | 复杂业务领域、长期维护、有领域专家协作 |
| 核心哲学 | 代码语言 = 业务语言 |
