# 第02章：DDD核心概念——领域、子域与限界上下文

## 学习目标

- 理解"领域"的含义与边界
- 区分不同类型的子域及其战略价值
- 深刻理解限界上下文的作用
- 识别上下文边界的常见信号

---

## 2.1 领域（Domain）

**领域**是DDD中最基础的概念，指的是**一个组织所从事的业务活动及其所对应的知识体系**。

简单说：领域就是"你的软件要解决的那个现实世界的问题"。

```
组织/公司
  └── 领域（Business Domain）
        ├── 银行：存款、贷款、风险控制、清算...
        ├── 电商：商品、订单、支付、物流...
        └── 医疗：挂号、诊疗、药品、保险...
```

领域包含两个层面：
- **问题空间（Problem Space）**：业务问题和挑战是什么
- **解决方案空间（Solution Space）**：我们如何用软件解决这些问题

DDD的战略设计主要在**问题空间**中工作，帮助我们理解业务结构；战术设计在**解决方案空间**中工作，指导我们如何编写代码。

---

## 2.2 子域（Subdomain）

现实中的业务领域通常很复杂，不可能用一个统一的模型描述所有问题。DDD将一个大领域拆分为多个**子域**。

### 子域的三种类型

```
           你的业务领域
          ┌─────────────────────────────────┐
          │                                 │
          │  核心域 ⭐       支撑域 🔧        │
          │  ┌──────────┐   ┌──────────┐   │
          │  │这是你的  │   │支持核心  │   │
          │  │竞争优势  │   │域工作    │   │
          │  │所在      │   │          │   │
          │  └──────────┘   └──────────┘   │
          │                                 │
          │  通用域 📦                       │
          │  ┌──────────────────────┐       │
          │  │通用功能，买现成的    │       │
          │  └──────────────────────┘       │
          └─────────────────────────────────┘
```

#### 核心域（Core Domain）⭐

- **定义**：公司的核心竞争优势所在，是业务成功的关键差异化因素
- **特征**：高度复杂，独一无二，不可外包，需要最好的人力投入
- **建议**：自主开发，精心设计，DDD战术模式全力应用

**例子（电商平台）**：
- 推荐算法（个性化推荐是差异化竞争力）
- 智能定价引擎（动态价格策略）
- 供应链优化（选品、补货算法）

#### 支撑域（Supporting Subdomain）🔧

- **定义**：支持核心域工作的业务功能，有一定业务规则但不是竞争优势
- **特征**：中等复杂度，有业务特定逻辑，但不是核心价值
- **建议**：自主开发，但设计精力适度投入即可

**例子（电商平台）**：
- 订单管理（有复杂规则，但所有电商都差不多）
- 商品目录管理
- 用户账户管理

#### 通用域（Generic Subdomain）📦

- **定义**：各行各业都需要的通用功能，没有业务特定逻辑
- **特征**：低复杂度（从业务角度），成熟方案多，外部可购买/开源
- **建议**：优先购买现成方案，不要自己造轮子

**例子（电商平台）**：
- 用户认证与授权（用Keycloak/Auth0）
- 发送邮件/短信（用第三方服务）
- 日志与监控（用ELK/Prometheus）
- 文件存储（用OSS/S3）

### 子域划分的战略意义

```python
# 错误的资源分配：把最好的人放在通用域
team_allocation_wrong = {
    "核心推荐算法": ["初级开发者"],        # ❌ 
    "订单管理": ["中级开发者"],            # 还好
    "权限系统": ["高级架构师（自研）"],    # ❌ 本可购买现成
}

# 正确的资源分配：聚焦核心域
team_allocation_right = {
    "核心推荐算法": ["资深算法专家", "高级架构师"],  # ✅
    "订单管理": ["中级开发者", "有经验的后端"],      # ✅
    "权限系统": "采购 Auth0 或开源方案",             # ✅
}
```

## 2.3 限界上下文（Bounded Context）

限界上下文是DDD中**最重要的战略工具**，也是最难理解的概念之一。

### 为什么需要限界上下文？

先看一个问题：

**"用户"在不同部门的含义相同吗？**

```
销售团队的"用户"：
  - 关心：联系方式、购买历史、消费能力、偏好
  - 关键属性：customerId, name, email, purchaseHistory, tier

运营团队的"用户"：
  - 关心：活跃度、功能使用情况、留存率
  - 关键属性：userId, registrationDate, lastLoginDate, featureUsage

风控团队的"用户"：
  - 关心：风险评分、异常行为、欺诈记录
  - 关键属性：userId, riskScore, flaggedBehaviors, fraudHistory
```

同样一个"用户"，在不同语境下含义截然不同。如果强行统一为一个大而全的User模型，会发生什么？

```python
# 反例：大而全的User模型（噩梦）
class User:
    # 基础信息
    user_id: str
    name: str
    email: str
    
    # 销售相关
    customer_tier: str          # VIP / 普通
    purchase_history: list
    credit_limit: Decimal
    
    # 运营相关
    last_login_date: datetime
    feature_usage: dict
    session_count: int
    
    # 风控相关
    risk_score: float
    fraud_flags: list
    blocked: bool
    
    # 物流相关
    default_shipping_address: Address
    preferred_courier: str
    
    # ... 越来越多的字段 ...
    # 这个类会变成一个永远无法理解的大泥球
```

### 限界上下文的定义

**限界上下文（Bounded Context）** 是一个**明确的语义边界**，在这个边界内：
- 领域模型是一致的、无歧义的
- 同一个术语有且只有一个含义
- 模型的内部一致性得到保护

```
┌──────────────────────────────────────────────────────┐
│                    电商平台                           │
│                                                      │
│  ┌──────────────┐    ┌──────────────┐               │
│  │  销售上下文  │    │  运营上下文  │               │
│  │              │    │              │               │
│  │  Customer    │    │  AppUser     │               │
│  │  - name      │    │  - userId    │               │
│  │  - tier      │    │  - lastLogin │               │
│  │  - orders    │    │  - sessions  │               │
│  └──────────────┘    └──────────────┘               │
│                                                      │
│  ┌──────────────┐    ┌──────────────┐               │
│  │  风控上下文  │    │  物流上下文  │               │
│  │              │    │              │               │
│  │  RiskProfile │    │  Recipient   │               │
│  │  - riskScore │    │  - address   │               │
│  │  - flags     │    │  - phone     │               │
│  └──────────────┘    └──────────────┘               │
│                                                      │
└──────────────────────────────────────────────────────┘
```

在各自的上下文中，虽然都代表"现实中的同一个人"，但**模型不同，语言不同，职责不同**。这正是限界上下文的价值。

### 限界上下文的内部结构

一个限界上下文通常包含：

```
限界上下文（Bounded Context）
├── 领域模型（Domain Model）
│   ├── 实体（Entities）
│   ├── 值对象（Value Objects）
│   ├── 聚合（Aggregates）
│   └── 领域事件（Domain Events）
├── 应用层（Application Layer）
│   └── 应用服务（Application Services）
├── 基础设施层（Infrastructure Layer）
│   ├── 仓储实现（Repository Implementations）
│   └── 外部服务适配器（External Service Adapters）
└── 接口层（Interface Layer）
    ├── REST API / GraphQL
    └── 消息监听器
```

## 2.4 识别限界上下文的边界

边界的划分是DDD中最需要判断力的部分。以下是一些常见的边界信号：

### 信号1：术语歧义

当同一个词在不同的业务讨论中有不同含义时，这是边界的强信号。

```
"Account"
  ├── 在财务部门：账目、账户余额，关联分类账
  ├── 在客服部门：客户账号、关联工单和联系历史  
  └── 在安全部门：登录账号、密码、权限、设备

→ 三个不同的限界上下文！
```

### 信号2：组织结构

Conway定律指出：**系统结构往往与组织结构一致**。不同部门、不同团队负责的业务往往自然形成边界。

```
电商公司组织 → 可能的限界上下文
├── 商品部门    → 商品目录上下文
├── 交易部门    → 订单上下文、支付上下文
├── 物流部门    → 物流上下文、仓储上下文
├── 用户增长    → 用户画像上下文
└── 风控部门    → 风险控制上下文
```

### 信号3：数据所有权

谁是某类数据的**权威来源（Source of Truth）**？数据的所有者往往就是边界的所在。

```python
# 问：订单的状态由谁决定？
# 答：订单上下文——它是状态数据的权威来源

# 问：用户的风险等级由谁决定？  
# 答：风控上下文——只有它才能更新风险评分

# 当一个服务需要另一个服务的数据时，是读取（查询）还是写入（命令）？
# 跨边界写入往往意味着边界划分有问题
```

### 信号4：变化频率不同

不同部分的代码以不同的频率和原因变化，可能属于不同的上下文。

## 2.5 一个完整的示例

让我们以一个简化的**保险公司**为例，进行子域和限界上下文的划分：

### 业务领域分析

```
保险公司业务领域
│
├── 核心域（保险公司的竞争优势）
│   ├── 精算定价（风险评估、保费计算）← 核心算法
│   └── 理赔调查（欺诈检测、理赔评估）← 差异化能力
│
├── 支撑域（支持核心业务）
│   ├── 保单管理（投保、续保、变更、退保）
│   ├── 客户管理（客户信息、联系方式）
│   └── 理赔处理（受理、审核、支付）
│
└── 通用域（买现成的）
    ├── 用户认证（OAuth2/OIDC）
    ├── 文件存储（投保材料、理赔材料）
    ├── 消息推送（短信、邮件通知）
    └── 支付（理赔款支付、保费收取）
```

### 限界上下文划分

```
┌─────────────────────────────────────────────────────────────┐
│                         保险平台                             │
│                                                             │
│  ┌───────────────┐      ┌───────────────┐                  │
│  │  精算上下文   │      │  承保上下文   │                  │
│  │               │      │               │                  │
│  │ RiskModel     │      │ Policy        │                  │
│  │ PremiumCalc   │◄─────│ Coverage      │                  │
│  │ ActuarialTable│      │ Insured       │                  │
│  └───────────────┘      └───────────────┘                  │
│                                  │                          │
│                                  ▼                          │
│  ┌───────────────┐      ┌───────────────┐                  │
│  │  欺诈检测上下文│      │  理赔上下文   │                  │
│  │               │◄─────│               │                  │
│  │ FraudProfile  │      │ Claim         │                  │
│  │ RiskIndicator │      │ Claimant      │                  │
│  └───────────────┘      │ Investigation │                  │
│                          └───────────────┘                  │
└─────────────────────────────────────────────────────────────┘
```

注意：在**承保上下文**中，被保险人叫 `Insured`；在**理赔上下文**中，同一个人叫 `Claimant`。这两个概念代表同一个人，但**建模的关注点不同**，放在各自的上下文中才是正确的做法。

## 2.6 常见误区

### 误区1：把子域和限界上下文混淆

```
子域 ≠ 限界上下文

子域：问题空间的概念，描述业务中的一块功能区域
限界上下文：解决方案空间的概念，是代码和模型的边界

两者之间的关系：
- 理想情况：一个子域对应一个限界上下文（1:1）
- 现实情况：
  - 一个子域可能被多个上下文实现（遗留系统迁移）
  - 多个子域可能在一个上下文中（小团队，合并处理）
```

### 误区2：边界划分越细越好

```
错误思维：微服务=好，所以限界上下文越多越好

正确思维：边界的目的是隔离变化，过细的边界带来
         不必要的分布式复杂度

原则：边界大小应与团队大小和业务复杂度匹配
     宁可边界大一点，也不要过度拆分
```

### 误区3：一旦划定就不能变

```
限界上下文是可以演化的！

随着对业务理解加深，边界会逐渐调整。
早期可以用"大上下文"，随着团队成长和业务复杂度增加，
再逐步细化拆分。这是正常且健康的演化。
```

## 本章小结

| 概念 | 所在空间 | 核心作用 |
|------|---------|---------|
| 领域 | 问题/解决方案空间 | 界定软件要解决的业务范围 |
| 子域（核心/支撑/通用） | 问题空间 | 识别业务价值，指导资源分配 |
| 限界上下文 | 解决方案空间 | 保护模型一致性，划定语义边界 |

**关键记忆点**：
- 核心域 = 投入最多的地方
- 通用域 = 买，不造轮子
- 限界上下文 = 语义边界，同一词汇在边界内唯一含义
