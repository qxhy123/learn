# 第05章：限界上下文（深入）

## 学习目标

- 深刻理解限界上下文的本质及其价值
- 掌握识别和划分限界上下文的系统方法
- 理解限界上下文的内部结构与外部接口
- 避免常见的边界划分错误

---

## 5.1 重新认识限界上下文

在第02章，我们初步了解了限界上下文。本章将深入探讨其本质。

**限界上下文（Bounded Context）** 的核心价值是：

> **在一个明确的边界内，维护一个一致的领域模型。**

让我们通过一个真实的例子理解"一致性"为何如此重要：

### 没有限界上下文的世界

```python
# 没有边界，所有代码共享一个巨大的"Product"模型
class Product:
    product_id: str
    name: str
    description: str
    
    # 目录相关
    category: Category
    tags: list[str]
    images: list[str]
    
    # 定价相关
    base_price: Decimal
    cost_price: Decimal           # 进货价（财务用）
    member_price: Decimal
    promotional_price: Decimal
    
    # 库存相关
    stock_count: int
    reserved_count: int
    warehouse_location: str
    reorder_threshold: int
    
    # 搜索相关
    search_keywords: list[str]
    search_ranking: float
    
    # 推荐相关
    recommendation_score: float
    co_purchase_items: list[str]
    
    # 合规相关
    safety_certificate: str
    import_license: str
    
    # ... 随着业务增长，还会不断增加字段
```

这个类变成了**上帝类（God Class）**，任何业务变更都可能影响所有使用该类的代码。

### 有限界上下文的世界

```python
# 商品目录上下文（Catalog Context）
class CatalogItem:
    item_id: CatalogItemId
    name: str
    description: str
    category: Category
    images: list[ImageUrl]

# 定价上下文（Pricing Context）
class PricedProduct:
    product_id: ProductId
    base_price: Money
    pricing_rules: list[PricingRule]  # 封装复杂定价逻辑
    
    def calculate_price(self, customer: Customer, date: date) -> Money: ...

# 库存上下文（Inventory Context）
class StockItem:
    sku: SKU
    available_quantity: int
    reserved_quantity: int
    
    def reserve(self, quantity: int) -> Reservation: ...

# 每个类都小而聚焦，每个上下文内部完全一致
```

## 5.2 限界上下文的本质特征

### 特征1：语义一致性（Semantic Consistency）

在边界内，每个术语有且仅有一个含义。

```
在"订单上下文"中：
  Customer = 下单的人（关注购买行为）
  
在"客服上下文"中：
  Customer = 需要服务的人（关注问题和工单）

同一个现实世界的"顾客"，在不同上下文中是不同的模型对象
这不是重复，而是针对不同关注点的不同建模
```

### 特征2：自治性（Autonomy）

每个限界上下文应该能够**独立演化**，不需要其他上下文的配合就能修改自己的模型：

```python
# 库存上下文可以独立改变其内部模型
# 无需影响订单上下文

# 改变前：
class StockItem:
    quantity: int

# 改变后（增加批次管理）：
class StockItem:
    batches: list[StockBatch]  # 内部结构变了
    
    @property
    def available_quantity(self) -> int:  # 但对外接口不变
        return sum(b.available for b in self.batches)

# 订单上下文完全感知不到这个变化
```

### 特征3：明确的接口（Explicit Interface）

上下文与外界的交互必须通过明确定义的接口：

```python
# 限界上下文对外暴露的接口（而非内部模型）
class InventoryService:  # 库存上下文的对外接口
    
    def check_availability(self, sku: str, quantity: int) -> bool:
        """检查是否有足够库存"""
        ...
    
    def reserve_stock(self, sku: str, quantity: int, order_id: str) -> ReservationId:
        """锁定库存"""
        ...
    
    def release_reservation(self, reservation_id: str) -> None:
        """释放库存锁定"""
        ...

# 外部调用者只看到这个接口，看不到 StockItem 的内部结构
```

## 5.3 识别限界上下文的系统方法

### 方法1：语言不一致分析

列出所有你能想到的"同名不同义"或"同义不同名"的情况：

```
同名不同义（需要拆分）：
┌──────────────────┬──────────────────────────────────────────────┐
│ 术语             │ 在不同地方的含义                              │
├──────────────────┼──────────────────────────────────────────────┤
│ "用户"           │ 买家视角 vs 卖家视角 vs 管理员视角            │
│ "账户"           │ 财务账户 vs 登录账号 vs 客服账号              │
│ "订单"           │ 购买订单 vs 采购订单 vs 工单                  │
│ "产品"           │ 商品目录 vs 库存单元 vs 财务资产              │
└──────────────────┴──────────────────────────────────────────────┘

→ 每一行都是潜在的边界信号
```

### 方法2：业务能力分析

将业务拆分为独立的能力域：

```
电商平台业务能力：

搜索与发现
  ├── 商品搜索
  ├── 商品推荐
  └── 商品目录浏览

购买流程
  ├── 购物车
  ├── 订单创建
  └── 支付

履约
  ├── 库存管理
  ├── 物流调度
  └── 配送跟踪

售后
  ├── 退换货
  ├── 投诉处理
  └── 评价管理

用户运营
  ├── 会员管理
  ├── 营销活动
  └── 积分体系
```

每个叶子节点可能是一个上下文，或几个叶子合并为一个上下文。

### 方法3：数据所有权分析

问自己：**谁是这类数据的主人（Source of Truth）？**

```python
# 数据所有权分析表
data_ownership = {
    "订单状态": "订单上下文",          # 只有订单上下文能更改
    "商品价格": "定价上下文",          # 定价引擎决定价格
    "库存数量": "库存上下文",          # 库存系统是权威
    "用户余额": "账户/钱包上下文",     # 财务上下文控制
    "物流状态": "物流上下文",          # 外部物流系统同步
}

# 如果发现某个服务需要"写入"别人拥有的数据
# 这是边界划分有问题的信号
```

### 方法4：变化频率分析

变化频率相近的功能往往属于同一个上下文：

```
频繁变化的核心业务逻辑 → 订单、支付、推荐算法
中等变化的业务功能 → 商品目录、用户管理
不常变化的基础功能 → 消息通知、文件管理

把变化频率差异极大的功能放在一个上下文里，
会导致该上下文频繁因为小原因而发布更新
```

## 5.4 限界上下文的内部层次

一个设计良好的限界上下文内部通常有以下层次：

```
┌────────────────────────────────────────────────────────┐
│                    限界上下文                            │
│                                                        │
│  ┌──────────────────────────────────────────────────┐  │
│  │                   接口层（Interface）             │  │
│  │    REST API / GraphQL / 消息监听器 / CLI          │  │
│  └──────────────────────────────────────────────────┘  │
│                         │                              │
│  ┌──────────────────────────────────────────────────┐  │
│  │                  应用层（Application）            │  │
│  │         应用服务（用例编排、事务管理）             │  │
│  └──────────────────────────────────────────────────┘  │
│                         │                              │
│  ┌──────────────────────────────────────────────────┐  │
│  │                  领域层（Domain）                 │  │
│  │    实体 / 值对象 / 聚合 / 领域服务 / 领域事件     │  │
│  │           ← 最核心的业务逻辑所在地 →              │  │
│  └──────────────────────────────────────────────────┘  │
│                         │                              │
│  ┌──────────────────────────────────────────────────┐  │
│  │                基础设施层（Infrastructure）       │  │
│  │    数据库 / 消息队列 / 外部服务 / 缓存            │  │
│  └──────────────────────────────────────────────────┘  │
│                                                        │
└────────────────────────────────────────────────────────┘
```

### 对外暴露的三种方式

```python
# 方式1：API接口（同步查询/命令）
@app.post("/orders")
def place_order(request: PlaceOrderRequest) -> OrderResponse:
    command = PlaceOrderCommand(...)
    return order_application_service.handle(command)

# 方式2：领域事件（异步通知）
# 订单上下文发布事件，库存上下文订阅
class OrderPlaced(DomainEvent):
    order_id: str
    items: list[dict]
    
# 方式3：共享内核（谨慎使用）
# 两个上下文共享同一个库中的某些基础类型
# 例如：共享一个通用的 Money 类型定义
```

## 5.5 上下文边界的物理形式

限界上下文在代码中的体现形式：

### 形式1：代码包/模块（单体应用）

```
monolith/
├── catalog/          ← 商品目录上下文
│   ├── domain/
│   ├── application/
│   └── infrastructure/
├── ordering/         ← 订单上下文
│   ├── domain/
│   ├── application/
│   └── infrastructure/
└── inventory/        ← 库存上下文
    ├── domain/
    ├── application/
    └── infrastructure/
```

**规则**：不同包之间只通过接口层交互，**禁止直接引用其他上下文的领域对象**。

### 形式2：独立服务（微服务）

```
catalog-service/       ← 独立进程
ordering-service/      ← 独立进程
inventory-service/     ← 独立进程

服务间通过：
  - REST/gRPC API 同步通信
  - 消息队列（Kafka/RabbitMQ）异步通信
```

### 形式3：混合（模块化单体 → 逐步拆分）

```
# 推荐的演化路径：
# 1. 先在单体中做好模块边界（逻辑隔离）
# 2. 证明边界正确后，按需提取为独立服务（物理隔离）

# 好处：
# - 无需一开始就承担分布式系统的复杂度
# - 边界设计有问题时，重构成本较低
# - 随业务增长自然演化
```

## 5.6 限界上下文的大小

多大的上下文是合适的？这是一个经常被问到的问题。

### 过大的上下文（Bad）

```
症状：
- 一个上下文的团队有10+人
- 模型中存在"上帝类"
- 同一个词在不同业务场景含义不同
- 部署困难，一个小改动需要测试大量代码

信号：内部的语义不一致性
```

### 过小的上下文（Bad）

```
症状：
- 大量跨上下文的"聊天模式"（chatty calls）
- 一个业务操作需要协调5+个上下文
- 分布式事务无处不在
- 难以找到某个功能在哪个服务里

信号：上下文之间的耦合太高
```

### 适当大小的判断标准

```
✅ 良好的上下文大小：
- 一个2-8人的团队能够独立拥有和维护它
- 上下文内部的变更不需要其他团队的批准
- 上下文边界与团队边界对齐（Conway定律正向利用）
- 测试套件在10分钟内能跑完
```

## 5.7 实战：划分一个电商平台

让我们实际操练一下，为一个中型电商平台划分限界上下文：

### 业务描述

某中型B2C电商平台，功能包括：商品浏览、搜索、购物车、下单、支付、发货、退货、会员积分、优惠券、客服工单。

### 分析过程

**步骤1：列出所有名词概念**
```
商品、品类、搜索关键词、库存、SKU、SPU、价格、折扣、优惠券、
购物车、订单、支付、流水、发货单、物流轨迹、退货申请、退款、
用户、会员、积分、等级、客服、工单、评价、投诉
```

**步骤2：按语义相关性分组**
```
[搜索与发现]：商品、品类、搜索关键词、搜索排名
[定价]：价格、折扣、优惠券、会员价
[库存]：库存、SKU、SPU、库存预警
[购物]：购物车、收藏
[订单]：订单、发货单
[支付]：支付、流水、退款
[物流]：物流轨迹、快递单
[售后]：退货申请、退款申请
[会员]：用户、会员、积分、等级
[客服]：工单、投诉、评价
```

**步骤3：考虑组织结构和团队**
```
技术团队：
├── 商品团队（3人）→ 商品目录、定价
├── 交易团队（5人）→ 购物车、订单、支付
├── 履约团队（4人）→ 库存、物流
├── 用户增长（3人）→ 会员、积分、营销
└── 用户体验（2人）→ 客服、评价
```

**步骤4：最终限界上下文划分**
```
┌─────────────────────────────────────────────────────────────────┐
│                         电商平台                                 │
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐         │
│  │ 商品目录上下文│  │   定价上下文  │  │  搜索上下文  │         │
│  │   Catalog   │  │   Pricing    │  │   Search    │         │
│  └──────────────┘  └──────────────┘  └──────────────┘         │
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐         │
│  │ 购物车上下文  │  │  订单上下文   │  │  支付上下文  │         │
│  │    Cart     │  │   Ordering   │  │   Payment   │         │
│  └──────────────┘  └──────────────┘  └──────────────┘         │
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐         │
│  │  库存上下文   │  │  物流上下文   │  │  售后上下文  │         │
│  │  Inventory  │  │   Shipping   │  │  AfterSale  │         │
│  └──────────────┘  └──────────────┘  └──────────────┘         │
│                                                                 │
│  ┌──────────────┐  ┌──────────────┐                           │
│  │ 会员/营销上下文│  │  客服上下文   │                           │
│  │  Marketing  │  │  CustomerSvc │                           │
```

## 本章小结

| 要点 | 内容 |
|------|------|
| 本质 | 语义一致的边界，保护模型完整性 |
| 核心特征 | 语义一致性、自治性、明确接口 |
| 识别方法 | 语言分析、业务能力、数据所有权、变化频率 |
| 内部结构 | 领域层 + 应用层 + 基础设施层 + 接口层 |
| 适当大小 | 2-8人团队，与组织边界对齐 |

## 思考练习

3. 如果让你重新划分当前系统的边界，你会如何权衡上下文的大小？
