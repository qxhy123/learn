# 第08章：事件风暴（Event Storming）

## 学习目标

- 理解事件风暴的核心思想和价值
- 掌握事件风暴的符号体系和流程
- 能够独立组织一场事件风暴工作坊
- 从事件风暴结果中提取设计信息

---

## 8.1 什么是事件风暴

**事件风暴（Event Storming）** 是由 Alberto Brandolini 发明的一种协作建模方法。它让领域专家和开发者在一面大白墙上，用便利贴共同探索业务流程。

```
事件风暴的核心理念：

  "The smartest person in the room is the room itself."
  最聪明的人，是整个房间里的所有人。

  传统方法：一个人或小组写需求文档 → 其他人执行
  事件风暴：所有相关人（业务+技术）共同建模

价值：
  ├── 快速获得对复杂领域的共同理解
  ├── 发现隐藏的业务规则和边界
  ├── 识别限界上下文的自然边界
  └── 建立团队的统一语言
```

---

## 8.2 事件风暴的便利贴符号系统

事件风暴使用不同颜色的便利贴代表不同概念：

```
🟠 橙色：领域事件（Domain Event）
  - 过去发生的事情（动词的过去时）
  - 例：OrderPlaced（订单已下）、PaymentCompleted（支付已完成）
  - 问法：这个系统中有哪些"重要的事发生了"？

🔵 蓝色：命令（Command）
  - 触发领域事件的指令（动词原形）
  - 例：PlaceOrder（下单）、ProcessPayment（处理支付）
  - 问法：什么操作导致了这个事件？

🟡 黄色：聚合（Aggregate）
  - 处理命令并产生事件的领域对象
  - 例：Order（订单）、Payment（支付）
  - 通常紧贴在命令和事件之间

🟣 紫色：策略/反应（Policy/Reaction）
  - 当某事件发生时，自动触发的后续动作
  - 例："当支付完成时 → 通知仓库发货"
  - 格式：When [Event], Then [Command]

🔴 红色热点（Hot Spot）
  - 争议点、问题、不确定的地方
  - 用于标记需要深入讨论的部分

🩵 浅蓝色：外部系统（External System）
  - 不在我们系统内的外部参与者
  - 例：支付宝、快递系统、短信服务

🟩 绿色：读模型/视图（Read Model / View）
  - 用户做决策时看到的信息展示
  - 例：订单详情页、库存查询结果

👤 人物贴纸：用户/角色（User/Actor）
  - 发起命令的人或系统角色
  - 例：买家、卖家、系统自动触发
```

## 8.3 事件风暴的三个阶段

### 阶段一：混沌探索（Chaotic Exploration）

**目标**：尽可能多地发现领域事件，不做评判。

```
操作步骤：
1. 每个人拿一沓橙色便利贴
2. 独立写下所有能想到的"已发生的重要事件"
3. 贴到墙上（任意位置，无需整理）
4. 大声读出来，让所有人听到

关键：
  - 不要讨论，只写和贴
  - 用过去时态命名（已xxx、xxx了）
  - 30-45分钟内完成

此阶段产出示例（电商场景）：
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ 商品已浏览   │  │ 订单已下单   │  │ 支付已完成   │
  └──────────────┘  └──────────────┘  └──────────────┘
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ 库存已锁定   │  │ 订单已取消   │  │ 退款已申请   │
  └──────────────┘  └──────────────┘  └──────────────┘
  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
  │ 商品已发货   │  │ 评价已提交   │  │ 优惠券已核销 │
  └──────────────┘  └──────────────┘  └──────────────┘
  ... （可能有100+张贴纸）
```

### 阶段二：时间轴排序（Timeline）

**目标**：将事件按时间顺序排列，讲述完整的业务故事。

```
操作步骤：
1. 找出"起始事件"（故事的开头）
2. 将事件从左到右按时间顺序排列
3. 发现重复的事件合并
4. 发现遗漏的事件补充
5. 用红色热点标记争议和问题

排序后的时间轴：

时间 ──────────────────────────────────────────────►

🟠         🟠           🟠          🟠          🟠
用户已注册  商品已加购物车  订单已创建   支付已完成   商品已发货

         🔴                  🟠          🟠
         争议：              库存已锁定   物流单已创建
         购物车逻辑
         不清楚
```

### 阶段三：深化建模（Deep Dive）

**目标**：为关键流程添加命令、聚合、策略等，完整建模。

```
对每个关键事件区域：
1. 添加蓝色命令（什么触发了这个事件？）
2. 添加黄色聚合（谁处理了这个命令？）
3. 添加紫色策略（这个事件触发了什么后续？）
4. 添加用户角色（谁发起了命令？）

示例（订单下单流程）：

👤 买家       🔵 下单          🟡 订单          🟠 订单已创建
              ──────────────►            ──────────────►
              
                                                    │ 策略
                                                    ▼
                                             🟣 当订单已创建
                                                触发：锁定库存
                                                
                                             🔵 锁定库存  🟡 库存   🟠 库存已锁定
```

## 8.4 事件风暴工作坊操作手册

### 准备工作

```
物料清单：
  □ 大卷白纸（最好有5-10米长）或大白板
  □ 橙色便利贴（1-2包）
  □ 蓝色便利贴（1包）
  □ 黄色便利贴（1包）
  □ 紫色便利贴（1包）
  □ 红色便利贴（热点用）
  □ 粗头马克笔（写字更清晰）
  □ 相机（拍照记录）

参与者：
  □ 领域专家（真正了解业务的人）：必须参加！
  □ 开发者（架构师/高级工程师）
  □ 产品经理/业务分析师
  □ 主持人（熟悉事件风暴的人）
  
  理想人数：6-10人
  太少：视角不够，遗漏多
  太多：协调困难，讨论失控

时间安排：
  混沌探索：30-45分钟
  时间轴排序：45-60分钟
  热点讨论：30-60分钟（每个热点5-10分钟）
  深化建模：60-90分钟
  总计：3-4小时
```

### 主持人指南

```
关键原则：
  ✅ 保持探索的开放态度，暂缓评判
  ✅ 确保所有人参与，不要让一人主导
  ✅ 鼓励"笨问题"（往往揭示假设）
  ✅ 记录分歧，不要强求立即解决
  
常用引导问题：
  - "然后发生了什么？"（推进时间轴）
  - "在这之前发生了什么？"（发现前置条件）
  - "谁触发了这个操作？"（识别角色）
  - "如果这里出错了，会怎样？"（发现异常路径）
  - "这个词的意思，大家理解一致吗？"（发现语言问题）
```

## 8.5 从事件风暴到设计

事件风暴不只是建模工具，它的产出可以直接指导设计决策：

### 提取限界上下文

```
观察时间轴，找自然的"簇"：

时间轴上的一段事件，如果涉及的概念高度相关，
可能属于同一个限界上下文：

  ┌─────────────────────────┐  ┌─────────────────────────────┐
  │   商品目录上下文？        │  │     订单/购物上下文？         │
  │                         │  │                             │
  │ 🟠商品已创建             │  │ 🟠购物车已创建               │
  │ 🟠品类已更新             │  │ 🟠商品已加入购物车           │
  │ 🟠商品已上架             │  │ 🟠订单已创建                │
  │ 🟠商品已下架             │  │ 🟠订单已确认                │
  └─────────────────────────┘  └─────────────────────────────┘
  
  边界通常出现在：
  - 参与角色改变的地方
  - 数据所有权改变的地方
  - 有明显概念断层的地方
```

### 识别聚合

```
当多个命令都指向同一个"黄色聚合"时，
这个聚合可能是重要的设计元素：

  🔵下单 ───────────►
  🔵确认订单 ────────►  🟡 Order（订单聚合）
  🔵取消订单 ────────►
  🔵修改地址 ────────►

这说明 Order 是一个核心聚合，需要精心设计
```

### 识别领域事件

```python
# 事件风暴中的橙色便利贴，直接对应领域事件类：

# 🟠 "订单已创建" → 
@dataclass(frozen=True)
class OrderCreated:
    order_id: OrderId
    customer_id: CustomerId
    items: tuple[OrderItem, ...]
    occurred_at: datetime

# 🟠 "支付已完成" →
@dataclass(frozen=True)
class PaymentCompleted:
    payment_id: PaymentId
    order_id: OrderId
    amount: Money
    occurred_at: datetime

# 🟠 "库存已锁定" →
@dataclass(frozen=True)
class StockReserved:
    reservation_id: ReservationId
    order_id: OrderId
    sku: SKU
    quantity: int
    occurred_at: datetime
```

### 识别策略（跨上下文集成）

```python
# 🟣 策略（紫色便利贴）对应事件处理器：

# "当支付完成时 → 通知仓库发货"
class WhenPaymentCompletedThenNotifyWarehouse:
    """策略：支付完成后触发发货流程"""
    
    def __init__(self, warehouse_service: WarehouseService):
        self._warehouse = warehouse_service
    
    def handle(self, event: PaymentCompleted) -> None:
        self._warehouse.prepare_shipment(
            order_id=event.order_id
        )

# 这个策略可能连接了两个限界上下文：
# 支付上下文（发出事件） → 物流/仓储上下文（响应事件）
```

## 8.6 一个完整的事件风暴示例

以**外卖点餐系统**为例，展示事件风暴结果转化为设计：

### 事件风暴结果（简化）

```
时间轴：

👤顾客                                                    👤骑手
  🔵搜索餐厅  🔵选择餐品  🔵提交订单  🔵支付订单
                                          │
  🟠餐厅已搜索 🟠餐品已选择 🟠订单已提交  🟠支付已完成
                                          │
                               ┌──────────┘
                               │  🟣策略：支付完成 → 通知商家
                               ▼
                           🔵接单                    👤商家
                           🟠订单已接受
                               │
                    ┌──────────┘
                    │  🟣策略：接单 → 通知骑手配送
                    ▼
               🔵取餐  🟠餐品已备好  🔵送餐  🟠订单已送达
```

### 提取的设计

```python
# 识别出的限界上下文：
contexts = [
    "搜索上下文（Search）",       # 餐厅搜索、排序
    "餐厅目录上下文（Catalog）",   # 菜单、餐品管理
    "订单上下文（Ordering）",      # 购物车、订单创建、状态管理
    "支付上下文（Payment）",       # 支付、退款
    "配送上下文（Delivery）",      # 骑手调度、配送跟踪
    "通知上下文（Notification）",  # 短信、推送通知
]

# 识别出的核心聚合：
aggregates = [
    "Order（订单）",          # 处理大量命令
    "DeliveryTask（配送任务）", # 骑手配送的核心
    "Restaurant（餐厅）",      # 接单/拒单的主体
]

# 识别出的跨上下文集成点（策略）：
integrations = [
    "支付完成 → 通知商家接单",
    "商家接单 → 调度骑手",
    "骑手接单 → 通知顾客",
    "送达确认 → 完成订单 → 释放支付",
]
```

## 8.7 远程事件风暴

不在同一地点时，可以用数字化工具进行：

```
推荐工具：
  ├── Miro（最流行，功能完整）
  ├── Mural（协作效果好）
  ├── FigJam（轻量级）
  └── EventStorming.com（专用工具）

远程注意事项：
  ├── 事先准备数字白板模板（颜色分区）
  ├── 用视频会议保持实时沟通
  ├── 给所有人分配颜色（防止手忙脚乱）
  └── 每30分钟一次"屏幕截图"存档
```

## 本章小结

| 要素 | 作用 |
|------|------|
| 橙色（领域事件） | 业务流程的骨架 |
| 蓝色（命令） | 是什么触发了事件 |
| 黄色（聚合） | 谁负责处理命令 |
| 紫色（策略） | 跨上下文的业务规则 |
| 红色（热点） | 需要深入讨论的问题 |

**事件风暴的最大价值**：让所有人——无论是业务还是技术——对同一个领域建立共同理解。

---

## 思考练习

1. 选择你系统中的一个核心业务流程，尝试独自写出所有领域事件
2. 观察这些事件，它们自然地聚集成哪几组？这与限界上下文的划分有何关联？
3. 找出其中的"策略"（当X发生时，触发Y），这些策略是否跨越了上下文边界？
