# 第07章：子域类型与战略决策

## 学习目标

- 深入理解三种子域类型的本质区别
- 掌握子域识别与分类的实践方法
- 理解子域类型如何指导设计和资源投入决策
- 避免常见的战略决策错误

---

## 7.1 为什么子域分类如此重要

在软件开发中，**投入哪里很重要**。如果把最好的工程师放在通用域（比如自研一套权限系统），而核心业务逻辑却写得粗糙，这是战略上的失败。

子域分类帮助我们回答：
- 在哪里应该**精心设计**？
- 在哪里可以**快速实现**？
- 在哪里应该**直接购买**？

---

## 7.2 三种子域详解

### 核心域（Core Domain）

**定义**：公司在市场竞争中的差异化优势所在，是真正创造业务价值的核心。

```
核心域的判断标准：
✅ 如果这个功能做得很好，会带来竞争优势
✅ 如果不自己做，业务就无法存在
✅ 它包含我们独特的业务规则和知识
✅ 竞争对手很难简单复制

典型例子（按行业）：
  ├── 电商：个性化推荐算法、动态定价引擎
  ├── 金融：风险评估模型、反欺诈系统
  ├── 物流：路径规划算法、运力调度
  ├── 医疗：诊断辅助算法、药品交互检测
  └── 游戏：核心玩法机制、匹配算法
```

**如何对待核心域**：
```
投入策略：
  - 招募最好的领域专家
  - 使用最完整的DDD战术模式
  - 持续精炼模型，不断深化理解
  - 绝对不外包，绝对不用现成方案替代

代码质量期望：
  - 领域模型高度表达业务语义
  - 充分测试（包括领域规则的单元测试）
  - 定期重构，保持模型清晰
```

### 支撑域（Supporting Subdomain）

**定义**：支持核心域运作的业务功能，有一定的领域特殊性，但不是差异化竞争力。

```
支撑域的判断标准：
✅ 支撑核心业务运作，没有它核心业务无法工作
✅ 有一定的业务特定规则，不能直接用通用方案替代
✅ 但它不是你的竞争优势
✅ 竞争对手也有类似的功能（虽然细节不同）

典型例子（电商平台）：
  ├── 订单管理（复杂的状态机，但所有电商都类似）
  ├── 商品目录管理（有业务规则，但不是竞争优势）
  ├── 用户账户管理（有点特殊，但也不是核心差异）
  └── 退换货处理（有规则，但竞争对手也有）
```

**如何对待支撑域**：
```
投入策略：
  - 中等质量的领域设计
  - 可以外包开发，但要保留知识产权
  - 使用部分DDD战术模式
  - 优先考虑快速实现

代码质量期望：
  - 基本的领域建模（实体、值对象）
  - 关键业务规则有测试
  - 接口清晰，但内部可以稍粗糙
```

### 通用域（Generic Subdomain）

**定义**：各行各业都需要的通用功能，有成熟的现成方案可用。

```
通用域的判断标准：
✅ 这个问题已经被无数公司解决过了
✅ 有成熟的开源项目或商业产品可用
✅ 它不含任何你的业务特定逻辑
✅ 做得再好也不会带来竞争优势

典型例子：
  ├── 用户认证/授权（Auth0, Keycloak, Okta）
  ├── 邮件/短信发送（SendGrid, 阿里云短信）
  ├── 文件存储（AWS S3, 阿里云OSS）
  ├── 搜索引擎（Elasticsearch）
  ├── 支付网关（Stripe, 支付宝, 微信支付）
  ├── 日志/监控（ELK, Prometheus+Grafana）
  └── CI/CD（GitHub Actions, Jenkins）
```

**如何对待通用域**：
```
投入策略：
  - 优先购买现成产品（Buy before Build）
  - 使用成熟开源项目
  - 如果必须自研，外包开发，聚焦接入而非创新

代码质量期望：
  - 最小化自己编写的代码
  - 专注于接入和配置，而非定制
  - 用接口（防腐层）隔离通用域依赖
```

## 7.3 子域分类的动态性

子域的分类不是永久不变的：

```
分类会随时间变化的情况：

1. 核心域 → 通用域（技术商品化）
   早期：地图导航算法是核心域（自研，很难）
   现在：直接用高德/百度地图SDK（通用域）

2. 通用域 → 核心域（战略转型）
   早期：支付只是必要功能（通用域，用第三方）
   转型后：做自己的支付产品（核心域，如微信支付）

3. 支撑域 → 核心域（竞争重心转移）
   早期：物流只是履约工具（支撑域，外包物流）
   战略后：物流变成核心竞争力（如京东物流、亚马逊FBA）

启示：定期重新评估你的子域分类！
```

## 7.4 实践：子域分析工作坊

以下是一个系统性分析子域的方法，可以组织团队一起做：

### 步骤1：列举所有业务能力

```markdown
业务能力清单（以在线教育平台为例）：

教学内容
  ├── 课程内容创作
  ├── 视频处理和CDN
  └── 课程目录管理

学习体验
  ├── 个性化学习路径推荐
  ├── 进度跟踪
  └── 测验和评估

社区与互动
  ├── 学员讨论区
  ├── 直播互动
  └── 作业提交和批改

商业变现
  ├── 课程定价
  ├── 订阅/购买
  └── 营销/优惠券

运营支撑
  ├── 讲师管理
  ├── 内容审核
  └── 数据分析

基础设施
  ├── 用户认证
  ├── 文件存储
  ├── 消息通知
  └── 支付
```

### 步骤2：评分矩阵

```
对每个业务能力打分（1-5分）：

                        竞争差异化  业务特定性  市场成熟方案
──────────────────────────────────────────────────────────
个性化学习路径推荐           5           5           1
课程内容创作工具             3           4           2
视频处理和CDN               1           1           5
订阅购买流程                2           3           3
用户认证                    1           1           5
直播互动                    3           3           3
数据分析报表                1           2           4

判断规则：
  高竞争差异化 + 高业务特定性 → 核心域
  中等竞争差异化/业务特定性   → 支撑域
  低竞争差异化 + 高市场成熟方案 → 通用域
```

### 步骤3：分类结果

```python
subdomain_classification = {
    # 核心域
    "core": [
        "个性化学习路径推荐",    # 这是差异化竞争力
        "直播互动体验",          # 质量直接影响用户体验
        "学习效果评估系统",      # 独特的教学方法
    ],
    
    # 支撑域
    "supporting": [
        "课程内容创作工具",      # 需要一定定制
        "订阅购买流程",          # 有业务规则但不是差异化
        "讲师管理",              # 特定于教育行业
        "内容审核",              # 有特定规则
    ],
    
    # 通用域
    "generic": [
        "视频处理和CDN",         # 用第三方（七牛、阿里云）
        "用户认证",              # 用Auth0/Keycloak
        "消息通知",              # 用第三方短信/邮件服务
        "支付",                  # 接入微信/支付宝
        "数据分析报表",          # 用BI工具（Metabase等）
        "文件存储",              # 用OSS/S3
    ]
}
```

### 支撑域：适度建模

```python
# 支撑域（订单管理）：基本建模，清晰但不过度
class CourseOrder:
    """课程购买订单"""
    
    def __init__(self, order_id: str, learner_id: str, course: Course):
        self._id = order_id
        self._learner_id = learner_id
        self._course = course
        self._status = OrderStatus.PENDING_PAYMENT
    
    def mark_paid(self, payment_id: str) -> None:
        if self._status != OrderStatus.PENDING_PAYMENT:
            raise ValueError("只有待支付订单才能标记为已付款")
        self._status = OrderStatus.PAID
        self._payment_id = payment_id
    
    # 基本的状态管理，不需要复杂的领域建模
```

### 通用域：最小化代码

```python
# 通用域（邮件通知）：最小化自己的代码，直接用第三方
class EmailNotificationService:
    """邮件通知服务 - 使用第三方服务，我们的代码只是薄薄的适配层"""
    
    def __init__(self, sendgrid_client: SendGridClient):
        self._client = sendgrid_client
    
    def send_course_enrollment_confirmation(
        self, learner_email: str, course_name: str
    ) -> None:
        """发送课程报名确认邮件"""
        # 我们只关心发什么，不关心如何发
        self._client.send(
            to=learner_email,
            template_id="d-xxxx-enrollment-confirmation",  # SendGrid模板
            data={"course_name": course_name}
        )
    
    # 这里没有任何业务逻辑，只是调用第三方服务
    # 如果需要更换邮件服务商，只改这个类
```

## 7.6 常见战略错误

### 错误1：自研通用域（重复造轮子）

```
真实案例：
  团队花了6个月自研了一套权限管理系统
  结果：功能比Keycloak少，bug比Keycloak多
  
  背后的问题：
  - 权限系统是通用域，没有差异化价值
  - 这6个月本可以用于核心推荐算法的优化
  
  教训：通用域 → 买，不造
```

### 错误2：把核心域外包出去

```
真实案例：
  某公司外包了核心的风险评估算法
  结果：外包团队不理解业务，模型质量差
  更糟的是：核心知识沉淀在了外部团队，后来形成了依赖
  
  背后的问题：
  - 核心域是业务知识的结晶
  - 外包出去就是把竞争壁垒拱手相让
  
  教训：核心域 → 自研，不外包
```

### 错误3：对所有子域投入相同的精力

```
错误做法：
  给所有功能分配相同的设计深度
  ├── 发邮件：精心设计了领域模型 ← 浪费！
  ├── 核心算法：快速实现，代码质量差 ← 危险！
  └── 订单管理：过度设计 ← 浪费！

正确做法：
  核心域：最高设计质量，最多人力投入
  支撑域：合理质量，适当投入
  通用域：买现成的，最小化自研
```

## 7.7 与技术决策的关联

子域类型应该直接影响技术选型：

```python
# 不同子域的技术选型策略
technology_decisions = {
    "core_domain": {
        "approach": "自研，精心设计",
        "design_depth": "完整DDD战术模式",

        "team": "最有经验的工程师",
        "outsourcing": "禁止",
    },
    
    "supporting_domain": {
        "approach": "自研或外包，但保留控制",

        "approach": "购买或使用开源",
        "design_depth": "接入层最小化",
        "testing": "集成测试确保接入正确",
        "team": "任何工程师",
        "outsourcing": "完全可以",
        "products": ["Auth0", "Stripe", "SendGrid", "S3", "Elasticsearch"]
```

## 本章小结

| 子域类型 | 战略价值 | 设计深度 | 投入策略 |
|---------|---------|---------|---------|
| 核心域 | 竞争差异化 | 完整DDD战术模式 | 最优工程师，精心建模 |
| 支撑域 | 支撑核心业务 | 基本领域建模 | 适度投入，保持控制 |
| 通用域 | 基础设施 | 最小化接入代码 | 购买现成，不重复造轮子 |

**核心原则**：把最好的资源放在真正重要的地方。
