# 第22章：多 GPU 与 NCCL 基础

> 当单卡已经无法容纳模型、数据或吞吐需求时，问题就不再只是“kernel 怎么写”，而变成“多张 GPU 如何协同、如何交换数据、通信与计算如何平衡”。

---

## 学习目标

学完本章，你将能够：

1. 理解为什么会需要多 GPU
2. 建立 device、peer access、通信与同步的第一层直觉
3. 理解数据并行场景下 all-reduce 的核心作用
4. 知道 NCCL 在多 GPU 深度学习和通信中的定位
5. 建立“多 GPU 优化不是单卡优化简单复制”的基本认识

---

## 22.1 为什么需要多 GPU？

常见原因有三类：

### 原因一：单卡显存不够

例如：

- 模型太大
- batch 太大
- 中间激活和优化器状态占用太高

### 原因二：单卡吞吐不够

即使模型放得下，也可能希望：

- 更快训练
- 更高推理吞吐
- 更高并发处理能力

### 原因三：问题天然可分布

例如：

- 数据并行
- 模型并行
- pipeline 并行
- 大规模模拟中的域分解

但要注意，多 GPU 并不是“多几张卡就线性更快”，因为一旦跨卡，就会引入：

- 通信
- 同步
- 数据划分
- 负载平衡

---

## 22.2 最基础的多 GPU 视角：每张卡都是一个独立 device

在 CUDA 中，多 GPU 的最基础思路是：

- 机器上有多个 device
- 每个 device 有自己的显存和执行上下文
- 需要显式指定当前操作针对哪张卡

典型思路是：

```cpp
cudaSetDevice(device_id);
```

你可以把它理解为：

- 后续很多分配、kernel launch、stream 创建等操作
- 都将关联到当前设置的那张 GPU

### 这意味着什么？

意味着多 GPU 编程首先不是“同时模糊地操作很多卡”，而是：

- 明确每张卡是谁
- 明确每份数据在哪张卡上
- 明确什么时候需要跨卡交换

---

## 22.3 Peer Access 的第一层直觉

在某些硬件和拓扑条件下，不同 GPU 之间可以建立 peer access，使一张 GPU 能更直接地访问另一张 GPU 的内存或进行相关传输。

初学阶段你不必记很多配置细节，只需知道：

- 多 GPU 不一定只能“经由 CPU 中转”
- 某些设备之间可以有更直接的通信路径

### 为什么这重要？

因为跨卡通信成本常常是多 GPU 性能的关键因素之一。

如果通信路径更直接，通常更有利于提升整体效率。

---

## 22.4 数据并行：最常见的多 GPU 模式之一

在深度学习里，最常见的多 GPU 模式之一就是 **数据并行**。

它的基本思路是：

1. 每张 GPU 持有一份模型副本
2. 不同 GPU 处理不同数据分片
3. 每张 GPU 独立完成前向与反向
4. 再把梯度做聚合
5. 更新后的参数保持一致

### 关键问题出在哪里？

关键不在于“每张卡都能算”，而在于：

- 梯度如何聚合
- 聚合成本有多大
- 聚合是否会成为训练瓶颈

这正是 all-reduce 的核心角色。

---

## 22.5 什么是 all-reduce？

你可以先把 all-reduce 理解为：

- 每张 GPU 都有一份局部数据
- 先做全局规约（例如求和）
- 再把规约后的结果分发回所有参与者

在数据并行训练中，最典型的例子就是：

- 每张 GPU 都算出一份本地梯度
- 用 all-reduce 求全局梯度和（或平均）
- 让每张卡都得到同样的最终结果

### 为什么它如此关键？

因为在数据并行里，如果没有这一步：

- 每张 GPU 的模型更新方向就会不一致
- 训练无法维持统一状态

---

## 22.6 NCCL 是做什么的？

NCCL 可以粗略理解为：

- NVIDIA 提供的面向 GPU 间高性能通信的库

它常用于：

- all-reduce
- all-gather
- reduce-scatter
- broadcast
- 点对点 / 集体通信相关场景

### 为什么深度学习工程中经常提到它？

因为分布式训练和多 GPU 训练里，通信几乎是绕不过去的，而 NCCL 正是这类 GPU 集体通信的核心基础设施之一。

很多你在高层框架中看到的多卡训练能力，底层都可能依赖 NCCL。

---

## 22.7 为什么多 GPU 不等于单卡代码复制 N 份？

这是一个关键认识。

你不能简单理解为：

- “单卡快，就把同样的代码在 8 张卡上同时跑。”

因为多 GPU 还要回答：

- 数据如何切分
- 参数 / 中间结果如何同步
- 通信何时发生
- 通信是否能与计算重叠
- 哪张卡会不会成为慢卡
- 拓扑是否影响性能

### 一句话总结

单卡优化关注的是：

- 局部计算效率

多卡优化则还必须面对：

- 全局协同效率

---

## 22.8 通信与计算的关系

在多 GPU 场景中，一个很重要的目标是：

- 尽量让通信与计算重叠

例如在训练中，有时会尝试：

- 一部分梯度刚算完，就开始通信
- 同时后续层还在继续计算

这样做的目标是减少：

- 纯等待通信的空档时间

这和你在第 16 章学到的“传输与计算重叠”思维本质上是相通的，只不过现在把对象从：

- host-device 传输

扩展到了：

- device-device / multi-device 通信

---

## 22.9 一个最小双 GPU 思维模型

假设你有 GPU0 和 GPU1。

### 最简单的数据并行图景

- GPU0 处理 batch 的前半部分
- GPU1 处理 batch 的后半部分
- 各自得到本地梯度
- 用 all-reduce 聚合梯度
- 两边应用相同更新

### 从这里你应该意识到什么？

即使完全不写复杂代码，你也已经需要考虑：

- 两张卡的数据分工
- 两份模型副本
- 一次跨卡通信
- 同步点

这就是多 GPU 的最小闭环。

---

## 22.10 常见误区

### 误区一：多 GPU 一定线性加速

不对。通信、同步、负载不平衡都可能破坏线性扩展。

### 误区二：多 GPU 的难点只是“多写几个 `cudaSetDevice`”

不对。真正难的是数据划分和跨卡协作。

### 误区三：只要用了 NCCL，通信问题就自动解决了

不对。NCCL 是高性能通信工具，但整体扩展效果仍然受：

- 拓扑
- 工作负载
- 通信量
- 重叠策略

影响。

### 误区四：多卡训练的瓶颈一定在计算

很多时候真正限制扩展的恰恰是通信。

---

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| 多 GPU 需求 | 来自显存、吞吐和问题规模的扩大 |
| device 视角 | 每张 GPU 都是独立 device，需要显式管理 |
| 数据并行 | 多卡训练中最常见模式之一 |
| all-reduce | 聚合各 GPU 局部结果并回传给所有参与者 |
| NCCL | 面向 GPU 集体通信的高性能库 |
| 核心挑战 | 多 GPU 优化的关键在通信与协同，而不只是单卡复制 |
