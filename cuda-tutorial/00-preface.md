# 前言：如何使用本教程

## 教程设计理念

CUDA 不是“会写几个 kernel”就结束的技能。真正掌握 GPU 编程，需要同时建立三层理解：

1. **执行直觉**：线程、线程块、warp、SM 到底是怎样组织工作的
2. **内存直觉**：数据放在哪里、怎样移动、为什么会慢
3. **工程直觉**：什么时候该自己写 kernel，什么时候该直接调用库，什么时候该先 profile 再优化

本教程围绕这三层能力展开，遵循以下设计原则：

1. **从零开始，但不绕远路**：默认你没有 CUDA 经验，但也不会花大量篇幅把教程写成完整 C++ 入门课
2. **先正确，后优化**：每个主题先建立正确模型，再讨论性能与工程折中
3. **实验驱动**：重要概念尽量通过可运行的 CUDA C++ 代码说明
4. **贴近真实工程**：不仅讲 kernel，还讲 profiler、debugger、库生态、PyTorch 接口、多 GPU
5. **中文优先**：尽量把术语说清，把直觉讲透，把坑点点明

## 为什么这套教程值得从头读完？

很多人接触 CUDA 的方式是：

- 看一篇博客，抄一个向量加法例子
- 会写 `<<<blocks, threads>>>`
- 知道 `cudaMalloc`、`cudaMemcpy`
- 然后马上去写自己的 kernel

结果通常是：

- 程序能跑，但结果不稳定
- 能算对，但性能很差
- 知道“共享内存更快”，却不知道为什么更快
- 听说过 occupancy、coalescing、Tensor Core、CUDA Graphs，却不知道它们各自解决什么问题

本教程的目标不是让你“见过很多名词”，而是让你在遇到一个新的 GPU 计算问题时，能系统地问出并回答下面这些问题：

1. 这个问题是否真的适合放到 GPU 上？
2. 并行粒度应该放在线程、warp 还是 block？
3. 数据应该如何布局，才能获得连续访存？
4. 是否需要共享内存、异步拷贝、多个 stream？
5. 应该自己写 kernel，还是直接调用 cuBLAS / cuDNN / CUB？
6. 现在的瓶颈是计算、内存、同步，还是 launch overhead？

## 代码约定

本教程示例主要使用 CUDA C++，并遵循以下约定：

```cpp
#include <cuda_runtime.h>
#include <iostream>
#include <vector>

// kernel 名称使用动词短语
__global__ void vector_add(const float* a, const float* b, float* c, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        c[idx] = a[idx] + b[idx];
    }
}

// host 侧变量使用小写下划线
int num_elements = 1 << 20;
int threads_per_block = 256;
```

约定说明：

- Kernel 使用 `__global__`
- 设备内联函数可使用 `__device__` 或 `__forceinline__ __device__`
- 宏只用于简短、明确的错误检查或配置场景
- 在首次出现的重要 API 处解释参数含义，不依赖读者自行猜测

## 输出格式说明

代码输出示例统一采用如下形式：

```cpp
std::cout << "max error = " << max_error << std::endl;
// 输出: max error = 0
```

命令行输出示例统一采用如下形式：

```bash
nvcc vector_add.cu -o vector_add
./vector_add
# 输出: result check passed
```

## 环境准备建议

### 平台选择

如果你要真正动手练习 CUDA，请优先选择：

1. **Linux + NVIDIA GPU**：最常见、最稳定、最接近生产环境
2. **Windows + NVIDIA GPU**：适合桌面开发者，但注意编译器和 PATH 配置
3. **远程 GPU 服务器 / 云 GPU**：如果本地没有 NVIDIA 显卡，这是最实际的方案

需要特别注意：

- **Apple Silicon 和近年的 macOS 设备不支持 CUDA 开发**
- 如果你使用 Mac，可以把教程当作理论阅读材料，并把实验放在远程 Linux 机器上完成

### 最低工具链认知

在进入第 2 章之前，你最好知道以下命令的作用：

```bash
nvidia-smi        # 查看 GPU 与驱动信息
nvcc --version    # 查看 CUDA 编译器版本
g++ --version     # 查看主机侧 C++ 编译器版本
```

## 学习建议

### 对于完全没有 CUDA 经验的学习者

1. 严格按顺序学习第 1-9 章，不要跳过执行模型和内存层次
2. 每写一个 kernel，都先手算 2-3 个线程的索引结果
3. 一旦结果不对，优先怀疑索引、边界判断和同步，而不是“GPU 很神秘”
4. 在开始优化前，先用 `cudaDeviceSynchronize()` 保证错误暴露出来

### 对于深度学习工程师

1. 不要只停留在 `tensor.to("cuda")` 的层面
2. 重点关注第 7-9 章和第 13-18 章，这些章节决定你是否真正理解 GPU 性能
3. 学第 19-21 章时，把 cuBLAS、Tensor Core、PyTorch Extension 与自己熟悉的框架联系起来

### 对于系统与性能导向的学习者

1. 每个实验都尽量做对照版本：基线版 vs 优化版
2. 记录 block 大小、共享内存使用量、寄存器压力、同步次数
3. 不要用单次运行结果下结论，学会做 warmup 和重复测量

## 常见误区

### 误区一：GPU 一定比 CPU 快
不对。小规模任务、强串行依赖任务、数据搬运大于计算量的任务，GPU 可能更慢。

### 误区二：能跑就算掌握了 CUDA
不对。很多 CUDA 程序“能跑”，但存在隐藏越界、未同步读写、异步错误未检查、访存极差等问题。

### 误区三：优化就是不停调 block 大小
不对。block 大小只是优化变量之一；访存模式、数据重用、同步策略、kernel 融合、库调用选择同样重要。

### 误区四：自己写 kernel 一定比库更快
通常不对。对于 GEMM、卷积、归约、scan 这类常见算子，优先考虑高质量库往往更合理。
