# 第11章：前缀和与流压缩

> 前缀和之所以重要，不是因为它本身多神秘，而是因为它常常充当 GPU 算法里的“位置生成器”——它告诉你某个元素最终应该写到哪里去。

---

## 学习目标

学完本章，你将能够：

1. 理解前缀和（scan）的基本定义，以及 inclusive / exclusive 的区别
2. 知道为什么 scan 是构建更复杂并行算法的基础模块
3. 理解 compaction（流压缩 / 过滤）与前缀和之间的关系
4. 看懂一个基于标记数组和 scan 的并行过滤思路
5. 建立“scan 是很多 GPU 数据重排算法底座”的工程直觉

---

## 11.1 什么是前缀和？

前缀和（prefix sum / scan）指的是：

- 对一个序列
- 计算每个位置之前某种累积结果

以加法为例，给定输入：

```text
[3, 1, 7, 0, 4, 1, 6, 3]
```

### Inclusive scan

inclusive 表示当前位置自己也算进去：

```text
[3, 4, 11, 11, 15, 16, 22, 25]
```

### Exclusive scan

exclusive 表示当前位置自己不算进去，而是只统计前面的元素：

```text
[0, 3, 4, 11, 11, 15, 16, 22]
```

这两种形式都很常见，关键是要明确：

- inclusive 和 exclusive 的含义不同
- 使用时不能混淆

---

## 11.2 为什么 scan 在 GPU 上特别重要？

因为很多并行问题都不是单纯“算一个值”，而是：

- 先判断某个元素是否保留
- 再决定它应该写到输出数组的哪个位置

而这个“写到哪里”的信息，scan 恰好能提供。

### 一个直观例子

假设我们要从数组中筛出所有正数：

```text
in = [-2, 5, 0, 7, -1, 3]
```

先为每个元素生成一个标记：

```text
flag = [0, 1, 0, 1, 0, 1]
```

如果对 `flag` 做 exclusive scan：

```text
pos = [0, 0, 1, 1, 2, 2]
```

那么：

- `flag[i] == 1` 的元素应该被保留
- 它在输出中的目标位置就是 `pos[i]`

这样就能得到：

```text
out = [5, 7, 3]
```

这就是 scan 在 compaction 中的核心作用。

---

## 11.3 Scan 和 reduction 有什么不同？

scan 和 reduction 都是经典并行模式，但它们的目标不同。

### Reduction

- 最终得到一个或少量汇总结果
- 例如总和、最大值

### Scan

- 要为每个位置都生成一个累积结果
- 输出大小通常和输入相同

所以从并行实现角度看，scan 往往比 reduction 更复杂，因为：

- 你不能只关心最终一个值
- 需要保留各位置的结构性信息

---

## 11.4 一个最小的 block 内 scan 直觉

初学阶段先不急着追求完整高性能实现，你可以先理解 block 内 scan 的基本过程：

- 把一段输入放到 shared memory
- 通过若干轮传播，把前缀信息逐步传到后面的元素

例如输入：

```text
[1, 0, 1, 1, 0, 1, 0, 1]
```

经过 scan 之后，每个位置都知道自己前面有多少个保留元素。

### 为什么实现起来比 reduction 更细腻？

因为：

- reduction 最终只保留一份结果
- scan 要保留所有位置的前缀关系

这要求你在中间步骤中更仔细地维护每个元素的状态。

---

## 11.5 compaction：为什么它是 scan 的经典应用？

流压缩（compaction）通常指：

- 从一批输入中筛出满足条件的元素
- 去掉无效项
- 把保留项紧密写入输出

### 三步基本结构

#### 第一步：生成标记

```cpp
flag[i] = predicate(in[i]) ? 1 : 0;
```

#### 第二步：对标记做 scan

得到每个保留元素该写到输出中的位置。

#### 第三步：scatter 写出

```cpp
if (flag[i]) {
    out[pos[i]] = in[i];
}
```

这三步结构非常重要，因为很多更复杂的问题，本质上也是它的变体：

- 过滤
- 稀疏数据整理
- 分区
- 构建索引列表
- 稀疏图 / 稀疏矩阵预处理

---

## 11.6 一个并行过滤的伪代码骨架

```cpp
// step 1: mark
flag[idx] = (in[idx] > 0) ? 1 : 0;

// step 2: exclusive scan on flag
pos[idx] = exclusive_scan(flag)[idx];

// step 3: scatter
if (flag[idx]) {
    out[pos[idx]] = in[idx];
}
```

### 最终输出长度怎么得到？

如果使用 exclusive scan，那么输出总长度通常可由：

- 最后一个位置的前缀值
- 加上最后一个元素的 `flag`

共同确定。

这点很重要，因为 compaction 的输出长度通常不是预先固定的。

---

## 11.7 为什么 scan 是很多 GPU 算法的“基础设施”？

因为它解决的是一个非常普遍的问题：

> 在并行环境里，如何为每个有效元素分配一个唯一、紧凑的位置？

这个问题一旦解决，就能支撑很多更高层操作，例如：

- 删除无效元素
- 生成稀疏索引
- 按条件划分数据
- 构造图遍历前沿
- 构建 radix sort 的中间步骤

所以 scan 不只是“一个算法题”，而是 GPU 数据组织中的基础构件。

---

## 11.8 自己写 scan 还是调用库？

从学习角度：

- 自己实现一个基础版本很有价值
- 它能帮助你理解并行数据依赖如何传播

但从工程角度：

- scan 是一个非常常见且细节很多的基础操作
- 高质量库实现通常更可靠，也更高效

因此在真实工程中，常常会优先考虑：

- CUB
- Thrust
- 更高层框架中的现成 scan / prefix-sum 原语

这和我们前面反复强调的原则一致：

- 学习时理解机制
- 工程时优先复用成熟高性能实现

---

## 11.9 常见错误与混淆点

### 11.9.1 把 inclusive 和 exclusive 混淆

这会直接导致输出位置偏一位。

### 11.9.2 忘记输出长度不是原输入长度

compaction 后输出通常更短，不能默认还按原数组长度来理解结果。

### 11.9.3 标记阶段和 scatter 阶段的条件不一致

如果 `flag` 用的是一个条件，写回输出时又用了另一个条件，就会导致错位或越界。

### 11.9.4 误以为 scan 只是“求和小技巧”

它真正的价值在于位置分配和结构重排，而不只是数值累积。

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| scan | 为每个位置生成前缀累积结果 |
| inclusive / exclusive | 是否包含当前位置自身，语义不同 |
| compaction | 通过标记 + scan + scatter 完成过滤与紧缩 |
| 与 reduction 区别 | reduction 关注最终汇总，scan 保留每个位置的信息 |
| 工程价值 | scan 是很多并行重排和过滤算法的基础模块 |
| 实践建议 | 学会原理后，工程中优先考虑成熟库实现 |
