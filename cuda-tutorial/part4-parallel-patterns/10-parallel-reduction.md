# 第10章：并行归约

> 并行归约的难点，不在“把一堆数加起来”这件事本身，而在“如何让大量线程把局部结果高效、正确地收缩成一个答案”。

---

## 学习目标

学完本章，你将能够：

1. 理解归约问题为什么是 GPU 上的经典基础模式
2. 看懂 block 内树形归约的基本过程
3. 知道为什么直接对单个全局变量做大量原子加通常不是理想方案
4. 理解分阶段归约的常见结构
5. 对性能演进方向建立第一层直觉

---

## 10.1 什么是归约？

归约（reduction）指的是：

- 把一批元素
- 通过某种结合型操作
- 缩并成更少的结果，通常最终变成一个值

最常见的例子包括：

- 求和 `sum`
- 求最大值 `max`
- 求最小值 `min`
- 求逻辑与 / 或
- 求 `argmax` / `argmin`

例如：

```text
[3, 1, 4, 2, 8, 5] -> 23   （sum）
[3, 1, 4, 2, 8, 5] -> 8    （max）
```

在 CPU 上，最直觉的写法通常是一个循环。

```cpp
float sum = 0.0f;
for (int i = 0; i < n; ++i) {
    sum += in[i];
}
```

但在 GPU 上，这种串行方式显然无法发挥大量线程的优势。

---

## 10.2 为什么归约不是“一个线程处理一个元素然后结束”那么简单？

像向量加法那样的问题，每个输出元素彼此独立，非常适合“一个线程处理一个元素”。

但归约不同，因为：

- 很多线程最终都要把结果合并到更少的位置
- 合并过程本身存在依赖关系

这意味着你通常不能简单写成：

```cpp
out[idx] = ...
```

然后就结束。

归约真正要解决的是：

- 如何先让很多线程做局部工作
- 再把局部结果逐步汇总
- 最终得到全局结果

---

## 10.3 最朴素的方法：所有线程原子加到一个全局变量

最容易想到的 GPU 版本往往是：

```cpp
__global__ void reduce_atomic(const float* in, float* out, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < n) {
        atomicAdd(out, in[idx]);
    }
}
```

这段代码的优点是：

- 好理解
- 代码短
- 结果通常正确

但问题也很明显：

- 所有线程都争抢同一个地址
- 冲突非常严重
- 并发度高时性能通常很差

因此它更适合作为“正确但低效的起点”，而不是高性能实现。

---

## 10.4 block 内树形归约的基本思想

更常见的做法是：

1. 每个 block 先对自己负责的数据做局部归约
2. 生成一个 block 级结果
3. 再把这些 block 级结果继续归约

### 为什么这更合理？

因为它把“全体线程争抢一个地址”改成了：

- 每个 block 先在本地协作
- 大幅减少全局冲突数量

这通常更符合 GPU 的执行结构。

---

## 10.5 一个最小的 block 内求和骨架

下面先看一个经典教学版本：

```cpp
__global__ void reduce_block(const float* in, float* block_sums, int n) {
    __shared__ float sdata[256];

    int tid = threadIdx.x;
    int idx = blockIdx.x * blockDim.x + tid;

    sdata[tid] = (idx < n) ? in[idx] : 0.0f;
    __syncthreads();

    for (int stride = blockDim.x / 2; stride > 0; stride >>= 1) {
        if (tid < stride) {
            sdata[tid] += sdata[tid + stride];
        }
        __syncthreads();
    }

    if (tid == 0) {
        block_sums[blockIdx.x] = sdata[0];
    }
}
```

### 这段代码在做什么？

#### 第一步：加载到 shared memory

每个线程先把自己的输入元素放到 `sdata` 中。

#### 第二步：逐轮折半

例如如果 block 有 256 个线程，那么归约轮次大致是：

```text
256 -> 128 -> 64 -> 32 -> 16 -> 8 -> 4 -> 2 -> 1
```

每轮中：

- 前一半线程把后一半的数据加到自己身上
- 参与线程数逐步减半

#### 第三步：输出 block 结果

最后 `sdata[0]` 就是这个 block 的局部和。

---

## 10.6 为什么每轮都需要同步？

因为 shared memory 中的数据会在每轮迭代中被更新。

例如：

- 第 1 轮写完 `sdata[0..127]`
- 第 2 轮才能基于这些新值继续归约

如果不同线程对轮次推进不同步，就可能出现：

- 有些线程已经开始读取下一轮所需数据
- 但另一些线程还没写完上一轮结果

所以在教学骨架里，每一轮都要配一个 `__syncthreads()`。

---

## 10.7 多 block 的全局归约怎么做？

block 内归约只能得到：

- 每个 block 一个局部结果

如果整个输入很大，通常还需要第二阶段。

### 常见做法

#### 方法一：多 kernel 分阶段归约

1. 第一个 kernel 输出 `num_blocks` 个局部和
2. 第二个 kernel 再对这些局部和继续归约
3. 直到结果足够小，最后回到 host 或再做一次小规模归约

#### 方法二：block 内归约 + 少量原子操作

1. 每个 block 在 shared memory 内完成局部和
2. block 的线程 0 用一次 `atomicAdd` 累积到全局结果

这通常比“每个线程都原子加”更好，因为原子操作数量从：

- 所有线程数

减少为：

- block 数量

---

## 10.8 归约为什么是性能优化教学的经典案例？

因为它几乎涵盖了 CUDA 优化中的很多核心点：

- shared memory
- 同步
- 分支控制
- 原子操作
- warp 级优化
- 内存访问模式
- launch 配置

所以很多 CUDA 教程都会用 reduction 展示“从能跑，到更快”的优化演进。

### 常见演进方向

- 从全局原子加改为 block 局部归约
- 从 global memory 多次访问改为 shared memory 缓存
- 减少同步开销
- 优化最后几个 warp 的处理方式
- 让每个线程先处理多个元素，提高吞吐

初学阶段不必一次掌握所有优化变体，但要理解：

- 归约不是只有一个版本
- 正确性模板和性能模板往往不是同一层复杂度

---

## 10.9 一个更现实的思考：每线程处理多个元素

在大规模归约中，常见改进是：

- 每个线程不只读取一个元素
- 而是先顺序处理若干元素，形成线程私有局部和
- 再把局部和写入 shared memory 做 block 内归约

这样做的好处通常是：

- 减少 launch 和同步相对开销
- 提高内存带宽利用
- 降低后续共享归约的数据量

这也是为什么高性能 reduction 往往不只是“树形加法”那么简单，而是“线程局部累积 + block 内汇总 + 全局合并”的组合。

---

## 10.10 归约中的常见错误

### 10.10.1 忘记处理越界输入

如果 `idx >= n` 仍然直接读 `in[idx]`，就可能越界。

### 10.10.2 忘记同步

尤其在 shared memory 归约里，漏掉同步很容易导致结果不稳定。

### 10.10.3 把所有线程都原子加到一个位置

虽然可能正确，但性能通常会很差。

### 10.10.4 只做了一层 block 内归约，却误以为已经得到全局结果

如果有多个 block，`sdata[0]` 只是当前 block 的结果，不是整个输入的总和。

---

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| 归约 | 把大量元素通过结合型操作缩并成更少结果 |
| 直接原子加 | 实现简单，但高冲突时性能差 |
| block 内归约 | 用 shared memory 做局部汇总，是常见基础模式 |
| 多阶段归约 | 大规模全局归约通常需要多轮或分阶段处理 |
| 同步 | shared memory 归约中通常每轮都需要同步 |
| 优化方向 | 减少冲突、提高复用、降低同步和访存成本 |
