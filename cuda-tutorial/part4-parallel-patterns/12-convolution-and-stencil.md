# 第12章：卷积、Stencil 与邻域计算

> 邻域计算之所以值得单独讲，不是因为它们公式复杂，而是因为它们天然包含“数据重用”和“边界处理”这两个 GPU 编程里的核心难点。

---

学完本章，你将能够：

1. 理解卷积、stencil、邻域计算的共同结构
2. 知道为什么这类问题常常适合 shared memory 优化
3. 理解 halo 区域的概念以及它为什么必要
4. 看懂一个最小的 1D stencil 和 2D 邻域访问骨架
5. 对边界处理、数据重用和访存模式建立工程直觉

## 12.1 什么是邻域计算？

邻域计算指的是：

- 一个输出位置的结果
- 不只依赖输入中的同一位置
- 还依赖它周围的一小片邻域

常见例子包括：

- 图像滤波
- 卷积
- 有限差分
- 热传导 / 波动方程中的 stencil 更新
- 网格上的局部平均、梯度、拉普拉斯算子

### 一个简单例子

例如 1D stencil：

```text
out[i] = in[i - 1] + in[i] + in[i + 1]
```

这里 `out[i]` 依赖输入的局部邻域，而不是单一位置。

---

## 12.2 为什么这类问题特别适合 GPU？

因为它们通常具有：

- 大量重复、结构相似的局部计算
- 明显的数据并行性
- 相邻输出之间共享大量输入邻域

这最后一点尤其关键：

- 相邻线程处理相邻输出位置时
- 它们往往会反复读取很多相同输入元素

这就意味着：

- 如果直接从 global memory 一次次取，会有大量重复访问
- 如果把局部输入块先搬到 shared memory，就有机会复用

这也是 stencil / convolution 成为 shared memory 经典应用场景的原因。

---

## 12.3 从 1D stencil 建立直觉

看一个最小 1D stencil：

```cpp
__global__ void stencil_1d(const float* in, float* out, int n) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx > 0 && idx < n - 1) {
        out[idx] = in[idx - 1] + in[idx] + in[idx + 1];
    }
}
```

### 这里已经出现了两个核心问题

#### 问题一：边界怎么办？

- `idx = 0` 时没有左邻居
- `idx = n - 1` 时没有右邻居

所以代码必须显式处理边界。

#### 问题二：数据重复读取

例如：

- `out[i]` 需要 `in[i]`
- `out[i+1]` 也可能需要 `in[i]`
- `out[i-1]` 也可能需要 `in[i]`

相邻输出之间共享输入，这就是优化空间来源。

---

## 12.4 什么是 halo？

当你把输入按 block 分块处理时，光把“当前 block 对应的中心区域”搬进 shared memory 往往还不够。

因为计算边缘输出时，还需要访问 tile 外侧的一圈邻域数据，这部分额外区域通常就叫 **halo**。

### 1D 情况下的 halo

如果 stencil 半径为 1，即需要：

```text
in[i - 1], in[i], in[i + 1]
```

那么一个 block 若要处理一段中心区间，就还需要额外加载：

- 左边 1 个 halo 元素
- 右边 1 个 halo 元素

### 2D 情况下更直观

例如做 `3 × 3` 卷积时，一个 tile 除了中心区域，还需要周围一圈像素，这一圈就是 halo。

### 为什么 halo 重要？

因为如果没有它：

- block 边缘线程在计算时就拿不到完整邻域
- 要么算错
- 要么被迫回退去直接读 global memory

---

## 12.5 一个使用 shared memory 的 1D stencil 骨架

下面是一个教学型骨架，用来展示 tile + halo 的结构：

```cpp
__global__ void stencil_1d_shared(const float* in, float* out, int n) {
    __shared__ float tile[258];

    int tid = threadIdx.x;
    int idx = blockIdx.x * blockDim.x + tid;

    if (idx < n) {
        tile[tid + 1] = in[idx];
    }

    if (tid == 0 && idx > 0) {
        tile[0] = in[idx - 1];
    }

    if (tid == blockDim.x - 1 && idx < n - 1) {
        tile[blockDim.x + 1] = in[idx + 1];
    }

    __syncthreads();

    if (idx > 0 && idx < n - 1) {
        out[idx] = tile[tid] + tile[tid + 1] + tile[tid + 2];
    }
}
```

### 这段代码的结构重点

- `tile[tid + 1]` 存的是中心区间
- `tile[0]` 和 `tile[blockDim.x + 1]` 是左右 halo
- 同步后，每个线程都能从 shared memory 读取自己的邻域

它还不是完整工业级实现，但已经很好地体现了：

- tile
- halo
- block 边界
- 邻域复用

## 12.6 2D 卷积为什么更能体现共享内存价值？

假设你做一个 `3 × 3` 图像卷积。

对于每个输出像素 `(row, col)`，需要访问输入中：

- 自己
- 上下左右
- 四个对角

也就是 9 个位置。

如果相邻线程处理相邻像素，那么它们需要的输入区域有大量重叠。

### 这意味着什么？

意味着如果每个线程都各自从 global memory 读取 9 次：

- 相同像素可能被读取很多次

而如果 block 先把一块图像 tile 及其 halo 一起搬到 shared memory：

- 多个线程可以共享这份局部输入
- 重复 global 读取显著减少

这就是图像卷积与 stencil 优化的核心直觉。

---

## 12.7 边界处理为什么总是麻烦但不能省？

邻域计算几乎一定会遇到边界问题。

常见策略包括：

- 不计算边界，只处理内部区域
- 边界补零（zero padding）
- 复制边界（clamp）
- 环绕（wrap around）
- 镜像（reflect）

### 初学阶段建议

为了先把核心结构搞清楚，初学阶段最适合先用：

- 只处理内部区域
- 或最简单的 zero padding

不要一开始就把所有边界策略混进同一个 kernel，否则更容易把索引和逻辑都写乱。

---

## 12.8 邻域计算中的常见性能问题

### 12.8.1 重复 global memory 访问太多

这是最典型问题，也是 shared memory 优化的主要动机。

### 12.8.2 边界分支很多

如果每个线程都频繁做复杂边界判断，可能增加控制流复杂度并带来 warp 分歧。

### 12.8.3 tile 和 halo 装载不合理

如果 halo 的加载逻辑太零碎、线程利用不均，可能影响效率。

### 12.8.4 数据布局与访问方向不匹配

例如二维访问中，线程组织、行列顺序和内存布局不协调，可能破坏访存合并。

---

## 12.9 什么时候该自己写 stencil / convolution，什么时候该用库？

如果只是学习 CUDA：

- 自己写一个基础版本很有帮助
- 能深入理解 shared memory、tile、halo、边界处理

但在工程中：

- 卷积往往有成熟高性能库实现
- 深度学习场景通常优先使用 cuDNN 等库

所以更合理的态度是：

- 学习时自己实现，理解机制
- 工程时优先使用成熟库，除非你有非常特殊的算子需求

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| 邻域计算 | 一个输出依赖输入中的局部邻域 |
| stencil / convolution | 都是典型的局部邻域计算模式 |
| 数据重用 | 相邻输出共享大量输入，是 shared memory 优化来源 |
| halo | tile 外侧为支持边界邻域计算而额外加载的数据 |
| 边界处理 | 几乎不可避免，初学阶段宜先选简单策略 |
| 工程建议 | 学原理时可手写，工程中常优先考虑成熟库 |
