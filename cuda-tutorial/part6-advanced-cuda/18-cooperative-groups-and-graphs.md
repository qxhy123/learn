# 第18章：Cooperative Groups 与 CUDA Graphs

> 当你开始觉得“block 内同步”已经不够表达更复杂协作关系，或者发现反复 launch 很多小 kernel 的开销越来越明显时，就会开始真正理解为什么 CUDA 需要更高层的协作与执行抽象。

---

## 学习目标

学完本章，你将能够：

1. 理解 Cooperative Groups 想解决什么问题
2. 知道 thread block、warp 等不同协作粒度为什么需要更明确的编程接口
3. 理解 CUDA Graphs 的基本思想以及它为什么能减少 launch 开销
4. 看懂“重复工作流捕获为图再复用”的高层执行模式
5. 建立“这些特性属于高阶工程工具而不是入门必需品”的合理认知

---

## 18.1 为什么还需要更高层抽象？

前面的 CUDA 编程中，你已经有了：

- thread / block / grid
- `__syncthreads()`
- stream / event

这些工具已经足以写很多程序，但随着工程复杂度上升，会出现两个常见问题：

### 问题一：协作粒度表达不够清晰

例如：

- 我只想在一个 warp 范围内协作
- 我想把 block 划分成更小的线程组
- 我想让代码更清楚地表达“哪些线程是一组”

### 问题二：大量重复 launch 的开销变得显著

例如：

- 一个迭代流程里每次都提交同样结构的一串 kernel / memcpy / event 操作
- 单次工作很小，但重复很多次
- host 端 launch 开销开始影响整体性能

Cooperative Groups 和 CUDA Graphs 分别就是针对这两类问题的重要工具。

---

## 18.2 Cooperative Groups 解决的是什么？

Cooperative Groups 的核心目标，可以先概括为：

- 更清楚地表达“哪些线程在协作”
- 让不同粒度的线程组拥有更明确的同步与通信接口

### 为什么这有价值？

因为在传统写法中，很多代码虽然逻辑上是在做：

- warp 级协作
- block 内子组协作

但表达方式不够显式，容易：

- 依赖隐含假设
- 可读性差
- 迁移或维护时更容易犯错

Cooperative Groups 的价值就在于把“协作组”从隐含概念变成更明确的编程对象。

---

## 18.3 可以先把 Cooperative Groups 理解成什么？

初学阶段，一个足够好的理解是：

- 它提供了一套接口，让你能更明确地获得某个线程组
- 然后在这个组的范围内做同步、通信或协同操作

例如你可能会遇到这样的思路：

- 当前 block 是一个组
- block 内再按固定大小划分出更小的 tile 组
- 每个子组独立做局部协作

这种表达在：

- warp 级算法
- 分层归约
- 更复杂的并行协作模式

中会很自然。

---

## 18.4 为什么它比“直接假设 warp 行为”更稳妥？

很多早期 CUDA 代码会直接依赖：

- warp 大小
- 某些隐式同步直觉
- 手工组织的线程分组

这在简单情况下可以工作，但表达上往往不够清晰。

Cooperative Groups 更强调：

- 把协作粒度显式写出来
- 让同步和组操作围绕这个粒度展开

这样做的好处是：

- 可读性更高
- 推理更直接
- 更适合构建复杂分层协作算法

### 你此时不必追求 API 细节全记住

现阶段更重要的是知道：

- 它是一种更明确描述线程组协作的工具
- 适合写更高阶的组内同步与协作逻辑

---

## 18.5 CUDA Graphs 解决的是什么？

CUDA Graphs 的核心关注点则不同。它主要解决的是：

- 大量重复提交相似工作流时
- host 发射开销和调度开销太高的问题

### 典型场景

假设你的每一轮迭代都执行：

1. H2D memcpy
2. kernel A
3. kernel B
4. kernel C
5. D2H memcpy

如果这个流程要重复成千上万次，而每轮工作本身又不大，那么：

- host 每次都重新发起这些操作
- 相关 launch / 调度开销就可能开始变得重要

CUDA Graphs 的思路是：

- 先把这整个工作流记录成一张图
- 后续反复复用这张图的执行实例

从而减少反复逐条提交命令的成本。

---

## 18.6 什么叫“把工作流表示成图”？

你可以先把 graph 想成：

- 一组节点
- 节点之间有依赖关系
- 节点类型可能是 kernel、memcpy、memset、host function 等

例如一张简单图可以表示：

```text
H2D -> kernel A -> kernel B -> D2H
```

这里箭头表示依赖关系。

### 为什么这比每次重新 launch 有意义？

因为对于重复结构，图的“拓扑关系”其实每次都一样。

既然如此，就没有必要让 host 每一轮都重新逐条构建和提交完全相同的工作流。

---

## 18.7 Graph capture 的高层直觉

一个很常见的使用方式是 graph capture。你可以先把它理解为：

- 在某个 stream 上按正常方式提交一串操作
- 运行时把这串操作记录下来
- 形成一张 graph
- 然后实例化并反复执行

### 高层流程

1. begin capture
2. 提交 memcpy / kernel / event 等操作
3. end capture，得到 graph
4. instantiate graph executable
5. 重复 launch graph executable

### 为什么它常用于迭代流程？

因为很多训练、仿真、迭代优化过程都有稳定重复的执行图结构。

---

## 18.8 CUDA Graphs 的收益边界是什么？

它最适合的往往不是：

- 一次性、结构多变的工作流

而是：

- 重复次数很多
- 图结构相对稳定
- 单次工作粒度较细，小 kernel 较多
- host launch overhead 已经成为可见成本

### 如果单个 kernel 已经非常大怎么办？

这时 launch 开销相对整体执行时间可能没那么显著，graph 的收益也就未必突出。

这再次说明一个重要原则：

- 高阶特性是否值得引入，要由瓶颈决定，而不是由“新特性看起来很高级”决定。

---

## 18.9 一个简化的 graph 直觉示例

假设每次迭代都固定执行：

```text
memcpy H2D
-> preprocess kernel
-> main compute kernel
-> postprocess kernel
-> memcpy D2H
```

不使用 graph 时：

- host 每次都要逐条提交这些操作

使用 graph 时：

- 先把它们捕获成一张图
- 后面主要重复执行图实例

你可以把它看成：

- 从“每次重新发命令”
- 变成“复用一个预先编好的执行计划”

这也是它为什么经常被类比为“减少 launch overhead”的工具。

---

## 18.10 什么时候不该急着引入这些高阶特性？

### 不该急着用 Cooperative Groups 的时候

如果你还没有把：

- block / warp 基础
- shared memory
- 同步边界

搞清楚，那么过早上更高层线程组抽象，往往只会增加认知负担。

### 不该急着用 CUDA Graphs 的时候

如果你甚至还没测出：

- 时间是不是花在 launch overhead 上
- 工作流是否真的高度重复

那就不应为了“高级感”而引入 graph。

---

## 18.11 一个实用判断框架

### 对 Cooperative Groups 问自己

1. 我的算法是否真的存在多粒度组协作需求？
2. 现在的手工线程分组是否已经难以维护？
3. 显式组语义是否能让代码更清楚？

### 对 CUDA Graphs 问自己

1. 我的执行流程是否高度重复？
2. 小 kernel / 多步骤 launch 开销是否已经成为瓶颈？
3. 图结构是否足够稳定，值得捕获和复用？

如果这些答案都不明确，就先不要急着引入复杂度。

---

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| Cooperative Groups | 更明确地表达不同粒度线程组的协作关系 |
| 价值 | 提升复杂协作逻辑的可读性和可维护性 |
| CUDA Graphs | 把重复工作流表示为图并复用，以减少 launch 开销 |
| graph capture | 捕获一串固定执行操作并实例化复用 |
| 适用边界 | 更适合稳定、重复、高频的小粒度工作流 |
| 工程原则 | 只有在真实瓶颈出现时才引入这些高阶特性 |
