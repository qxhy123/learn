# 第19章：cuBLAS、cuDNN 与 CUB

> 学 CUDA 不是为了把所有算子都重写一遍，而是为了知道什么时候该自己写，什么时候该果断把问题交给成熟高性能库。

---

## 学习目标

学完本章，你将能够：

1. 理解 cuBLAS、cuDNN、CUB 各自主要解决什么问题
2. 知道为什么很多常见算子优先用库比手写 kernel 更合理
3. 理解“调用库”本身也是 CUDA 工程能力的一部分
4. 建立“自己写 kernel 的合理边界”意识
5. 看懂一个典型 GPU 工程中“库 + 自定义 kernel”混合使用的思路

---

## 19.1 为什么高性能 CUDA 工程离不开库？

到这里你已经学过：

- kernel 编写
- shared memory
- reduction / scan / stencil
- profiling
- 异步执行

这会让人产生一种危险错觉：

> 既然我已经会写 kernel，是不是所有东西都应该自己写？

通常不是。

真实工程中，一个非常重要的能力是：

- 判断问题是否已经有成熟高性能库实现
- 在有现成优质方案时，不重复造轮子

原因很简单：

- 这些库通常经过大量架构优化
- 覆盖了复杂边界情况
- 在不同 GPU 架构上做过长期调优
- 维护成本远低于自己长期手写和维护一套版本

---

## 19.2 cuBLAS：线性代数高性能库

cuBLAS 是 NVIDIA 提供的高性能 BLAS 库，主要面向：

- 向量运算
- 矩阵运算
- GEMM（矩阵乘法）
- batched GEMM
- 常见线性代数基础操作

### 为什么它重要？

因为很多高性能计算和深度学习工作负载，本质上都可以归结为：

- 矩阵乘法
- 向量-矩阵计算
- 矩阵-矩阵计算

而 GEMM 正是 GPU 上最核心、最值得深度优化的算子之一。

### 工程上的基本判断

如果你的需求是：

- 标准 dense GEMM
- 常见矩阵 / 向量操作

那么通常应优先考虑 cuBLAS，而不是自己手写一个矩阵乘法 kernel 去“挑战”它。

---

## 19.3 cuDNN：面向深度学习核心算子的高性能库

cuDNN 可以粗略理解为：

- 面向神经网络核心操作的高性能库

它常覆盖的内容包括：

- 卷积
- 激活函数
- pooling
- normalization
- RNN 某些组件
- 张量变换与部分融合操作

### 为什么深度学习工程离不开它？

因为深度学习中的许多高频算子：

- 结构固定
- 数据规模大
- 优化空间极深

用通用手写 kernel 虽然可以做一个“能跑版”，但和经过大量工程优化的库相比，通常差距很大。

### 一个重要现实

很多你在 PyTorch、TensorFlow 中调用的高层算子，底层很可能就间接依赖 cuDNN 或类似高性能后端。

也就是说：

- 框架并不是“绕开 CUDA”
- 它们往往是在更高层封装这些 CUDA 库能力

---

## 19.4 CUB：基础并行原语库

CUB 更像是一组高质量并行基础构件集合，典型包括：

- reduction
- scan
- sort 的部分基础模块
- histogram
- select / partition 等并行原语

### 它的角色和 cuBLAS / cuDNN 有什么不同？

- cuBLAS 更偏线性代数
- cuDNN 更偏深度学习核心算子
- CUB 更偏通用并行算法基础原语

### 为什么它在工程里很重要？

因为很多你自己想写的“中间算法模块”，其本质可能就是：

- 归约
- 前缀和
- 过滤
- 重新排列

这些正是 CUB 擅长的领域。

因此如果你遇到的问题不是 GEMM，也不是卷积，而是大量通用数据并行原语，那么 CUB 往往很值得优先考虑。

---

## 19.5 什么情况下应优先调用库？

这是本章最重要的问题之一。

### 情况一：问题是标准算子

例如：

- 矩阵乘法
- 标准卷积
- 基础 reduction / scan

如果库已经很好覆盖，优先调用通常最合理。

### 情况二：性能要求很高，但没有必要定制特殊数据流

高质量库往往已经把：

- 架构适配
- Tensor Core 利用
- 分块策略
- 调度细节

做得比通用手写实现更成熟。

### 情况三：你更需要工程可靠性，而不是教学价值

自己写 kernel 对学习很有帮助，但在工程里，维护成本也是真实成本。

---

## 19.6 什么情况下可能需要自己写 kernel？

### 情况一：算子非常特殊，库没有直接覆盖

例如：

- 特殊数据布局
- 自定义融合逻辑
- 非标准数学操作
- 稀疏或结构化稀疏的特殊模式

### 情况二：你需要把多个小操作融合成一个更高效的专用 kernel

有时候单独调用多个库函数虽然都很快，但：

- 中间结果落到 global memory
- launch 次数过多
- 数据搬运冗余

这时专门写一个 fusion kernel 可能更好。

### 情况三：研究与教学

为了理解 CUDA 原理，自己实现 reduction / matmul / scan 非常有价值。

但要清楚：

- 学习版实现 ≠ 工业级最优实现

---

## 19.7 “调用库”也需要你理解底层吗？

非常需要。

因为即使你不自己写底层算子，也仍然要理解：

- 数据布局是否符合库的高效路径
- 精度类型是否会触发 Tensor Core
- batch / stride / shape 是否影响性能
- 调用顺序是否引入不必要同步
- 是否需要 stream / event 协作

也就是说：

- 不写 kernel 不代表不需要 CUDA 素养
- 恰恰相反，只有理解底层，才能把库用好

---

## 19.8 一个现实中的混合模式

真实 GPU 工程中，常见的结构不是：

- 全部用库
- 或全部自己写

而是：

- 核心标准算子交给库
- 自定义 glue logic、预处理、后处理、特殊融合部分自己写 kernel

例如：

```text
数据准备 -> 自定义预处理 kernel -> cuBLAS GEMM -> 自定义后处理 kernel -> CUB reduction
```

这种“库 + 自定义 kernel”混合使用，其实是非常典型的工业模式。

---

## 19.9 一个最小的 cuBLAS 调用直觉

你在工程里看到 cuBLAS 时，常常并不是一大段复杂 CUDA kernel，而是：

- 创建 handle
- 配置参数
- 调用某个 GEMM / BLAS 接口

高层直觉上，你可以把它理解为：

- 你负责准备数据和调用上下文
- 库负责执行已经高度优化的底层计算

### 初学阶段的重点不是记全 API

而是建立这个判断：

- 对标准 dense 线性代数，优先库实现通常更合理

---

## 19.10 常见误区

### 误区一：会写 kernel 之后，就不该再用库

不对。高水平 CUDA 工程师往往更知道什么时候应该直接调用库。

### 误区二：用库就不需要理解底层

不对。你仍然需要理解数据布局、精度、流、同步、形状和性能路径。

### 误区三：库一定覆盖所有需求

不对。对于特殊算子、融合逻辑或研究型操作，你仍然可能需要自己写 kernel。

### 误区四：自己手写 matmul 是通往高性能的默认道路

对学习有价值，但在工程上，标准 dense GEMM 通常优先考虑 cuBLAS。

---

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| cuBLAS | 面向高性能线性代数，尤其是 GEMM |
| cuDNN | 面向深度学习核心算子，如卷积等 |
| CUB | 面向通用并行基础原语，如 reduction、scan、select |
| 工程原则 | 标准算子优先考虑成熟库 |
| 自定义边界 | 特殊算子、融合逻辑和研究需求常需要手写 kernel |
| 真实模式 | 工程中常见“库 + 自定义 kernel”混合方案 |
