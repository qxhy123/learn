# 第20章：WMMA 与 Tensor Core

> Tensor Core 真正重要的地方，不在于它是一个“更快的矩阵乘法按钮”，而在于它改变了你理解 GPU 矩阵计算路径的方式：数据类型、分块形状、指令级矩阵乘加、精度权衡都会因此进入视野。

---

## 学习目标

学完本章，你将能够：

1. 理解 Tensor Core 想加速的核心工作负载是什么
2. 知道 WMMA 在 CUDA 中扮演什么角色
3. 建立混合精度、矩阵 tile、warp 级矩阵乘加的第一层直觉
4. 理解为什么不是所有矩阵乘法都会自动走到理想 Tensor Core 路径
5. 知道 Tensor Core 的工程价值和使用边界

---

## 20.1 为什么 Tensor Core 会成为深度学习时代的关键硬件？

深度学习中最核心的工作负载之一就是：

- 大规模矩阵乘法
- 卷积可转化出的矩阵计算
- attention 中的大量 GEMM

这些操作具有共同特点：

- 规则
- 批量大
- 可分块
- 计算密集

Tensor Core 就是为了让这类矩阵乘加工作负载在硬件上走更高吞吐路径而设计的。

### 一句话直觉

你可以先把 Tensor Core 理解为：

- 面向小矩阵 tile 的高吞吐矩阵乘加专用执行路径

它不是通用“所有算术都更快”的单元，而是对特定模式特别强。

---

## 20.2 WMMA 是什么？

WMMA 通常可理解为：

- Warp Matrix Multiply-Accumulate 的编程接口

也就是说，它是 CUDA 提供的一组更接近 Tensor Core 编程模型的接口，帮助你以 warp 为粒度组织：

- 矩阵片段的装载
- 矩阵乘加
- 结果存回

### 为什么是 warp 级？

因为 Tensor Core 路径通常不是按“单个线程独立算一个标量”的方式设计的，而更像是：

- 一个 warp 协同处理一个小矩阵 tile 的矩阵乘加

这和你在前面看到的“一个线程一个元素”的基础模型非常不同。

---

## 20.3 从“标量乘加”到“tile 级矩阵乘加”

在最基础的 CUDA kernel 中，你习惯的计算像是：

```cpp
out[idx] = a[idx] * b[idx] + c[idx];
```

而在 WMMA / Tensor Core 的思维中，更自然的单位是：

- 一个小的矩阵块 `A_tile`
- 一个小的矩阵块 `B_tile`
- 一个累积块 `C_tile`

然后做：

```text
C_tile = A_tile × B_tile + C_tile
```

### 这意味着什么？

意味着：

- 你不再主要思考单个线程的标量任务
- 而是开始思考 warp 如何协作处理一个矩阵片段

这也是 Tensor Core 编程“难度跳跃”的来源之一。

---

## 20.4 混合精度为什么总和 Tensor Core 一起出现？

因为 Tensor Core 的高吞吐路径通常和特定数据类型强相关。

在深度学习实践中，常见模式包括：

- 输入使用较低精度，如 FP16 / BF16
- 累积使用更高精度，如 FP32

### 为什么这样做？

因为它在很多场景中能兼顾：

- 更高吞吐
- 较低存储带宽压力
- 足够好的数值表现

这就是常说的 **混合精度** 思路。

### 但要注意

混合精度不是“白送性能”，你仍然要关心：

- 精度误差
- 数值稳定性
- 算法对低精度的敏感程度

---

## 20.5 为什么不是所有 GEMM 都会自动获得理想 Tensor Core 加速？

这是一个非常重要的现实判断。

影响因素通常包括：

- 数据类型是否匹配
- 矩阵形状是否适合高效 tile 化
- 内存布局是否合适
- 对齐和 stride 是否友好
- 库是否选择到对应实现路径

### 这意味着什么？

意味着你不能简单认为：

- “只要在 GPU 上做矩阵乘法，就一定天然享受 Tensor Core 的最佳加速。”

更准确的理解是：

- Tensor Core 提供了强大的潜在加速路径
- 但是否真正走上这条路径，仍取决于数据类型、布局、尺寸和调用方式

---

## 20.6 一个 WMMA 高层执行直觉

即使暂时不深究 API 细节，你也应理解 WMMA 程序大致会经历：

1. 定义矩阵 fragment
2. 从内存装载 fragment
3. 执行矩阵乘加
4. 将结果 fragment 写回内存

高层上，它是把：

- warp 级协作
- tile 级矩阵计算
- 混合精度支持

组合成一个更接近 Tensor Core 工作方式的编程模型。

---

## 20.7 为什么很多工程里更常“间接使用” Tensor Core？

虽然可以手写 WMMA，但在大量真实工程中，更常见的情况是：

- 通过 cuBLAS / cuDNN / 高层框架间接触发 Tensor Core 路径

原因包括：

- API 更成熟
- 性能实现更复杂
- 调优空间很深
- 手写 WMMA 的维护成本高

### 工程上的常见路径

- 先通过高性能库获得 Tensor Core 加速
- 只有在非常特殊的自定义算子场景下，才考虑自己深入 WMMA 级实现

这和前一章的核心原则完全一致：

- 先判断问题是不是标准算子
- 标准算子优先交给成熟库

---

## 20.8 Tensor Core 的工程价值

### 在深度学习中

- 训练吞吐提升
- 推理吞吐提升
- 对大规模矩阵计算极其关键

### 在高性能数值计算中

对于适合混合精度和 tile 矩阵乘加的问题，也可能获得显著收益。

### 但它不是所有算法的答案

如果你的问题：

- 控制流复杂
- 数据访问极不规则
- 不是矩阵乘加主导

那么 Tensor Core 可能根本不是主要优化方向。

---

## 20.9 常见误区

### 误区一：Tensor Core = 所有 CUDA 程序都会自动更快

不对。它主要加速特定矩阵乘加工作负载。

### 误区二：用了低精度就一定没问题

不对。数值稳定性和误差边界必须结合具体问题评估。

### 误区三：想用 Tensor Core 就必须自己手写 WMMA

不对。很多工程里更合理的方式是优先通过库和框架间接使用。

### 误区四：学了 WMMA 就说明已经掌握了 GPU 高性能矩阵计算

不对。真正高性能的矩阵计算还涉及：

- 数据布局
- pipeline
- shared memory
- 寄存器分块
- 异步装载
- 库级调优策略

WMMA 只是其中一个重要切面。

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| Tensor Core | 面向小矩阵 tile 的高吞吐矩阵乘加专用路径 |
| WMMA | Warp 级矩阵乘加编程接口 |
| 思维变化 | 从标量线程编程转向 warp 协作的 tile 级矩阵计算 |
| 混合精度 | 常见于 Tensor Core 场景，需权衡性能与数值稳定性 |
| 工程现实 | 很多场景更常通过库间接使用 Tensor Core |
| 使用边界 | 不适合所有工作负载，核心仍是矩阵乘加密集问题 |
