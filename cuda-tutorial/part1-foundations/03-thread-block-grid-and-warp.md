# 第3章：线程、线程块、网格与 Warp

> 真正开始理解 CUDA，不是在你第一次写出 `<<<blocks, threads>>>` 的时候，而是在你能准确说出“这个线程到底在处理哪一块数据”的时候。

---

## 学习目标

学完本章，你将能够：

1. 理解线程、block、grid 在 CUDA 中的层级关系
2. 知道为什么 CUDA 使用线程组而不是“无限平铺的独立线程”
3. 建立 warp 的第一层执行直觉
4. 熟练写出 1D 和 2D 索引计算公式
5. 理解“线程组织方式会影响性能”这件事的根本原因

## 3.1 CUDA 的线程层级长什么样？

CUDA 不只是“启动很多线程”这么简单，它提供了分层组织：

- **thread**：最小执行单位
- **block**：一组线程构成线程块
- **grid**：多个 block 构成一次 kernel 启动的网格

可以用一个简单结构理解：

```text
grid
├── block 0
│   ├── thread 0
│   ├── thread 1
│   └── ...
├── block 1
│   ├── thread 0
│   ├── thread 1
│   └── ...
└── ...
```

每个线程都能通过内建变量知道自己在这个层级中的位置。

常见内建变量有：

- `threadIdx.x`
- `blockIdx.x`
- `blockDim.x`
- `gridDim.x`

后面你会看到，几乎所有基础 kernel 都围绕这些变量展开。

---

## 3.2 为什么要有 block 这一层？

很多人刚接触 CUDA 时会问：

> 为什么不直接让所有线程都是一个大集合，而非要分 block？

因为 block 不只是管理单位，它还决定了很多关键行为。

### 3.2.1 block 是协作边界

同一个 block 内的线程：

- 可以使用共享内存
- 可以用 `__syncthreads()` 做 block 内同步
- 通常会被安排在同一个 SM 上执行

而不同 block 之间：

- 默认不能直接同步
- 不能共享同一块 shared memory
- 执行顺序也不保证

这意味着：

- 你可以在 block 内设计协作
- 但不能假设整个 grid 是全局同步运行的

### 3.2.2 block 也是调度粒度

GPU 会把 block 分配到不同的 SM 上运行。一个 block 可以被看作“调度给某个 SM 的一份工作单元”。

这也是为什么 block 大小、共享内存使用量、寄存器压力等，都会影响并发度和吞吐。

## 3.3 先建立 1D 索引直觉

最常见的场景是 1D 数据，例如向量。

假设：

- 每个 block 有 `blockDim.x` 个线程
- 当前 block 编号为 `blockIdx.x`
- 当前线程在 block 内的位置为 `threadIdx.x`

那么全局线程索引通常写成：

```cpp
int idx = blockIdx.x * blockDim.x + threadIdx.x;
```

### 这个公式为什么成立？

因为：

- 前面所有 block 一共提供了 `blockIdx.x * blockDim.x` 个线程位置
- 再加上当前线程在本 block 内的偏移 `threadIdx.x`

例如：

- `blockDim.x = 256`
- `blockIdx.x = 3`
- `threadIdx.x = 10`

则：

```cpp
idx = 3 * 256 + 10 = 778
```

也就是说，这个线程负责全局第 778 个元素。

---

## 3.4 为什么总是需要边界判断？

假设你要处理 `n = 1000` 个元素，而每个 block 有 256 个线程。

那么通常会计算：

```cpp
int blocks = (n + threads_per_block - 1) / threads_per_block;
```

代入：

```cpp
blocks = (1000 + 255) / 256 = 4
```

4 个 block 共提供：

```text
4 × 256 = 1024 个线程
```

但你的数据只有 1000 个元素，所以最后 24 个线程其实“没有合法工作”。

因此 kernel 里通常要写：

```cpp
if (idx < n) {
    out[idx] = in[idx] * 2.0f;
}
```

如果没有这层判断，就可能发生越界访问。

## 3.5 2D 线程组织为什么重要？

当问题从向量变成矩阵、图像或二维网格时，2D 线程组织就很自然。

### 2D block 与 2D grid

CUDA 允许你使用：

- `threadIdx.x`, `threadIdx.y`
- `blockIdx.x`, `blockIdx.y`
- `blockDim.x`, `blockDim.y`

例如二维矩阵中，可以写：

```cpp
int col = blockIdx.x * blockDim.x + threadIdx.x;
int row = blockIdx.y * blockDim.y + threadIdx.y;
```

如果矩阵按 row-major 存储，那么线性地址通常是：

```cpp
int idx = row * width + col;
```

### 为什么这比手写一维映射更直观？

因为很多二维问题天然就是“按行列定位”的：

- 图像像素 `(row, col)`
- 矩阵元素 `(i, j)`
- stencil 邻域中心点 `(x, y)`

当线程组织和数据结构在形状上更一致时，代码会更容易理解，也更容易发现索引错误。

## 3.6 Warp 到底是什么？

前面讲的是逻辑层级：thread、block、grid。

但在硬件执行时，还有一个非常关键的概念：**warp**。

### 第一层直觉

在 NVIDIA GPU 上，线程不是完全一个一个独立执行的，而是通常以 **32 个线程为一组** 进行调度和执行。这一组就叫 warp。

你可以先把它理解为：

- CUDA 代码看起来像“每个线程都独立运行”
- 但硬件常常按 warp 为单位推进执行

### 为什么 warp 很重要？

因为很多性能现象都和 warp 有关：

- 分支分歧
- 内存访问合并
- warp-level primitive
- 指令执行效率

例如，如果一个 warp 中有些线程走 `if` 分支，有些线程走 `else` 分支，那么硬件往往需要分批执行这些路径，效率就会下降。

这就是后面要详细讲的 warp divergence。

---

## 3.7 逻辑独立线程 vs 硬件分组执行

CUDA 编程模型的一个核心优点是：

- 你以“线程独立”的方式思考程序
- 硬件再把这些线程组织成 warp 去执行

这种模型通常被称为 **SIMT**（Single Instruction, Multiple Threads）。

它和传统 SIMD 有相似之处，但不完全相同。初学阶段你只需抓住这层直觉：

- 代码层面像很多线程各自写自己的逻辑
- 执行层面 GPU 会尽量把相似线程一起推进

这也是为什么：

- 索引规则更重要
- 分支规则更重要
- 数据布局更重要

## 3.8 一个完整的线程映射例子

假设你要处理长度为 10 的数组，block 大小设为 4。

那么需要：

```cpp
int blocks = (10 + 4 - 1) / 4;  // 3
```

会启动 3 个 block，共 12 个线程。

线程映射关系如下：

```text
block 0: thread 0 1 2 3  -> idx 0 1 2 3
block 1: thread 0 1 2 3  -> idx 4 5 6 7
block 2: thread 0 1 2 3  -> idx 8 9 10 11
```

其中：

- `idx = 0..9` 是合法数据
- `idx = 10, 11` 超出范围，必须被 `if (idx < n)` 过滤掉

这个小例子非常重要，因为你以后看到更复杂的 kernel，本质上也还是在做类似映射，只不过从 1D 变成了 2D/3D，或者从“一个线程处理一个元素”变成“一个线程处理一个子任务”。

## 3.9 block 大小应该怎么理解？

初学者常常会问：

- 为什么很多示例喜欢用 128、256、512？
- 为什么不能随便设一个 37？

先说结论：

- 从“正确性”角度，只要不超过硬件限制，很多值都可以
- 从“性能”角度，block 大小通常会影响调度、warp 利用率、寄存器使用和共享内存利用

### 为什么常用 32 的倍数？

因为 warp 通常是 32 个线程。

如果 block 大小是 256，那么它正好包含：

```text
256 / 32 = 8 个 warp
```

这样通常更符合硬件调度粒度。

### 但 32 的倍数不代表一定最优

真正最优的 block 大小还和这些因素有关：

- kernel 的寄存器使用量
- shared memory 使用量
- 访存模式
- 是否受 occupancy 限制

所以初学阶段可以先记住：

- **256 是常见起点，不是神奇数字**

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| thread / block / grid | CUDA 用分层结构组织并行工作 |
| block 的意义 | block 是协作边界，也是调度粒度 |
| 1D 索引 | `blockIdx.x * blockDim.x + threadIdx.x` |
| 边界判断 | 用来防止覆盖范围外线程发生越界访问 |
| warp | 硬件通常以 32 线程为组推进执行 |
| block 大小 | 影响性能，但先保证映射正确更重要 |
