# 第1章：为什么需要 GPU 计算

> 如果一个问题可以被拆成海量、相互独立、结构相似的小任务，那么它往往就已经在向 GPU 招手了。

---

## 学习目标

学完本章，你将能够：

1. 理解 CPU 与 GPU 在设计目标上的核心差异
2. 判断一个问题为什么适合或不适合放到 GPU 上
3. 理解“吞吐导向”和“延迟导向”的区别
4. 建立对 CUDA 软件栈的总体认识
5. 用向量加法、矩阵运算等案例理解并行问题的基本特征

---

## 1.1 CPU 与 GPU 的设计目标不同

很多初学者第一次接触 CUDA 时，会把 GPU 想成“很多核心的 CPU”。这个类比有帮助，但不准确。

### CPU：更擅长低延迟、复杂控制流

CPU 的设计目标通常是：

- 尽快完成单个线程上的工作
- 处理复杂的分支与不规则逻辑
- 具备更强的缓存层次和分支预测能力
- 对系统调用、文件 IO、网络协议、数据库逻辑等通用任务友好

典型场景：

- Web 服务
- 编译器
- 数据库查询
- 解释器与运行时
- 强串行依赖的算法

### GPU：更擅长高吞吐、规则并行

GPU 的设计目标通常是：

- 同时处理大量结构相似的计算任务
- 用更多计算单元换取更高总吞吐量
- 接受单个线程更弱，但整体并行更强的架构选择
- 通过大量线程隐藏内存访问延迟

典型场景：

- 向量与矩阵运算
- 图像处理
- 物理仿真
- 深度学习训练与推理
- 大规模数值计算

### 一个常见比喻

可以把 CPU 和 GPU 分别想成：

- **CPU**：少数经验丰富的专家，擅长处理复杂、变化多、决策密集的任务
- **GPU**：一个规模庞大的流水线工厂，擅长把统一流程复制到大量数据上

这个比喻不是严格的体系结构描述，但足够帮助你建立第一层直觉。

## 1.2 吞吐导向 vs 延迟导向

### 什么是延迟？

延迟（latency）关心的是：

> 从任务开始到得到结果，需要多久？

例如一次数据库查询希望尽快返回；一次交互式点击希望尽快响应。CPU 在这类任务上通常有天然优势。

### 什么是吞吐？

吞吐（throughput）关心的是：

> 单位时间内，总共能处理多少任务？

例如：

- 每秒处理多少张图片
- 每秒完成多少次矩阵乘法
- 每秒更新多少个粒子状态

GPU 的设计重点通常是提升整体吞吐，而不是让某一个线程跑得极快。

### 为什么这对 CUDA 很重要？

因为当你把代码迁移到 GPU 时，真正要问的问题不是：

> “这一行代码能不能放到 GPU 上？”

而是：

> “我是不是有足够多、足够规则、足够相似的工作，让 GPU 的大量并行单元保持忙碌？”

如果答案是否定的，那么即使程序可以写成 CUDA，也可能不会得到理想加速。

## 1.3 什么样的问题适合 GPU？

一般来说，适合 GPU 的问题具备以下特征。

### 1.3.1 数据并行明显

最常见的模式是：

- 输入是一大批元素
- 对每个元素执行相同或相似的计算
- 元素之间依赖较少

例如向量加法：

$$c_i = a_i + b_i$$

对每个位置 $i$ 来说，计算彼此独立，非常适合并行。

### 1.3.2 计算密度较高

如果数据刚搬到 GPU 就做一点点运算，然后又搬回 CPU，那么 PCIe / NVLink 传输开销可能会吞掉收益。

更适合 GPU 的情况通常是：

- 一次拷贝后能做很多计算
- 数据能在 GPU 上停留更久
- 同一份数据可被多个 kernel 重复利用

### 1.3.3 访问模式相对规则

GPU 更喜欢：

- 连续、规整的内存访问
- 相似的控制流
- 大量线程执行同一类操作

GPU 不喜欢：

- 大量随机访问
- 极强的数据相关分支
- 很难并行化的递归或链式依赖

### 1.3.4 常见适合 GPU 的例子

| 问题类型 | 为什么适合 GPU |
|----------|----------------|
| 向量/矩阵运算 | 每个元素或子块可并行处理 |
| 图像卷积与滤波 | 像素或邻域计算高度相似 |
| 神经网络训练 | 大规模张量运算和规整访存 |
| Monte Carlo 仿真 | 多个样本路径彼此独立 |
| 粒子系统 | 每个粒子状态更新模式类似 |

## 1.4 什么样的问题不适合 GPU？

理解“不适合”同样重要，因为这能避免你把时间花在错误的方向上。

### 1.4.1 串行依赖很强

例如某些递推算法中，第 $i$ 步必须等待第 $i-1$ 步完成。虽然有些问题可以通过 scan、分块、重写数据结构来并行化，但不是所有串行算法都能自然迁移到 GPU。

### 1.4.2 工作量太小

如果只有几百个元素、几千次简单运算，GPU 的启动与传输开销可能比计算本身还大。

### 1.4.3 控制流极不规则

如果每个线程都走不同分支，warp 内会出现明显分歧，实际执行效率可能大幅下降。

### 1.4.4 需要频繁与 CPU 往返交互

如果程序流程是：

1. CPU 做一点事
2. 把数据发给 GPU
3. GPU 做一点事
4. 立刻把结果传回 CPU
5. 再重复

那么频繁的数据往返很可能使整体性能很差。

## 1.5 从向量加法理解并行思维

CUDA 教程几乎都会从向量加法开始，不是因为它重要，而是因为它足够简单，可以把“一个线程处理一个元素”的基本模型讲清楚。

设有两个长度为 $n$ 的向量：

$$\mathbf{a} = [a_0, a_1, \dots, a_{n-1}]$$
$$\mathbf{b} = [b_0, b_1, \dots, b_{n-1}]$$

我们要计算：

$$c_i = a_i + b_i, \quad i = 0, 1, \dots, n-1$$

### CPU 的直觉写法

```cpp
for (int i = 0; i < n; ++i) {
    c[i] = a[i] + b[i];
}
```

### GPU 的直觉写法

你可以把它理解为：

- 启动很多线程
- 每个线程负责一个 `i`
- 每个线程独立完成 `c[i] = a[i] + b[i]`

```cpp
int idx = blockIdx.x * blockDim.x + threadIdx.x;
if (idx < n) {
    c[idx] = a[idx] + b[idx];
}
```

这一行索引公式在 CUDA 中极其重要。它把“线程层级结构”映射成“数据元素位置”，也是后续几乎所有 kernel 的基础模板。

## 1.6 CUDA 软件栈全景图

CUDA 不只是一个编译器，也不只是几个 API，而是一整套围绕 NVIDIA GPU 计算建立的软件栈。

```text
应用程序
├── 你的 CUDA C++ 代码
├── PyTorch / TensorFlow / JAX 等框架
└── 高性能库（cuBLAS / cuDNN / NCCL / CUB）

CUDA 运行时与驱动
├── CUDA Runtime API
└── CUDA Driver API

编译与工具链
├── nvcc
├── Nsight Systems
├── Nsight Compute
└── Compute Sanitizer

硬件层
└── NVIDIA GPU（SM、warp、寄存器、共享内存、全局内存等）
```

### 初学阶段你最需要先理解什么？

在刚入门时，优先理解以下四件事：

1. 线程、block、grid 如何组织
2. host 内存和 device 内存如何分配与传输
3. 一个 kernel 是如何被 launch 的
4. 为什么访存模式和同步会影响性能与正确性

只要这四件事真正建立起来，后面的 shared memory、streams、Tensor Core、CUDA Graphs 才不会显得神秘。

## 1.7 深度学习为什么离不开 GPU？

如果你来自 AI 背景，GPU 的重要性可能比图像处理或数值仿真更容易直观感受到。

原因很简单：

- 深度学习的核心运算大量依赖矩阵乘法与卷积
- 这些运算具有很强的规则性
- 数据规模大、计算密度高
- 可以被批处理、分块、流水线化

从高层框架看，你可能只是写了：

```python
y = x @ w
```

但在底层，真正高效的实现依赖：

- 高度优化的 GEMM kernel
- 共享内存分块
- 连续访存
- Tensor Core
- 混合精度
- 多 stream / 多 GPU 协作

这也是为什么理解 CUDA，会帮助你更深入地理解 PyTorch、Transformer 和现代大模型工程。

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| CPU vs GPU | CPU 更偏低延迟与复杂控制流；GPU 更偏高吞吐与规则并行 |
| 适合 GPU 的问题 | 数据并行强、计算密度高、访存较规则 |
| 不适合 GPU 的问题 | 串行依赖强、工作量小、控制流极不规则、频繁往返传输 |
| CUDA 的第一层直觉 | 让大量线程把相似工作映射到大量数据元素上 |
| 初学重点 | 线程层级、内存模型、kernel launch、同步与访存 |

## CUDA实验

### 实验 1：分析哪些任务适合 GPU

请判断下面这些任务更偏向 CPU 还是 GPU，并说明原因：

1. 对长度为一亿的浮点数组做逐元素加法
2. 处理一个包含大量 `if/else` 分支的解释器主循环
3. 对 4096×4096 矩阵做乘法
4. 每处理一个元素都要查询磁盘文件
5. 对一批图像做相同卷积滤波

建议你从三个角度分析：

- 是否容易数据并行
- 是否有足够计算量
- 是否存在大量不规则分支或 IO

### 实验 2：把串行循环改写成“线程视角”

观察下面的 CPU 代码：

```cpp
for (int i = 0; i < n; ++i) {
    out[i] = in[i] * 2.0f + 1.0f;
}
```

思考如果把它放到 GPU 上：

- 每个线程负责哪个元素？
- 线程的全局索引如何计算？
- 为什么必须做边界判断？
