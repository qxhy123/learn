# 附录D：练习答案汇总

本附录提供的是各章练习题的答案提示与思路要点，不追求写成唯一标准答案。更推荐的使用方式是：

1. 先独立回答每章练习题
2. 再对照本附录检查是否抓住了核心边界
3. 如果答案里只剩“会配某段 YAML”或“会抄某段代码”，通常说明对工程设计的理解还不够扎实

---

## 第一部分：OpenTelemetry 基础认知

### 第1章：为什么需要 OpenTelemetry

答题要点：

- OpenTelemetry 首先解决的是统一观测模型、统一接入方式和统一语义问题，而不是单纯“多装一个 SDK”。
- 它不是某个 tracing 产品，因为它同时覆盖 traces、metrics、logs、context propagation、semantic conventions 和 Collector。
- traces 负责回答“经过了哪里、慢在哪”，metrics 负责回答“整体是否退化”，logs 负责回答“这次到底发生了什么”。
- “有监控但不可观测”通常指只有零散面板或日志，却无法把问题缩小到具体服务、阶段、依赖或请求类型。
- 厂商中立的现实收益包括：降低应用耦合、方便多后端并存、便于平台统一治理，也降低后续迁移成本。

### 第2章：三种信号与 OTel 架构

答题要点：

- 三种信号是互补关系，不是替代关系。trace 看链路，metric 看趋势，log 看证据。
- OTel 架构里，应用负责产生语义，SDK 负责采集实现，Collector 负责管道治理，backend 负责存储、查询与展示。
- OTLP 的价值在于提供统一传输协议，让应用与后端之间不必直接强耦合。
- 一个成熟系统的关键不是“有没有三种信号”，而是三种信号能否围绕同一套身份与上下文协同工作。
- 回答架构题时，最好明确每一层“该负责什么”和“不该负责什么”。

### 第3章：第一个端到端链路

答题要点：

- 最小闭环通常是：应用 -> OTLP -> Collector -> debug exporter 或 backend。
- 初学时应先验证数据是否真的从应用发出、进入 Collector，再看 backend 展示，不要一上来就怀疑查询界面。
- Resource 负责服务身份，span attributes 负责本次操作上下文，两者不要混用。
- 调试最小闭环时，`debug` exporter 往往是最可靠的第一检查点。
- 真正的“跑通”不是后端里有一条 trace，而是服务名、span 层级、属性和错误状态都可解释。

## 第二部分：Trace 与上下文传播

### 第4章：Trace、Span 与 Context

答题要点：

- Trace 是一次完整请求或事务的链路；span 是其中的单个阶段；context 是当前执行流里携带 trace/span 信息的运行时上下文。
- span 的价值在于表达边界、阶段和错误，而不是把所有函数都机械变成 span。
- span 常见元素包括：名称、开始/结束时间、attributes、events、status、parent-child 关系。
- Context 的主要作用是让当前请求在同步与异步代码中继续携带“我属于哪条 trace”。
- 回答时要区分“数据模型概念”和“代码运行机制”，不要把 trace_id 当成 context 的全部含义。

### 第5章：上下文传播与 Baggage

答题要点：

- 传播的目标是让跨进程、跨服务、跨协议的调用能延续同一条 trace，而不是每到下游就重新开始。
- W3C Trace Context 的核心是标准化 trace 上下文头部，减少不同语言、框架和中间件之间的不兼容。
- Baggage 适合传播少量、轻量、有限的上下文分类信息，不适合传大对象、敏感信息或高变化字段。
- 链路断裂时，先检查 header 是否传递、propagator 是否一致、网关/消息系统是否丢头，再看业务代码。
- 传播成功不等于建模正确，异步系统还要考虑是否该用 link 而不是强行 parent-child。

### 第6章：Span 设计与链路建模

答题要点：

- 好的 span 设计围绕责任边界、业务阶段和依赖调用，而不是围绕函数名或类名。
- 关键阶段更适合建成 span，例如鉴权、算价、库存预留、支付授权、消息投递。
- parent-child 更适合同步调用；消息、批处理和多输入聚合场景常更适合 link。
- 错误应区分业务拒绝与技术失败，并通过 status、event、exception 和稳定字段表达出来。
- 常见误区是 span 太碎、命名不稳定、把高基数主键直接当指标维度使用。

## 第三部分：Metrics、Logs 与语义规范

### 第7章：Metrics 模型与常见 Instrument

答题要点：

- Counter 适合累计次数，Gauge 适合当前值，Histogram 适合延迟和大小分布。
- 选择 instrument 时要先问“这个量是累计、瞬时还是分布”，而不是看哪个 API 最顺手。
- 指标的核心用途是长期趋势、聚合和告警，因此维度要稳定、有限、可解释。
- 高基数字段通常不适合作为高频指标 attributes，例如 `user.id`、`order.id`、`session.id`。
- 回答题目时要强调“指标是为了回答什么问题”，而不是只给出类型定义。

### 第8章：Logs 与跨信号关联

答题要点：

- 结构化日志的核心价值是让日志字段可检索、可聚合、可关联，而不只是打印文本。
- trace 与 log 关联最基本的要求是日志中稳定保留 `trace_id`、`span_id` 以及服务身份字段。
- logs 更适合记录详细证据和错误细节，不适合承担长期趋势分析职责。
- span event 和日志不是一回事：event 更像某个 span 生命周期内的关键瞬时注记，log 是独立记录。
- 答题时要注意区分“日志很多”和“日志可关联、可用于排障”这两件事。

### 第9章：Resources、Attributes 与 Semantic Conventions

答题要点：

- Resource 更偏身份层，回答“数据来自谁”；attributes 更偏本次 span、metric、log 的上下文信息。
- `service.name`、`service.version`、`deployment.environment.name` 通常属于 Resource，而不是每个 span 都重复写的普通属性。
- Semantic Conventions 的价值在于统一命名和语义，降低多团队、多语言和多后端之间的理解成本。
- 字段设计时要区分稳定类别字段和高基数实例字段，后者通常不适合长期聚合。
- 好的 schema 设计本质上是一种接口设计，直接影响查询、告警、迁移和治理能力。

## 第四部分：Instrumentation 与 SDK 使用

### 第10章：API、SDK 与 Library Guidelines

答题要点：

- API 负责调用接口和抽象入口，SDK 负责真实采集、处理和导出实现。
- 通用业务库通常只依赖 API，因为 exporter、采样、OTLP 地址和 provider 初始化都应由应用决定。
- 如果库内部直接初始化 SDK，容易破坏复用性、厂商中立和应用层统一治理。
- 自动 instrumentation 库负责基础框架和客户端 patch，业务库负责表达领域语义，两者职责不同。
- “只依赖 API”不等于不做设计，库仍然要对 span 名称、属性和错误语义的稳定性负责。

### 第11章：自动注入、手动埋点与混合策略

答题要点：

- zero-code 更强调少改业务代码；自动 instrumentation 是实现 zero-code 的重要手段；手动埋点负责补齐业务语义。
- 自动 instrumentation 擅长覆盖 HTTP、数据库、消息客户端等通用路径，但不理解业务阶段。
- 业务指标、关键业务 span、领域错误分类通常必须手动补充。
- 最推荐的方案通常是混合策略：自动覆盖基础组件，手动补关键业务路径。
- 常见误区包括重复埋点、所有函数都加 span、把健康检查等噪音请求也完整纳入高频观测。

### 第12章：用 Node.js / TypeScript 接入 OTel

答题要点：

- Node.js 中启动顺序很重要，通常要先初始化 telemetry，再导入框架和驱动。
- Resource 负责服务身份，span attributes 负责当前业务上下文，不应混在一起使用。
- `order_id` 之类字段通常不适合作为 metrics 标签，但在某些 trace 或日志里可用于单次定位。
- 排查“本地有数据、容器没数据”时，应先查 OTLP 地址、协议、容器网络、Collector 监听与优雅退出。
- 自动 instrumentation 给基础链路，业务 span 给语义，结构化日志给证据，三者是分工协作关系。

## 第五部分：Collector 与 Telemetry Pipeline

### 第13章：OpenTelemetry Collector 基础

答题要点：

- receiver 负责输入，processor 负责处理中间环节，exporter 负责输出，pipeline 负责把它们串起来。
- Collector 的价值不只是转发，而是统一做批处理、过滤、变换、采样、路由和出口治理。
- agent 更靠近应用，偏接入层；gateway 更偏集中治理层。
- 很多生产环境会采用 agent + gateway 双层模式，以兼顾近端接入和集中治理。
- 学习 Collector 时应先做最小闭环，再逐步增加 processor，不要一开始就堆复杂配置。

### 第14章：Processor、Sampling 与 Routing

答题要点：

- 常见 processor 设计思路是先保护自己，再做清理和标准化，最后再批量导出。
- `batch`、`memory_limiter` 常是基础组件；`filter`、`transform`、`tail_sampling` 更偏治理策略。
- head sampling 更轻量，tail sampling 更智能，但更耗状态和资源；采样主要影响 traces，不会自动修复 metrics 高基数问题。
- routing 适合按环境、租户、信号类型或数据特征做分流，但前提是路由规则清晰且可验证。
- 应用应专注生成正确语义，复杂管道治理更适合放到 Collector，而不是复制到每个服务里。

### 第15章：在生产环境部署 Collector

答题要点：

- 侧车、DaemonSet、节点 agent、集中 gateway 各有边界，应按网络路径、运维复杂度和治理需求选择。
- 生产部署要优先考虑高可用、资源限制、版本升级、配置变更和自监控，而不是只看功能能否跑通。
- 接入层 Collector 更适合近端缓冲和统一入口，治理层 Collector 更适合集中策略和多后端出口。
- Collector 自己也是关键基础设施，必须观测其 CPU、内存、队列积压、导出失败和丢弃量。
- 回答部署题时，要把拓扑选择、可靠性、成本和团队运维能力一起讨论。

## 第六部分：生产环境运维与治理

### 第16章：Cardinality、成本与性能

答题要点：

- trace 采样率下降通常只能降低 traces 成本，不能自动解决 metrics 高基数问题，因为后者取决于时间序列数量。
- 更适合高频 histogram 维度的通常是 `service.name`、`http.route`、`deployment.environment.name`、`user.tier` 这类稳定有限字段。
- `user.id`、`session.id`、`order.id` 一般不适合作为高频指标维度，因为会制造大量唯一 series。
- 成本控制应分层做：应用层控制 schema，SDK 层控制批量和上限，Collector 层做过滤和保护，backend 层做保留与索引策略。
- `order.id` 在 trace 或日志中可能对单次排障有价值，但通常不适合作为 metrics 标签，因为聚合价值低、基数高、成本大。

### 第17章：安全、隐私与治理

答题要点：

- “先不采集，再脱敏”比“先采集，再统一清洗”更可靠，因为能从源头减少扩散和副本风险。
- 典型不应直接进入 telemetry 的字段包括：认证头、密码或 token、完整身份信息、支付敏感信息、原始请求体中的高风险内容。
- 多租户平台应通过统一资源属性、路由隔离、访问控制和环境隔离避免数据串租户。
- Collector 是重要治理点，但它不能完全替代源头修复，尤其当敏感数据已经先写入本地日志或中间副本时。
- 可观测系统本身需要审计和变更记录，因为采集策略、字段新增和访问行为都可能带来安全与合规影响。

### 第18章：调试断裂链路与缺失信号

答题要点：

- 上下游各自有 span 但不在同一条 trace 中时，应先查 propagation、header 传递、propagator 配置和异步边界。
- “指标不存在”和“指标查不到”是两类问题：前者通常是代码没记录或 Collector 丢了，后者常是 schema 变更、查询条件过旧或可见延迟问题。
- 日志很多但无法回到具体 trace，最可能缺少 `trace_id`、`span_id` 或日志采集链路中这些字段被改名/丢弃。
- `debug` exporter 的价值在于把问题切分为“应用到 Collector”还是“Collector 到 backend”两个层次。
- 排障时还要考虑时钟偏移、采样和批量导出延迟，它们会制造“像丢数据”的迷惑性症状。

## 第七部分：生态、异步系统与平台整合

### 第19章：Exporter、Backend 与 OTLP 策略

答题要点：

- OTLP 的价值在于提供统一、厂商中立的传输入口，使应用更容易与具体 backend 解耦。
- 应用直连 backend 的优点是路径简单，但经 Collector 中转更适合统一治理、重试、脱敏、采样和多后端导出。
- exporter 解决“怎么发”，backend 解决“怎么存、怎么查、怎么展示”，两者职责不同。
- 多 backend 并存是现实常态，关键是应用层不要被多种私有 exporter 和字段绑死。
- 做 backend 策略题时，应从查询能力、信号类型、成本、保留、团队技能和迁移性综合考虑，而不是只比较产品名。

### 第20章：消息队列、批处理与异步系统

答题要点：

- 异步系统不意味着无法追踪，而是必须区别同步 parent-child 与异步 link / producer-consumer 模型。
- 消息元数据中的上下文传播非常关键，否则生产和消费会自然断链。
- 批处理任务往往对应一批输入和一个批次结果，因此更适合用 batch span、batch size 指标、links 等方式表达。
- 除 trace 外，还应设计消息积压、消费延迟、失败重试、死信数量等指标。
- 定时任务、后台 worker、补偿任务通常不应伪装成同步请求的子 span，应按实际触发关系建模。

### 第21章：Kubernetes 与平台级可观测性

答题要点：

- Kubernetes 场景下，平台侧最重要的是统一注入或探测 `k8s.cluster.name`、`k8s.namespace.name`、`k8s.pod.name` 等资源字段。
- DaemonSet、sidecar、gateway、Operator 等部署方式各有边界，应结合平台治理与应用团队能力选择。
- 平台可观测性不仅是应用 traces，还包括节点、容器、调度、网络和资源配额等信号。
- `k8s.pod.name`、`container.id` 这类字段对排障有价值，但高变化，不适合滥做长期高频聚合维度。
- 平台题的核心是“统一入口、统一身份、统一治理”，而不是把每个业务团队都变成平台运维专家。

## 第八部分：高级设计与完整项目

### 第22章：为微服务系统设计可观测性

答题要点：

- 微服务可观测性设计应先从业务路径、故障模式和治理需求出发，而不是先把自动 instrumentation 打开。
- `order-api` 与 `order-worker` 是否应分不同 `service.name`，关键看部署边界、运行职责、故障定位与消费视图是否明显不同；大多数情况下应分开。
- “下单 -> 发事件 -> 异步扣减库存”这类流程，通常同步部分用 parent-child，异步消费部分更常结合 producer/consumer 语义或 links。
- `user.id`、`order.id` 常不适合作为高频指标维度，因为会导致 series 膨胀，但在 trace 或日志中可用于单次问题定位。
- 评估 `payment.channel.raw_response` 这类日志字段时，要同时看敏感性、体积、稳定性、诊断价值、索引需求和是否可用更抽象字段替代。

### 第23章：从旧监控/日志体系迁移到 OpenTelemetry

答题要点：

- 最大难点往往不是 SDK 替换，而是字段兼容、旧告警与看板迁移、组织协作和治理收敛。
- “不可动资产”通常包括核心告警、关键仪表盘、审计日志、SLA 报表等，一旦退化会直接影响生产运行。
- 双写适合高风险强验证场景，桥接适合降低改造成本，灰度适合逐步启用；三者的共同代价是复杂度上升，需要明确退出条件。
- 迁移验收不能只看“后端里有数据”，还要看数据正确性、排障闭环、核心消费面和治理质量是否达标。
- 如果只是把数据都送到 OTel Collector，但命名、日志关联、告警口径和 runbook 仍然混乱，只能说明“接入开始了”，不能说明“迁移完成了”。

### 第24章：搭建一个观测栈

答题要点：

- 观测栈不是某个具体后端，而是从信号生产、处理、存储、查询到告警和排障动作的完整链路。
- 应用层负责生成正确语义，接入层 Collector 负责统一入口和近端缓冲，治理层 Collector 负责策略和出口，backend 负责查询、存储和消费体验。
- 接入层与治理层分离的好处在于：应用配置更简单，局部抖动更容易吸收，组织级治理更集中，后端替换更容易。
- 一个成熟的排障闭环通常是：指标告警 -> 锁定服务/路由/依赖 -> 打开相关 trace -> 跳转日志 -> 结合版本与变更信息定位问题。
- 观测栈本身也要被观测，否则最容易忽略 Collector 丢数、exporter 失败、backend 查询退化和告警链路本身异常等盲区。

## 使用本附录时的三个提醒

### 1. 不要把“知道名词”当成“会设计系统”

如果你的答案只能复述定义，而不能说明：

- 为什么字段放在这里
- 为什么这层负责这件事
- 为什么这类数据不该进 metrics
- 为什么迁移和治理需要分阶段

那么说明还没有真正掌握工程边界。

### 2. 不要追求唯一标准答案

很多题目本身就在考察权衡，例如：

- 何时直连 backend，何时经 Collector
- 何时 parent-child，何时 link
- 何时用 sidecar，何时用 DaemonSet
- 何时双写，何时桥接

真正重要的是：你的答案是否能说清边界、代价与适用条件。

如果你在做题时常卡住，可以优先回看这些章节：

| 你卡住的问题 | 建议回看 |
|--------------|----------|
| traces、metrics、logs 分工不清 | 第2、4、7、8章 |
| Resource、Attributes、语义字段容易混淆 | 第9章与附录A、附录C |
| API、SDK、Collector 边界不清 | 第10、13章 |
| 自动埋点、手动埋点如何配合 | 第11、12章 |
| 成本、采样、字段治理拿捏不准 | 第14、16、17章 |
| 迁移、平台治理、完整观测栈难以串起来 | 第22、23、24章 |

如果你能把本附录中的每章提示都说成自己的语言，并且能联系到自己系统里的真实服务、字段和故障场景，那么这套教程的主要目标就基本达成了。
