# 第8章：Logs 与跨信号关联

## 学习目标

- 理解结构化日志在 OpenTelemetry 体系中的位置与边界
- 说清 logs 为什么不能简单替代 traces 或 metrics
- 掌握 trace/log correlation 的核心价值与基本做法
- 理解跨信号关联依赖哪些稳定字段与上下文机制
- 建立“日志要有解释力，但也要克制采集与控制风险”的工程意识

## 1. 为什么系统已经有日志，仍然还需要跨信号关联

很多团队对日志并不陌生，甚至在接入 OTel 之前就已经有很成熟的日志体系。但真正到了排障现场，大家仍然经常遇到几个痛点：

- trace 里看到某个 span 失败了，却不知道当时具体报了什么错误
- metrics 告警提示错误率升高，却不知道哪类请求受影响
- 日志很多，但找不到和某次请求对应的上下文

这说明问题不在于“有没有日志”，而在于日志是否和其他信号真正形成了可关联的系统。

OpenTelemetry 的一个关键价值，就是让 logs 不再只是孤立文本，而是更容易与 trace、metric 一起使用。

### 1.1 logs 擅长什么

日志最擅长的是保留事件细节，例如：

- 具体错误信息
- 某个业务分支为什么被触发
- 请求参数的大致类别
- 某次回退、重试、降级的决策原因

这些内容往往很难完全靠 metrics 表达，也不一定适合全塞进 trace span。

### 1.2 logs 不擅长什么

日志并不擅长：

- 直接做长期趋势聚合
- 表达完整调用结构
- 自动说明父子关系与执行边界

所以日志虽然重要，但它更像“高细节补充层”，而不是单独就能构成完整可观测体系的中心。

## 2. 什么样的日志才算结构化日志

很多团队说“我们有日志”，其实说的是文本日志。而真正适合进入现代可观测体系的，通常是结构化日志。

### 2.1 结构化日志的核心

结构化日志的重点不是“长得像 JSON”，而是：

- 字段有稳定命名
- 同类事件的 schema 尽量一致
- 关键上下文可机器解析和查询

例如下面这条日志就比一行纯文本更容易被系统使用：

```ts
console.log(JSON.stringify({
  level: 'error',
  message: 'payment gateway timeout',
  service_name: 'checkout-service',
  operation: 'payment.charge',
  order_channel: 'web',
}))
```

它的价值在于：

- 能按 `level`、`service_name`、`operation` 做检索
- 可以和 trace span 的业务阶段名称形成映射
- 可以逐步建立稳定 schema，而不是每条日志都自由发挥

### 2.2 结构化不等于字段越多越好

结构化日志常见的误区是：

- 看到请求对象就整包序列化
- 把所有 header 全量打出来
- 把所有业务主键和原始正文都写进去

这样做短期看似“信息很全”，长期通常会引发：

- 成本升高
- 敏感信息风险
- 噪音变多
- 难以统一 schema

好的结构化日志不是最大化采集，而是最小必要且稳定地表达上下文。

## 3. Trace / Log Correlation：为什么日志要带 trace_id

跨信号关联里，最有代表性的就是 trace/log correlation。它解决的问题非常直接：

> 当你看到一条日志时，能不能快速知道它属于哪次请求、哪条 trace、哪个 span？

### 3.1 为什么只靠 message 不够

假设日志里只有：

```text
payment gateway timeout
```

你很难知道：

- 这是哪个服务打的
- 属于哪个用户请求
- 对应哪个业务步骤
- 是否发生在某次已知慢请求中

如果它带上了 trace 上下文，情况会完全不同。

### 3.2 关联日志的关键字段

最常见的关联字段包括：

- `trace_id`
- `span_id`
- `service.name` 或等价服务身份字段
- 业务阶段字段，例如 `operation` 或 span 对应动作名

一个简化的 Node.js 示例：

```ts
import { context, trace } from '@opentelemetry/api'

export function logError(message: string, extra: Record<string, unknown> = {}) {
  const span = trace.getSpan(context.active())
  const spanContext = span?.spanContext()

  console.log(JSON.stringify({
    level: 'error',
    message,
    trace_id: spanContext?.traceId,
    span_id: spanContext?.spanId,
    ...extra,
  }))
}
```

这段代码表达的关键直觉是：**日志不应只记录业务文本，也应该尽量记录当前执行上下文。**

### 3.3 有了 trace_id 之后能做什么

一旦日志与 trace 建立了稳定关联，你就可以：

- 从 trace 跳到相关日志
- 从日志反查整条请求链路
- 把 metrics 告警定位到某一类 trace，再看对应日志细节

这就是“跨信号关联”的真正工程价值：减少人工拼接和猜测。

## 4. Logs、Traces、Metrics 三者如何分工协作

可以用一个支付失败场景来理解三者协同：

### 4.1 metrics 告诉你是否整体异常

例如：

- `payment_failures_total` 明显升高
- `checkout_submit_duration_ms` 的 p95 变高

这会告诉你问题不是个别请求，而是某种趋势正在发生。

### 4.2 traces 告诉你失败发生在哪个阶段

例如在 trace 中你能看到：

- `checkout.submit_order` 很慢
- 其中 `payment.charge` 子 span 报错

这会告诉你问题落在支付阶段，而不是库存或风控。

### 4.3 logs 告诉你具体错误细节是什么

例如日志进一步说明：

- 下游支付网关返回超时
- 本地重试策略触发了两次
- 某类卡种请求更容易失败

这一步常常是最后把问题解释清楚的关键。

### 4.4 为什么三者缺一不可

- 只有 metrics，你知道“有问题”，但不知问题如何发生
- 只有 traces，你能看结构，但细节可能不够
- 只有 logs，你有细节，但难以快速回到整体路径和趋势

因此，真正可用的跨信号关联不是“把三类数据都存起来”，而是让它们围绕同一套上下文和语义协作。

## 5. 日志设计中的工程边界

### 5.1 日志不是原始数据回收站

很多日志问题不是太少，而是太多、太乱、太不稳定。日志不应该承担：

- 完整请求体永久归档
- 所有内部对象的无差别序列化
- 任意调试信息的长期保留

这些做法会迅速让日志系统变成高成本、低信噪比的黑箱。

### 5.2 结构化字段命名要稳定

例如关联字段最好统一为：

- `trace_id`
- `span_id`
- `service.name` 或你们约定的稳定字段

如果同一组织里：

- 有的服务写 `traceId`
- 有的写 `trace_id`
- 有的放在 `context.trace.id`

那跨服务关联与统一查询会很脆弱。

### 5.3 敏感信息与高基数字段要克制

日志比 metric 更宽容一些，因为它本来就更适合细节记录。但这不代表可以无节制记录：

- 身份证号、Token、完整手机号
- 全量 SQL
- 用户原始输入正文
- 高频主键字段的全量扩散

日志系统同样是治理对象，而不是随手丢东西的地方。

## 6. 常见误区：把日志当成唯一真相来源

### 6.1 误区一：只要日志够多，排障就一定够快

实际情况往往相反。没有关联字段、没有结构、没有统一 schema 的海量日志，只会让排障更慢。

### 6.2 误区二：trace 里已经有错误了，就不需要日志

trace 很擅长告诉你哪一步失败了，但不一定适合容纳全部业务错误上下文和详细文本。日志仍然是关键补充层。

### 6.3 误区三：把所有业务上下文都塞进日志

日志确实更适合细节，但它也有成本、隐私和查询噪音问题。应该围绕可解释性和最小必要原则设计。

### 6.4 误区四：只有在出错时才需要 trace/log 关联

实际上，即使是正常请求，关联能力也很有价值，因为很多性能问题、重试问题和分支决策问题都需要同时查看链路与日志。

## 本章小结

| 主题 | 核心结论 |
|------|----------|
| logs 的定位 | 保留细节事件和业务上下文，是 trace 与 metric 的补充层 |
| 结构化日志 | 重点是稳定字段与可机器处理，不只是 JSON 形式 |
| trace/log correlation | 通过 trace_id、span_id 等字段把日志与链路关联起来 |
| 三种信号协同 | metrics 找趋势，traces 看结构，logs 补细节 |
| 设计边界 | 日志要有解释力，但不能变成原始数据回收站 |
| 工程重点 | 稳定字段名、上下文注入、控制敏感与高基数内容 |

## OTel实验

### 实验目标

让一个本地 Node.js 服务同时输出 trace 和结构化日志，并验证日志如何通过 trace_id 与 span_id 回到同一条请求链路。

### 实验步骤

1. 准备一个带手动 span 的 `POST /checkout` 接口。
2. 在业务逻辑中输出两类结构化日志：
   - 正常提交日志
   - 支付失败日志
3. 在日志输出函数中读取当前活动 span，并写入 `trace_id` 与 `span_id`。
4. 发送几次正常请求和一次故意失败请求。
5. 观察日志、trace 与 metrics 的协同方式：
   - metrics 是否能反映错误率变化
   - trace 是否能反映失败阶段
   - 日志是否能提供具体错误文本
6. 再移除日志里的 `trace_id`，比较排障体验变化。

### 预期现象

- 带 `trace_id` 的结构化日志能显著降低人工拼接成本。
- trace 能告诉你失败在什么阶段，日志能告诉你失败细节是什么。
- 去掉关联字段后，日志仍然存在，但它和 trace 的协作价值会明显下降。
