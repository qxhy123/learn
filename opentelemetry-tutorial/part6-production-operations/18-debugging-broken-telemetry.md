# 第18章：调试断裂链路与缺失信号

## 学习目标

- 识别 traces、metrics、logs 在生产环境中最常见的断裂与缺失模式
- 建立从应用、传播、Collector 到 backend 的分层排障思路
- 学会判断问题是“没有产生数据”“数据没送到”“数据送到了但不可查询”，还是“数据不能关联”
- 理解时钟偏移、采样、异步边界和 schema 不一致对排障的影响
- 形成一套可复用的 telemetry 故障诊断清单

## 1. 先把问题分类，而不是一上来改配置

当团队说“OpenTelemetry 出问题了”时，背后可能是完全不同的故障类型。为了避免盲目修改，应先把问题归类。

常见故障至少有五类：

1. **trace 断裂**：上下游服务各自有 span，但不在同一条 trace 中
2. **metrics 缺失**：预期的指标不存在，或某些 label 组合没有数据
3. **logs 无法关联**：日志里没有 trace_id/span_id，无法回到请求上下文
4. **数据延迟或间歇性消失**：有时有，有时没有，通常与队列、采样、重试有关
5. **查询看起来像缺失，实则是 schema 变了**：字段名、资源标签、环境标签不一致导致面板查不到

因此，一个好的排障入口不是“先去看某个产品界面”，而是先问四个问题：

- 数据有没有在应用里被创建出来？
- 数据有没有沿着网络和 Collector 管道传下去？
- 数据有没有被采样、过滤、丢弃或改名？
- 数据是否以你查询时期待的结构落到了后端？

## 2. 调试 trace 断裂：先看传播，再看建模

### 2.1 最常见的断裂原因

trace 断裂常见于这些边界：

- HTTP / gRPC 请求头没有正确传递 trace context
- 异步任务、消息队列、后台 job 没有显式传播上下文
- 框架中间件顺序错误，导致 span 在错误作用域中创建
- 代理、网关、消息中间件把传播头丢掉或重写
- 某个服务使用了不同传播格式，或没有启用相同 propagator

如果上游和下游都有 span，但 trace id 不同，优先检查传播。如果下游压根没有服务端 span，再去检查自动注入、路由匹配和 SDK 初始化。

### 2.2 如何系统检查一条链路

一个实用方法是沿请求路径逐跳确认：

1. 入口服务是否创建了根 span
2. 出站客户端调用是否注入了 trace context
3. 网络边界是否保留这些头
4. 下游服务是否提取上下文并建立子 span
5. 异步任务是否把上下文转移到新的执行单元

如果你能拿到日志，可以重点关注：

- `trace_id`
- `span_id`
- `parent_span_id`
- 请求头中的 `traceparent`
- 消息元数据中的追踪字段

### 2.3 parent-child 与 link 别混用

在异步系统里，很多人误以为所有后续处理都必须挂在原请求 span 下面。其实并不是。

- 如果是严格的同步调用链，通常适合 parent-child
- 如果是一个消息被多个消费者异步处理，或批处理任务追溯到多个来源，通常更适合 span links

如果建模错了，你可能会看到“链没断，但结构非常奇怪”，例如一个长达几小时的父 span 挂着大量离散任务。这虽然不一定是传播失败，但同样会让 trace 失去解释性。

## 3. 调试 metrics 缺失：先确认 instrument，再确认聚合与查询

指标缺失并不总是 exporter 坏了。常见原因有：

- 应用代码根本没有执行到记录逻辑
- 选择了不合适的 instrument 类型
- 观察周期内没有产生数据点
- 属性名或资源标签变更导致查询条件不再匹配
- Collector transform/filter 把指标删掉了
- backend 只保留了聚合后的视图，而你在查不存在的原始维度

### 3.1 先问“指标为什么应该出现”

例如你预期有一个请求延迟 histogram，就先确认：

- 请求路径是否真实被访问
- 记录逻辑是否在错误分支、成功分支都能执行
- 路由模板是否稳定
- 指标导出间隔是否导致你观察窗口太短

一个常见错误是把 `ObservableGauge` 用在“每个请求都记录一次”的场景，或者把 Counter 当作可以随意回退的值。结果不是完全没有数据，就是数据语义和预期不符。

### 3.2 缺失有时来自 schema 漂移

例如原来 dashboard 用的是：

- `service.name = checkout`
- `deployment.environment.name = production`

后来某个服务改成：

- `service.name = checkout-service`
- `env = production`

这时指标并不是没上报，而是你原有查询条件已经对不上了。这个问题在多团队长期演进中非常常见，因此 schema 治理是排障能力的一部分，而不是文档附属品。

## 4. 调试 logs 无法关联：日志本身正确，不代表可关联

很多团队说“日志没问题”，实际上只是日志内容存在，但缺少跨信号关联字段。

### 4.1 要确认的不是有没有日志，而是有没有上下文

如果你希望通过日志回到 trace，至少要确认：

- 产生日志时当前上下文里存在活动 span
- 日志框架有把 `trace_id`、`span_id` 注入结构化字段
- 异步执行时上下文没有丢失
- 日志采集链路没有把这些字段丢掉或重命名

示例：

```ts
logger.info({
  trace_id: currentTraceId,
  span_id: currentSpanId,
  orderId,
}, 'payment authorized')
```

如果日志只包含 `message` 和 `level`，即使日志很多，排障时也仍然要手工拼接上下文。

### 4.2 结构化日志字段名要稳定

如果有的服务叫 `trace_id`，有的叫 `traceId`，有的放在 `context.trace_id`，后端即使能收进来，查询与关联也会变得脆弱。统一命名是日志可关联的基础条件之一。

## 5. Collector 与后端：数据可能在中间层丢失、延迟或变形

### 5.1 Collector 侧常见问题

Collector 常见故障包括：

- receiver 没有监听正确协议或端口
- processor 顺序错误，导致先过滤后变换或先采样后关联
- memory limiter 触发后开始丢弃数据
- exporter 重试积压，队列满后丢弃
- 多 pipeline 路由不正确，导致某类信号没有被导出

一个最小排障配置通常会临时加入 `debug` exporter，用来确认数据是否进入 Collector：

```yaml
receivers:
  otlp:
    protocols:
      grpc:
      http:

processors:
  batch:

exporters:
  debug:
    verbosity: detailed

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [batch]
      exporters: [debug]
```

如果 debug exporter 能看到数据，而目标 backend 看不到，问题更可能出在 exporter、认证、路由或后端查询层。

### 5.2 后端侧常见误判

有时数据已经到了，但用户仍然会报告“没数据”，原因包括：

- 查询时间范围过窄
- 查询条件使用了旧字段名
- 采样后只剩部分 trace
- backend 对 logs、traces、metrics 的可见延迟不同
- 某些高基数字段默认不建索引，导致查不到想象中的精确过滤结果

所以排障时不要只问“后端有没有”，还要问“后端会如何展示和索引”。

## 6. 时钟偏移、采样和异步边界会制造迷惑性症状

有些问题看起来像丢数据，其实是时序被扭曲了。

### 6.1 时钟偏移

如果不同主机时间不同步，可能出现：

- 子 span 看起来早于父 span 开始
- logs 与 trace 时间线对不上
- 指标窗口与事件发生时间不一致

### 6.2 采样

采样可能导致：

- 错误日志存在，但相关 trace 不存在
- 上游 trace 有记录，下游因为策略不同未保留
- 单条请求的局部 span 存在，但整条 trace 不完整

### 6.3 异步缓冲与批量导出

批处理会引入可见延迟，因此“刚发生的问题立刻看不到”并不一定意味着丢失。你需要区分：

- 正常导出延迟
- 队列积压
- 超时重试
- 真正的数据丢弃

## 本章小结

| 故障类型 | 优先检查点 |
|----------|------------|
| trace 断裂 | 传播头、上下文作用域、异步边界、propagator 配置 |
| metrics 缺失 | instrument 类型、代码路径、schema 变化、Collector 过滤 |
| logs 无法关联 | trace_id/span_id 注入、字段命名、日志采集链路 |
| Collector 丢数 | receiver、processor 顺序、memory limiter、exporter 队列 |
| 后端误判 | 查询时间窗、字段名、采样效果、索引与可见延迟 |

## OTel实验

### 实验目标

人为制造三类典型故障：trace 断裂、指标缺失、日志无法关联，并练习分层定位。

### 实验步骤

1. 准备一个简单服务，包含 HTTP 入口、数据库调用和后台异步任务。
2. 第一轮实验，去掉客户端传播头注入或让下游服务关闭提取逻辑，观察 trace 如何断裂。
3. 第二轮实验，把某个请求指标的 attributes 从 `http.route` 改成原始路径，或直接改掉 `service.name`，观察 dashboard 查询为什么“像是没数据”。
4. 第三轮实验，关闭日志中的 `trace_id` 注入，只保留 message 和 level，观察日志与 trace 关联能力如何下降。
5. 在 Collector 中临时加入 `debug` exporter，分别确认三类数据是否进入 Collector。
6. 用排障记录表描述每个故障属于哪一层、证据是什么、修复点在哪。

### 预期现象

- 传播断裂时，上下游各有 span，但 trace id 不一致
- schema 变化时，数据可能仍然存在，但原有查询与告警失效
- 日志缺少 trace_id 后，排障仍可看日志，但很难快速回到完整请求上下文
