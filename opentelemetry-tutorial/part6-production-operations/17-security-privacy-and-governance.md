# 第17章：安全、隐私与治理

## 学习目标

- 理解 OpenTelemetry 数据为什么既是诊断资产，也是安全与合规风险源
- 区分 resource、span attribute、log field、metric attribute 中常见的敏感信息类型
- 掌握在应用、Collector、存储与访问控制四个层面治理 telemetry 数据的方法
- 学会设计最小必要采集、脱敏、保留与审计策略
- 建立多团队、多环境下的可观测性治理基线

## 1. 为什么观测数据需要治理

很多团队一谈安全，首先想到的是业务数据库、对象存储和访问令牌，却忽略了观测系统本身也会持续收集大量上下文。实际上，telemetry 往往天然包含：

- 请求路径、客户端地址、用户代理
- SQL、HTTP header、消息体片段
- 异常栈、文件路径、容器环境变量
- 业务主键、租户标识、内部服务拓扑

这些信息对排障很有帮助，但也可能构成安全与合规风险。原因在于观测系统具有几个特殊属性：

1. **覆盖面广**：几乎所有服务都会上报数据
2. **保留链长**：数据会经过 SDK、Collector、后端、归档层
3. **消费方多**：开发、SRE、平台、数据、审计团队都可能访问
4. **复制范围大**：同一条数据可能被导出到多个后端

因此，观测治理不是“上线前清理一下字段”，而是一个贯穿采集、传输、处理、存储、访问和删除全链路的问题。

## 2. 哪些数据最容易出问题

### 2.1 明确敏感信息类型

在 OpenTelemetry 场景中，常见高风险数据包括：

- **直接识别信息**：姓名、手机号、邮箱、身份证号、精确地址
- **认证与授权信息**：密码、Token、Session Cookie、API Key、JWT 原文
- **支付与财务信息**：银行卡号、支付单号、账单明细
- **医疗与合规敏感信息**：病历号、诊断信息、受法规约束的属性
- **内部安全信息**：内网拓扑、密钥路径、主机名模式、调试开关

### 2.2 最常见的泄漏入口

最容易被忽视的不是显式埋点，而是“顺手记录”：

- 自动采集时把完整 URL 查询参数写入 span
- 结构化日志把整个请求对象序列化
- 异常处理直接输出原始请求体和响应体
- 记录 SQL 时包含用户输入与敏感字段值
- 把 HTTP header 全量转成 attributes

下面是一个典型的反例：

```ts
span.setAttributes({
  'http.request.header.authorization': req.headers.authorization,
  'enduser.id': user.id,
  'app.request.body': JSON.stringify(req.body),
})
```

这类写法的问题不只是“字段太多”，而是它会把认证信息、PII 和高基数字段一次性送入整条观测链路。

更合理的思路是记录类别、状态和必要上下文，而不是原始敏感内容：

```ts
span.setAttributes({
  'auth.scheme': 'bearer',
  'enduser.role': user.role,
  'app.request.has_body': Boolean(req.body),
  'app.request.payload_type': req.headers['content-type'] ?? 'unknown',
})
```

## 3. 最小必要采集：先决定什么不该进来

安全治理最有效的一步，通常不是后置脱敏，而是前置约束。也就是说，先判断哪些数据根本不应进入 telemetry。

### 3.1 按用途设计字段

可以用一个简单问题做判断：**这个字段进入观测系统，是为了长期聚合、短期排障，还是根本没有必要？**

- 适合长期保留的，通常是稳定、抽象、低风险的分类字段
- 适合短期排障的，可以进入日志或 span，但要限制范围和保留时长
- 不应采集的，通常是原始密钥、完整身份信息、原始正文

### 3.2 建立字段分级

一个实用做法是给组织内部的 telemetry 字段分级：

| 等级 | 示例 | 处理建议 |
|------|------|----------|
| L1 公共运行信息 | `service.name`、`deployment.environment.name` | 可默认采集 |
| L2 内部运维信息 | `k8s.namespace.name`、错误码、路由模板 | 控制访问范围 |
| L3 业务敏感信息 | 租户标识、内部订单类型、部分客户标签 | 需审批和用途说明 |
| L4 高敏感信息 | Token、手机号、身份证、支付信息 | 默认禁止进入 telemetry |

有了这个分级，团队就不再只依赖个人经验，而是可以把“哪些字段允许出现”变成制度化约束。

## 4. 在 Collector 中做统一脱敏与策略执行

虽然最好的做法是在应用侧不采集敏感数据，但真实系统里，总会有老服务、第三方库或错误配置把不该出现的字段带进来。这时 Collector 是组织级治理的重要控制点。

示例：

```yaml
receivers:
  otlp:
    protocols:
      grpc:
      http:

processors:
  memory_limiter:
    check_interval: 1s
    limit_mib: 512
  transform/redact:
    trace_statements:
      - context: span
        statements:
          - delete_key(attributes, "http.request.header.authorization")
          - delete_key(attributes, "app.request.body")
    log_statements:
      - context: log
        statements:
          - delete_key(attributes, "user.email")
          - replace_match(body, "[0-9]{11}", "[REDACTED_PHONE]")
  batch:

exporters:
  debug:
    verbosity: basic

service:
  pipelines:
    traces:
      receivers: [otlp]
      processors: [memory_limiter, transform/redact, batch]
      exporters: [debug]
    logs:
      receivers: [otlp]
      processors: [memory_limiter, transform/redact, batch]
      exporters: [debug]
```

这个例子强调的是治理思路：

- 删除绝不应保留的字段
- 对日志正文中的模式做脱敏
- 在统一入口执行策略，而不是指望每个团队都手工一致

不过也要认识边界：如果敏感数据已经在应用日志里落盘，Collector 只能减少后续传播，不能抹掉之前已经写出的副本。

## 5. 访问控制、租户隔离与审计

即使数据本身已经被清理，谁能看到什么，仍然是治理的一部分。

### 5.1 最小权限访问

常见原则包括：

- 开发者默认查看自己服务的运行信息，而不是全组织原始日志
- 安全或审计团队可以访问更广范围，但需要审计记录
- 平台团队关注管道健康和资源使用，不必默认查看全部业务正文

### 5.2 多租户与环境隔离

如果一个平台服务多个团队、多个客户或多个环境，至少要把以下边界做清楚：

- 生产与测试环境分开
- 不同租户的数据有隔离标签与访问策略
- 共享 Collector 时，路由、认证与资源属性不能混淆

### 5.3 审计与可追责

可观测系统自身也应被审计，例如：

- 谁修改了采集策略
- 谁新增了高风险字段
- 谁导出了原始日志
- 谁访问了高敏感索引

只有这样，治理才不是一纸规范，而是可落地、可追踪的运行机制。

## 6. 保留、删除与合规边界

很多团队会认真讨论“怎么采”，却很少讨论“什么时候删”。但对合规来说，保留策略同样关键。

应至少分别考虑：

- traces 保留多久才还有排障价值
- metrics 长期趋势分析需要保留多久
- logs 中哪些类别需要短保留，哪些属于审计日志需要更长保留
- 归档副本、备份、副本导出是否与主存储保持一致删除策略

这里有一个常见误区：以为只要主后端删掉了数据，就完成了删除。实际上，如果数据曾被多路导出、落到对象存储或测试环境副本里，那么删除流程必须覆盖这些下游位置。

## 本章小结

| 主题 | 核心结论 |
|------|----------|
| 观测治理目标 | 既要保留诊断能力，也要降低泄漏、滥用与合规风险 |
| 主要风险入口 | 完整正文、认证头、主键、PII、异常原文、全量 header |
| 最佳治理点 | 优先应用侧不采集，其次 Collector 统一脱敏与过滤 |
| 访问控制 | 最小权限、环境隔离、租户隔离、审计可追踪 |
| 保留策略 | 不同信号、不同敏感级别应采用分层保留与删除 |
| 常见误区 | 只依赖后端权限，不管采集源头与中间副本 |

## OTel实验

### 实验目标

模拟一个含敏感字段的 HTTP 服务，验证应用侧约束与 Collector 脱敏的区别。

### 实验步骤

1. 构造一个登录接口，请求头带 `Authorization`，请求体带手机号或邮箱。
2. 第一轮实验，在应用中直接把这些字段写入 span attributes 和结构化日志。
3. 通过本地 Collector 的 `debug` exporter 观察输出，确认敏感字段已经进入管道。
4. 第二轮实验，在应用中移除这些字段，只保留 `auth.scheme`、`app.request.payload_type` 等抽象信息。
5. 第三轮实验，故意恢复错误埋点，但在 Collector 中增加脱敏 transform，观察输出差异。
6. 记录三轮实验中：
   - 数据是否可用于排障
   - 敏感内容是否暴露
   - 哪一层最适合做根因修复

### 预期现象

- 应用侧不采集是最稳妥的治理方式
- Collector 脱敏能减少扩散，但不能替代源头治理
- 抽象字段通常仍然足够支撑大多数排障场景

3. 如果一个共享平台服务多个业务租户，你会如何设计资源属性、访问控制和路由，避免不同租户数据混淆？
4. 为什么可观测系统本身也需要审计日志和变更记录？
