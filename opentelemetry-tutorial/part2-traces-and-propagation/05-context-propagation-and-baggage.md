# 第5章：上下文传播与 Baggage

## 学习目标

- 理解为什么一条 trace 能跨进程、跨服务持续关联，核心在于上下文传播
- 认识 W3C Trace Context 在分布式追踪中的基本作用
- 说清 trace context 与 baggage 分别适合承载什么信息
- 理解 HTTP、RPC、消息队列与异步任务中最常见的传播断点
- 建立“传播的是关联语义，不是随便塞业务数据”的工程边界意识

## 1. 为什么 trace 会断

很多团队第一次接入 tracing 时，最典型的问题不是 span 不会创建，而是：

- 服务 A 有 trace
- 服务 B 也有 trace
- 但它们不在同一条 trace 里

这就是所谓的“链路断裂”。

一条 trace 能跨服务持续成立，前提不是“所有服务都装了 SDK”，而是：

1. 上游在发起下游调用时把 trace 关联信息带出去
2. 下游收到请求时把这些关联信息提取出来
3. 下游创建的新 span 正确挂到已有上下文下面

这整个过程，就叫上下文传播。

如果没有传播，你看到的只是很多孤立的本地 span，而不是一条真正跨边界的 trace。

### 1.1 为什么传播是分布式追踪的核心，而不是附属功能

单进程内创建 span 相对容易，真正难的是跨边界时保持因果关系。因为一旦请求经过：

- HTTP 服务
- RPC 调用
- 消息队列
- 后台异步 worker
- 网关、代理或任务调度器

执行路径就不再是同一个进程里的连续函数调用，而是分散到多个独立运行单元中。传播的作用，就是让这些独立单元仍然知道自己属于同一条更大的请求路径。

## 2. W3C Trace Context：跨服务传什么

在 OpenTelemetry 体系里，跨服务传播最常见的基础标准是 W3C Trace Context。

你不必一开始就背所有字段格式，但要理解它解决的问题：

> 用一种跨语言、跨框架、跨供应商都能理解的方式，把 trace 的核心关联信息带到下游。

最常见的传播载体是请求头，例如：

- `traceparent`
- `tracestate`

### 2.1 traceparent 的作用

`traceparent` 最重要的价值是告诉下游：

- 这次请求属于哪条 trace
- 当前父级 span 是谁
- 采样相关的基础决策是什么

你可以把它理解成“这次调用的链路身份证”。

下游服务拿到这个信息后，才能把自己创建的 server span 接到正确的 trace 上，而不是重新开一条新的 trace。

### 2.2 tracestate 的作用

`tracestate` 更像可扩展的附加状态，用于在标准 trace 关联之外，保留某些跨系统需要的附加信息。初学阶段不必展开太深，但要知道它的定位不是替代业务参数，更不是随便塞自定义大字段的地方。

### 2.3 为什么标准化很重要

如果每个服务都自定义一套追踪头：

- 某个叫 `x-trace-id`
- 某个叫 `request-trace`
- 某个又带自家私有格式

那么跨团队、跨语言、跨平台集成就会非常脆弱。W3C Trace Context 的价值，正是尽量消除这种碎片化。

## 3. Context propagation 在不同边界上如何发生

传播的本质，是把当前上下文从一个执行单元交给下一个执行单元。不同边界上的传播方式会有差异。

### 3.1 HTTP / RPC：最直观的传播场景

在 HTTP 或 RPC 场景中，传播通常通过请求头或 metadata 完成。

例如一个 Node.js 服务向下游发起请求时，当前活动 span 的上下文会被注入到请求头中；下游服务接收请求后，再把这些头提取出来，建立自己的 server span。

可以用一个简化示意理解：

```text
service-a
  active context
    -> inject to HTTP headers
      -> service-b extract headers
        -> create child/server span
```

如果自动 instrumentation 生效，这一步往往能自动完成；但只要你碰到自定义客户端、代理改写头部、特殊中间件顺序问题，就仍然可能断。

### 3.2 消息队列：传播不是天然连续的

消息系统里的传播比 HTTP 更容易让初学者困惑，因为它不是“我发一个请求，对方立刻处理并返回”的同步模式。

常见路径是：

- 生产者发送消息
- 消息在 broker 中停留一段时间
- 消费者稍后拿到消息并处理

这意味着传播通常依赖消息 metadata、header 或 envelope 字段，而不是 HTTP 头。

这里最容易犯的错误是：

- 只记录了消费侧 span，却没有把生产时的上下文带过去
- 把消息处理错误建模成同步 parent-child，而忽略了它其实是异步边界

### 3.3 异步任务与后台 worker：上下文经常在这里丢

例如：

- 把请求放入一个 job queue
- 由后台 worker 稍后处理
- 或者在 Node.js 中切换到某些特殊异步执行路径

这时如果没有显式传播上下文，worker 创建出来的 span 很可能会另起一条 trace。

所以一个实用原则是：**只要执行权从当前调用栈切换到另一个独立执行单元，就应该主动思考上下文如何被传递。**

## 4. Baggage 是什么，和 trace context 有什么不同

很多人会把 baggage 和 trace context 混在一起。实际上，它们解决的问题不同。

### 4.1 trace context 负责关联链路

trace context 的核心目标是：

- 让上下游知道自己属于同一条 trace
- 确保父子关系或关联关系成立
- 支撑 trace id、span id 等基础链路信息的延续

### 4.2 baggage 负责传播少量跨服务元数据

baggage 更适合承载一些需要跨边界传递、且多个服务都可能用到的辅助元数据，例如：

- 租户类别
- 实验分组
- 业务优先级
- 用户分层标签

可以把它理解为：

> 和请求一起走，但不属于 trace 结构本身的一小组上下文标签。

### 4.3 baggage 不是通用业务参数容器

这是一个非常重要的边界。

baggage 不适合承载：

- 大体积数据
- 敏感信息
- 高频变化主键
- 完整用户资料
- 原始请求体

因为它会随着跨服务传播一路复制，如果滥用，会带来：

- 网络开销上升
- 安全与隐私风险
- 调试困难
- 上下游耦合加重

### 4.4 一个简化的 Node.js 例子

```ts
import { context, propagation } from '@opentelemetry/api'

const carrier: Record<string, string> = {}
const currentContext = context.active()

propagation.inject(currentContext, carrier)

console.log(carrier)
// 输出: 可能包含 traceparent 等传播字段
```

这段代码的重点不是让你背 API，而是帮助你建立直觉：传播本质上就是把当前上下文注入到一个可以跨边界携带的载体里，再由下游提取出来。

## 5. 传播为什么会断：最常见的工程原因

### 5.1 中间件顺序或初始化顺序错误

在 Node.js 中，如果自动 instrumentation 初始化太晚，或中间件包裹顺序不对，可能导致上下文作用域管理失效。结果是：

- 服务本地有 span
- 但出站请求没有携带正确的 trace context

### 5.2 自定义客户端或封装层绕过了自动注入

自动 instrumentation 只对它已知的框架和库生效。如果你封装了一个特殊 HTTP 客户端、任务调度器或消息发送器，传播可能不会自动完成。

### 5.3 代理、网关、消息中间件丢头或改写

现实系统里，请求很少是服务直连。网关、反向代理、服务网格、中间件插件都可能：

- 丢掉未知 header
- 改写 metadata
- 只保留白名单字段

结果就是应用明明“已经注入了 traceparent”，但下游根本收不到。

### 5.4 服务之间没用同样的 propagator 或标准

如果一边用 W3C Trace Context，另一边还停留在别的私有格式，而且没有兼容配置，那么链路也会断。

### 5.5 异步边界建模错误

很多时候 trace 不是完全断了，而是被错误地建成了不合理的父子关系，或者干脆完全脱离上游上下文。这在 job、batch、consumer 系统里尤其常见。

## 6. 如何调试传播问题

传播问题通常要按边界逐跳检查，而不是一口气怀疑整个系统。

### 6.1 沿路径逐跳确认

可以按以下顺序检查：

1. 入口服务是否有根 span
2. 出站请求是否带上了 `traceparent`
3. 网络边界是否保留了这些头或 metadata
4. 下游是否成功提取上下文
5. 下游新建 span 的 trace id 是否与上游一致

### 6.2 用日志辅助观察传播字段

在排障阶段，适度记录这些字段会很有帮助：

- `trace_id`
- `span_id`
- `parent_span_id`
- `traceparent`
- 消息 header 中的传播字段

但要记住，这些用于调试，不代表你应该长期无节制记录所有传播原文。

### 6.3 先确认传播，再讨论 span 设计

如果 trace id 根本没有延续，先解决传播问题；如果 trace 连上了但结构不好读，再进入 span 建模问题。把这两个层次分开，会大幅减少排障混乱。

## 本章小结

| 主题 | 核心结论 |
|------|----------|
| 上下文传播 | 决定一条 trace 能否跨服务持续关联 |
| W3C Trace Context | 提供跨系统统一传播 trace 关联信息的标准 |
| trace context | 主要承载 trace id、span 关系等链路关联信息 |
| baggage | 传播少量辅助元数据，但不应被滥用成业务参数容器 |
| 常见断点 | 初始化顺序、自定义封装、代理丢头、异步边界、标准不一致 |
| 调试方法 | 沿调用路径逐跳检查注入、传输、提取与挂接是否成功 |

## OTel实验

### 实验目标

观察一次跨两个 HTTP 服务的请求，理解 trace context 如何被传播，以及 baggage 与普通业务参数有什么边界区别。

### 实验步骤

1. 准备两个本地 Node.js 服务：`gateway-service` 和 `checkout-service`。
2. 在 `gateway-service` 中接收外部请求后，再向 `checkout-service` 发起一次 HTTP 请求。
3. 开启自动 instrumentation，观察在正常情况下两边 span 是否出现在同一条 trace 中。
4. 刻意移除或改写出站请求中的传播头，观察 trace 是否断裂。
5. 再实验性地添加一个很小的 baggage 字段，例如 `user.tier=premium`，并在下游读取它。
6. 最后思考：如果把 `user.id`、邮箱或完整请求体放进 baggage，会带来什么问题。

### 预期现象

- 正常传播时，上下游 span 会出现在同一条 trace 中。
- 去掉传播头后，下游往往会另起一条 trace。
- baggage 适合小而稳定的辅助元数据，不适合承载敏感或大体积业务信息。
