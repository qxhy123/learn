# 第22章：为微服务系统设计可观测性

## 学习目标

- 理解微服务可观测性设计为什么应从业务路径、故障模式与治理边界出发
- 学会为服务身份、资源字段、span 边界、指标维度和日志 schema 设计稳定约束
- 掌握同步请求、异步消息、批处理任务等不同调用形态下的 trace 建模方法
- 能围绕 SLI、容量与排障闭环设计 traces、metrics、logs 的协同关系
- 建立“可观测性即接口契约”的工程意识，把埋点设计纳入长期治理

## 1. 可观测性设计要从系统问题出发，而不是从埋点清单出发

很多团队在进入微服务阶段后，会自然地产生一种冲动：

- 每个服务先装上自动 instrumentation
- 看到 trace 后再慢慢补字段
- 哪个问题排不到，再临时加日志或 metric

这种方式在早期能跑起来，但很难支撑长期维护。原因很简单：微服务系统的复杂性，不在于“某个服务有没有数据”，而在于这些数据能否共同回答系统级问题。

一个更稳的顺序通常是：

1. 先定义关键业务路径
2. 再定义关键故障模式
3. 然后决定不同信号分别承担什么责任
4. 最后才进入具体埋点与 Collector 配置

可以把微服务可观测性理解成对三类问题的回答能力：

| 问题类型 | 你真正想知道什么 | 主要依赖的信号 |
|----------|------------------|----------------|
| 业务路径问题 | 一个请求经过了哪些服务，在哪个阶段变慢或失败 | traces |
| 稳定性问题 | 某个服务、依赖或队列整体是否退化 | metrics |
| 证据与细节问题 | 某次失败到底发生了什么、输入条件是什么 | logs |

例如，一个下单请求可能经过：

```text
gateway -> auth -> cart -> pricing -> inventory -> payment -> order
```

如果你只想着“每个服务都打一堆 span”，很可能会得到很多局部信息，却仍然回答不了下面的问题：

- 到底是 `pricing` 算价慢，还是 `payment` 重试导致尾延迟放大？
- 为什么只有 `mobile` 渠道、只有 `cn-north` 区域出现高失败率？
- 同一个错误码到底是来自业务拒绝、依赖超时还是消息消费积压？

因此，第 22 章真正讨论的不是“怎么把 OTel 接上去”，而是：**怎么为一个微服务系统设计一套长期可解释、可治理、可迁移的 telemetry 结构。**

## 2. 先把服务身份与资源边界设计清楚

在微服务系统里，第一件必须稳定下来的事情不是 span 名，而是“谁是谁”。如果服务身份不稳定，后面的 trace 查询、指标聚合、日志检索都会非常混乱。

### 2.1 Resource 是系统身份契约，而不是补充信息

Resource 更适合承载“数据来自哪个实体”的信息，例如：

- `service.name`
- `service.namespace`
- `service.version`
- `deployment.environment.name`
- `cloud.region`
- `k8s.namespace.name`
- `k8s.cluster.name`

这些字段的作用，不是为了“多打点”，而是为了建立稳定的聚合与过滤维度。一个实用原则是：

- **稳定身份字段** 用来长期聚合与治理
- **瞬时实例字段** 用来短期排障与定位

下面是一份适合微服务体系的最小资源字段清单：

| 字段 | 建议用途 | 设计提醒 |
|------|----------|----------|
| `service.name` | 服务主身份 | 全组织唯一且稳定，不要频繁改名 |
| `service.namespace` | 业务域或平台域分组 | 用于降低重名概率，反映组织边界 |
| `service.version` | 版本对比、发布回溯 | 用于灰度与回滚分析 |
| `deployment.environment.name` | production / staging / development 区分 | 建议全组织统一枚举值 |
| `cloud.region` / `k8s.cluster.name` | 多区域、多集群定位 | 用于跨区域故障分析 |
| `service.instance.id` | 单实例定位 | 仅用于排障，不要当核心聚合维度 |

### 2.2 服务边界决定 Resource 边界

微服务体系里最常见的错误之一，是把“代码仓库”“进程”“服务”“任务”这几个概念混在一起。

例如：

- 一个仓库里可能有多个可独立部署服务
- 一个服务可能有多个 worker 角色
- 一个 API 服务和一个异步消费者可能共享代码，但不应该共享完全相同的身份模型

更实用的做法是按**部署与责任边界**来定义资源身份。

比如订单域可以这样区分：

| 组件 | 是否建议独立 `service.name` | 原因 |
|------|----------------------------|------|
| `order-api` | 建议独立 | 对外提供同步请求入口 |
| `order-worker` | 建议独立 | 消费消息、处理异步补偿 |
| `order-migrator` | 视情况而定 | 如果只是低频运维任务，可用 job identity |

### 2.3 资源身份要服务于查询与治理

设计资源字段时，建议反复问两个问题：

1. 这个字段是否帮助长期聚合？
2. 这个字段是否应该由平台统一注入，而不是业务手写？

通常：

- `service.name`、`deployment.environment.name` 更适合平台统一约束
- `app.feature_flag`、`business.scenario` 更适合业务侧谨慎补充

这与第 9 章讨论的 Resources、Attributes 和 Semantic Conventions 是直接衔接的：**资源字段不是杂项集合，而是整个系统的身份层。**

## 3. Trace 设计的重点是“解释系统”，而不是“生成很多 span”

微服务 trace 设计的核心，不是把所有函数都变成 span，而是让链路结构能够解释：

- 请求经过了哪些责任边界
- 关键延迟花在什么阶段
- 错误到底发生在业务逻辑、依赖调用还是异步处理

### 3.1 Span 应围绕边界与阶段来建模

在微服务里，最值得建模的 span 往往是这些位置：

- 对外请求入口
- 出站依赖调用
- 关键业务阶段
- 异步投递与消费
- 批处理任务的批次边界

一个下单流程可以设计成：

| 业务阶段 | 推荐 span | 说明 |
|----------|-----------|------|
| API 接入 | `orders.create` | 根 span 或 server span 下的关键业务 span |
| 价格计算 | `orders.calculate_price` | 表达业务阶段，而不是底层函数名 |
| 库存预留 | `inventory.reserve` | 跨服务依赖边界 |
| 支付授权 | `payment.authorize` | 关键下游依赖与失败源 |
| 发送异步事件 | `orders.publish_created_event` | 进入异步链路的边界 |

比起 `controller.handle`、`service.run`、`helper.call` 这种名字，上面的命名更容易回答“系统正在做什么”。

### 3.2 同步、异步、批处理的链路模型不能混用

同步 HTTP / RPC 请求通常适合 parent-child 模型；而消息队列、事件总线和批处理系统则更常需要 `link` 或拆分为多个根链路。

实用判断如下：

| 场景 | 更常见的建模方式 | 原因 |
|------|------------------|------|
| API 调用下游服务 | parent-child | 调用因果关系清晰，时间窗口连续 |
| 消息生产与消费 | link 或 producer/consumer 关系 | 消费时间可能延后，且可被多个消费者处理 |
| 批处理任务汇总多个输入 | links | 一个结果可能依赖多条上游记录 |
| 定时任务 | 新根 trace 或与触发事件 link | 不应伪装成同步请求的子 span |

这与第 20 章关于异步系统的内容会形成直接衔接：**异步链路不是“不支持 trace”，而是不能机械套用同步调用模型。**

### 3.3 错误、重试和超时要可解释

微服务故障中最常见的误判之一，是只看到“请求失败”，却看不清失败在哪里被放大。

因此，trace 设计至少要让下面三种事情可以区分：

- 业务拒绝，例如库存不足、风控拒绝
- 技术失败，例如超时、连接错误、协议错误
- 重试放大，例如下游 200ms 超时被重试三次，最终表现为 900ms 尾延迟

一个简单原则是：

- 用 span status 表达该阶段是否失败
- 用 exception / event 记录关键异常与重试信息
- 用 attributes 记录有限、稳定、可聚合的上下文类别

例如，比起直接记录整段错误文本，更适合记录：

- `error.type=timeout`
- `retry.count=2`
- `dependency.name=payment-gateway`

### 3.4 一个好的 trace 应支持跨角色阅读

一个好的微服务 trace，不应该只有写代码的人看得懂。平台工程师、值班工程师、技术负责人也应该能看出：

- 服务边界是否合理
- 哪类依赖是系统瓶颈
- 哪些 span 代表业务阶段，哪些代表基础设施调用

这也是为什么 span 名和属性设计不能只以“开发实现便利”为标准。

## 4. Metrics 设计要围绕 SLI、容量与依赖，而不是围绕“想筛选什么”

微服务体系的 metrics 设计，经常在两个极端之间摇摆：

- 要么只有 CPU、内存、QPS 这类太粗的系统指标
- 要么把大量请求属性全做成标签，最后 series 数量失控

一个成熟的做法，是围绕三类观测目标来设计指标：

1. 服务对外行为
2. 关键依赖健康度
3. 资源与容量状态

### 4.1 每个微服务至少要有一层“稳定服务指标”

一个服务最基础的指标集合通常应覆盖：

| 观测面 | 典型指标 | 常见维度 |
|--------|----------|----------|
| 请求量 | request count | `service.name`、`http.route`、`method` |
| 错误率 | error count / rate | `status_code_class`、`error.type` |
| 延迟 | request duration histogram | `http.route`、`dependency.class` |
| 饱和度 | queue depth、worker concurrency、CPU | `service.name`、`instance.role` |

这里的重点不是抄某种固定命名，而是明确：这些指标应该能支持 SLI/SLO 与容量规划，而不是变成随意堆标签的查询素材。

### 4.2 指标维度优先选择“有限类别”

在微服务场景里，更适合作为指标维度的往往是：

- `deployment.environment.name`
- `http.route`
- `rpc.system`
- `messaging.system`
- `dependency.type`
- `user.tier`
- `tenant.plan`

而不适合作为高频指标维度的通常是：

- `user.id`
- `order.id`
- `session.id`
- 原始 URL
- 动态错误文本

这与第 16 章的 cardinality、成本与性能是直接衔接的。对于微服务系统来说，**指标不是为了还原单次请求，而是为了看整体退化模式。**

### 4.3 依赖指标和业务指标要分层

很多系统只监控“自己的请求延迟”，却不监控“自己对外部依赖造成或承受的压力”。结果一旦下游故障，就只能看到自己变慢，难以区分根因。

建议至少分成三层：

| 层次 | 例子 | 作用 |
|------|------|------|
| 服务层 | `orders.request.duration` | 看服务整体健康度 |
| 依赖层 | `orders.dependency.duration` | 看数据库、缓存、支付网关等瓶颈 |
| 业务层 | `orders.created_total`、`payments.authorized_total` | 看业务结果与漏斗变化 |

### 4.4 指标要与告警和排障闭环对应

如果一个指标不能支撑以下任一动作，它的长期价值通常有限：

- 做告警阈值或 burn rate 判断
- 区分发布问题与流量结构变化
- 缩小 trace / log 排查范围

换句话说，指标设计不能脱离运维动作本身。

## 5. Logs、Events 与跨信号关联要有明确分工

微服务系统里，日志最容易退化成“什么都往里塞”的垃圾桶。真正成熟的做法，是先明确日志与其他信号分别解决什么问题。

### 5.1 日志更适合承载证据，而不是承载聚合职责

日志更适合回答：

- 某次异常的详细输入条件是什么
- 某个业务决策为什么被拒绝
- 某个补偿任务执行到了哪一步

而不适合承担：

- 长期趋势统计
- 高频低成本聚合
- 替代 metrics 的告警职责

### 5.2 结构化日志至少要和 trace 能互相跳转

在微服务系统里，日志至少应保证以下关联字段能被保留或注入：

- `trace_id`
- `span_id`
- `service.name`
- `deployment.environment.name`
- 稳定的业务类别字段，如 `order.channel`、`payment.method_type`

一个简单的职责区分如下：

| 信息类型 | 更适合放哪里 | 原因 |
|----------|--------------|------|
| 请求全链路阶段 | trace / spans | 用于回答“经过了哪里” |
| 聚合趋势与告警 | metrics | 用于回答“整体是否退化” |
| 失败细节与输入证据 | structured logs | 用于回答“这次到底发生了什么” |
| 某个 span 内的关键瞬时注记 | span event | 用于补充阶段内关键时刻 |

### 5.3 日志 schema 也需要治理

常见的坏味道包括：

- 不同服务同一语义字段写成不同名字
- 错误日志里直接打印动态大对象
- trace_id 只在部分服务里有
- 业务主键被随意复制到所有日志和指标中

更实用的方式是建立统一日志约定，例如：

| 字段类别 | 建议 |
|----------|------|
| 身份字段 | 与 Resource 对齐，如 `service.name`、`deployment.environment.name` |
| 关联字段 | `trace_id`、`span_id`、`request_id` |
| 业务字段 | 只保留有限、解释明确的类别字段 |
| 错误字段 | `error.type`、`error.code`、`error.message`，避免把动态堆栈当主索引 |

## 6. 把可观测性当成接口设计与迁移治理的一部分

一个系统是否拥有“设计过的可观测性”，不在于它有没有图表，而在于它有没有可持续的变更约束。

### 6.1 可观测性契约要有人负责

在微服务体系里，最好明确以下责任：

| 角色 | 负责内容 |
|------|----------|
| 业务服务团队 | 关键业务 span、业务 metrics、结构化日志字段 |
| 平台团队 | Resource 规范、Collector 入口、公共 processor、出口策略 |
| SRE / 可观测性团队 | 告警标准、保留策略、成本治理、统一查询范式 |

### 6.2 变更应经过评审，而不是“上线后再看”

以下变更最好进入评审清单：

- 新增高频指标及其维度
- 新增跨服务关键 span
- 调整 `service.name`、`service.namespace`
- 新增可能包含 PII 的日志字段
- 在 Collector 中新增高成本 processor 或导出路径

一个简化评审清单可以是：

1. 这个字段属于 resource、span、metric 还是 log？
2. 它是稳定类别，还是高基数实例值？
3. 它的主要消费者是谁：开发、SRE、业务分析还是审计？
4. 它是否会影响告警、查询、保留或成本？
5. 它和已有语义规范是否冲突？

### 6.3 可观测性设计必须可迁移

如果一个系统的 telemetry 结构高度依赖某家后端的私有字段、私有 query 语义或私有 agent，那么迁移成本会非常高。

因此，微服务可观测性设计要尽量保持：

- Resource 和 attributes 尽量贴近语义规范
- 应用侧埋点保持 vendor-neutral
- 与后端强耦合的逻辑更多放到 Collector 与平台层

这也正好为下一章的迁移与 rollout 主题做铺垫：**好的可观测性设计，不只是今天能查问题，还要保证明天能演进。**

## 本章小结

| 主题 | 核心结论 |
|------|----------|
| 设计起点 | 应从业务路径、故障模式与治理需求出发，而不是从埋点数量出发 |
| Resource 设计 | 先稳定服务身份与环境边界，再谈具体 span 与日志字段 |
| Trace 设计 | 重点是解释边界、阶段、错误与异步关系，而不是制造大量低价值 span |
| Metrics 设计 | 围绕 SLI、依赖和容量，优先选择有限、稳定的维度 |
| Logs 设计 | 日志负责证据与细节，必须与 trace 形成稳定关联 |
| 长期治理 | 可观测性应被视为接口契约，需要评审、版本意识和迁移意识 |

## OTel实验

### 实验目标

为一个包含 `gateway`、`orders`、`inventory`、`payment` 四个组件的微服务系统设计一份最小可观测性契约，而不是直接写代码。

### 实验步骤

1. 先画出一条核心业务路径，例如“用户提交订单直到支付完成”。
2. 列出这条路径中最重要的 3 类故障模式，例如下游超时、库存不足、消息投递失败。
3. 为每个服务定义最小 Resource 字段集合，至少包括 `service.name`、`service.version`、`deployment.environment.name`。
4. 为链路中的关键阶段设计 span 名称，并说明哪些场景使用 parent-child，哪些场景更适合 link。
5. 为每个服务设计 3 组指标：服务指标、依赖指标、业务指标，并检查是否存在高基数风险。
6. 设计一份结构化日志字段清单，要求能够从日志跳转到 trace，并能区分技术失败与业务拒绝。
7. 最后用一页表格总结：每种信号分别帮助回答什么问题，谁负责维护，如何评审变更。

### 预期收获

- 你会发现“画出契约”比“多打几个点”更能暴露设计问题。
- 你会更清楚哪些字段适合聚合，哪些字段只适合单次定位。
- 你会意识到 Resource、trace、metrics、logs 必须一起设计，不能分散决定。

## 练习题

1. 为什么说微服务可观测性设计应该先从业务路径和故障模式开始，而不是先从自动 instrumentation 开始？
2. 假设一个系统同时有 `order-api` 和 `order-worker`，它们共用同一代码仓库。请说明什么时候应该使用不同的 `service.name`，为什么？
3. 对一个“下单后发送事件，再由异步 worker 处理库存扣减”的流程，你会如何设计 trace 关系？请说明 parent-child 与 link 的取舍。
4. 为什么 `user.id`、`order.id` 常常不适合作为高频指标维度，却仍可能在某些 trace 或日志中有价值？
5. 如果团队计划新增一个 `payment.channel.raw_response` 字段到日志中，你会从哪些维度评估它是否应该上线？
