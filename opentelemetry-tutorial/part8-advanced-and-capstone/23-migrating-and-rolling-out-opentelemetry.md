# 第23章：从旧监控/日志体系迁移到 OpenTelemetry

## 学习目标

- 理解从旧有监控、日志与 tracing 体系迁移到 OpenTelemetry 时最常见的工程阻力与风险来源
- 学会设计分阶段 rollout 路线，区分试点、并行期、治理期与收口期的目标
- 掌握双写、桥接、兼容、灰度与回滚等迁移策略背后的适用条件与代价
- 能建立迁移验收标准，避免“数据看起来有了”却无法支撑真实排障与治理
- 建立以 schema、组织协作和风险控制为中心的迁移治理思路

## 1. 迁移的难点通常不在 SDK，而在系统惯性

很多团队在评估 OpenTelemetry 迁移时，会把问题理解成“把旧 SDK 换成新 SDK”。但在真实工程里，迁移难点通常来自更深层的系统惯性：

- 已有仪表盘、告警、查询语句大量依赖旧字段
- 各个团队对服务名、错误码、业务维度没有统一约定
- 不同语言、不同框架使用着不同年代的接入方式
- 平台层已经依赖某些私有 agent、sidecar 或 exporter 机制
- 组织里没有明确的 schema owner 和 rollout owner

因此，迁移工作首先是一次**观测治理重构**，其次才是技术替换。

一个常见的误区是：

1. 先让几个服务接上 OTel
2. 看到后端里有 trace 了
3. 就宣布迁移已经开始成功

问题在于，真正决定迁移能否成功的不是“能不能发出数据”，而是下面这些能力是否保住了，甚至变得更好：

- 核心告警是否仍可靠
- 关键仪表盘是否仍可用
- trace / metrics / logs 是否能比以前更好地协同
- schema 是否更统一，而不是更混乱
- 成本与治理边界是否更清晰

### 1.1 迁移对象通常不止一套

所谓“旧体系”，现实中可能包括很多不同来源：

| 旧能力来源 | 常见情况 |
|------------|----------|
| 旧 metrics SDK | 直接暴露自定义指标，命名与标签长期历史包袱较重 |
| 旧 tracing SDK | 字段、采样、传播方式与 OTel 不兼容 |
| 旧日志 agent | 只负责采集文本日志，没有统一 trace 关联 |
| 私有 exporter | 应用直接耦合某家后端入口 |
| 手工 request-id 方案 | 部分串联调用链，但没有标准 propagation |

所以迁移前要先回答：**你迁移的是一个“工具”，还是一整套“信号生产与消费方式”？**

### 1.2 迁移目标不能只写“统一到 OpenTelemetry”

更可操作的目标通常应拆成：

- 应用侧接入统一到 OTel API / SDK 或兼容方式
- OTLP 成为主要数据出口或中间协议
- Collector 成为统一治理入口
- 服务身份、关键字段和语义规范趋于统一
- 旧体系中的关键消费面逐步迁移，而不是突然断掉

这与第 22 章强调的“可观测性契约”直接相连。**如果没有契约，迁移只会把历史混乱搬到新管道里。**

## 2. 迁移前先做清点：现状、依赖与不可动资产

在真正 rollout 前，最重要的一步不是改代码，而是资产清点。你需要知道哪些东西必须保、哪些东西可替换、哪些东西该趁机清理。

### 2.1 至少建立一份迁移基线清单

建议先整理以下内容：

| 类别 | 需要清点的内容 |
|------|----------------|
| 服务清单 | 哪些服务使用哪种语言、框架、运行环境 |
| 信号清单 | 哪些服务已有 traces、metrics、logs，质量如何 |
| 传播方案 | 当前是否使用 W3C Trace Context、B3 或私有 header |
| 仪表盘与告警 | 哪些是关键资产，依赖哪些字段与指标 |
| 日志关联 | 是否已有 trace_id / request_id，能否跨服务检索 |
| 成本现状 | 哪些服务是高数据量、高成本来源 |
| 风险资产 | 哪些场景涉及计费、审计、合规，不允许观测退化 |

### 2.2 找出“不可动资产”

迁移时常见的一类风险，是误把一些看起来“老旧”的东西当成可以立刻删除的遗留物。实际上其中有些是不可动资产，例如：

- 值班团队依赖的核心告警
- 财务或计费链路依赖的审计日志
- 发布看板依赖的延迟指标
- 大客户 SLA 周报依赖的统计口径

这些资产不一定要永久保留旧实现，但迁移时必须有明确承接方案。

### 2.3 把字段兼容问题显式化

旧体系里常见的问题不是“没有字段”，而是同一语义在不同地方叫法不同。例如：

| 旧字段 | 新字段方向 |
|--------|------------|
| `app` | `service.name` |
| `env` | `deployment.environment.name` |
| `status` | 需要区分 HTTP 状态、业务状态、span status |
| `requestId` | `trace_id` / `span_id` / `request.id` 视语义拆分 |

在迁移前把映射关系写清楚，能大幅降低并行期混乱。

## 3. 分阶段 rollout：先试点，再并行，再收口

最稳妥的 OTel 迁移几乎都不是一次性切换，而是分阶段推进。一个常见的四阶段模型如下。

### 3.1 阶段一：试点期

目标不是覆盖率，而是验证最小闭环是否可靠。

建议试点服务满足以下条件：

- 有明确业务路径，容易观察效果
- 不要太核心，以免第一轮失败影响过大
- 同时具备 traces、metrics、logs 的实际需求
- 团队愿意配合改 schema、改日志和做对照验证

试点期重点验证：

- OTel 接入是否稳定
- Collector 路线是否可行
- trace / metric / log 关联是否真的改善
- 与旧体系相比，诊断体验是否提升

### 3.2 阶段二：并行期

并行期通常意味着：

- 旧体系继续产出关键数据
- 新体系开始稳定接收同类或映射后的数据
- 告警、仪表盘、排障流程逐步做等价验证

这是最容易产生额外成本的阶段，但也是最重要的安全带。并行期的重点不是“双倍时间越短越好”，而是确保对照验证足够扎实。

### 3.3 阶段三：治理期

当多个服务已接入后，真正的挑战开始从“接入”转向“统一”：

- 统一服务命名
- 统一资源字段
- 统一日志关联字段
- 统一采样与保留策略
- 统一 Collector pipeline 与出口治理

如果缺少治理期，试点成功往往会在规模化后重新退化成“每个团队各接各的”。

### 3.4 阶段四：收口期

收口期的目标是让旧路径真正退出，而不是长期并存成技术债。

常见收口动作包括：

- 移除旧 SDK 与旧 agent 依赖
- 下线旧 exporter 和旧 collector 路线
- 迁移或重建遗留仪表盘、告警、查询模板
- 更新运维 runbook 和开发接入指南

如果到了这一步还没有明确“哪些旧能力可以删除”，通常说明前面的验收标准没有定义清楚。

## 4. 双写、桥接、灰度与回滚：常见迁移策略怎么选

迁移不是单选题，不同信号、不同服务、不同风险等级，策略常常不同。

### 4.1 双写适合高风险、强验证场景，但成本最高

双写的典型做法是：

- 应用同时向旧路径与新路径发送数据
- 或应用发到 Collector，再由 Collector 桥接到旧后端与新后端

双写的优点：

- 便于做同口径对照
- 告警与仪表盘迁移更稳妥
- 一旦新路径异常，旧路径仍在

双写的代价：

- 成本上升
- 应用或 Collector 复杂度上升
- 字段映射与去重问题会更明显

因此，双写更适合：

- 核心业务链路
- 核心告警依赖路径
- 组织尚未完全信任新体系时

### 4.2 桥接适合降低应用改造成本，但要注意语义损失

很多团队会先用桥接策略过渡，例如：

- 旧 instrumentation 的数据先汇到 Collector
- 或通过兼容 exporter / receiver 暂时接入 OTLP 管道
- 或在日志管道中注入 trace 关联字段，再逐步标准化

桥接的价值在于：

- 可以先统一传输与治理层
- 不必一次性改完所有业务代码

但桥接的限制也很明显：

- 旧字段语义可能无法完全映射到 OTel
- 旧 trace 模型可能与新 propagation 不一致
- 某些私有概念难以长期保留

所以桥接更像缓冲带，而不是长期目标。

### 4.3 灰度 rollout 应按服务批次和消费面批次同时设计

很多人理解灰度，只想到“先给 10% 服务开关”。实际上可观测性迁移至少有两条灰度线：

1. **生产侧灰度**：哪些服务、哪些环境、哪些实例先启用
2. **消费侧灰度**：哪些仪表盘、哪些告警、哪些值班流程先切换到新数据

这两条线如果不同步，就会出现：

- 数据已经接入，但没人使用
- 或者告警已经切过去，但新数据仍不稳定

### 4.4 回滚不只是关掉新 SDK

成熟的回滚方案至少要覆盖：

- 应用侧：关闭 OTel 接入或降级 exporter
- Collector 侧：回退 pipeline 配置、关闭新 processor、切回旧出口
- 消费侧：告警和看板恢复到旧数据源
- 治理侧：保留必要的并行数据，方便事后对比

如果没有预先定义“回滚后哪些能力必须立刻恢复”，回滚很容易变成混乱手工操作。

## 5. 迁移验收要看“可用性”和“可治理性”，不只看“有数据”

可观测性迁移最容易犯的错误，就是把“后端里看见了数据”误当成迁移完成。真正的验收至少要覆盖四个层次。

### 5.1 数据正确性验收

要确认：

- trace 是否完整，关键边界是否不断裂
- `service.name`、环境字段、版本字段是否正确
- 指标维度是否符合预期，没有明显高基数污染
- 日志是否能稳定关联 trace_id / span_id

### 5.2 诊断能力验收

要确认：

- 出现慢请求时，是否能比旧体系更快定位阶段
- 出现错误峰值时，是否能从指标跳到 trace，再跳到日志
- 异步场景是否仍可解释，不会只剩下孤立 span

### 5.3 消费资产验收

要确认：

- 核心仪表盘是否完成迁移或重建
- 核心告警是否已校准阈值与数据口径
- runbook 是否更新为新查询方式
- 值班人员是否知道新旧差异

### 5.4 治理质量验收

要确认：

- 新接入服务是否遵循统一命名与 schema 约束
- Collector pipeline 是否具备统一入口与出口控制
- 新体系是否降低了应用与后端的直接耦合
- 成本、保留、采样策略是否已纳入治理

一个简化验收表可以是：

| 验收维度 | 验收问题 |
|----------|----------|
| 数据正确性 | 关键信号是否正确、完整、可关联 |
| 诊断能力 | 排障闭环是否比旧体系更顺畅 |
| 业务安全 | 核心告警、审计与 SLA 视图是否保住 |
| 治理质量 | 是否真的统一，而不是换壳后继续分裂 |

## 6. 组织协作、文档与长期收敛同样属于迁移的一部分

迁移到 OpenTelemetry 不是一次性工程任务，它还会改变组织协作方式。

### 6.1 需要明确三类 owner

| owner 类型 | 负责内容 |
|------------|----------|
| 平台 owner | Collector、OTLP 入口、通用资源字段、出口治理 |
| 服务 owner | 本服务埋点、日志、关键业务指标与验证 |
| 消费 owner | 仪表盘、告警、runbook、排障流程迁移 |

### 6.2 文档必须覆盖“新接入”和“旧资产退役”

迁移文档至少应包含：

- 新服务如何接入 OTel
- 旧服务如何迁移与对照验证
- 字段映射与命名规范
- 告警、看板、日志查询的迁移指引
- 何时可以下线旧路径

### 6.3 不要让并行期永久化

很多组织在双写和并行期停留太久，最后形成新的技术债：

- 开发要维护两套埋点
- 平台要维护两套 Collector / agent
- 值班要看两套看板
- 成本翻倍，却没有获得额外治理收益

因此，从试点开始就要定义退出条件，例如：

1. 某类服务的核心告警全部迁移完成
2. 新体系连续稳定运行一段时间
3. 关键指标与 trace 在新体系中通过验收
4. 旧路径没有剩余消费方依赖

只有这样，迁移才会真正形成收敛，而不是永远“正在迁移中”。

## 本章小结

| 主题 | 核心结论 |
|------|----------|
| 迁移本质 | 首先是观测治理重构，其次才是 SDK 或 agent 替换 |
| 迁移前准备 | 先清点服务、信号、告警、字段映射和不可动资产 |
| rollout 路线 | 推荐经历试点期、并行期、治理期、收口期 |
| 常见策略 | 双写最稳但最贵，桥接适合过渡，灰度与回滚必须提前设计 |
| 验收重点 | 不只看有没有数据，更要看诊断能力、消费资产和治理质量 |
| 长期成功条件 | 明确 owner、补齐文档、定义旧路径退出条件 |

## OTel实验

### 实验目标

为一个已有旧 metrics SDK、旧日志 agent 和部分私有 tracing 接入的系统，设计一份 OpenTelemetry 迁移计划草案。

### 实验步骤

1. 选定 3 个代表性服务，分别假设为同步 API、异步 worker、批处理任务。
2. 盘点它们当前已有的 traces、metrics、logs、告警和关键仪表盘，标出哪些属于不可动资产。
3. 设计字段映射表，把旧的服务名、环境名、请求标识映射到 OTel 语义或统一 schema。
4. 选择一个试点服务，说明为什么它适合先迁移。
5. 为试点服务设计并行期方案，明确哪些信号走双写，哪些信号走桥接，哪些只做灰度观察。
6. 列出回滚方案，至少包括应用、Collector、告警与看板四个层面。
7. 写出验收标准，要求覆盖数据正确性、排障闭环、核心告警与治理质量。

### 预期收获

- 你会意识到迁移计划的主体不是配置文件，而是依赖关系与验证标准。
- 你会更清楚双写、桥接、灰度的适用边界。
- 你会发现“旧资产退出条件”如果不提前定义，迁移很难真正完成。

## 练习题

1. 为什么说从旧监控/日志体系迁移到 OpenTelemetry 时，最大的难点往往不是 SDK 替换，而是观测治理和组织协作？
2. 什么是迁移中的“不可动资产”？请举三个例子，并说明为什么必须显式识别它们。
3. 双写、桥接、灰度三种策略分别更适合什么场景？它们各自的主要代价是什么？
4. 为什么迁移验收不能只看“后端里已经有数据”？请至少从两个不同层次说明。
5. 如果一个团队已经让所有服务都把数据发到了 OTel Collector，但核心告警、日志关联和 schema 仍然混乱，为什么还不能说迁移完成？
