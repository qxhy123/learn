# 第1章：为什么需要 OpenTelemetry

> 如果一个系统只能看到零散日志、几个孤立指标和一堆彼此对不上的报警，那么它不是“缺一个工具”，而是缺一个统一的观测模型。

---

## 学习目标

学完本章，你将能够：

1. 理解现代分布式系统为什么越来越需要统一的可观测性入口
2. 说清 OpenTelemetry 想解决的核心问题，而不把它误解成某个 tracing 产品
3. 区分“有监控”“有日志”和“真正可观测”之间的差别
4. 理解厂商中立与统一数据模型在工程中的现实价值
5. 对 OpenTelemetry 在 traces、metrics、logs 三类信号中的位置建立第一层直觉

---

## 1.1 为什么传统观测方式越来越不够用

在单体应用时代，很多问题都可以靠以下方式解决：

- 登录服务器看本地日志
- 查一个 CPU / 内存面板
- 用 request ID 手工串几段调用链

但在今天的系统里，情况通常不再这么简单。一个请求可能会穿过：

- API gateway
- 鉴权服务
- 订单服务
- 库存服务
- 支付服务
- 消息队列
- 后台异步 worker
- 数据库与缓存

如果每个组件都用自己的日志格式、自己的指标命名、自己的 trace 方案，那么真正的问题就不是“数据有没有”，而是：

- 数据彼此能不能对上
- 谁负责传播上下文
- 谁来定义字段名与语义
- 以后换后端或接多个后端怎么办

这也是为什么很多团队明明已经有：

- 监控平台
- 日志平台
- tracing 产品

却仍然觉得排障很慢、迁移很痛、接入很碎。

---

## 1.2 OpenTelemetry 想解决的到底是什么问题？

很多人第一次接触 OpenTelemetry，会把它理解成：

- 一个 tracing SDK
- 一个 Collector
- 某种 OTLP 协议
- 某个后端接入方式

这些都只是一部分。更准确地说，OpenTelemetry 想解决的是：

> 在应用、运行时、Collector 和后端之间，建立一套统一、标准、可移植的观测数据模型与接入方式。

它主要试图解决以下几类问题：

### 问题一：数据模型碎片化

不同工具、不同语言、不同团队对同一种概念可能有不同命名。

例如：

- 一个系统叫 `service`
- 一个系统叫 `app`
- 一个系统叫 `svc_name`

从短期看，这只是命名差异；从长期看，它会直接影响查询、聚合、告警和跨团队协作。

### 问题二：接入方式碎片化

如果每种后端都有一套完全不同的 SDK、字段名和接入方式，那么：

- 应用很难保持厂商中立
- 库作者很难写出可复用埋点
- 平台团队很难统一治理

### 问题三：三种信号无法自然关联

很多系统里 metrics、logs、traces 各自存在，但：

- 日志找不到 trace_id
- 指标维度和 trace 属性对不上
- traces 能看到请求链路，但无法和业务日志统一解释

OpenTelemetry 的价值之一，就是尝试把这三类信号放在更统一的模型中看待。

## 1.3 OpenTelemetry 不是什么

理解边界和理解能力同样重要。

### 它不是某个具体观测后端

OpenTelemetry 可以把数据发往多种后端，但它本身不是 Jaeger、Prometheus、Tempo、Loki，也不是某家云厂商的私有产品。

### 它不是“装了就自动一切都好”的魔法层

即使你用自动注入接上了 OpenTelemetry，以下问题仍然要自己面对：

- span 设计是否合理
- metrics 维度是否会爆 cardinality
- logs 是否真的可关联
- schema 是否长期稳定
- Collector 部署拓扑是否适合你的系统

### 它也不是只等于 tracing

OpenTelemetry 同时覆盖：

- traces
- metrics
- logs
- context propagation
- semantic conventions
- Collector pipeline

如果只把它看成 tracing SDK，就会忽略它在统一观测体系中的真正位置。

---

## 1.4 为什么厂商中立在工程里真的重要？

“厂商中立”听起来像抽象原则，但它在工程里有非常现实的价值。

### 1.4.1 降低应用与后端的耦合

如果应用代码直接写满某个后端的专有 SDK、专有字段和专有配置，那么：

- 迁移成本会非常高
- 多后端输出会变得别扭
- 共享库很难在不同团队复用

而如果应用主要依赖 OpenTelemetry API / SDK，把后端差异更多放到 Collector 和 exporter 层，系统就更容易治理。

### 1.4.2 更适合平台统一接入

大型组织通常不希望每个团队都自己选一套完全不同的埋点模型。平台更希望：

- 统一 service identity
- 统一 trace propagation
- 统一关键资源字段
- 统一 Collector 入口

OpenTelemetry 恰好更容易支撑这种统一化能力。

### 1.4.3 更适合多后端并存

现实中很常见的情况是：

- traces 给 A 后端
- metrics 给 B 系统
- logs 给 C 平台

如果数据入口不统一，多路输出会很快变得混乱。OpenTelemetry + Collector 的组合，则更适合做统一接入与后续分流。

## 1.5 “有监控”和“真正可观测”有什么区别？

这是初学阶段非常重要的分界。

### 只有监控，不一定可观测

如果你只有：

- CPU 利用率
- 请求量
- 错误率

那么你只能知道“系统看起来有问题”，却未必知道问题在哪一段链路、哪类请求、哪个版本、哪个租户。

### 只有日志，也不一定可观测

如果日志很多，但：

- 没有 trace_id
- 没有稳定字段
- 没有结构化 schema

那么排障时依旧要大量手工检索、猜测和拼接。

### 只有 trace，也不一定可观测

如果 trace 有了，但：

- 没有关键业务 span
- 没有稳定的 resource 和属性
- 没有对应的 metrics 和 logs

你仍然可能看得到链路，却解释不了系统为什么慢、为什么贵、为什么某类用户更容易失败。

### 真正可观测的核心

真正可观测，不是某一类数据量特别多，而是：

- 三类信号能互相补充
- 语义命名足够稳定
- 上下文能正确传播
- 数据足够支持诊断与决策

## 1.6 一个现实例子：为什么 trace 能纠正“错怪模型”的直觉

假设你维护一个 RAG 问答系统，用户反馈：

- “回答很慢”

如果你没有完整观测，最容易做出的粗糙判断是：

- 模型太慢
- GPU 不够

但如果有较好的 OpenTelemetry 体系，你可能会看到：

```text
gateway -> auth -> embed -> vector search -> rerank -> llm -> safety filter
```

然后发现：

- 真正慢的是向量检索
- 或者是下游 metadata 回源
- 或者是网关重试导致整体时延放大

这就是 OpenTelemetry 真正的价值之一：

- 它不是让你“多看一个面板”
- 而是让你更少误判系统瓶颈

## 1.7 什么时候值得引入 OpenTelemetry？

不是所有系统都必须一开始就上完整 OpenTelemetry 体系，但以下场景通常会越来越值得引入：

### 场景一：服务边界增多

一旦请求开始跨多个服务、多个进程、多个协议边界，就会越来越依赖统一传播与统一 schema。

### 场景二：你已经有多种观测工具，但仍然排障缓慢

这说明问题可能不在“工具不够多”，而在“观测模型不统一”。

### 场景三：团队需要统一治理

例如你需要：

- 统一 service.name
- 统一 trace propagation
- 统一 Collector 入口
- 统一 schema 和成本控制

### 场景四：希望保持后端可替换性

如果你希望未来能够：

- 更换后端
- 多路导出
- 统一接不同团队已有平台

那更早采用 OpenTelemetry 模型通常更稳妥。

## 本章小结

| 核心概念 | 要点 |
|----------|------|
| 引入动机 | 现代分布式系统需要统一的观测模型，而不是零散工具拼接 |
| OpenTelemetry 目标 | 统一 traces、metrics、logs 的接入与数据模型 |
| 它不是什么 | 不是某个具体后端，也不是只等于 tracing SDK |
| 厂商中立价值 | 降低耦合、支持平台统一治理、便于多后端并存 |
| 可观测性的关键 | 不是数据越多越好，而是数据能支持诊断与决策 |

### 实验 1：拆解一个“排障很慢”的系统

请回忆一个你熟悉的系统，列出它至少 5 个关键组件，例如：

- gateway
- auth
- order service
- payment service
- queue worker

然后思考：

1. 如果只有 metrics，你能定位到哪一层？
2. 如果只有 logs，你需要如何手工拼接？
3. 如果有完整 trace，你最希望看到哪几个边界？

### 实验 2：判断哪些问题更适合用 OTel 统一解决

对下面每个问题判断是否更适合通过 OpenTelemetry 体系来改善，并说明原因：

1. 单机脚本报错，只有一个进程
2. HTTP 请求跨 6 个微服务，偶发高延迟

4. 多个团队分别用不同字段表示服务名
