# 第2章：三种信号与 OTel 架构

## 学习目标

- 理解 traces、metrics、logs 三种信号分别擅长回答什么问题
- 建立“单次请求视角”与“长期聚合视角”之间的区别
- 说清 OTel API、SDK、OTLP、Collector、backend 在架构中的职责边界
- 看懂一条 telemetry 数据从应用到后端的大致流动路径
- 识别把 OTel 误解成“某个 exporter 配置”时最容易忽略的工程问题

## 1. 为什么 traces、metrics、logs 必须一起看

很多团队并不是完全没有观测数据，而是三类数据各自存在，却彼此像三套平行系统。

例如一个下单请求变慢时，你可能同时看到：

- metrics 面板上 `request_duration` 变高了
- tracing 页面里某些请求超过 2 秒
- 日志里出现库存服务超时与重试信息

如果这三类数据不能互相解释，排障时仍然要靠人工来回切换页面、猜测因果关系。OpenTelemetry 的意义之一，就是让你用更统一的模型理解这三类信号，而不是把它们当成完全独立的工具功能。

可以先建立一个最核心的直觉：

- **trace** 更像一次请求的执行叙事
- **metric** 更像一段时间内的聚合趋势
- **log** 更像某个时刻发生的离散事件或上下文记录

它们关注的是同一个系统，但观察尺度不同。

如果只有 metrics，你知道“系统整体变慢了”，却未必知道是哪一类请求、哪一段依赖导致的。
如果只有 trace，你能看懂某一次请求怎么走，但未必容易回答“今天 30 分钟内错误率是否整体升高”。
如果只有 logs，你可能有最细碎的上下文，但需要大量检索和拼接，才能还原整体路径。

因此，OpenTelemetry 不是把三种信号简单放在一起，而是试图让它们形成一种分工明确、可以互相补位的工程体系。

## 2. 三种信号分别擅长回答什么问题

把三种信号讲清，最好的方式不是背定义，而是看它们各自最擅长回答的问题。

| 信号 | 更擅长回答的问题 | 常见形态 | 主要优势 | 主要限制 |
|------|------------------|----------|----------|----------|
| traces | 某一次请求为什么慢、为什么失败、经过了哪些边界 | trace、span、event | 能还原单次执行路径 | 不适合直接做长期趋势统计 |
| metrics | 系统最近 1 小时、1 天、1 周整体表现如何 | counter、histogram、gauge | 聚合高效，适合告警和趋势分析 | 难以解释某个具体请求发生了什么 |
| logs | 某个时刻到底记录了什么上下文、错误详情和业务信息 | 结构化日志、事件日志 | 细节丰富，适合精确检索 | 噪音大，容易失去统一结构 |

### 2.1 traces：面向单次执行路径

trace 适合回答的问题通常是：

- 一个请求经过了哪些服务
- 哪个依赖调用最慢
- 错误发生在链路的哪一段
- 上下游的父子关系是否正确

它的优势是因果链清晰，但代价是数据更离散，也更依赖正确的上下文传播。

### 2.2 metrics：面向时间窗口与聚合

metric 更适合回答：

- 当前 QPS 是否上升
- 过去 15 分钟的 p95 延迟是否变高
- 某个消费者组的积压是否在扩大
- 某个依赖的错误率是否持续超阈值

它最大的价值是稳定、可聚合、适合告警，但它回答问题的方式是统计意义上的，而不是逐请求叙事式的。

### 2.3 logs：面向细节与解释

log 更适合回答：

- 这次失败时，业务输入大致是什么类别
- 某个分支为什么走到了回退逻辑
- 某个异常的原始错误文本是什么
- 某个租户、某个订单或某次任务到底发生了哪些离散事件

日志的长处是细节，但如果没有统一字段、没有 trace_id、没有稳定 schema，它就会很快变成“有很多信息，但很难系统使用”的噪音堆。

### 2.4 三种信号的协同关系

可以把它们想象成三种不同焦距的镜头：

- metrics 告诉你“哪里值得看”
- traces 告诉你“这次是怎么发生的”
- logs 告诉你“当时的细节是什么”

一个成熟的可观测体系，通常不是三类信号平均用力，而是让它们各自在最适合的位置承担职责。

## 3. OTel 架构里的几个核心部件

OpenTelemetry 容易让初学者困惑，主要是因为它既不是单一 SDK，也不是单一服务，而是一组协同工作的层次。

一个最常见的心智模型如下：

```text
业务代码
  -> OTel API
  -> OTel SDK
  -> OTLP exporter
  -> Collector
  -> backend 或 debug exporter
```

### 3.1 API：代码层的统一入口

API 提供的是调用接口，例如：

- 获取 tracer、meter、logger 的入口
- 上下文与传播接口
- span、attribute、baggage 等核心概念的访问方式

API 的重点是“如何表达 telemetry 语义”，而不是“如何把数据发出去”。

### 3.2 SDK：把语义变成真正的数据

SDK 负责把 API 层表达出来的 span、metric、log record 变成真正可采集、可导出的数据。它通常负责：

- provider 初始化
- exporter 配置
- 批处理与导出节奏
- 采样与聚合
- 资源信息装配

如果说 API 是接口，SDK 就是实现。

### 3.3 OTLP：一种统一传输方式

OTLP 是 OpenTelemetry 协议，用来在应用、Collector、后端之间传输 telemetry 数据。它的重要性在于：

- 让应用不需要针对每个后端学习完全不同的发送方式
- 让 Collector 可以成为统一中间层
- 让 traces、metrics、logs 都有相对一致的传输入口

需要特别注意的是：**OTLP 是协议，不是后端本身。**

### 3.4 Collector：独立的 telemetry pipeline

Collector 位于应用与后端之间，负责：

- 接收数据
- 批处理与限流保护
- 字段过滤、变换、脱敏
- 采样、路由、多路导出

它不是必须永远存在，但在真实工程里往往非常有价值，因为它把很多运行时治理能力从应用中抽离了出来。

### 3.5 backend：存储、查询、展示与告警

backend 负责把采集到的数据变成可查询、可展示、可告警的系统。它解决的是：

- 数据如何存储
- 查询怎么执行
- 图表和链路怎么展示
- 谁可以消费这些数据

OTel 本身不等于 backend。它更像 backend 之前的一层统一观测标准和管道体系。

## 4. 一条 telemetry 数据是怎么流动的

理解 OTel 架构，最重要的不是记住名词，而是知道数据为什么会按这个顺序流动。

假设一个 Node.js 服务处理一次 `POST /checkout` 请求，最小路径通常如下：

```text
客户端请求
  -> 应用中的 OTel SDK 创建 span / metric / log
  -> OTLP exporter 发送到 Collector
  -> Collector 做批处理与导出
  -> backend 接收并展示
```

把这条链路再拆细一点，可以看到每层解决的是不同问题：

1. **业务代码与框架层**：决定什么时候产生 span、记录哪些属性、输出哪些日志
2. **SDK 层**：决定如何采样、如何聚合、如何导出
3. **网络传输层**：通过 OTLP 把数据送到下游
4. **Collector 层**：决定是否过滤、变换、路由、批量发送
5. **backend 层**：决定如何存储、查询、展示与告警

这也解释了为什么很多问题不能只在一个地方解决。

例如：

- 业务 span 名称错了，Collector 无法凭空替你设计正确语义
- 指标标签高基数，backend 只能承受后果，修复点通常仍在应用或 Collector
- trace 传播断了，光改 exporter 地址通常不会有帮助

换句话说，OTel 架构的价值不只是“数据能走通”，而是让不同层次各司其职。

## 5. 看待 OTel 架构时最容易混淆的边界

### 5.1 OTel 不等于 tracing

OTel 覆盖的不只是 trace，还包括 metrics、logs、context propagation、semantic conventions 与 Collector pipeline。如果只把它当成 tracing SDK，就会误解它的大部分设计目标。

### 5.2 Collector 不等于必须，但经常值得有

理论上，应用也可以直接把 OTLP 数据发往兼容后端。但只要你需要：

- 多后端导出
- 统一脱敏与治理
- 集中采样与路由
- 更平滑地替换 backend

Collector 往往就会变得非常值得。

### 5.3 backend 不负责定义应用语义

backend 很擅长查询和展示，但它不应该决定你的 `service.name` 应该叫什么，也不应该替你决定业务 span 的边界。schema 设计和埋点建模的责任仍然主要在应用与平台侧。

### 5.4 应用不应该背负所有治理逻辑

如果每个服务都自己处理：

- 多路导出
- 过滤敏感字段
- 路由不同环境
- 控制批处理参数

那么应用会越来越重，也越来越难统一治理。Collector 存在的一个关键原因，就是让这些共性职责尽量从应用中剥离。

### 5.5 架构正确不代表观测自然就有价值

即使 API、SDK、Collector、backend 全都接上了，你仍然可能得到一套很难排障的数据。常见原因包括：

- span 设计没有业务解释力
- metric 维度失控
- logs 没有 trace_id
- 资源字段不稳定
- 语义命名混乱

所以真正重要的不是“我用了 OTel”，而是“我是否建立了可解释、可治理、可长期演进的 telemetry 体系”。

## 本章小结

| 主题 | 核心结论 |
|------|----------|
| 三种信号分工 | trace 看单次链路，metric 看聚合趋势，log 看细节上下文 |
| OTel API | 负责统一调用入口与语义表达 |
| OTel SDK | 负责采集、聚合、采样与导出实现 |
| OTLP | 是统一传输协议，不是后端本身 |
| Collector | 是独立的 telemetry pipeline，负责治理与中转 |
| backend | 负责存储、查询、展示与告警 |
| 架构理解重点 | 先看职责边界，再看配置细节 |

## OTel实验

### 实验目标

把一个你熟悉的分布式请求路径拆成 traces、metrics、logs 三种视角，并画出应用到 Collector 再到 backend 的数据流。

### 实验步骤

1. 选择一个真实或假想场景，例如“用户提交订单”或“用户发起搜索”。
2. 画出请求经过的 4-6 个组件，例如 gateway、auth、checkout、inventory、payment。
3. 对每个问题判断更适合依赖哪类信号：
   - 某一次请求为什么超时
   - 最近 10 分钟错误率是否升高
   - 支付失败时的原始错误详情是什么
4. 再画出 telemetry 管道：应用代码、SDK、OTLP、Collector、backend 分别位于哪里。
5. 标注每一层最应该负责的事情，以及最不应该承担的事情。

### 预期现象

- 你会发现三种信号虽然都在描述同一个系统，但观察尺度完全不同。
- 你会更容易看清应用、Collector、backend 之间的职责边界。
- 你会发现很多“数据没价值”的问题，其实不是后端问题，而是上游建模问题。
