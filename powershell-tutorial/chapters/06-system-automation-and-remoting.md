# 第6章：系统自动化与远程执行

## 本章解决什么问题

PowerShell 真正经常被使用的场景，不只是“查一下进程”或“改一下文件”，而是：

- 批量处理系统对象
- 在多台机器上执行任务
- 做只读检查、状态收集、定时任务和自动化运维

这时你最需要的，不只是命令会不会写，而是是否能判断：

- 作用范围
- 权限边界
- 远程目标
- 失败后如何观测

---

## 学习目标

完成本章后，你应能：

1. 使用 PowerShell 处理常见系统对象，如进程、服务和任务
2. 理解远程执行和本地执行的基本边界
3. 使用作业和脚本块组织异步或批量任务
4. 在自动化场景里保持对作用域和风险的清晰判断

---

## 6.1 常见系统管理对象

PowerShell 对系统管理友好的一个原因是：很多系统资源都能以对象形式拿到。

例如：

```powershell
Get-Process
Get-Service
Get-Date
Get-ChildItem
```

在 Windows 场景里，你还会经常碰到：

- 计划任务
- 注册表
- 事件日志

这意味着你的自动化脚本通常不是在处理“命令输出文本”，而是在处理一组结构明确的系统对象。

---

## 6.2 处理进程和服务时，先只读观察

例如先看 CPU 占用高的进程：

```powershell
Get-Process |
    Sort-Object CPU -Descending |
    Select-Object -First 5 Name, Id, CPU
```

再例如看运行中的服务：

```powershell
Get-Service |
    Where-Object Status -eq Running |
    Select-Object Name, Status
```

这类只读命令应该成为你所有自动化动作之前的默认起点。

---

## 6.3 管理命令越有副作用，越要先缩小范围

例如服务操作：

```powershell
Get-Service -Name Spooler
Stop-Service -Name Spooler -WhatIf
```

这里的稳妥顺序是：

1. 先确认对象是否找对
2. 再看是否支持 `-WhatIf`
3. 最后才真正执行

这套顺序以后在删除文件、停服务、远程执行时都适用。

---

## 6.4 远程执行的最小心智模型

PowerShell 远程执行常见的几种形式：

```powershell
Invoke-Command -ComputerName Server01 -ScriptBlock { Get-Service }
Enter-PSSession -ComputerName Server01
New-PSSession -ComputerName Server01
```

你现在先建立三层判断：

- `Invoke-Command`：像“远程运行一段命令再拿结果回来”
- `Enter-PSSession`：像“进入一台机器的交互式远程会话”
- `New-PSSession`：像“创建可复用的远程会话对象”

重点不是先全记住，而是先知道：

> 远程执行意味着你的脚本影响范围已经从“当前机器”扩展到了“其它机器”。

---

## 6.5 脚本块在自动化里的地位

很多自动化入口最终都依赖脚本块：

```powershell
Invoke-Command -ComputerName Server01 -ScriptBlock {
    Get-Process | Sort-Object CPU -Descending | Select-Object -First 3
}
```

这段脚本块本质上就是：

- 你希望在远端执行的逻辑
- 你的自动化单元

所以脚本块写得清不清楚，直接决定远程自动化是否可维护。

---

## 6.6 作业：把任务放到后台

最基础的后台作业示例：

```powershell
$job = Start-Job -ScriptBlock { Get-Process }
Receive-Job -Id $job.Id -Keep
```

作业适合：

- 时间较长的任务
- 不想阻塞当前交互式窗口的任务
- 需要分步收集结果的任务

但你要知道：

- 作业会引入额外状态管理
- 结果读取和错误查看都要显式处理

不能因为它“能后台跑”就把它当成自动化万能解法。

---

## 6.7 计划任务和定时执行

很多真实自动化需求都不是“现在跑一次”，而是“每天跑一次”。

在 Windows 场景下，你通常会把 PowerShell 脚本接到计划任务体系中。这个时候你需要额外考虑：

- 运行身份是谁
- 工作目录是什么
- 日志写到哪里
- 脚本失败如何被发现

也就是说，自动化一旦进入定时执行，脚本本身就不再是全部问题，运行环境也成了脚本的一部分。

---

## 6.8 自动化不是越自动越好，而是越可控越好

你写自动化脚本时，最应该问自己的问题是：

1. 我现在读的是哪些对象？
2. 我将修改哪些对象？
3. 影响范围只在本机，还是会扩散到远端？
4. 如果失败，我能看到什么日志和错误？

PowerShell 真正的高级，不是“敢远程批量改”，而是“知道什么时候先只读、什么时候先预演、什么时候根本不该立刻执行”。

---

## 常见误区

- **误区 1：能远程执行，就说明自动化成熟了。**
  远程只是能力扩大，不代表风险可控。

- **误区 2：作业就是更高级的脚本。**
  作业只是执行模型不同，状态和错误也更复杂。

- **误区 3：计划任务只要脚本能手工跑通就行。**
  定时执行时，身份、路径、日志都可能变化。

- **误区 4：系统管理脚本先写动作，再补观察。**
  更稳妥的顺序永远是先观察，再缩小范围，再执行。

## 本章小结

本章最重要的判断标准是：

> 系统自动化不是把命令变长，而是把作用域变大；作用域一旦变大，观察、日志和风险控制就必须同步升级。
