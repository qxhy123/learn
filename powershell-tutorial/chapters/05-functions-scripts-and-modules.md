# 第5章：函数、脚本与模块

## 本章解决什么问题

到这里，你已经能查命令、看对象、写基础逻辑了。下一步的问题是：

- 怎么把一组命令封装起来反复用
- 怎么给它传参数
- 怎么把个人脚本整理成可复用工具

这就是函数、脚本和模块的分工。

---

## 学习目标

完成本章后，你应能：

1. 编写带参数的函数和脚本
2. 区分函数、脚本、模块各自的职责
3. 设计更清晰的输入和输出
4. 理解 Profile 和模块加载的基本用途

---

## 5.1 函数是最小复用单元

一个简单函数示例：

```powershell
function Get-LargeFile {
    param(
        [string]$Path = ".",
        [int]$Top = 5
    )

    Get-ChildItem -Path $Path -File |
        Sort-Object Length -Descending |
        Select-Object -First $Top Name, Length, LastWriteTime
}
```

这个函数的价值在于：

- 你不必每次重新拼同样的管道
- 你把变化点显式做成了参数
- 输出仍然是对象，而不是难处理的拼接字符串

---

## 5.2 参数设计比“能跑”更重要

好的参数设计会让脚本更像工具。

你至少应优先考虑：

- 哪些值应该参数化
- 是否应该有默认值
- 参数名是否清楚
- 输出对象是否稳定

例如：

```powershell
Get-LargeFile -Path .\logs -Top 10
```

这类调用一眼就能看懂，比把路径和数量写死在脚本里强得多。

---

## 5.3 脚本是交付入口，不只是“更大的函数”

当你把命令保存成 `.ps1` 文件时，你就有了脚本入口。

例如：

```powershell
param(
    [string]$Path = ".",
    [int]$Top = 5
)

Get-ChildItem -Path $Path -File |
    Sort-Object Length -Descending |
    Select-Object -First $Top Name, Length
```

然后这样运行：

```powershell
.\Get-LargeFile.ps1 -Path .\logs -Top 10
```

脚本的职责更偏向：

- 对外提供执行入口
- 承载流程
- 组织多个函数和步骤

---

## 5.4 模块解决的是“边界”和“复用”

当你的函数开始变多时，继续把所有内容堆在一个脚本里就会很乱。

这时应该考虑模块：

- `.psm1` 放实现
- `.psd1` 放模块清单

哪怕你暂时不写完整清单，也可以先用最小模块形态：

```powershell
Import-Module .\MyTools\MyTools.psm1
Get-Command -Module MyTools
```

模块的真正价值不是“更高级”，而是：

- 给功能分边界
- 让导入与复用更清楚
- 更适合团队共享

---

## 5.5 输出设计：尽量输出对象，而不是只输出字符串

例如这两种方式：

```powershell
"File: $($file.Name), Size: $($file.Length)"
```

和：

```powershell
[pscustomobject]@{
    Name   = $file.Name
    Length = $file.Length
}
```

第二种通常更适合进入后续管道，因为它保留了结构。

这会直接影响后面的：

- 排序
- 筛选
- 导出 CSV / JSON
- 单元测试和团队复用

---

## 5.6 Profile 是个人环境入口，但不要把业务逻辑全塞进去

查看你的 Profile 路径：

```powershell
$PROFILE
```

Profile 适合做：

- 常用别名
- 提示符样式
- 自动导入个人模块
- 开发环境初始化

不适合做：

- 大量业务脚本逻辑
- 高风险系统改动
- 团队共享核心脚本的唯一入口

Profile 更像“个人环境配置”，而不是正式工具仓库。

---

## 5.7 帮助和注释也属于工具设计的一部分

如果你希望未来的自己或团队成员看得懂，你至少应做到：

- 函数名清楚
- 参数名清楚
- 输出对象字段稳定
- 有最基本的注释或帮助说明

PowerShell 的长期可维护性，不是从高级语法开始的，而是从这些基本约定开始的。

---

## 5.8 一个从命令到模块的能力升级路径

你可以把能力升级理解成 4 步：

1. 单条命令能跑
2. 多条命令能串起来
3. 串好的逻辑能变成函数
4. 函数能被模块化并复用

真正的进步通常不是“学会更多零散命令”，而是让这 4 步越来越自然。

---

## 常见误区

- **误区 1：函数只是为了少敲字。**
  更重要的是抽象变化点和稳定输出。

- **误区 2：脚本越长越像正式工具。**
  真正的工具需要边界、参数和输出设计。

- **误区 3：模块只有团队才需要。**
  个人项目也非常受益于清晰的模块边界。

- **误区 4：Profile 可以当万能启动脚本。**
  它更适合个人环境，而不是承载正式业务逻辑。

---

## 本章练习

1. 写一个 `Get-LargeFile` 函数，支持 `-Path` 和 `-Top` 参数。
2. 把它改成 `.ps1` 脚本入口，并运行一次。
3. 用 `[pscustomobject]` 输出结果对象，而不是只输出字符串。
4. 把函数放进一个最小 `.psm1` 模块中并导入。
5. 思考题：函数、脚本、模块三者分别在什么层面上提供复用？

本章最重要的判断标准是：

> PowerShell 进入进阶阶段的标志，不是会写更长的命令，而是开始认真设计参数、输出和复用边界。
