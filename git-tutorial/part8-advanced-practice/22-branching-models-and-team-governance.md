# 第22章：分支模型与团队治理

## 学习目标

完成本章后，你应能：

1. 理解 GitFlow、Trunk-Based 等常见分支模型的差异
2. 根据团队规模、发布节奏和风险偏好选择策略
3. 建立最基本的主干保护、审查、热修治理规则
4. 明白 Git 策略问题往往是工程治理问题

---

## 22.1 分支模型不是术语选择，而是交付策略选择

分支模型真正回答的问题包括：

- 日常开发如何集成
- 发布如何控制节奏
- 热修如何处理
- 谁可以直接改主干
- 历史是否允许重写

所以它不是“团队偏好题”，而是交付模型题。

---

## 22.2 GitFlow

GitFlow 通常包括：

- `main`
- `develop`
- feature 分支
- release 分支
- hotfix 分支

优点：

- 分工明确
- 发布路径清晰

代价：

- 分支较多
- 长生命周期分支容易积累集成成本

它更适合发布节奏明确、流程较重、版本线区分明显的团队。

---

## 22.3 Trunk-Based Development

Trunk-Based 更强调：

- 主干频繁集成
- feature 分支尽量短生命周期
- 小步提交、快速合并
- 自动化测试强

优点：

- 减少长期分叉
- 更适合持续交付

代价：

- 对 CI、测试通过率、回滚能力要求更高
- 团队纪律要求更强

它更适合交付频繁、测试基础较好的团队。

---

## 22.4 一个更决策化的比较方式

| 团队条件 | 更可能适合的策略 |
|----------|------------------|
| 发布节奏较慢、版本线清晰、热修频繁 | GitFlow 类方案 |
| 发布频繁、主干常态可部署、CI 强 | Trunk-Based |
| 多维护线并存、合规要求高 | 更重的分支与发布治理 |
| 小团队、变化快、回滚能力强 | 更轻的主干式策略 |

真正重要的不是名词，而是：

- 你的测试能不能撑得住频繁集成
- 你的回滚 SLA 能不能支撑主干快速前进
- 你的团队能否承受长期分叉带来的合并成本

---

## 22.5 治理不只是选模型，还包括规则

无论选哪种模型，都建议明确：

- 主干是否受保护
- 谁可以直接 push
- 是否必须通过 PR
- 是否要求 CI 通过
- 是否允许 force push
- 提交签名是否必须

没有这些规则，模型很快会失效。

---

## 22.6 一个最小团队治理模板

对多数中小团队，一个务实起点通常是：

1. 主干受保护，不允许直接 push
2. 所有改动通过 PR 合并
3. 要求 CI 通过
4. feature 分支短生命周期
5. 禁止对共享主干 force push
6. 统一合并策略

这套规则的收益，通常比争论“到底该叫 GitFlow 还是 Trunk-Based”更直接。

---

## 22.7 常见失败路径：模型说得很好，规则没落地

很多团队的问题并不是模型选错，而是：

- 名义上 trunk-based，实际上没有稳定 CI
- 名义上 GitFlow，实际上分支长期不合并
- 说保护主干，但所有人都有直推权限

这说明：分支模型只有和权限、检查、发布节奏一起落地，才真正有意义。

---

## 常见误区

- **误区 1：分支模型只是个人偏好。**
  它会直接影响交付节奏与事故处理成本。

- **误区 2：团队小就不用治理。**
  小团队更容易因为“大家都随便来”积累隐患。

- **误区 3：只定模型，不定规则。**
  没有配套规则，模型很快失效。

- **误区 4：只看优点，不看前提。**
  每种模型都有适用条件。
