# 第24章：签名、CI 与灾难恢复手册

## 学习目标

完成本章后，你应能：

1. 理解提交签名和标签签名的意义
2. 明白 Git 与 CI/CD 如何协同工作
3. 针对误操作、坏提交、回滚和历史定位建立灾难恢复手册
4. 知道何时使用 `reflog`、`revert`、`bisect`、`force-with-lease`

---

## 24.1 签名解决的不是“好不好看”，而是来源可信度

签名常见于：

- 提交签名
- 标签签名

它的核心价值不是“让 Git 更高级”，而是：

- 提高来源可信度
- 满足合规要求
- 在组织内建立更明确的变更责任链

换句话说，签名面对的是一种 threat model：

- 你是否需要证明某次提交确实来自某个身份
- 你是否需要区分“能 push”与“来源可信”是两回事

对个人项目来说未必刚需，但在高合规团队里往往重要。

---

## 24.2 Git 与 CI 的关系

Git 历史通常是 CI 的触发源：

- 提交触发构建
- PR 触发测试
- 标签触发发布

这也是为什么 Git 质量和发布质量高度相关：

- 历史乱，CI 定位难
- PR 边界乱，测试范围难判断
- 标签混乱，发布追踪难

CI 不是一个独立于 Git 的外部系统；很多时候它就是“团队对 Git 历史做自动化约束”的一部分。

---

## 24.3 灾难恢复不是“出事再想”，而是提前写手册

一个最小 Git 灾难恢复手册，至少应回答：

- 提交错了怎么办
- 合并错了怎么办
- 主干坏了怎么办
- 谁可以 force push
- 回滚优先 `revert` 还是改历史
- 如何定位是哪次提交引入故障

如果这些问题平时没有共识，事故时通常会更乱。

---

## 24.4 四类常见事故与分流入口

### 症状一：本地误删最近提交

观察命令：

```bash
git reflog
```

安全动作：

- 找回位置
- 先建救援分支

不要先做：

- 继续盲目 reset

### 症状二：共享主干引入坏提交

观察命令：

```bash
git log --oneline --graph --decorate --all
```

安全动作：

- 优先考虑 `git revert <commit>`

不要先做：

- 在主干上直接重写已共享历史

### 症状三：提交 PR 前本地历史太乱

安全动作：

- `git rebase -i`

前提：

- 这段历史尚未被团队广泛依赖

### 症状四：不确定哪次提交引入了问题

安全动作：

- `git bisect`

---

## 24.5 `git bisect`：在历史中二分定位故障

`git bisect` 的思路非常直接：

- 指定一个“好版本”
- 指定一个“坏版本”
- Git 在中间不断二分，让你测试
- 最终定位最可能引入问题的提交

基本流程：

```bash
git bisect start
git bisect bad
git bisect good <known-good-commit>
```

之后每一步根据测试结果标记：

```bash
git bisect good
git bisect bad
```

结束后记得：

```bash
git bisect reset
```

它之所以高阶，不是因为命令难，而是因为它要求你把“问题范围”转成“好/坏提交边界”。

---

## 24.6 force push 的底线

如果组织确实允许在某些分支改历史，建议至少遵守：

1. 优先 `--force-with-lease`
2. 禁止对受保护主干直接强推
3. 改历史前先通知相关协作者
4. 改完后明确说明如何同步

这说明 force push 真正的风险，不在于命令本身，而在于它会改变团队对远程历史的共同假设。

---

## 24.7 一份最小团队 Git 应急卡片

你可以把下面这组规则写进团队文档：

1. 误删本地历史先看 `reflog`
2. 共享分支回滚优先 `revert`
3. 定位引入问题的提交优先 `bisect`
4. 强推必须使用 `--force-with-lease`
5. 主干必须受保护并绑定 CI
6. 发布必须有标签，重大版本建议签名

这类卡片在真正出事故时比长文档更有价值，因为它提供的是分流入口，而不是背景科普。

---

## 24.8 一个高阶视角：Git 能力的终点不是更多命令，而是更稳的组织反应

从第 13 章到本章，你应该逐渐看到一件事：

- 高阶 Git 不是更会炫命令
- 而是更会在复杂状态下选择安全动作
- 更会把个人操作放进团队约束里理解

所以 Git 学到最后，本质上是在学：

- 状态判断
- 恢复策略
- 组织边界

---

## 常见误区

- **误区 1：签名只是形式主义。**
  在高合规团队里，它可能是必要约束。

- **误区 2：CI 是平台的事，和 Git 历史质量无关。**
  它们强相关。

- **误区 3：`bisect` 太高级，平时用不上。**
  一旦问题跨多次提交难以肉眼定位，它就非常有价值。

- **误区 4：事故处理主要靠记住某条命令。**
  真正关键的是先观察、再分流、最后执行。
