# 第23章：Monorepo、LFS 与大仓库性能

## 学习目标

完成本章后，你应能：

1. 理解 monorepo 与多仓库方案的主要权衡
2. 知道 Git LFS 解决什么问题
3. 识别大仓库常见的性能瓶颈
4. 建立适合大仓库的提交、同步和清理习惯

---

## 23.1 仓库变大后，问题会变

小仓库里你几乎感觉不到 Git 成本，但仓库足够大以后，问题会逐渐出现：

- clone 太慢
- fetch 太慢
- 历史太重
- 二进制资源膨胀
- CI 拉仓库时间过长

这时 Git 使用方式就必须升级，不然工具层面的不适会迅速变成团队层面的摩擦。

---

## 23.2 Monorepo 的优势与代价

Monorepo 的优势：

- 统一版本和提交历史
- 跨模块改动更自然
- 工具链、CI、代码搜索更统一

代价：

- 仓库规模增大
- 权限与边界治理更难
- 无关改动的噪音更高

所以 monorepo 不是“更先进”，而是另一种协作边界选择。

---

## 23.3 Git LFS 解决什么问题

Git 对文本和普通源码很擅长，但对大二进制文件不友好。

Git LFS（Large File Storage）的核心思路是：

- 在 Git 历史中保存指针
- 把大文件内容放到专门存储中

典型适用对象：

- 设计资源
- 模型权重
- 视频、音频、数据集

如果你直接把大量大文件塞进普通 Git 历史，仓库体积和同步成本会迅速恶化。

---

## 23.4 大仓库慢，通常先是治理问题

很多团队的 Git 变慢，不一定是工具到了极限，而是使用习惯出了问题：

- 把构建产物提交进历史
- 大量无意义格式化造成巨型 diff
- 二进制文件直接放进普通 Git 历史
- 分支和远程引用长期不清理

所以优化仓库性能的第一步，常常不是换工具，而是先修使用规范。

---

## 23.5 一个更决策化的比较视角

| 因素 | 倾向 monorepo | 倾向多仓库 |
|------|---------------|------------|
| 跨模块改动频繁 | 是 | 否 |
| 权限边界强 | 否 | 是 |
| 工具链希望统一 | 是 | 否 |
| 团队之间发布节奏差异大 | 否 | 是 |

类似地，是否需要 LFS，也不应只看“文件大不大”，还应看：

- 是否长期版本化
- 是否频繁变更
- 是否需要和源码一起严格同步

---

## 23.6 高阶习惯：不要等仓库已经很痛再补救

对大仓库来说，最有价值的日常习惯通常是：

1. 避免提交构建产物
2. 尽量不把大二进制直接放进普通 Git 历史
3. 减少无意义的巨型提交
4. 定期清理无用远程跟踪分支
5. 明确哪些内容应进 LFS、哪些应进对象存储、哪些根本不该版本化

这些习惯的效果，往往比事后补复杂基础设施更立竿见影。

---

## 常见误区

- **误区 1：monorepo 一定比多仓库高级。**
  它只是不同边界下的选择。

- **误区 2：Git 变慢只能换工具。**
  很多时候先要修治理和习惯。

- **误区 3：二进制文件和源码一样直接进 Git 就行。**
  这是最常见的历史膨胀来源之一。

- **误区 4：性能问题是纯技术问题。**
  仓库治理和团队行为往往是更深层的原因。
