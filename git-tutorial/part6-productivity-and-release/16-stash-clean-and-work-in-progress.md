# 第16章：stash、clean 与进行中的工作

## 学习目标

完成本章后，你应能：

1. 使用 `git stash` 临时保存未完成工作
2. 理解 `stash` 适合什么，不适合什么
3. 使用 `git clean` 清理未跟踪文件，并理解其风险
4. 在切换任务、修复紧急问题时保持仓库整洁

---

## 16.1 `stash` 解决的是“临时切走”，不是“长期保存”

`git stash` 适合的典型场景是：

- 正做一半，突然要切去修紧急 bug
- 当前改动还没整理成合适提交
- 想快速回到干净工作区做对比

最常用写法：

```bash
git stash push -m "wip: retry refactor"
```

但要记住：

> stash 是临时收纳，不是版本控制替代品。

如果一段工作已经能被说成“一件明确的事”，它通常更适合提交到分支。

---

## 16.2 一个实用决策树：stash、提交，还是 worktree

### 改动还没成型，但必须立刻切任务

优先考虑：

- `git stash push -m ...`

### 改动已经是完整小步

优先考虑：

- 直接提交到当前分支或临时分支

### 你需要长时间并行处理多个任务

后续更值得考虑：

- `git worktree`

也就是说，stash 不是“所有多任务切换问题”的通用答案。

---

## 16.3 stash 的基本工作流

保存当前工作：

```bash
git stash push -m "wip: login form"
```

查看 stash 列表：

```bash
git stash list
```

恢复但保留记录：

```bash
git stash apply stash@{0}
```

恢复并删除记录：

```bash
git stash pop
```

给 stash 写说明很重要，否则多了以后你很快会忘记自己存的是什么。

---

## 16.4 为什么 stash 也可能冲突

很多人误以为 stash 是绝对安全保险箱。其实不是。

如果你 stash 后，分支状态已经发生变化，再应用 stash 时也可能出现冲突。

这意味着：

- stash 保存的是一组改动
- 它不是冻结整个世界
- 应用回来时仍然要面对目标环境已经变化的现实

所以 stash 冲突时，正确动作仍然是：

```bash
git status
```

先看状态，再解决冲突。

---

## 16.5 `git clean`：清理未跟踪文件前，先预演

查看会删什么：

```bash
git clean -nd
```

真正删除：

```bash
git clean -fd
```

这个命令只处理未跟踪文件和目录，不会改已跟踪文件，但依然危险，因为：

- 被删内容通常不在 Git 历史里
- 很多实验目录、临时脚本、生成结果都可能属于未跟踪文件

所以推荐默认习惯是：

> 先 dry-run，再决定是否真的删。

---

## 16.6 一个常见失败路径：长期 stash

把所有 WIP 都丢进 stash，短期看省事，长期看会有几个问题：

- 不知道哪条 stash 对应什么工作
- 不知道哪些应该恢复、哪些可以丢弃
- 分支状态一变，应用冲突越来越多

如果你发现自己 stash 列表越来越长，通常意味着：

- 该提交的时候没提交
- 或者这个问题本该交给 worktree、临时分支来解决

---

## 16.7 建议练的三个实验

### 实验 1：中断当前工作去修 bug

1. 做一组未提交改动
2. `stash push -m`
3. 切分支修 bug
4. 回来 `apply` 或 `pop`

### 实验 2：清理未跟踪目录

1. 创建一个未跟踪目录
2. 先 `git clean -nd`
3. 确认后再 `git clean -fd`

### 实验 3：stash 冲突

1. stash 一组改动
2. 在目标分支上修改同一文件
3. 应用 stash
4. 观察冲突与 `git status`

---

## 常见误区

- **误区 1：所有未完成工作都该 stash。**
  很多完整的小步更适合直接提交。

- **误区 2：`git clean -fd` 很安全。**
  对未跟踪文件来说，它可能是不可逆删除。

- **误区 3：stash 一定不会冲突。**
  目标状态变了，照样会冲突。

- **误区 4：stash 可以长期替代分支管理。**
  它更适合短期切换，不适合长期并行开发。

1. 制造一组未提交改动，用 `git stash push -m` 暂存并恢复。

3. 创建一个未跟踪目录，先用 `git clean -nd` 预览，再决定是否删除。
4. 思考题：什么时候一个问题更适合用分支或 worktree，而不是 stash？
