# 第18章：blame、grep 与历史排障

## 学习目标

完成本章后，你应能：

1. 使用 `git blame`、`git grep`、`git log --grep` 辅助定位问题
2. 理解 Git 历史在排障中的价值
3. 区分“定位最近修改来源”和“定位真正根因”的不同
4. 建立从现象回溯到提交的排障习惯

---

## 18.1 Git 历史是排障线索，不只是档案

当某个问题出现时，你通常想知道：

- 它从什么时候开始坏的
- 哪次改动引入了它
- 这段代码是谁、在什么上下文下改的
- 是否有相关提交说明、PR 或 issue

Git 的强项，不在于“保存过去”，而在于让你从当前现象回到历史上下文。

---

## 18.2 一个实用的排障决策流

### 我知道关键词、配置项或字符串

优先：

```bash
git grep "keyword"
```

### 我知道具体文件或某一行可疑

优先：

```bash
git blame path/to/file
git show <commit>
```

### 我怀疑某个提交说明提到过相关修复

优先：

```bash
git log --grep="keyword"
```

### 我已经怀疑是历史回归，但线索不够

下一步考虑：

- `git bisect`

也就是说，`grep`、`blame`、`show`、`log --grep` 最好被看成一条链，而不是散乱命令。

---

## 18.3 `git blame`：看某一行最近是谁改的

```bash
git blame src/config.ts
```

它会显示每一行最近一次修改对应的提交和作者。

但要注意：

- blame 只能告诉你“最近谁改了这一行”
- 它不等于这人就是问题根因
- 它更适合当“线索入口”，而不是责任归属系统

正确用法通常是：

1. 先 `blame`
2. 找到相关提交
3. 再 `git show <commit>` 看上下文

---

## 18.4 `git grep`：在仓库内容里找入口

```bash
git grep "timeout"
git grep "feature_flag" HEAD~3
```

它特别适合快速定位：

- 某个配置项定义在哪
- 某个接口名出现在哪里
- 某个字符串在某个版本里是否存在

与普通 `grep` 相比，`git grep` 更贴近仓库视角，尤其适合配合历史版本一起查。

---

## 18.5 `git log --grep` 与按文件查看历史

按提交说明搜索：

```bash
git log --grep="timeout"
```

按文件查看：

```bash
git log -- src/auth.ts
```

这能把“好像最近改过”变成：

- 改的是哪次提交
- 提交说明写了什么
- 是否能在 PR、issue 或设计讨论中找到背景

---

## 18.6 为什么 blame 不是“甩锅工具”

很多团队把 `git blame` 用坏了，因为他们把它变成了“找谁背锅”。

更合理的用法是：

- 找到相关提交上下文
- 理解当时为什么这么改
- 追踪问题链路
- 关联 PR、issue、设计讨论

Git 历史是帮助你理解系统，不是帮助你制造对立。

---

## 18.7 一个更完整的历史排障实验

建议你做下面这组实验：

1. 在一个文件里多次修改同一行
2. 用 `git blame` 找最近修改来源
3. 用 `git show` 看该提交上下文
4. 用 `git grep` 搜索某个配置键
5. 用 `git log --grep` 搜索某类修复说明

练完之后，你应该能把“我觉得最近改坏了”变成“是这次提交在这个上下文下引入了变化”。

---

## 常见误区

- **误区 1：`git blame` 能直接告诉你根因。**
  它只能提供最近修改线索。

- **误区 2：排障只看当前代码就够了。**
  很多问题只有放回历史里才说得通。

- **误区 3：`git grep` 和普通 `grep` 完全一样。**
  `git grep` 更贴近仓库内容和版本语境。

- **误区 4：提交说明写得很差也没关系。**
  后面做历史排障时会付出代价。
