# 第17章：标签、发布与版本语义

## 学习目标

完成本章后，你应能：

1. 理解标签和分支的区别
2. 创建轻量标签与附注标签
3. 用标签标记发布版本
4. 理解为什么正式版本不该依赖“记忆中的某次提交”

---

## 17.1 标签是什么

分支是会继续向前移动的引用；

标签（tag）更像一个稳定路标，用来标记“某个重要提交”。

典型用途：

- 发布版本
- 标记里程碑
- 记录可回滚基线

如果分支表示“开发线”，那么标签更像“历史上的固定坐标”。

---

## 17.2 轻量标签与附注标签

### 轻量标签

```bash
git tag v1.0.0
```

更像简单别名。

### 附注标签

```bash
git tag -a v1.0.0 -m "Release v1.0.0"
```

包含作者、日期和说明，更适合正式发布。

大多数团队项目里，正式版本建议优先使用附注标签。

---

## 17.3 标签为什么比“记住某次提交”更可靠

如果你只说“1.0 版大概是 main 上某个位置”，后续会出现很多问题：

- 无法稳定回溯
- 自动化脚本难以绑定
- 回滚和审计成本增加

标签的价值在于：

- 发布点清晰
- CI/CD 更容易触发
- 发布说明更容易关联
- 回滚与二分定位更可靠

---

## 17.4 查看和推送标签

查看所有标签：

```bash
git tag
```

查看某个标签指向什么：

```bash
git show v1.0.0
```

推送标签：

```bash
git push origin v1.0.0
```

推送所有标签：

```bash
git push origin --tags
```

一个很好的习惯是：打完标签后先 `git show` 检查，不要想当然地认为自己标对了位置。

---

## 17.5 标签与分支的边界

| 对象 | 特性 |
|------|------|
| 分支 | 会继续前进，表示持续开发线 |
| 标签 | 应尽量稳定，表示某个固定版本点 |

如果你把标签当成可随便移动的分支来用，会直接破坏：

- 发布可追溯性
- 自动化发布稳定性
- 回滚准确性

所以正式发布后的标签，不应轻易改动。

---

## 17.6 错误打标签时该怎么处理

如果标签打错了位置，不要装作没事。正确做法通常是：

1. 明确当前错误标签是否已经推送
2. 如果已推送，确认团队和自动化是否已依赖它
3. 再决定是删除重打，还是发布新版本号

这也是为什么正式发布流程里，标签应该被当成受控动作，而不是“最后顺手打一条命令”。

---

## 17.7 一个最小发布实验

建议在练习仓库中做：

1. 创建 `v1.0.0` 附注标签
2. 用 `git show v1.0.0` 检查它的内容
3. 再做一个修复提交并打 `v1.0.1`
4. 观察历史图中标签分别挂在哪里

如果想再往前一步，可以模拟：

- 从旧标签切出 hotfix 分支
- 修复问题
- 打新 patch 版本标签

---

## 常见误区

- **误区 1：标签和分支差不多。**
  分支会移动，标签通常不该随意移动。

- **误区 2：发布版本只要记住某次提交就够了。**
  团队协作和自动化更需要稳定锚点。

- **误区 3：打标签后就万事大吉。**
  标签是发布锚点，不是完整发布流程本身。

- **误区 4：轻量标签和附注标签只是写法不同。**
  它们承载的正式程度和元数据量不同。

## 本章练习

1. 在练习仓库中创建一个附注标签，并用 `git show` 查看其内容。
2. 连续做两次提交并分别打标签，比较它们在历史图中的位置。
3. 为一个虚拟项目设计 `1.0.0 -> 1.1.0 -> 1.1.1` 的发布节奏。
4. 思考题：为什么正式发布更需要稳定标签，而不是“差不多就是这一版”？

## 本章小结

标签让 Git 历史第一次真正具备“发布语义”。下一章我们继续学习如何用历史排障，而不是只把 Git 当作存档系统。
