# 第19章：objects、refs 与 packfiles

## 学习目标

完成本章后，你应能：

1. 理解 Git 中 blob、tree、commit、tag 四类核心对象
2. 理解 refs、branches、HEAD 在底层是如何关联的
3. 知道 packfiles 为什么存在
4. 用内部模型解释“为什么提交是快照、分支是引用”

---

## 19.1 Git 对象模型的四个角色

### blob

保存文件内容本身，不直接保存文件名。

### tree

描述目录结构，把文件名映射到 blob 或子 tree。

### commit

记录：

- 指向哪个 tree
- 父提交是谁
- 作者与提交者信息
- 提交说明

### tag

用于给某个对象（通常是提交）打稳定标签。

这四类对象的组合，构成了 Git 历史的基本语言。

---

## 19.2 为什么说 Git 管理的是快照

一次提交最终会指向一个 tree，而 tree 会递归描述整个目录在那个时刻的组织方式。

因此从学习者视角说：

- blob 是内容
- tree 是结构
- commit 是一个项目状态的入口点

这也是为什么：

- commit 更像项目状态快照
- branch 只需移动引用，不必复制目录
- rebase 会生成新 commit，而不是“改旧 commit”

---

## 19.3 refs：给对象起人类可读的名字

哈希值稳定，但不适合人类直接记忆。

所以 Git 用 refs 给对象起名字，例如：

- `refs/heads/main`
- `refs/heads/feature/login`
- `refs/tags/v1.0.0`

分支本质上是一种 ref；HEAD 则通常指向某个 ref。

你在高阶使用 Git 时，很多现象本质上都是“对象没变，但引用变了”。

---

## 19.4 用底层视角重新看 branch 和 HEAD

当你在 `main` 上新建提交时，本质上发生的是：

1. 创建新 commit 对象
2. 更新 `refs/heads/main`
3. HEAD 继续指向 `main`

所以“分支是指针”不是为了方便教学的比喻，而是很接近底层事实的说法。

而 detached HEAD 则意味着：

- HEAD 不再通过 branch ref 间接指向提交
- 它直接指向某个 commit 对象

---

## 19.5 把对象和引用“看出来”

如果你想让这一章不止停留在概念层，可以做一些只读观察：

```bash
git rev-parse HEAD
git cat-file -p HEAD
git show-ref --heads --tags
```

这些命令会帮助你看到：

- 当前提交 ID
- 提交对象内部包含哪些字段
- 分支和标签究竟对应哪些引用

当你真看过这些内容后，“对象 + 引用”的模型会比背概念稳得多。

---

## 19.6 packfiles：为什么仓库不会无限膨胀

Git 在存储上会做压缩和打包，把多个对象组织成 packfiles，以提高：

- 存储效率
- 网络传输效率
- 历史遍历效率

这提醒你一件事：

> 学习者按“快照”理解 Git 是对的，但实现层会用更高效的方式保存这些快照关系。

所以“Git 是快照模型”与“Git 存储很高效”并不矛盾。

---

## 19.7 为什么懂一点内部原理会显著提高判断力

知道对象与引用关系后，下面这些问题都会更容易解释：

- 为什么创建分支几乎是瞬时的
- 为什么 rebase 会生成新提交
- 为什么标签和分支性质不同
- 为什么对象失去引用后不一定立刻消失
- 为什么 reflog 能帮你找回“曾经指向过的地方”

你不需要一开始就深入源码，但不理解对象与引用，很难真正进入 Git 的高阶层。

---

## 常见误区

- **误区 1：commit 直接保存了一堆补丁。**
  对使用者来说，它更接近“一个项目状态入口”。

- **误区 2：分支是对象副本。**
  更准确地说，它是对象的命名引用。

- **误区 3：packfiles 推翻了快照模型。**
  不，快照是理解模型，packfiles 是实现优化。

- **误区 4：底层原理没必要学。**
  不懂底层也能用，但很难解释边界。
