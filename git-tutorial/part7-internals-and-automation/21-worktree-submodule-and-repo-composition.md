# 第21章：worktree、submodule 与仓库组合

## 学习目标

完成本章后，你应能：

1. 理解 `git worktree` 解决什么问题
2. 知道 submodule 的基本机制与使用成本
3. 区分 submodule、subtree、monorepo 等仓库组织方式
4. 根据团队边界选择更合适的仓库组合策略

---

## 21.1 `git worktree`：同一仓库，多份工作目录

`worktree` 的用途是：

> 让同一个 Git 仓库同时拥有多个检出工作目录。

典型场景：

- 一边修 hotfix，一边继续功能开发
- 需要同时对比多个分支状态
- 不想反复 stash / switch

例如：

```bash
git worktree add ../project-hotfix hotfix/payment
```

这会在新目录中检出指定分支。

---

## 21.2 为什么 worktree 往往比频繁 stash 更稳

如果你经常在多个任务间来回切：

- stash 会堆积
- 切换分支会被未提交改动阻塞
- 脑内上下文切换成本高

`worktree` 的优势是把不同任务直接分到不同工作目录，让状态更清晰。

所以：

- **短期临时切走**：stash 够用
- **长时间并行工作**：worktree 往往更稳

---

## 21.3 submodule：把另一个仓库嵌进当前仓库

submodule 的典型场景是：

- 某个组件有独立生命周期
- 主项目只想固定引用它的某个版本

本质上，主仓库记录的是：

- 子仓库当前指向哪个提交

它的好处是边界明确，但代价也很明显：

- 克隆和更新更复杂
- 新成员容易忘记初始化
- 日常操作心智负担更高

---

## 21.4 什么时候不该轻易上 submodule

如果你的真实需求只是：

- 想共享一段内部代码
- 想减少仓库大小
- 想让目录结构看起来更模块化

submodule 往往未必是最佳答案。

很多时候：

- `worktree` 解决的是并行开发
- `subtree` 解决的是代码引入
- monorepo 解决的是统一演化

它们比 submodule 更接近问题本质。

---

## 21.5 仓库组合方式其实是在选择协作边界

| 方式 | 更适合解决什么 | 主要代价 |
|------|----------------|----------|
| `worktree` | 同仓库多任务并行 | 额外工作目录管理 |
| submodule | 组件独立演化、版本绑定明确 | 操作复杂、维护心智负担高 |
| subtree | 想引入外部仓库代码但保持主仓体验 | 同步策略要明确 |
| monorepo | 强调统一演化、统一工具链 | 仓库规模与治理成本上升 |

这张表最重要的信息是：这些方案不是层级关系，而是不同边界下的折中。

---

## 21.6 高阶视角：机制之外的长期维护债务

真正让 submodule、subtree、monorepo 变难的，往往不是命令，而是长期维护问题：

- 谁负责升级依赖版本
- 谁决定组件边界
- 权限和发布节奏如何配合
- 跨团队改动如何审查

所以仓库组合方式从来不只是目录结构设计，它几乎总是组织边界设计。

---

## 常见误区

- **误区 1：有多个任务就只能频繁 stash。**
  `worktree` 往往更清晰。

- **误区 2：submodule 是组织仓库的默认高级方案。**
  它只是特定边界下的折中。

- **误区 3：仓库结构只是目录布局问题。**
  实际上它决定协作方式、版本边界和长期维护成本。

- **误区 4：技术上能做，就说明组织上适合。**
  组合策略必须同时看团队结构和演化频率。
