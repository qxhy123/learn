# 第11章：pull、push 与同步策略

## 学习目标

完成本章后，你应能：

1. 理解 `git pull` 本质上做了什么
2. 区分本地领先、远程领先、双方都前进等典型状态
3. 处理常见的 push 被拒绝问题
4. 根据团队策略选择 merge 或 rebase 同步方式

---

## 11.1 `git pull` 其实是两个动作

默认情况下：

```bash
git pull
```

大致等于：

```bash
git fetch
git merge
```

如果配置或参数要求 rebase，则第二步可能变成 rebase。

所以理解 `pull` 的关键，不是把它背成一个原子动作，而是先拆开理解：

- 先同步远程状态
- 再整合历史

---

## 11.2 先判断，再同步：四种典型状态

### 远程领先

你本地没新提交，远程有新提交。通常只需要把本地追上去。

### 本地领先

你有新提交，远程没有。通常可以直接 `push`。

### 双方都前进

你和远程都有新提交。此时需要 merge 或 rebase。

### 远程历史被改写

远程可能已经不是你以为的那条线了，此时需要更谨慎地判断是谁改了历史、为什么改、你是否应该跟着重写。

---

## 11.3 一条实用同步决策树

在执行 `pull` 之前，建议先过一遍：

1. 我当前在哪个分支？
2. 这个分支是否共享给团队？
3. 本地和远程谁领先？
4. 团队默认是 merge pull 还是 rebase pull？
5. 如果同步后要 push，是否会涉及重写历史？

所以一个稳妥流程通常是：

```bash
git status -sb
git fetch
git log --oneline --graph --decorate --all
```

看清楚再决定要不要 `pull`、用 merge 还是 rebase。

---

## 11.4 push 被拒绝时，Git 在保护什么

常见的 push 拒绝，本质是在告诉你：

> 远程分支上有你本地没有的历史，Git 不允许你直接把它覆盖。

这叫 non-fast-forward 拒绝。

它保护的是：

- 别人已经 push 上去的提交
- 团队共享的历史完整性
- 远程分支不会被你无意间向后改写

所以看到拒绝后，第一反应不该是：

```bash
git push --force
```

而应该是先 `fetch`，再看图，最后判断。

---

## 11.5 merge 同步与 rebase 同步

### merge 同步

```bash
git pull
```

优点：

- 保留真实分叉历史
- 对共享分支更保守

缺点：

- 会产生更多同步型 merge commit

### rebase 同步

```bash
git pull --rebase
```

优点：

- 本地历史更线性
- 减少同步型 merge commit

缺点：

- 会重写本地未推送提交
- 冲突时对初学者更不透明

所以是否默认 `pull --rebase`，本质上是团队历史风格与风险偏好的选择。

---

## 11.6 `--force-with-lease` 的边界

如果你确实在个人分支上做了 rebase，后续 push 可能需要：

```bash
git push --force-with-lease
```

相比 `--force`，它多了一层检查：远程是否还是你以为的状态。

但它也不是免责按钮。使用前仍要确认：

- 这是允许改写的分支吗
- 有人已经基于旧历史工作了吗
- 你是否已经和协作者沟通过

---

## 11.7 团队需要统一哪些默认习惯

如果团队里有人默认 merge pull，有人默认 rebase pull，主干历史风格会长期漂移。

建议至少统一：

- `pull.rebase` 默认值
- 是否允许 feature 分支 force push
- 主干是否受保护
- 何时允许改写已开 PR 的历史

很多所谓“Git 很乱”，其实不是工具乱，而是团队默认值没统一。

---

## 常见误区

- **误区 1：`pull` 只是“下载一下最新代码”。**
  它通常还会整合历史。

- **误区 2：push 被拒绝说明仓库坏了。**
  多数只是远程历史已经前进。

- **误区 3：`--force` 是万能修复按钮。**
  它可能破坏共享历史。

- **误区 4：同步策略是个人喜好。**
  在团队里，它会直接影响主干历史形态。

## 本章练习

1. 模拟一个远程仓库，让另一个本地克隆先提交并 push，再观察第一个仓库 push 被拒绝的场景。
2. 分别尝试 merge 同步和 rebase 同步，比较历史图差异。
3. 使用 `git fetch` 后，再用 `git branch -vv` 观察领先/落后状态。
4. 思考题：为什么说 non-fast-forward 拒绝是在保护共享历史？
