# 第10章：远程仓库、跟踪分支与 fetch

## 学习目标

完成本章后，你应能：

1. 理解远程仓库在 Git 中扮演什么角色
2. 分清本地分支和远程跟踪分支
3. 使用 `git remote`、`git fetch`、`git branch -vv` 查看同步关系
4. 明白为什么 `fetch` 是最安全的同步起点

---

## 10.1 远程仓库不是“Git 本体”，而是协作同步点

由于 Git 是分布式版本控制系统，本地仓库本身就有完整历史。远程仓库的核心作用是：

- 作为团队共享入口
- 提供协作、备份、审查和 CI 触发点
- 让不同开发者交换历史

所以远程仓库很重要，但它不是 Git 唯一的真相源，也不是本地状态的自动镜像。

---

## 10.2 `origin` 只是一个名字，不是语法关键字

当你执行：

```bash
git clone <url>
```

Git 通常会默认给这个远程仓库起名为 `origin`。

查看远程：

```bash
git remote -v
```

你可能会看到：

```text
origin  git@github.com:team/project.git (fetch)
origin  git@github.com:team/project.git (push)
```

这个名字可以改，也可以同时存在多个远程。

---

## 10.3 本地分支与远程跟踪分支

这里最容易混淆：

- `main`：你的本地分支
- `origin/main`：你本地记录的“上次同步到的远程 main 状态”

`origin/main` 不是远程服务器里的真实分支对象本身，而是你本地保存的一份**远程状态快照**。

这也意味着：

> 如果你很久没 `fetch`，本地的 `origin/main` 也可能已经过期。

---

## 10.4 `git fetch` 做了什么，不做什么

```bash
git fetch origin
```

它会：

- 从远程下载新的对象和引用信息
- 更新 `origin/*` 这类远程跟踪分支

它不会：

- 自动改动你的工作区
- 自动改动当前本地分支
- 自动创建 merge commit

这就是为什么 `fetch` 常被视为最安全的同步起点。

---

## 10.5 fetch 之后你应该看什么

推荐立即看两类信息：

```bash
git branch -vv
git log --oneline --graph --decorate --all
```

它们会帮你看出：

- 当前分支是否领先或落后远程
- 本地分支跟踪的是哪个远程分支
- 当前历史图中，本地和远程分别站在哪

也就是说，`fetch` 的价值不只是“拿更新”，更是让你获得**判断同步策略的观察前提**。

---

## 10.6 不只一个远程：fork 与 upstream 场景

在开源协作或 fork 模式中，常见状态可能是：

- `origin`：你的个人 fork
- `upstream`：原始主仓库

这时如果你搞混远程，很容易出现：

- 本来想同步主仓，结果推到了自己 fork
- 本来想给个人分支 push，结果误把东西推到错误远程

所以远程名字不是装饰，而是协作语义的一部分。

---

## 10.7 upstream tracking branch 是怎么建立的

当本地分支和远程分支建立跟踪关系后，你才能更自然地使用：

```bash
git pull
git push
git branch -vv
```

否则 Git 并不知道“你当前分支默认对应哪个远程分支”。

这一点在你开始使用多个远程或多个个人功能分支时会特别明显。

---

## 常见误区

- **误区 1：`origin/main` 就是远程服务器本身。**
  它是你本地记录的远程状态。

- **误区 2：只要本地有 `origin/main`，就说明自己知道远程最新状态。**
  如果没 `fetch`，它可能早就过期了。

- **误区 3：`fetch` 会自动把代码并到当前分支。**
  不会，它只更新远程状态快照。

- **误区 4：一个仓库里通常只会有一个远程。**
  fork、镜像、多源同步场景里并不成立。

---

## 本章练习

1. 在一个克隆仓库中执行 `git remote -v`，确认 `origin` 地址。
2. 执行 `git fetch` 后，用 `git branch -vv` 观察本地分支与远程跟踪分支的关系。
3. 给一个仓库添加第二个远程，思考它在协作中可能扮演什么角色。
4. 思考题：为什么说 `fetch` 是最安全的同步动作，而不是“半个 pull”？

---

## 本章小结

到这里你应该建立一个重要意识：远程状态不是自动最新，本地跟踪分支也不是远程本体。下一章我们继续处理真正高频、也最容易出错的同步动作：pull 与 push。
