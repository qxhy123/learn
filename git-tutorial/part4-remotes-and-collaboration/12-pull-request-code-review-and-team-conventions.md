# 第12章：Pull Request、Code Review 与团队约定

## 学习目标

完成本章后，你应能：

1. 理解 Pull Request 在团队协作中的作用
2. 知道如何准备一个可 review 的分支历史
3. 建立常见的分支命名、提交规范和合并约定
4. 理解 PR 是历史治理接口，而不只是审代码页面

---

## 12.1 PR 是协作接口，不只是“提个页面”

PR（或 Merge Request）通常承担几件事：

- 展示这次改动的范围和上下文
- 触发 CI 检查
- 让同事进行代码审查
- 记录为什么要这样改
- 决定以什么方式并入主干

所以一个好的 PR 不是“代码写完了”，而是“这组历史已经整理成可讨论、可验证、可合并的单元”。

---

## 12.2 PR 为什么也是历史治理接口

PR 不只是给人看代码，它还会长期影响：

- 主干历史如何演化
- 哪些提交被保留，哪些被压缩
- review 记录与提交之间如何对应
- 后续回滚和排障时能否看懂上下文

如果你把 PR 只当成“网页上点个按钮”，就会忽略它在治理层面的价值。

---

## 12.3 什么样的 PR 更容易被 review

通常具备这些特征：

- 主题单一
- 提交数适中
- 历史干净，明显调试提交已整理
- 描述中说明背景、方案、风险和验证方式
- 没有顺手带入无关重构

PR 太大时，即使代码是对的，review 质量也会显著下降。

---

## 12.4 PR 前自检：不要把整理成本转嫁给 reviewer

在发起 PR 之前，建议过一遍：

1. 是否基于最新主干同步过
2. 是否删除了明显无意义的试验提交
3. `git diff main...HEAD` 的内容是不是就是本次主题
4. 测试、格式化、静态检查是否已通过
5. PR 描述是否说明背景、验证方法和风险点

一个 reviewer 最不应该替你做的事情，就是帮你从杂乱历史里猜“你到底想提交什么”。

---

## 12.5 常见团队约定

### 分支命名

例如：

- `feature/login-page`
- `fix/session-timeout`
- `hotfix/payment-rollback`
- `docs/git-tutorial`

### 提交规范

常见形式：

- `feat: ...`
- `fix: ...`
- `docs: ...`
- `refactor: ...`

### 合并策略

团队通常会统一选择：

- merge commit
- squash merge
- rebase merge

没有普适唯一答案，但必须统一边界。

---

## 12.6 三种常见合并策略的长期影响

### merge commit

- 保留真实分叉历史
- 更适合强调主题汇入点的团队

### squash merge

- 主干更整洁
- 但会丢失功能分支内的细粒度提交结构

### rebase merge

- 历史线性
- 但会重写提交身份，review 期间需要更明确的约定

所以合并策略不是“偏好设置”，而是主干历史治理选择。

---

## 12.7 Review 期间可以改历史到什么程度

这也是团队常见分歧来源。

需要明确的问题包括：

- PR 打开后，是否允许继续 force push
- reviewer 是否按“增量差异”阅读，还是只看最终 diff
- 评论是否要求逐条 resolve
- 哪些受保护分支上任何人都不能改历史

如果这些不讲清楚，很多团队会在“技术上能做”和“协作上该不该做”之间反复摩擦。

---

## 12.8 Protected Branch、必过检查与合并权限

一个最小团队治理模板通常包括：

1. 主干受保护，不允许直接 push
2. 所有改动通过 PR 合并
3. CI 必须通过
4. 合并权限有明确边界
5. 是否允许 squash / rebase / merge commit 有统一规则

没有这些规则，PR 很容易退化成“流程存在，但治理不存在”。

---

## 常见误区

- **误区 1：PR 只是流程，不用花心思整理。**
  它是团队协作和历史治理的关键接口。

- **误区 2：CI 绿了，PR 就一定准备好了。**
  CI 只能证明某些检查通过，不能替代历史质量和边界清晰度。

- **误区 3：分支命名和提交规范只是形式主义。**
  在多人团队里，它们直接决定沟通成本。

- **误区 4：PR 打开后随便 force push 没关系。**
  如果 reviewer 正在按旧历史阅读，代价会立刻体现出来。

---

## 本章练习

1. 为一个虚拟功能设计分支名、3 条提交说明和一段 PR 描述。
2. 用 `git diff main...HEAD` 检查你的功能分支变更范围。
3. 比较 merge commit、squash merge、rebase merge 三种策略对主干历史的影响。
4. 思考题：为什么说 PR 是“历史治理接口”，而不只是“审代码页面”？

---

## 本章小结

到这里，你应该开始意识到：Git 不只是个人命令工具，它还是团队协作协议的一部分。下一部分我们进入很多人最关心、也最容易搞混的一组主题：撤销、回退与恢复。
