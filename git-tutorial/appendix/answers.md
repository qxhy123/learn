# 附录D：练习答案索引

本附录不追求替代你亲手操作，而是给出每章练习的关键观察点、答案方向和恢复提示。

---

## 基础段：第 1-6 章

### 第1章

- 关键点：版本控制不只是备份，还要记录差异、历史和协作关系。
- 观察点：解释清楚仓库、提交、分支、HEAD 的角色。
- 更进一步：能说明 Git 的分布式模型为什么让本地也拥有完整历史。

### 第2章

- 关键点：`git init` 只初始化仓库，不会自动提交。
- 观察点：`.git/` 目录出现，`git log --oneline` 能看到第一条提交。
- 更进一步：用 `git config --show-origin --list` 看出配置来源。

### 第3章

- 关键点：`git add` 是进索引，`git commit` 是把索引写入历史。
- 观察点：`git diff` 与 `git diff --cached` 看到的内容不同。
- 更进一步：用 `git ls-files --stage` 证明索引真实存在。

### 第4章

- 关键点：提交前一定要检查 staged diff。
- 观察点：`git add -p` 可以只暂存部分代码块。
- 更进一步：能说明 `git diff`、`git diff --cached`、`git diff HEAD` 分别回答什么问题。

### 第5章

- 关键点：`git log --oneline --graph --decorate --all` 是阅读历史的高频入口。
- 观察点：能解释提交图中分叉和汇合是什么意思。
- 更进一步：能根据问题切换图视图、文件视图、单提交视图。

### 第6章

- 关键点：`.gitignore` 不会自动取消已跟踪文件。
- 观察点：`git rm --cached` 只移出索引，不删除工作区文件。
- 更进一步：会用 `git check-ignore -v` 排查忽略规则。

---

## 协作段：第 7-12 章

### 第7章

- 关键点：分支是可移动引用，不是项目副本。
- 观察点：在功能分支提交后，`main` 指针不会自动前进。
- 更进一步：能解释 HEAD 指向分支和 HEAD 直接指向提交的区别。

### 第8章

- 关键点：冲突是 Git 无法替你做业务判断，不是仓库损坏。
- 观察点：冲突解决后 `git status` 应回到可继续提交状态。
- 更进一步：能解释 merge 的输入为什么是“两段历史”，而不是“两个文件”。

### 第9章

- 关键点：rebase 会重写提交 ID。
- 观察点：rebase 前后，功能提交会以新身份出现在新的基线上。
- 更进一步：能说明为什么“内容相同”不等于“历史没变”。

### 第10章

- 关键点：`origin/main` 是本地记录的远程状态。
- 观察点：`git fetch` 后 `origin/*` 更新，但当前本地分支不自动变化。
- 更进一步：能解释远程跟踪分支为什么会过期。

### 第11章

- 关键点：`pull` 本质上是 `fetch + merge/rebase`。
- 观察点：push 被拒绝时先同步和观察，而不是直接强推。
- 更进一步：能把同步问题分成远程领先、本地领先、双方都前进三类。

### 第12章

- 关键点：PR 是协作接口，也是历史治理接口。
- 观察点：PR 前应能说清背景、变更范围、验证方式和风险。
- 更进一步：能比较 merge commit、squash merge、rebase merge 的长期影响。

---

## 恢复与排障段：第 13-18 章

### 第13章

- 关键点：先判断你要恢复的是工作区、索引还是历史。
- 观察点：`restore`、`reset`、`revert` 各自作用层级不同。
- 更进一步：能说明为什么共享分支更适合 `revert`。

### 第14章

- 关键点：`amend`、`rebase`、`cherry-pick` 都属于历史编辑工具，但边界不同。
- 观察点：`cherry-pick` 搬运的是提交改动，不是分支关系。
- 更进一步：能说明为什么历史编辑几乎总要先确认“是否已共享”。

### 第15章

- 关键点：误操作后先看 `reflog`。
- 观察点：恢复前先建救援分支通常更稳。
- 更进一步：能说明 `log` 和 `reflog` 的本质差别。

### 第16章

- 关键点：stash 适合临时保存未完成工作，不适合长期替代提交。
- 观察点：`git clean` 前应先使用 `-n` 预览。
- 更进一步：能判断什么时候该 stash，什么时候更该分支或 worktree。

### 第17章

- 关键点：标签是稳定路标，分支是可移动引用。
- 观察点：正式发布更适合附注标签。
- 更进一步：能说明为什么正式发布后的标签不应随意移动。

### 第18章

- 关键点：`blame` 是定位上下文，不是甩锅工具。
- 观察点：`git blame` 最好配合 `git show` 和 PR 上下文一起看。
- 更进一步：能把 `grep -> blame/show -> log --grep -> bisect` 串成排障链路。

## 高阶段：第 19-24 章

### 第19章

- 关键点：Git 的核心是对象和引用。
- 观察点：能解释 blob、tree、commit、tag 的区别。
- 更进一步：会用 `git cat-file -p HEAD` 和 `git show-ref` 做只读观察。

### 第20章

- 关键点：本地 hook 适合快速反馈，CI 负责团队一致性。
- 观察点：alias 应提升观察效率，而不是制造迷惑。
- 更进一步：能说明哪些规则适合 hook，哪些必须放 CI。

### 第21章

- 关键点：`worktree` 适合并行任务，submodule 适合独立组件边界。
- 观察点：仓库组合策略本质上是协作边界选择。
- 更进一步：能解释为什么 submodule 的困难常常来自长期维护而不是单条命令。

### 第22章

- 关键点：分支模型没有唯一正确答案，治理规则比名词更重要。
- 观察点：主干保护、PR、CI、force push 规则应明确成文。
- 更进一步：能把分支模型和发布节奏、测试能力、回滚要求联系起来。

### 第23章

- 关键点：大仓库问题先是使用习惯和治理问题，再是工具问题。
- 观察点：大文件、构建产物和无意义巨型提交是常见性能杀手。
- 更进一步：能说明 monorepo、多仓库、LFS 分别在解决什么问题。

### 第24章

- 关键点：高阶 Git 能力是恢复能力和治理能力。
- 观察点：知道何时优先用 `reflog`、`revert`、`bisect`、`force-with-lease`。
- 更进一步：能写出一份最小 Git 事故应急卡片。

## 通用答题标准

如果你在练习时只写出命令但解释不出状态变化，说明还没有真正掌握。一个合格答案通常至少包含：

1. 你执行了什么命令
2. 这个命令改了哪一层
3. 执行后 `git status` 或历史图应该怎么变化
4. 如果操作错了，应该如何恢复

真正的高质量答案，通常还会补一句：

- 为什么当前场景选这条命令，而不是另一条看起来也像能做类似事情的命令
