# 附录B：命令安全矩阵

本附录按“风险等级 + 影响层级 + 恢复入口”组织，而不是按命令分类。

---

## 1. 低风险：默认可先尝试

| 命令 | 主要作用层 | 作用 | 风险说明 |
|------|------------|------|----------|
| `git status -sb` | 只读 | 查看状态 | 无副作用 |
| `git log --oneline --graph --decorate --all` | 只读 | 查看历史图 | 无副作用 |
| `git show <commit>` | 只读 | 查看提交详情 | 无副作用 |
| `git diff` | 只读 | 看工作区相对索引差异 | 无副作用 |
| `git fetch` | 远程跟踪引用 | 同步远程状态 | 不改当前分支与工作区 |

这些命令的共同点是：它们优先帮你看清楚，而不是直接改变仓库状态。

---

## 2. 中风险：会改变本地状态，但通常容易恢复

| 命令 | 主要作用层 | 使用前先确认 | 常见恢复入口 |
|------|------------|--------------|--------------|
| `git add` | 索引 | 是否明确知道要提交什么 | `git restore --staged` |
| `git commit` | 提交历史 + 当前分支 | staged diff 是否正确 | `git reset --soft` 或 `--mixed` |
| `git merge` | 历史 + 工作区/索引中间态 | 是否站在正确目标分支 | `git merge --abort` 或手动解决 |
| `git rebase` | 提交身份 + 当前分支 | 是否是未共享历史 | `git rebase --abort` |
| `git stash` | stash 栈 + 工作区 | 是否真的只是临时切走 | `git stash list` / `apply` |

---

## 3. 高风险：可能覆盖、删除或重写历史

| 命令 | 主要作用层 | 风险 | 第一观察动作 |
|------|------------|------|--------------|
| `git reset --hard` | 分支引用 + 索引 + 工作区 | 可能丢失未提交工作 | `git status -sb`、`git reflog` |
| `git clean -fd` | 未跟踪文件 | 删除内容通常不在 Git 历史里 | `git clean -nd` |
| `git push --force` | 远程共享历史 | 可能覆盖团队已依赖的历史 | `git fetch`、看图 |
| `git rebase -i` | 一串提交身份 | 易重写已共享历史 | 先确认是否已共享 |

---

## 4. 常见症状的分流入口

| 症状 | 先看什么 | 默认安全动作 | 不该先做什么 |
|------|----------|--------------|---------------|
| 文件改坏了但没提交 | `git status -sb` `git diff` | `git restore` | `reset --hard` |
| 已经 add 了但不想提交 | `git diff --cached` | `git restore --staged` | `reset --hard` |
| 本地提交不见了 | `git reflog` | 先建救援分支 | 继续乱 reset |
| push 被拒绝 | `git fetch` + 历史图 | 判断谁领先、是否共享 | 直接 `push --force` |
| stash 应用冲突 | `git status` | 手工解决冲突 | 假装 stash 不会冲突 |

---

## 5. 共享分支优先规则

对共享分支，优先遵循下面这组经验规则：

1. 回滚优先 `git revert`
2. 强推优先 `git push --force-with-lease`
3. 历史整理尽量发生在 PR 前、本地分支上
4. 不确定时先 `fetch`、先看图、先沟通

---

## 6. 用前必问的三个问题

在执行高风险命令前，先问自己：

1. 这个命令会改工作区、索引、提交历史、引用中的哪一层？
2. 它影响的是我一个人，还是会影响团队共享历史？
3. 如果结果不对，我知道第一恢复入口是什么吗？

如果三个问题里有一个答不上来，先停下来。
