# 第1章：为什么需要 Git 与版本控制

## 学习目标

完成本章后，你应能：

1. 解释版本控制系统解决的核心问题
2. 理解 Git 相对于“手动备份”和集中式系统的优势
3. 建立“快照 + 提交图 + 引用”的基础直觉
4. 说清仓库、提交、分支、HEAD 这些基本概念

---

## 1.1 如果没有版本控制，会发生什么

很多人最早管理文件版本的方式是这样的：

```text
report-final.docx
report-final-v2.docx
report-final-v2-final.docx
report-final-v2-final-really-final.docx
```

这类方式的问题不只是命名难看，而是它会直接损失几个关键能力：

- **差异能力**：你不知道两个版本到底改了什么
- **恢复能力**：你很难准确回到“昨天还能工作”的状态
- **协作能力**：多人同时改时，谁覆盖了谁很难判断
- **追溯能力**：你不知道某个改动为什么出现、是谁引入的

也就是说，版本控制真正管理的不是“文件数量”，而是“变更的可解释性”。

---

## 1.2 Git 比“网盘备份”强在哪

网盘、压缩包、复制目录能做的是“保留副本”；Git 能做的是“管理演化”。

Git 不只保存文件内容，还记录：

- 谁在什么时候做了修改
- 每次修改之间的父子关系
- 某个改动属于哪个分支
- 两条历史何时分叉、何时汇合
- 哪个版本被标记为正式发布

所以 Git 的价值不只是“有历史”，而是**历史可以被比较、被推理、被恢复、被协作**。

---

## 1.3 集中式与分布式：差别不只是“能不能离线”

在集中式版本控制系统里，中心服务器往往是唯一主要真相源，本地更像工作副本；Git 则是分布式版本控制系统：

- 本地仓库拥有完整历史
- 可以离线提交、看日志、比较差异
- 远程仓库更像同步点，而不是唯一真相源

这带来几个很实际的后果：

1. **离线可工作**
   没网时依然可以提交、回看历史、恢复改动。

2. **本地实验成本低**
   你可以本地先整理好历史，再决定是否共享。

3. **远程协作更灵活**
   同一个仓库可以同时接多个远程，不一定只有一个“中心”。

但也别误解成“分布式就等于大家可以随便改远程历史”。Git 允许本地强大，不等于团队不需要远程规范。

---

## 1.4 Git 最重要的视角：快照，而不是补丁堆

理解 Git 时，最重要的一点是：

> Git 的提交更像“某一时刻整个项目的快照”，而不是“按顺序保存的一堆手写补丁”。

这对你后面理解很多现象都很重要：

- 提交不是“追加一行记录”，而是“记录当前准备好的项目状态”
- 分支不是“复制一份目录”，而是“给某个提交起一个可移动的名字”
- 合并不是“把两个文件随便拼起来”，而是“把两段历史重新汇合”
- 回滚不是“删除过去”，而是“在历史上追加一个反向动作，或移动引用”

你现在不必理解所有内部实现，但必须先接受这件事：Git 管的是**项目状态的演化图**。

---

## 1.5 仓库、提交、分支、HEAD 的第一层定义

### 仓库（Repository）

Git 管理的项目目录。初始化后会出现 `.git/`，其中保存历史和元数据。

### 提交（Commit）

一次被永久记录的项目快照，带有唯一标识和父提交关系。

### 分支（Branch）

指向某个提交的可移动引用。新提交产生时，当前分支会向前移动。

### HEAD

表示“当前检出位置”。通常它指向某个分支，有时也会直接指向提交。

这四个概念在本章只先建立定义，后面第 3 章会把它们真正拆开。

---

## 1.6 一个最小观察实验：仓库和普通目录的区别

请在一个空目录中执行：

```bash
mkdir git-demo
cd git-demo
git init
ls -a
git status
```

如果你愿意再往下看一点：

```bash
find .git -maxdepth 2 | sort | sed -n '1,20p'
```

你会第一次直观看到：

- 普通目录初始化成仓库后，多了 `.git/`
- Git 并不是“隐形魔法”，它真的在目录里维护了一套历史元数据
- 仓库的存在与否，不取决于有没有远程仓库，而取决于 `.git/` 是否存在

这一步看起来简单，但它能帮助你尽早建立一个很重要的事实：**Git 仓库首先是本地对象。**

---

## 1.7 Git 解决的是“并行演化”，不是“单机存档”

真正的软件开发不是静态存文件，而是持续演化：

- 新功能不断加入
- Bug 会被修复
- 实验性改动可能被丢弃
- 多个人会并行修改同一套代码

Git 之所以重要，不是因为它能“保存很多版本”，而是因为它能把这些并行演化组织成一个可读、可比较、可恢复的历史图。

---

## 常见误区

- **误区 1：Git 就是网盘同步盘。**
  它真正管理的是历史和协作语义。

- **误区 2：分布式等于远程不重要。**
  远程依然是团队协作和审查的关键同步点。

- **误区 3：快照等于每次复制整个目录。**
  对学习者来说可以按快照理解，但底层实现会做去重和压缩。

- **误区 4：branch 就是复制项目。**
  更准确地说，branch 是对提交位置的命名。

## 本章小结

本章最重要的收获不是命令，而是一个判断标准：

> 以后看到任何 Git 操作，都优先问自己：它是在记录快照、移动内容，还是移动引用？
