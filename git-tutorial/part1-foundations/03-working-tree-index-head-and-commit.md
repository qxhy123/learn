# 第3章：工作区、暂存区、HEAD 与提交

## 学习目标

完成本章后，你应能：

1. 清楚区分工作区、暂存区、提交历史三层
2. 理解 `git add` 和 `git commit` 分别做了什么
3. 知道 HEAD 为什么会移动，以及它什么时候不再指向分支
4. 用快照视角解释“一部分改动已暂存，另一部分未暂存”的现象

---

## 3.1 Git 最重要的三层

### 工作区（Working Tree）

你当前目录里真实存在、正在编辑的文件。

### 暂存区（Index / Staging Area）

下一次提交的候选快照。它不是“临时垃圾桶”，而是提交前的精确清单。

### 提交历史（Commit History）

已经被 Git 正式记录下来的历史快照。

可以先把它理解成：

```text
工作区        -> 你正在改
暂存区        -> 你准备提交
提交历史      -> 你已经提交
```

很多 Git 困惑，本质上都来自于把这三层混成一层。

---

## 3.2 `git add` 到底做了什么

很多初学者以为 `git add` 等于“提交文件”。这是错误的。

`git add` 的真实作用是：

> 把工作区中的当前内容复制进暂存区，作为下一次提交的一部分。

所以一个文件即使已经 `git add` 过，你后面继续修改它，工作区和暂存区也可能再次分离。

例子：

```bash
echo "line 1" > app.txt
git add app.txt
echo "line 2" >> app.txt
git status -sb
```

此时常见状态是：

- 暂存区里有“只含 `line 1`”的版本
- 工作区里有“`line 1 + line 2`”的版本

这就是“同一文件同时 staged 和 unstaged”的来源。

---

## 3.3 为什么说索引不是“临时篮子”

索引真正重要的地方，不在于它能“先放一下”，而在于它可以让你**精确决定下一次提交的内容**。

它带来的能力包括：

- 把一个大改动拆成多个小提交
- 只提交已经完成的部分
- 保留实验代码在工作区，但不让它进入提交
- 在提交前做最后一次挑选和整理

没有索引，提交往往会退化成“整个目录一起打包”，历史质量会迅速下降。

---

## 3.4 `git commit` 记录的到底是什么

`git commit` 并不是把整个工作目录强行打包，而是把**暂存区当前的内容**记录为新的提交。

所以：

- 工作区里没暂存的改动，不会自动进入本次提交
- 暂存区里已经准备好的内容，会被写进提交

如果你只记住本章一句话，请记住：

> `add` 更新索引，`commit` 把索引写入历史。

---

## 3.5 HEAD 到底是什么

HEAD 表示“你当前检出的位置”。

最常见的情况是：

```text
HEAD -> main -> C3
```

意思是：

- 你当前附着在 `main` 分支上
- `main` 指向提交 `C3`
- HEAD 通过 `main` 间接指向 `C3`

但 HEAD 也可能直接指向某个提交：

```text
HEAD -> C2
```

这就是后面会讲到的 detached HEAD。它不是仓库损坏，只是当前没有附着在某个分支上。

---

## 3.6 一次完整提交在三层之间如何流动

假设你编辑了 `README.md`：

1. 修改文件后，变化先出现在**工作区**
2. 执行 `git add README.md` 后，变化进入**暂存区**
3. 执行 `git commit` 后，暂存区快照进入**提交历史**

这个过程会决定你后面如何理解：

- `restore`
- `reset`
- `revert`
- `rebase`
- `stash`

如果这条主线没建立，后面很多命令都会像彼此无关的魔法。

---

## 3.7 把索引“看出来”：几个特别值得用的观察命令

### 看索引

```bash
git ls-files --stage
```

这条命令很重要，因为它能让“索引真实存在”这件事从抽象概念变成可见事实。

### 看 HEAD 当前指向

```bash
git rev-parse HEAD
```

### 看最近一次提交的大致结构

```bash
git show --stat --summary HEAD
```

如果你愿意再稍微靠近底层一点：

```bash
git cat-file -p HEAD
```

你会看到提交对象包含作者、提交说明、树对象和父提交信息。

## 3.8 一个最关键的实验：同一文件同时 staged 和 unstaged

在练习仓库中执行：

```bash
echo "v1" > demo.txt
git add demo.txt
git commit -m "feat: add demo"

echo "v2" >> demo.txt
git add demo.txt
echo "v3" >> demo.txt
```

然后依次运行：

```bash
git status -sb
git diff
git diff --cached
git ls-files --stage
```

你应该观察到：

- `git diff` 看的是工作区相对索引的差异
- `git diff --cached` 看的是索引相对 HEAD 的差异
- 同一个文件确实可能同时在两层发生不同变化

这个实验建议你亲手做一遍，否则后面很多撤销命令只是口头知识。

---

## 常见误区

- **误区 1：`git add` 之后文件就安全保存了。**
  它只是进入索引，还没有形成提交。

- **误区 2：`git commit` 是对工作区拍照。**
  更准确地说，它是对索引拍照。

- **误区 3：HEAD 就等于当前分支名。**
  更准确地说，HEAD 是当前检出位置，通常指向分支，但也可能直接指向提交。

- **误区 4：索引只是中转站，没有必要理解。**
  理不清索引，几乎一定会在 `reset`、`restore`、`rebase` 上混乱。

3. 运行 `git ls-files --stage`，观察索引里的条目。
4. 思考题：为什么说索引是“下一次提交的精确清单”，而不是一个临时垃圾桶？
