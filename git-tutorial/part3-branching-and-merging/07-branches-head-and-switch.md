# 第7章：分支、HEAD 与 switch

## 学习目标

完成本章后，你应能：

1. 用“可移动引用”解释分支是什么
2. 理解 HEAD 和当前分支的关系
3. 使用 `git branch`、`git switch` 创建和切换分支
4. 看懂 detached HEAD、切换失败和分支职责差异

---

## 7.1 分支不是复制项目，而是给提交起名字

初学者常常把“创建分支”理解成“复制一份代码”。这个比喻不完全对。

更准确的说法是：

> 分支是一个指向提交的可移动引用。

例如：

```text
main -> C3
```

如果你从这里创建 `feature/login`，更像是多了一个名字：

```text
main -> C3
feature/login -> C3
```

此时并没有真的复制整套项目内容，只是两个名字同时指向同一个提交。

---

## 7.2 HEAD 到底指向谁

最常见的情况是：

```text
HEAD -> main -> C3
```

这表示：

- 你当前附着在 `main`
- `main` 指向 `C3`
- HEAD 通过 `main` 间接指向 `C3`

如果你执行：

```bash
git switch -c feature/x
```

则会变成：

```text
HEAD -> feature/x -> C3
main -> C3
```

如果你直接切到历史提交：

```bash
git switch --detach C2
```

则会变成：

```text
HEAD -> C2
main -> C3
feature/x -> C3
```

此时 HEAD 不再附着在分支上，这就是 detached HEAD。

---

## 7.3 新提交为什么会让分支“向前移动”

假设当前状态是：

```text
HEAD -> feature/x -> C3
main -> C3
```

你在 `feature/x` 上提交一次，产生 `C4`：

```text
HEAD -> feature/x -> C4
main -> C3
```

变化的不是旧提交，而是：

1. 新建了一个提交对象
2. 当前分支 `feature/x` 指向新提交
3. HEAD 继续跟着当前分支

这就是为什么说“分支不是内容副本，而是历史位置的名字”。

---

## 7.4 `git branch` 与 `git switch`

查看分支：

```bash
git branch
```

创建分支：

```bash
git branch feature/login
```

切到已有分支：

```bash
git switch main
```

创建并切换：

```bash
git switch -c feature/login
```

与旧命令 `git checkout` 相比，`git switch` 更专门，也更适合新手形成清晰的心智模型。

---

## 7.5 为什么切分支有时会失败

切分支失败，通常不是 Git “抽风”，而是它在保护你。

典型场景：

- 你当前工作区有未提交改动
- 这些改动在目标分支上会被覆盖

这时 Git 宁可拒绝切换，也不愿意默默把工作区改坏。

所以切分支前的默认动作应该是：

```bash
git status -sb
```

如果状态不清楚，先别切。

---

## 7.6 detached HEAD 不是异常，而是“暂时不在分支上”

如果你直接检出某个历史提交，HEAD 会直接指向它。你仍然可以：

- 查看代码
- 运行测试
- 临时做实验
- 甚至创建新提交

但要注意：

- 这些提交不会自动被某个分支记住
- 如果你想保留它们，最好尽快新建分支

例如：

```bash
git switch -c rescue-experiment
```

所以 detached HEAD 不是坏事，它只是要求你更明确地决定“要不要给这段历史起名字”。

---

## 7.7 分支的真正用途：隔离工作，而不是堆更多名字

分支的价值在于隔离不同工作流：

- 主分支保持相对稳定
- 功能开发在 feature 分支上进行
- 紧急修复可以在 hotfix 分支独立处理
- 实验性改动可以安全丢弃

但不要把分支误当成“权限边界”或“个人草稿无限仓库”。

### 短生命周期分支

适合功能开发、小型修复、短周期协作。

### 长生命周期分支

适合少数明确场景，例如发布线、维护线。

如果一个团队把所有开发都长期挂在多条共享分支上，后续合并和同步成本会迅速上升。

---

## 常见误区

- **误区 1：分支会复制整个仓库。**
  更准确地说，分支只是移动中的提交引用。

- **误区 2：HEAD 永远等于当前分支名。**
  detached HEAD 时它会直接指向提交。

- **误区 3：切分支失败说明仓库坏了。**
  更多时候是 Git 在保护你的未提交改动。

- **误区 4：可以先在 `main` 上乱做实验，之后再整理。**
  这是制造混乱历史的典型路径。
