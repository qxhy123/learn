# 第8章：merge 与冲突解决

## 学习目标

完成本章后，你应能：

1. 理解快进合并与三方合并的区别
2. 理解 merge 的输入不是“两个文件”，而是“两段历史”
3. 使用 `git merge` 合并功能分支并解决常见冲突
4. 知道冲突发生时 Git 处于什么中间状态

---

## 8.1 merge 合并的不是文件，而是历史

很多初学者会把 merge 理解成“把两个版本的文件拼起来”。这太窄了。

更准确的说法是：

> merge 处理的是两段历史如何重新汇合，而文件冲突只是汇合失败时暴露出来的表象。

这也是为什么 merge 前你应该先看历史图，而不是只盯某个文件。

---

## 8.2 快进合并：没有真正分叉时，只移动指针

假设当前图是：

```text
main -> C1
feature -> C3
```

如果 `main` 自从 `feature` 分出后没有新提交，那么：

```bash
git switch main
git merge feature
```

很多时候只会把 `main` 直接移动到 `C3`。这叫快进合并（fast-forward）。

它的特点是：

- 没有新的 merge commit
- 历史保持线性
- 本质上只是“目标分支追上了前方那条线”

---

## 8.3 三方合并：两边都前进时，Git 需要找共同祖先

如果历史变成：

```text
      C3 feature
     /
C1-C2
     \
      C4 main
```

此时 merge 往往需要：

1. 找共同祖先
2. 看两边各自改了什么
3. 尝试自动生成汇合结果
4. 必要时创建 merge commit

这就是三方合并。

它提醒你一件事：merge 的核心不是“谁覆盖谁”，而是“怎样把两段演化重新收束”。

---

## 8.4 冲突为什么会发生

冲突不是 Git 笨，而是 Git 无法替你做业务判断。

典型场景包括：

- 两个分支都改了同一段代码
- 一个分支删文件，另一个分支继续改该文件
- 一边重命名，另一边修改
- 并非同一行冲突，但语义上仍然矛盾

所以不要把冲突理解成“工具失败”，它更像是：

> Git 把自动决策的边界交还给了人。

---

## 8.5 冲突发生时，Git 正在保护什么

当 merge 过程中发生冲突时，仓库通常进入一种中间状态：

- 工作区中写入冲突标记
- 索引里存在未解决条目
- merge 尚未完成

这时最重要的动作不是“赶紧删标记”，而是先看：

```bash
git status
```

它会告诉你哪些文件仍未解决。

---

## 8.6 冲突标记怎么看

文件里常见的冲突标记大致是：

```text
<<<<<<< HEAD
当前分支内容
=======
被合并分支内容
>>>>>>> feature/login
```

正确的处理流程通常是：

1. 理解双方改动意图
2. 保留正确结果
3. 删除冲突标记
4. `git add <file>`
5. 再次查看 `git status`

不要把“解决冲突”简化成“把标记删掉就行”。真正的问题是：**最终结果语义是否正确。**

---

## 8.7 一个更稳的冲突处理流程

```bash
git status
# 打开冲突文件并手工修改
git add <resolved-files>
git status
git commit
```

如果你在没理解业务语义前就直接保留“自己这一边”，很容易把表面冲突解决成更深的逻辑错误。

---

## 8.8 什么时候该保留 merge commit

merge commit 的好处：

- 保留“这组提交作为一个主题汇入主干”的历史语义
- 更容易看出分支的真实分叉与汇合

缺点：

- 历史图更复杂
- 如果只是同步型 merge，可能引入噪音

所以是否保留 merge commit，往往是团队策略问题，而不是“某一种永远更高级”。

---

## 8.9 一个必须练的实验：故意制造冲突

建议你在练习仓库里执行：

1. 从同一提交创建两个分支
2. 在同一文件同一位置分别做不同修改
3. 尝试 merge
4. 观察冲突标记
5. 用 `git status` 确认何时从“未解决”回到“可提交”

这比只看教程截图有用得多，因为它能让你真正感受到 merge 的中间状态。

---

## 常见误区

- **误区 1：merge 的对象是文件。**
  更准确地说，merge 处理的是两段历史。

- **误区 2：冲突说明仓库坏了。**
  冲突只是自动决策到达边界。

- **误区 3：冲突解决就是删掉标记。**
  真正重要的是最终业务语义是否正确。

- **误区 4：为了避免冲突，直接整文件覆盖。**
  这通常会把另一边合理改动一并抹掉。

---

## 本章练习

1. 新建两个分支，让它们都修改同一文件的同一行，再尝试 merge。
2. 观察冲突发生时 `git status` 的变化。
3. 解决冲突后再次查看历史图，确认 merge commit 的位置。
4. 思考题：为什么说冲突本质上是“业务判断缺失”，而不是“工具失败”？

---

## 本章小结

merge 让你第一次真正面对“历史汇合”的问题。下一章我们继续学另一条高频路径：不是把两条历史汇合，而是把一组提交重新放到新基线上，也就是 rebase。
