# 第9章：rebase 与历史整形

## 学习目标

完成本章后，你应能：

1. 理解 rebase 与 merge 的语义差异
2. 使用 `git rebase` 让分支基于新的主干继续工作
3. 使用交互式 rebase 整理本地提交历史
4. 知道哪些场景不应随意 rebase

---

## 9.1 rebase 在做什么

可以把 rebase 理解成：

> 把一组提交重新播放到另一条基线上。

例如：

```text
main:    A-B-C
feature: A-B-D-E
```

如果你对 `feature` 执行：

```bash
git rebase main
```

那么结果大致会变成：

```text
main:    A-B-C
feature: A-B-C-D'-E'
```

`D'` 和 `E'` 不是原来的提交，而是**新的提交对象**。

---

## 9.2 rebase 改写的是身份，不只是顺序

这点非常关键：

- 改动内容可能类似
- 但提交 ID 会变化
- 父子关系会变化
- 历史语义也会变化

所以 rebase 不是“把图拉直一下”这么简单，它是在**重写历史身份**。

这也是为什么：

- 它经常需要 force push
- 它会打断某些 review 上下文
- 它不适合随意作用于共享历史

---

## 9.3 merge 和 rebase 的区别

### merge

- 保留真实分叉与汇合历史
- 适合共享分支和真实协作轨迹
- 不改已有提交身份

### rebase

- 让历史更线性
- 适合整理个人分支、本地未共享提交
- 会重写提交身份

二者不是谁更高级，而是回答不同问题：

- 你要保留真实分叉？
- 还是要整理个人历史，减少汇合噪音？

---

## 9.4 一个常见的 rebase 场景

当你的功能分支落后于主干时：

```bash
git switch feature/login
git rebase main
```

这会尝试把 `feature/login` 上独有的提交，重新基于当前 `main` 播放一遍。

如果中途发生冲突，Git 会停在某一条正在重放的提交上。

这时最重要的不是慌，而是先确认：

- 当前重放到哪一条提交
- 冲突是在哪一步发生的
- 你是要继续、跳过，还是放弃

---

## 9.5 `--continue`、`--skip`、`--abort` 分别意味着什么

### `git rebase --continue`

冲突已经解决，继续重放后续提交。

### `git rebase --skip`

跳过当前这条提交。只有在你明确知道它已无必要时才考虑。

### `git rebase --abort`

放弃本次 rebase，回到开始前状态。

对多数人来说，默认安全顺序是：

1. 先理解冲突
2. 能解决就 `--continue`
3. 不确定时宁可 `--abort`

`--skip`` 不应该成为习惯动作。

---

## 9.6 交互式 rebase：整理自己的提交

交互式 rebase 是高质量历史整理的重要工具：

```bash
git rebase -i HEAD~3
```

常见操作：

- `pick`：保留提交
- `reword`：修改提交说明
- `squash`：与前一个提交合并
- `drop`：丢弃提交

它特别适合在提交 PR 前，把零碎提交整理成一个 reviewer 能读懂的结构。

---

## 9.7 审查期和共享分支上的 rebase 禁区

一条重要原则：

> 不要轻易 rebase 已经共享给团队、且别人可能已经基于它工作的历史。

原因包括：

- 提交 ID 变化会让别人的本地分支同步复杂化
- PR 评论和引用可能失去锚点
- 若再配合强推，协作成本会突然升高

所以比较稳的经验规则是：

- 本地个人分支：可以整理
- PR 刚打开后：谨慎整理，避免频繁改写
- 已共享的发布分支或主干：一般不要 rebase

---

## 9.8 一个值得亲手做的对照实验

在相同场景下分别试一次 merge 和 rebase：

1. 从 `main` 分出 `feature`
2. 两边各自提交
3. 一次用 `merge`
4. 一次用 `rebase`
5. 用 `git log --oneline --graph --decorate --all` 对比结果

你会直观看到：

- merge 保留分叉与汇合
- rebase 把一组提交重新放到新基线上

这会比任何抽象定义都更有帮助。

---

## 常见误区

- **误区 1：rebase 只是把图弄好看。**
  它本质上是在重写提交身份。

- **误区 2：rebase 一定比 merge 高级。**
  二者语义不同，适用场景不同。

- **误区 3：内容没变就不算改历史。**
  提交 ID 和父子关系变化，本身就是改历史。

- **误区 4：共享历史上也可以想 rebase 就 rebase。**
  这往往会提高团队同步成本。
