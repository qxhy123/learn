# 第6章：`.gitignore` 与提交卫生

## 学习目标

完成本章后，你应能：

1. 理解哪些文件适合忽略、哪些不适合忽略
2. 正确编写 `.gitignore` 规则并排查忽略是否生效
3. 分清“忽略规则”和“已跟踪状态”是两套机制
4. 形成小步提交、单一主题提交的习惯

---

## 6.1 什么文件应该忽略

典型应该忽略的内容包括：

- 编译产物
- 依赖目录
- 临时文件和编辑器缓存
- 本地日志
- 含密钥的环境配置副本

例如：

```gitignore
node_modules/
dist/
.DS_Store
.env.local
*.log
```

但“适合忽略”和“看起来麻烦所以全忽略”是两回事。像下面这些往往不该轻易忽略：

- 配置模板
- 锁文件
- 迁移脚本
- 真实源码

---

## 6.2 `.gitignore` 规则是路径匹配，不是“删文件”

`.gitignore` 支持路径模式匹配：

```gitignore
*.log
build/
!build/keep.txt
```

常见理解方式：

- `*`：通配多个字符
- `/`：目录边界
- `!`：反向包含

这里最重要的不是记语法，而是理解：`.gitignore` 只是告诉 Git “哪些未跟踪文件默认不进入跟踪候选”，它不是删除机制，也不是安全机制。

---

## 6.3 忽略规则和跟踪状态是两套机制

这是本章最重要的一点：

> `.gitignore` 影响的是“未跟踪文件是否应被忽略”，并不会自动取消已经被跟踪的文件。

如果某个文件已经提交过，之后再把它写进 `.gitignore`，Git 不会自动停止跟踪它。

你需要显式把它从索引里移出：

```bash
git rm --cached .env.local
```

这条命令只会从版本控制中移除，不会删掉工作区文件。

---

## 6.4 不只一种忽略层：仓库级、个人级、局部级

很多初学者只知道仓库里的 `.gitignore`，其实常见忽略层有三种：

1. **仓库级 `.gitignore`**
   团队共享，适合写所有成员都应忽略的内容。

2. **个人级全局 ignore**
   适合编辑器缓存、操作系统垃圾文件等个人习惯。

3. **仓库局部排除**
   适合只想在本地临时排除、但不想提交给团队的情况。

这三层的存在提醒你：**忽略策略本身也是协作协议的一部分。**

---

## 6.5 把规则“看出来”：最该用的排查命令

如果你怀疑某个文件为什么被忽略或没被忽略，先用：

```bash
git check-ignore -v <path>
```

它会告诉你：

- 是哪条规则生效了
- 规则来自哪个文件

其他很有用的观察命令包括：

```bash
git status --ignored
git ls-files
git rm --cached <file>
```

如果你只会改 `.gitignore`，不会用这些排查命令，复杂项目里很快就会陷入“为什么不生效”的困惑。

---

## 6.6 什么叫“提交卫生”

提交卫生的核心是：让历史可读、可 review、可回滚。

它通常包括：

- 一次提交只做一件事
- 不把格式化、重构、功能、调试残留混在一起
- 提交说明能描述变化主题
- 提交前确认没有误带敏感文件和无关变更

这不是形式主义。它直接影响：

- PR 审查效率
- 回滚风险
- `git blame` 的可读性
- 排障速度

---

## 6.7 一个“坏提交”为什么会让后面所有工作变差

假设某条提交同时包含：

- 全项目格式化
- 一个真实功能修复
- 一段调试日志
- 配置项改动

这条提交的直接问题是：

- reviewer 很难判断风险
- 回滚功能修复时可能把格式化一起回掉
- 后面用 `blame` 时，很难知道某一行真正为什么被改

所以“提交卫生”不是审美问题，而是**未来维护成本问题**。

---

## 6.8 提交前检查清单

建议每次提交前快速过一遍：

1. `git status -sb` 是否清楚
2. `git diff --cached` 是否只包含本次主题
3. 是否混入日志、密钥、临时文件
4. 提交说明是否能概括本次变化

只要这四项能做到，历史质量通常不会太差。

---

## 常见误区

- **误区 1：把敏感文件写进 `.gitignore` 就等于安全了。**
  如果它已经进过历史，问题根本没有自动消失。

- **误区 2：`.gitignore` 会清除已经提交过的文件。**
  不会，必须显式从索引移除。

- **误区 3：提交越少越省事。**
  杂糅提交会把成本留给 review、回滚和排障。

- **误区 4：只要规则写对了，就不用会排查。**
  大项目里不会用 `git check-ignore -v` 几乎一定会卡住。

---

## 本章练习

1. 为一个 Node.js 或 Python 小项目写一份最小 `.gitignore`。
2. 制造一个“已经提交了 `.env.local`，后来才想忽略”的场景，并把它从索引中移除。
3. 用 `git check-ignore -v` 排查一条忽略规则。
4. 思考题：为什么“提交卫生”最终会影响 PR、回滚和排障质量？
