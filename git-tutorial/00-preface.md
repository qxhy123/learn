# 前言：如何学习 Git，如何避免“命令背了还是不会用”

Git 的学习门槛并不在于命令数量，而在于它和很多人之前接触过的工具很不一样。

很多初学者会把 Git 理解成：

- 一个“自动备份文件夹”的工具
- 一个“把代码传到远程仓库”的工具
- 一个“出了问题就 `reset --hard` 一把梭”的工具

这三种理解都不够准确。

更准确的说法是：

> Git 是一个记录项目状态演化、维护提交图、通过引用移动来组织协作历史的版本控制系统。

如果你从这个角度去看，很多原来看起来“神秘”的命令都会变成对同一套模型的不同操作。

---

## 一张先看懂的 Git 心智图

建议你先记下面这张图，再开始读正文：

```text
工作区 files
   │   ↑
   │   │ restore
   │ add / diff
   ▼   │
暂存区 index
   │   ↑
   │   │ reset --mixed / restore --staged
   │ commit
   ▼
提交历史 commits
   │
   ├─ 分支 refs/heads/*
   ├─ HEAD
   ├─ tags
   └─ 远程跟踪 refs/remotes/*
```

你可以把 Git 中的大部分动作分成三类：

1. **查看**
   例如 `status`、`diff`、`log`、`show`
2. **移动内容**
   例如 `add`、`restore`、`commit`
3. **移动引用或重写历史**
   例如 `switch`、`reset`、`merge`、`rebase`

真正的高阶能力，不是记住更多命令，而是看到一个命令时，立刻判断：

- 它会改哪一层
- 它是否可逆
- 它会不会影响共享历史

## Git 的五个观察对象

前 6 章建议你始终围绕这五个对象观察：

1. **工作区**
   真实存在于目录中的文件内容。

2. **索引（暂存区）**
   下一次提交的候选快照，而不是“顺手放一放的篮子”。

3. **HEAD**
   当前检出位置。它通常指向某个分支，但也可能直接指向某个提交。

4. **分支引用**
   像 `main`、`feature/login` 这样的可移动名字。

5. **远程跟踪分支**
   像 `origin/main` 这样的本地记录，表示“你上次同步到的远程状态”。

只盯前三个对象，会在远程协作和 rebase 上卡住；只盯后两个对象，又容易在基础的 `add/commit/diff` 上失真。

## 推荐的训练方法：预测 -> 执行 -> 观察 -> 恢复

“边学边操作”是对的，但还不够。建议你固定使用下面这套训练法：

### 1. 预测状态

在敲命令之前，先口头回答：

- 这条命令会改哪一层
- `git status` 预计会怎么变
- 提交图是否会变化

### 2. 执行命令

尽量在最小仓库里做，不要一上来就在真实项目里试错。

### 3. 观察图形

每次实验至少看这几样：

```bash
git status -sb
git diff
git diff --cached
git log --oneline --graph --decorate --all
```

如果当前章节涉及索引、对象或配置，再加：

```bash
git ls-files --stage
git config --show-origin --list
git rev-parse HEAD
```

### 4. 复述恢复路径

最后再问自己：

- 如果结果不对，第一步我该看什么
- 应该用哪条命令恢复
- 哪条命令这时绝对不能先用

只有这一步做了，Git 学习才不只是“成功路径演示”。

## 实验记录模板

建议你每做一次实验都记 4 件事：

1. **初始状态**
   当前分支、工作区是否干净、最近几次提交是什么。

2. **执行命令**
   不只是记命令本身，也记下执行顺序。

3. **状态变化**
   `status`、`diff`、历史图分别发生了什么变化。

4. **恢复方式**
   如果刚才操作错了，最安全的恢复入口是什么。

一个最简单的实验记录可以像这样：

```text
实验：同一文件同时 staged 和 unstaged
初始状态：main 分支，工作区干净
命令：echo "v1" > demo.txt -> add -> 修改 demo.txt
观察：diff 看工作区相对索引；diff --cached 看索引相对 HEAD
恢复：restore 丢工作区；restore --staged 丢暂存；reset 会动引用
```

这会让你从“记命令”变成“记状态变化”。

## 为什么很多人学了 Git 还是容易慌

常见原因不是记不住命令，而是下面这些问题没有想清楚：

- 不知道 `git add` 实际上是把内容放进索引，而不是“立刻提交”
- 不知道 `git commit` 记录的是索引快照，而不是整个目录的当前状态
- 不知道分支本质上只是“指向某个提交的名字”
- 不知道 `HEAD` 有时指向分支，有时会直接指向提交
- 不知道 `pull` 其实是 `fetch + merge/rebase`
- 不知道 `revert`、`reset`、`restore` 改的是不同层
- 不知道 `reflog` 记录的是引用移动，而不是团队共享历史

所以本教程的顺序是故意设计过的：

1. 先讲模型
2. 再讲本地操作
3. 再讲分支与协作
4. 再讲撤销、恢复和排障
5. 最后讲内部原理与治理

## 常见误区对照表

| 常见说法 | 更准确的说法 |
|----------|--------------|
| `add` 就是提交 | `add` 只是更新索引，`commit` 才写入历史 |
| branch 是复制目录 | branch 是可移动引用 |
| pull 就是下载最新代码 | pull 通常还会 merge 或 rebase |
| revert 会删除旧提交 | revert 会新建反向提交 |
| reflog 是备份系统 | reflog 是本地引用移动记录 |
| detached HEAD 说明仓库坏了 | detached HEAD 是正常状态，只是当前不附着在分支上 |

如果你发现自己经常说出左边那列，说明你更需要回到模型，而不是继续学更多命令。

## 危险命令真正危险在哪

危险通常不来自命令名字，而来自它**作用在哪一层**以及**是否影响共享历史**。

### `git reset --hard`

- 会同时改引用、索引和工作区
- 如果你没先确认状态，很容易把本地工作一起抹掉

### `git push --force`

- 不是“我本地看起来没问题就能推”
- 真正风险在于它可能覆盖团队已经依赖的远程历史

### `git clean -fd`

- 只删未跟踪文件，但这些文件往往不在 Git 历史里
- 不先 `-n` 预览，很容易误删重要实验文件

### `git rebase -i`

- 很强大，但本质上是在重写提交身份
- 本地整理通常合理，共享历史上则风险很高

一个很实用的习惯是：

> 先回答“它动哪一层、能不能恢复、会不会影响别人”，再决定要不要执行。

## 本教程默认采用的约定

为了减少初学阶段的混乱，本教程默认采用：

- 主分支名为 `main`
- 以命令行为主，GUI 只作为辅助
- 先讲 `merge`，再讲 `rebase`
- 强调小步提交、提交前自检、冲突后先读状态
- 不鼓励对共享分支随意改历史

如果你的团队默认 `pull --rebase`、采用 trunk-based、要求强制签名，也没有问题。等你掌握基础后，第 20-24 章会解释这些差异背后的原因。

## 如何判断自己真的学会了

当你能不用死记硬背，而是解释下面这些问题时，说明你开始真正掌握 Git：

- 为什么一个文件改了但提交里没有它？
- 为什么 `git add` 之后还可以继续修改同一个文件？
- 为什么 `pull` 有时快进、有时 merge、有时 rebase、有时冲突？
- 为什么 `revert` 往往比 `reset` 更适合共享历史？
- 为什么 `reflog` 能找回“看似丢掉”的提交？
- 为什么一个团队必须统一某些默认配置和分支策略？

## 本教程不追求什么

本教程不会把 Git 源码逐函数讲一遍，也不会把所有冷门参数都做成字典。它更重视：

- 建立稳定的状态转移模型
- 解释高频场景与真实误区
- 给出可重复的实验路线
- 帮你从“会敲几个命令”进化到“看到状态后知道怎么判断”

建议你从第 1 章开始，按顺序完成前 12 章；如果已经有基础，也可以按 README 中的路径选择器跳读。
