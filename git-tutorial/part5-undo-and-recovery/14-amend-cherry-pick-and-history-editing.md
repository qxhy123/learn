# 第14章：amend、cherry-pick 与历史编辑

## 学习目标

完成本章后，你应能：

1. 使用 `commit --amend` 修改最近一次提交
2. 理解 `cherry-pick` 的语义与适用场景
3. 明白“编辑历史”不等于“随便篡改共享历史”
4. 在 amend、rebase、cherry-pick、merge 之间做出判断

---

## 14.1 先分清你想编辑哪一段历史

历史编辑最容易出事的原因，是不同动作被混成一类。

先问自己：

- 我只想修最近一次提交吗
- 我想整理一串本地提交吗
- 我只是想把某一条修复搬到别的分支吗
- 这段历史有没有已经共享给别人

问题不同，动作不同。

---

## 14.2 `git commit --amend`：修最近一次提交

最常见的两个场景：

- 提交说明写错了
- 漏加了一个本该属于最近提交的文件

例如：

```bash
git add missing-file.txt
git commit --amend
```

或者只改说明：

```bash
git commit --amend -m "fix: handle empty state correctly"
```

要记住的是：

> amend 不是“修改原提交内容”，而是生成一个新的最近提交。

所以它同样属于改历史。

---

## 14.3 `cherry-pick`：搬一条提交，而不是合并一条分支

`cherry-pick` 的典型用途是：

- 某个修复先发生在功能分支，现在要补到主干
- 某个 hotfix 需要从主干同步到发布分支
- 你不想 merge 整个分支，只想要其中一条提交

例如：

```bash
git switch release/1.0
git cherry-pick <commit>
```

这会把目标提交的改动，以一个**新提交**的形式应用到当前分支。

---

## 14.4 amend、rebase、cherry-pick、merge 的边界

| 动作 | 更适合做什么 |
|------|--------------|
| `commit --amend` | 修最近一次本地提交 |
| `rebase -i` | 整理一串本地提交 |
| `cherry-pick` | 搬单个补丁到另一条分支 |
| `merge` | 汇合整段分支历史 |

最关键的不是命令名字，而是：

- 它会不会改提交身份
- 它是在整理历史，还是在搬补丁，还是在合并分支
- 这段历史有没有已经共享

---

## 14.5 什么时候不该 amend

如果别人已经基于你的最近提交继续工作，再 `amend` 会导致：

- 提交 SHA 改变
- 旧链接、旧 review 锚点失效
- 协作者同步变复杂

所以 amend 适合：

- 本地未共享历史
- PR 前最后整理

不适合：

- 已经稳定共享、被别人依赖的提交

---

## 14.6 `cherry-pick` 的好处与代价

好处：

- 精准搬运单个补丁
- 不引入整个分支历史
- 热修和 backport 非常实用

代价：

- 可能引发冲突
- 容易制造“内容相同、提交不同”的重复历史
- 如果反复多分支拣选，会让追踪变复杂

所以它适合补丁级动作，不适合长期替代清晰的分支策略。

---

## 14.7 一个值得练的场景：先 amend，再 cherry-pick

建议你在练习仓库中：

1. 连续做 3 个提交
2. 用 `commit --amend` 修正最近一次提交说明
3. 把中间那条提交 `cherry-pick` 到另一条分支
4. 用 `git log --oneline --graph --decorate --all` 看历史差异

你会很直观地看到：

- amend 改的是“最近提交的身份”
- cherry-pick 是“把某个补丁复制一份带过去”

---

## 14.8 什么时候宁可停手，也不要继续编辑历史

下面这些场景，建议先停一下：

- 你不确定历史是否已共享
- 你已经在 review 期间多次 force push
- 你不确定应该 amend、rebase 还是 cherry-pick
- 你只是“想让图好看”，但没想清楚协作代价

历史编辑的价值是提升可读性，不是制造更多上下文断裂。

---

## 常见误区

- **误区 1：`--amend` 只是改说明，不算改历史。**
  它会生成新的提交对象。

- **误区 2：`cherry-pick` 等于 merge 的小号版本。**
  不一样，它搬运的是单个补丁。

- **误区 3：内容没变，改 SHA 也无所谓。**
  对协作、review 和同步来说并非如此。

- **误区 4：历史整理一定是越多越好。**
  过度整理反而会打断团队协作语境。

---

## 本章练习

1. 故意写错一次提交说明，然后使用 `git commit --amend` 修正。
2. 在一个分支上做 2 次提交，只把其中 1 次 `cherry-pick` 到另一个分支。
3. 对比 `amend`、`rebase -i`、`cherry-pick` 和 `merge` 各自回答什么问题。
4. 思考题：为什么历史编辑的前提几乎总是“先确认是否已共享”？
