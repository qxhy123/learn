# 第13章：restore、reset 与 revert

## 学习目标

完成本章后，你应能：

1. 区分 `restore`、`reset`、`revert` 的作用层级
2. 根据场景选择“撤工作区”“撤暂存”“撤历史”的正确命令
3. 理解为什么 `revert` 往往更适合共享历史
4. 避免把 `reset --hard` 当成万能修复按钮

---

## 13.1 先别敲命令，先回答：你到底想撤销哪一层

很多 Git 事故来自一个错误习惯：

> 还没想清楚目标，就先敲命令。

你应该先明确自己要撤的是：

- 工作区里的未提交修改
- 暂存区里的准备内容
- 最近的一次或若干次提交
- 已经共享给团队的历史

场景不同，命令就不同。

---

## 13.2 一张够用的选择表

| 你想做什么 | 首选动作 | 为什么 |
|------------|----------|--------|
| 丢掉工作区未提交修改 | `git restore <file>` | 只改文件内容，不动历史 |
| 取消暂存 | `git restore --staged <file>` | 只把内容从索引拿出来 |
| 本地重写最近未共享提交 | `git reset` | 会动引用，适合个人未共享历史 |
| 撤销已共享历史里的某次提交 | `git revert <commit>` | 通过新提交撤销旧效果，不重写共享历史 |

这不是死规定，但对大多数日常场景足够稳定。

---

## 13.3 `git restore`：恢复文件状态

### 丢弃工作区改动

```bash
git restore app.js
```

### 取消暂存

```bash
git restore --staged app.js
```

它最大的优点是语义清晰：我是在恢复文件或索引状态，而不是移动分支指针。

这也是为什么，在现代 Git 里，很多基础恢复场景更建议先想到 `restore`，而不是上来就 `reset`。

---

## 13.4 `git reset`：移动引用，并且可能影响索引与工作区

`reset` 是威力很强、也最容易误用的命令之一。

常见形式：

```bash
git reset --soft HEAD~1
git reset --mixed HEAD~1
git reset --hard HEAD~1
```

区别大致是：

- `--soft`：移动当前分支，保留索引和工作区
- `--mixed`：移动当前分支，并重置索引
- `--hard`：移动当前分支，并重置索引和工作区

也就是说，`--hard` 最危险，因为它同时会动：

- 分支引用
- 索引
- 工作区

---

## 13.5 `git revert`：通过新提交撤销旧提交

如果某次提交已经进入共享历史，通常更稳妥的方式是：

```bash
git revert <commit>
```

它不会删除原提交，而是创建一个“反向修改”的新提交。

这带来几个好处：

- 历史公开透明
- 不重写已共享历史
- 团队协作更安全

这也是为什么很多团队更鼓励在主干上使用 `revert`，而不是 `reset`。

---

## 13.6 什么时候绝对不要先用 `reset --hard`

下面这些情形，`reset --hard` 都不应该成为你的第一反应：

1. 你还没看 `git status`
2. 你不确定这条分支是否已经共享给团队
3. 你本地还有未备份的工作
4. 你只是想“撤某个文件”或“撤索引”，而不是重写整个当前位置

很多人不是不会用 Git，而是把最有杀伤力的命令当成默认方案。

---

## 13.7 一个更像事故分流的处理方式

### 症状：文件改乱了，但还没提交

先看：

```bash
git status -sb
git diff
```

安全动作：

- `git restore <file>`

不要先做：

- `git reset --hard`

### 症状：我已经 `add` 了，但现在不想把它带进提交

先看：

```bash
git diff --cached
```

安全动作：

- `git restore --staged <file>`

### 症状：我本地提交错了，但还没推送

先看：

```bash
git log --oneline --graph --decorate -n 5
```

安全动作：

- `git reset --soft` 或 `--mixed`

### 症状：错误提交已经在共享历史里

安全动作：

- `git revert`

不要先做：

- 改写主干共享历史

---

## 13.8 一个必须动手做的对照实验

在练习仓库中：

1. 先改文件但不提交，练 `restore`
2. 再 `add` 后练 `restore --staged`
3. 再做两次本地提交，分别试 `reset --soft`、`reset --mixed`、`reset --hard`
4. 再模拟“已共享提交”，用 `revert` 撤销

每一步都观察：

```bash
git status -sb
git diff
git diff --cached
git log --oneline --graph --decorate --all
```

这会让“不同命令到底动哪一层”真正落地。

---

## 常见误区

- **误区 1：所有撤销都用 `reset --hard`。**
  这是最危险的坏习惯之一。

- **误区 2：`revert` 会删除旧提交。**
  不会，它会创建一个反向提交。

- **误区 3：`restore` 和 `reset` 只是新旧写法差异。**
  它们的语义边界不同。

- **误区 4：只要本地看起来没问题，就可以在共享分支上 reset。**
  这会直接把个人操作变成团队事故。

## 本章练习

1. 修改一个文件但不提交，使用 `git restore` 丢弃改动。
2. 修改一个文件并 `git add`，然后用 `git restore --staged` 取消暂存。
3. 做两次本地提交，比较 `reset --soft`、`--mixed`、`--hard` 的效果差异。
4. 思考题：为什么共享主干上通常更适合 `revert` 而不是 `reset`？
