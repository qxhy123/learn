# 第32章：如何定位 DNS、TCP、TLS、HTTP 问题

## 本章目标

- 学会把“网站打不开”这类笼统问题拆成分层问题
- 理解 DNS、TCP、TLS、HTTP 各层出问题时常见表现有什么不同
- 建立一条可重复使用的排查路径，而不是每次靠猜
- 明白排障的关键不是记住所有命令，而是先判断问题发生在哪一层

## 先讲结论

面对网络问题，最有效的思路通常不是：

- 见报错就随机搜索
- 换几个工具碰碰运气

而是先问自己一句话：

- **问题卡在 DNS、TCP、TLS，还是 HTTP？**

因为这几层虽然最后都可能表现成“网页打不开”，但本质完全不同：

- DNS 问题：连目标 IP 都不知道
- TCP 问题：知道目标是谁，但底层连接建不起来
- TLS 问题：TCP 通了，但安全握手失败
- HTTP 问题：连接和握手都成功了，但应用请求或响应有异常

只要把问题先分层，排查就会从混乱变成有路径。

## 核心概念

### 1. 为什么“网页打不开”不是一个足够好的诊断结论

“打不开”只是用户视角的表面现象，背后可能是很多不同原因：

- 域名解析失败
- 目标端口不通
- 证书错误
- 请求被重定向循环
- 服务器返回 500
- 某个关键静态资源加载失败

如果不先分层，后续所有动作都容易变成无效尝试。

所以真正专业的排障第一步不是修，而是：

- **把现象翻译成协议层次**

### 2. DNS 层问题通常长什么样

DNS 出问题时，常见表现包括：

- 域名无法解析
- 解析结果和预期不一致
- 解析非常慢
- 不同网络环境下结果不同

用户层面可能看到：

- 浏览器提示找不到服务器
- 请求根本没有进入建连阶段

这类问题的关键词通常是：

- 域名
- 解析
- 递归 DNS
- 缓存
- TTL

### 3. TCP 层问题通常长什么样

TCP 问题意味着：

- 域名已经能解析到 IP
- 但和目标主机的端口连不起来，或者连接很不稳定

常见表现包括：

- 建连超时
- 连接被拒绝
- 频繁重传
- 三次握手走不完

这类问题通常和这些因素相关：

- 防火墙
- 安全组
- 目标服务未监听
- 网络丢包严重
- 路由异常

### 4. TLS 层问题通常长什么样

TLS 问题意味着：

- TCP 已经通了
- 但 HTTPS 安全握手没成功

常见表现包括：

- 证书不受信任
- 域名不匹配
- 握手失败
- 协议版本不兼容
- 某些客户端能访问，另一些客户端不行

这时重点就不该再盯着“能不能 ping 通”，而要转去看：

- 证书链
- SNI
- TLS 版本
- 密码套件协商

### 5. HTTP 层问题通常长什么样

HTTP 问题意味着：

- DNS 正常
- TCP 正常
- TLS 也正常
- 但应用层语义或服务器处理过程有问题

常见表现包括：

- `404`
- `401`
- `403`
- `500`
- 重定向循环
- CORS 失败
- 缓存异常
- 接口返回的数据格式不对

这时候问题已经不在“通不通”，而在：

- 请求发得对不对
- 服务器理解得对不对
- 响应是否符合预期

## 过程拆解

### 一条从现象到定位的实用路径

假设用户反馈：

- 网站打不开

可以按照下面顺序判断。

#### 第一步：先确认是不是 DNS 问题

先问：

- 域名能不能解析出 IP
- 解析到的 IP 是不是预期目标
- 同一域名在不同环境结果是否一致

如果这里就失败了，后面 TCP、TLS、HTTP 都还没开始谈。

所以 DNS 层一旦有问题，要优先解决：

- 域名配置
- 解析链路
- 缓存污染或延迟

#### 第二步：如果 DNS 正常，再看 TCP 能否连上

此时重点变成：

- 目标 IP 的目标端口是否可达
- 是超时、拒绝，还是间歇性失败

如果 TCP 建连都失败，那么：

- TLS 根本还没开始
- HTTP 更无从谈起

这一步常常能把问题定位到：

- 网络访问控制
- 服务监听状态
- 负载均衡或网关配置

#### 第三步：如果 TCP 正常，再看 TLS 是否握手成功

如果是 HTTPS，下一步重点看：

- 证书是否有效
- 域名是否匹配
- 证书链是否完整
- 客户端和服务端支持的 TLS 版本是否兼容

这一步非常关键，因为很多团队会误把 TLS 失败当成“服务端挂了”。

实际上它往往是：

- 证书部署错误
- 中间证书漏配
- SNI 问题
- 老客户端和新配置不兼容

#### 第四步：如果 TLS 也成功，再看 HTTP 请求本身

此时就要重点关注：

- 请求方法对不对
- 路径对不对
- 头部有没有带齐
- Cookie、Token 是否正确
- 响应状态码和响应体是否符合预期

如果浏览器里出问题而 curl 没问题，还要进一步考虑：

- 缓存
- CORS
- 重定向策略
- Cookie SameSite 或 Secure 属性
- 前端资源加载链路

### 一个简单判断表

你也可以先用下面这种心智模型快速判断：

- **域名都解不出来**：先查 DNS
- **能解析但端口不通**：先查 TCP / 网络连通性
- **端口通但 HTTPS 报证书或握手错**：先查 TLS
- **连接都正常但返回 4xx/5xx 或页面行为异常**：先查 HTTP 和应用层

## 常见误区

### 误区 1：能 ping 通就说明网站没问题

不对。ping 只说明某种 ICMP 层面的可达性，不能证明：

- TCP 443 一定可用
- TLS 握手一定成功
- HTTP 服务一定正常

### 误区 2：证书报错就是服务器宕机

不对。很多证书错误说明的是：

- 服务在
- TCP 也通
- 但身份验证或 TLS 配置有问题

### 误区 3：HTTP 返回 500 说明是网络问题

通常不是。`500` 更接近应用服务器处理请求时发生了内部错误。

### 误区 4：只要浏览器打不开，就必须先抓包

不一定。很多问题先通过分层判断和简单工具就能快速缩小范围，抓包通常是后续取证手段。

## 小结

- 网络排障最核心的动作，是先把问题落到 DNS、TCP、TLS、HTTP 的某一层
- DNS 解决“去哪儿”，TCP 解决“能不能连上”，TLS 解决“能不能安全建立会话”，HTTP 解决“请求和响应是否正确”
- 分层后再选工具，效率会远高于盲目尝试
- 真正会排障的人，不是记命令最多的人，而是最会判断问题层次的人
