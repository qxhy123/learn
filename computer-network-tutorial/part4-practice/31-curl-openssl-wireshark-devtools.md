# 第31章：curl、OpenSSL、Wireshark 与 DevTools

## 本章目标

- 理解几个常见网络排查工具分别适合看什么层面
- 明白命令行工具、抓包工具、浏览器调试工具之间的分工
- 知道学习网络不只是背协议，还要学会观察真实流量和真实请求
- 建立“先定位层次，再选工具”的排查思路

## 先讲结论

学习网络和排查问题时，最容易犯的错之一是：

- 上来就乱试工具
- 但根本不知道自己要看哪一层

更好的做法是先想清楚：

- 是 DNS 问题
- 是 TCP 建连问题
- 是 TLS 证书或握手问题
- 还是 HTTP 请求本身的问题

然后再选合适工具：

- **curl**：适合快速看 HTTP/HTTPS 请求与响应
- **OpenSSL**：适合看证书、TLS 握手和链路细节
- **Wireshark**：适合从数据包层面看整个过程
- **浏览器 DevTools**：适合观察真实页面加载中的网络行为

这四类工具结合起来，几乎能覆盖网络学习和 Web 排障里的大部分基础场景。

## 核心概念

### 1. curl 更像“最小可复现客户端”

curl 的价值不只是“发一个请求”，而是：

- 不经过浏览器复杂渲染流程
- 可以精确控制方法、头部、请求体
- 可以直观看到响应头、状态码、重定向等信息

例如你想知道：

- 一个接口到底返回了什么状态码
- 重定向跳到了哪里
- 服务端有没有回 `Content-Type`
- 某个鉴权头带上后行为有没有变化

这时 curl 往往比浏览器更直接。

很多 HTTP 层问题，先用 curl 复现，会比盯着页面报错更快。

### 2. OpenSSL 更适合看 TLS 和证书

如果你怀疑问题在 HTTPS 这一层，OpenSSL 往往是非常有价值的。

它能帮你观察：

- 服务器证书是什么
- 证书链是否完整
- 域名和证书是否大致对应
- TLS 版本、密码套件协商结果是什么

例如：

```bash
openssl s_client -connect example.com:443 -servername example.com
```

这个命令的学习价值在于，它能让你把“浏览器里一个笼统的证书错误”，拆成更具体的问题，例如：

- 是链不完整
- 是证书过期
- 是 SNI 没带对
- 还是 TLS 协商本身失败

### 3. Wireshark 适合看“真实发生了什么包交换”

如果你已经怀疑：

- DNS 查询发没发出去
- TCP 三次握手到底停在哪一步
- TLS 握手有没有被重置
- HTTP 重传、乱序、RST 是否存在

Wireshark 就很有用。

它不是最适合第一步就打开的工具，但一旦问题需要看到包级证据，它非常强。

你可以把 Wireshark 理解成：

- 把“我感觉请求没发出去”变成“我亲眼看到这里确实没收到 ACK”
- 把“我怀疑证书有问题”变成“服务器在握手阶段返回了哪些消息”

也就是说，它能把猜测变成证据。

### 4. DevTools 适合看真实浏览器上下文

很多问题只有在浏览器里才会出现，例如：

- 页面真实加载顺序
- 资源瀑布图
- 缓存命中情况
- CORS 报错
- Cookie 是否被带上
- 某些重定向和预检请求行为

此时浏览器 DevTools 中的 Network 面板就很关键。

它的优势在于：

- 贴近真实用户访问路径
- 能看到每个资源请求的时序和依赖关系
- 能结合控制台错误一起看

所以，如果问题和“浏览器行为”强相关，DevTools 往往比纯命令行工具更贴近真相。

### 5. 这四类工具是互补关系，不是替代关系

可以先这样记：

- curl：最适合最小化复现 HTTP 行为
- OpenSSL：最适合直查 TLS 与证书
- Wireshark：最适合拿到底层流量证据
- DevTools：最适合观察真实页面加载与浏览器策略

真正高效的排查，通常不是只靠一个工具，而是：

- 先用简单工具缩小范围
- 再用更底层工具确认细节

## 过程拆解

### 遇到一个 HTTPS 页面打不开时，可以怎么用这些工具

假设你访问：

`https://example.com`

浏览器里表现为“页面打不开”或“请求失败”，可以有这样一条实用排查路径。

#### 第一步：先用 DevTools 看浏览器到底报了什么

先看 Network 面板和控制台，重点看：

- 请求有没有真正发出
- 状态码是多少
- 是 DNS 失败、连接失败、TLS 错误，还是 HTTP 返回异常
- 是否有重定向循环、CORS 报错、Mixed Content 问题

这一步的意义是：

- 先别猜，先收集现象

#### 第二步：用 curl 做最小化复现

例如：

```bash
curl -v https://example.com/
```

可以重点观察：

- 是否能连上
- 最终状态码是什么
- 响应头里有没有异常
- 是否发生了重定向

如果 DevTools 里页面加载很复杂，而 curl 访问首页却很正常，这通常说明：

- 问题可能不在基础连通性
- 而在某个静态资源、跨域请求、Cookie、缓存或浏览器特定行为上

#### 第三步：如果怀疑证书或握手问题，用 OpenSSL 深挖

例如：

```bash
openssl s_client -connect example.com:443 -servername example.com
```

可以关注：

- 服务端返回了哪些证书
- 证书链是否完整
- 握手有没有中途失败
- 服务器选用了什么 TLS 版本和密码参数

这一步尤其适合排查：

- 域名证书不匹配
- 中间证书缺失
- 某些客户端握手失败但另一些成功

#### 第四步：如果仍不清楚，就抓包看证据

这时可以用 Wireshark 观察：

- DNS 查询是否成功返回
- TCP 三次握手是否完成
- TLS ClientHello 和 ServerHello 是否正常往返
- 是否存在 RST、重传、超时

Wireshark 特别适合回答这种问题：

- 请求是没发出去，还是发出去了但没回
- 是客户端主动断开，还是服务端主动断开
- 是网络中间节点在丢包，还是应用层返回错误

## 常见误区

### 误区 1：curl 能访问，就说明浏览器一定也没问题

不对。浏览器还涉及：

- Cookie
- CORS
- 缓存
- 重定向策略
- 资源加载顺序
- 安全策略

curl 正常只能说明基础请求层面可能没问题，不代表浏览器上下文一定正常。

### 误区 2：OpenSSL 只是给搞加密的人用的

不对。只要你排查 HTTPS、证书、握手失败，OpenSSL 就非常实用。

### 误区 3：Wireshark 一定是排障第一步

不一定。很多问题先用 DevTools、curl、OpenSSL 就能快速定位。Wireshark 更适合在需要包级证据时再上。

### 误区 4：DevTools 只是前端开发工具

不对。它对理解 HTTP 请求时序、缓存、Cookie、CORS 和资源加载问题都非常重要，本质上也是网络调试工具。

## 小结

- 网络排查要先分层，再选工具
- curl 适合最小化复现 HTTP/HTTPS 请求
- OpenSSL 适合查看 TLS 握手和证书问题
- Wireshark 适合拿到底层流量证据，DevTools 适合观察真实浏览器行为
- 真正高效的排查，不是迷信某个工具，而是会让工具配合起来工作
