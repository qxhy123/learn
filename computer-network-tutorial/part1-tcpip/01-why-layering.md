# 第1章：为什么要分层

## 本章目标

- 理解网络协议为什么不能杂糅成一个大系统来设计
- 明白“分层”解决的是复杂度、协作和兼容性问题
- 认识每一层大致负责什么，以及层与层之间如何配合
- 能用“浏览器访问网页”这个例子解释分层的价值

## 先讲结论

网络分层的核心目的，不是为了考试好背，而是为了把复杂系统拆成多个相对稳定、职责明确的模块。

简单说：

- **上层关心“我要什么服务”**，下层关心“怎么把它送出去”
- **每层只暴露有限接口**，减少系统整体耦合
- **某一层升级时，最好不要把所有层都改一遍**
- **不同厂商、不同设备、不同操作系统**，只要遵守同一层的协议，就能互联互通

如果没有分层，浏览器、操作系统、网卡、路由器、交换机都会绑死在一起，任何小改动都可能影响整个通信系统。

## 核心概念

### 1. 分层是在对抗复杂度

一次网络通信，表面上看只是“发一条消息”，实际上背后至少包含这些问题：

- 应用怎么组织消息格式
- 怎么标识目标机器
- 怎么在多个网络之间转发
- 怎么在同一局域网找到目标设备
- 电信号、无线信号如何真正发送

把这些问题揉成一个协议，会非常难维护。分层的做法是：**按职责拆开，每层解决一类问题**。

### 2. 每层都在为上一层提供服务

可以把分层想成“层层外包”：

- 应用层：我想传输一段业务数据
- 传输层：我负责端到端传输
- 网络层：我负责跨网络寻址和转发
- 链路层：我负责同一链路上的交付
- 物理层：我负责把比特真正发出去

因此，上层通常**不需要直接理解下层全部细节**。例如浏览器通常不关心网卡芯片怎么发电信号。

### 3. 分层不是完全隔离，而是“有边界地协作”

分层不是说每层都与世隔绝，而是说协作要通过明确规则进行。

比如：

- HTTP 不直接操心 IP 路由，但它依赖 TCP 提供可靠传输
- TCP 不直接决定交换机怎么转发，但它依赖 IP 提供主机到主机的送达能力
- IP 不直接存储网页内容，但它要把承载这些内容的分组送到目标主机

### 4. 分层带来的收益

分层最重要的收益有四个：

1. **降低理解成本**：学习时可以一层层掌握
2. **降低实现成本**：不同团队可以分工实现不同层
3. **提高兼容性**：协议稳定后，不同实现也能互通
4. **方便替换和演进**：例如应用协议可以从 HTTP/1.1 演进到 HTTP/2、HTTP/3，而底层并不需要全部推倒重来

## 过程拆解

### 用“打开一个网页”理解分层

假设你在浏览器里输入一个网址并回车，大致会发生：

1. **应用层**：浏览器构造 HTTP 请求，例如“请给我首页内容”
2. **传输层**：HTTP 请求交给 TCP 或其他传输协议，负责端到端发送
3. **网络层**：传输层数据被交给 IP，IP 负责决定目标主机地址以及跨网络转发
4. **链路层**：IP 数据再交给以太网或 Wi‑Fi，负责在当前链路上发给下一跳设备
5. **物理层**：网卡把这些比特变成电信号、光信号或无线电波发出去

接收方则反过来一层层拆开，最终把请求交给 Web 服务器程序处理。

这个过程说明：

- 分层不是多余步骤，而是**让每一步各司其职**
- 同一个应用请求，会在不同层被看成不同形式的数据
- 每层都只处理自己那一层最关心的信息

## 常见误区

### 误区 1：分层越多越好

不是。层数太多会增加额外开销和理解负担。合理分层的关键是：**边界清晰、职责稳定、接口简单**。

### 误区 2：分层后就完全互不影响

不是。虽然每层尽量解耦，但上层业务特征仍会影响下层表现。比如短连接、长连接、大文件传输、实时音视频，对传输层和网络层的压力都不一样。

### 误区 3：真实网络一定严格按教材图工作

不是。教材中的分层是**理解世界的模型**，现实实现中常会出现合并、优化、跨层协作等情况，但总体思想仍然有效。

## 小结

- 分层的本质，是把复杂通信系统拆成职责明确的模块
- 每层向上提供服务，向下提出需求
- 分层的价值在于降低复杂度、支持分工、增强兼容性和可演进性
- 学网络时，不要只背层名，更要理解“这一层到底解决什么问题”
