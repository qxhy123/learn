# 第7章：DNS 基础

## 本章目标

- 理解 DNS 为什么存在，以及它解决什么问题
- 明白域名、IP、DNS 服务器之间的关系
- 认识递归查询、迭代查询、缓存等基本概念
- 能用“浏览器访问网站”说明 DNS 查询的大致流程

## 先讲结论

DNS 的核心作用是：**把人更容易记住的域名，转换成机器通信需要的 IP 地址。**

如果没有 DNS，用户就得直接记住大量 IP 地址，这几乎不现实。

但 DNS 不只是一个“电话本”，它还是一个分布式、分层、带缓存的名字解析系统。

## 核心概念

### 1. 域名和 IP 的关系

我们更习惯记：

- `www.example.com`
- `api.example.com`

而网络真正转发时更依赖：

- `93.184.216.34`
- `203.0.113.10`

DNS 就在中间做转换。

### 2. DNS 服务器不是只有一台

DNS 是分层分布式系统，常见角色包括：

- 本地缓存 / 操作系统缓存
- 递归解析器
- 根服务器
- 顶级域服务器
- 权威 DNS 服务器

它们共同协作完成解析，而不是所有请求都问同一台总服务器。

### 3. 递归查询与迭代查询

可以先粗略理解：

- **递归查询**：你把问题完整交给递归解析器，让它帮你一路问到底
- **迭代查询**：某台 DNS 服务器告诉你“我不知道最终答案，但你下一步可以去问谁”

普通用户设备通常更像是向递归解析器发起递归请求。

### 4. DNS 缓存为什么重要

DNS 如果每次都从根开始问，效率会很低。

因此大量使用缓存：

- 浏览器缓存
- 操作系统缓存
- 递归解析器缓存

缓存能显著减少查询次数、降低延迟，但也意味着记录更新后不一定马上全网同步，这和 TTL 有关。

## 过程拆解

### 浏览器访问一个域名时，DNS 大致怎么工作

假设你访问：

`https://www.example.com`

大致过程：

1. 浏览器先查自己有没有缓存结果
2. 没有的话，操作系统也会查本地缓存
3. 还没有，就把查询发给配置好的递归 DNS 服务器
4. 递归 DNS 如果也没有缓存，可能去问：
   - 根服务器：`.com` 该问谁
   - `.com` 顶级域服务器：`example.com` 该问谁
   - `example.com` 的权威 DNS：`www.example.com` 的 IP 是多少
5. 递归 DNS 拿到结果后返回给客户端，并缓存一段时间
6. 浏览器得到 IP 后，才继续发起后续 TCP/HTTPS 连接

所以，很多人以为“浏览器直接连域名”，其实真正连的仍然是 IP，只不过在连接之前先做了域名解析。

## 常见误区

### 误区 1：DNS 就是一张静态表

不对。DNS 是分布式系统，有层级、有缓存、有动态更新，不是一张固定名单。

### 误区 2：拿到一个域名就一定只对应一个 IP

不对。一个域名可能对应多个 IP，用于负载均衡、容灾和就近接入。

### 误区 3：DNS 查询一定很慢

不一定。很多时候缓存命中后会非常快，真正慢的是缓存失效后的完整解析路径，或者网络质量不佳时。

## 小结

- DNS 的核心作用是把域名解析成 IP 地址
- DNS 是分层、分布式、带缓存的系统
- 普通用户通常向递归解析器发请求，再由它帮助完成后续查询
- DNS 往往是浏览器发起网络访问前的重要第一步
