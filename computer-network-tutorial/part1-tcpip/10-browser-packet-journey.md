# 第10章：浏览器发出一个包的旅程

## 本章目标

- 把前面学过的 DNS、TCP、IP、MAC、路由等知识串起来
- 能从浏览器输入网址开始，描述一次典型网络访问流程
- 形成对“应用到网络底层再到对端返回”的整体直觉
- 理解一个网页请求背后其实是多层协议协作的结果

## 先讲结论

浏览器访问一个网页，不是“直接把网址发出去”这么简单，而是一个多阶段流程：

1. 解析域名
2. 建立连接
3. 发送 HTTP 请求
4. 数据逐层封装并经由网卡、交换机、路由器传输
5. 服务器处理后再把响应按相反方向返回

如果把网络学习拆成很多章节，那么这一章的意义就是把它们重新拼成一条完整路径。

## 核心概念

### 1. 一次访问涉及的不只是 HTTP

用户看到的是网页，开发者常想到的是 HTTP，但真正完成访问的协议通常不止一个：

- DNS：把域名解析成 IP
- TCP：建立可靠连接
- IP：跨网络寻址和转发
- ARP / 以太网 / Wi‑Fi：在本地链路上完成发送
- HTTP：表达“我要哪个资源”

### 2. 浏览器、操作系统、网卡、路由器都在参与

一次请求不是只有浏览器工作：

- 浏览器构造应用层请求
- 操作系统协议栈负责 TCP/IP 处理
- 网卡负责实际发帧
- 路由器负责跨网络转发
- 服务器协议栈和应用程序负责接收与响应

### 3. “一个包的旅程”通常不是只发一个包

实际中常常会有很多包：

- DNS 查询包
- TCP 握手包
- HTTP 请求包
- HTTP 响应包
- 可能还有多个资源的并发请求

所以“一个网页加载”通常是许多网络交互共同组成的。

## 过程拆解

### 以访问 `https://www.example.com` 为例

#### 第一步：输入 URL

浏览器先解析 URL，知道：

- 协议是 `https`
- 域名是 `www.example.com`
- 默认端口通常是 `443`
- 路径可能是 `/`

#### 第二步：DNS 解析

浏览器需要先知道目标服务器 IP。

- 先查浏览器缓存
- 再查系统缓存
- 没有就去问递归 DNS
- 最终拿到目标 IP

#### 第三步：确定怎么把数据发出去

操作系统查看路由信息：

- 如果目标在同一子网，直接找目标主机 MAC
- 如果不在同一子网，就找默认网关 MAC

此时可能通过 ARP 获得下一跳 MAC 地址。

#### 第四步：建立传输连接

浏览器通常要先和目标服务器建立 TCP 连接：

- 客户端发 SYN
- 服务器回 SYN+ACK
- 客户端回 ACK

这样双方建立了连接状态。

#### 第五步：发送应用层请求

浏览器构造 HTTP 请求，例如：

- 请求方法
- 路径
- Host 头
- 其他请求头

然后把这份请求交给 TCP。

#### 第六步：逐层封装并发送

- TCP 加上传输层头部
- IP 加上源/目标 IP
- 以太网或 Wi‑Fi 加上链路层头部
- 网卡把比特发到网络中

数据先到交换机，再可能到路由器，再跨多个网络抵达服务器。

#### 第七步：服务器接收并处理

服务器从链路层一路解封装到应用层：

- 网卡收到帧
- 内核协议栈处理 IP 和 TCP
- 数据交给监听端口上的 Web 服务
- Web 服务生成 HTTP 响应

#### 第八步：响应返回浏览器

响应会按相反方向返回，再经过浏览器解析。如果是 HTML 页面，浏览器还可能继续请求：

- CSS
- JavaScript
- 图片
- 字体等静态资源

这也是为什么一个网页往往对应很多后续网络请求。

## 常见误区

### 误区 1：输入网址后，浏览器直接把字符串发送给服务器

不对。浏览器要先解析 URL、解析 DNS、建立连接，再构造真正的 HTTP 请求。

### 误区 2：一次网页访问只发一个数据包

不对。真实情况下往往会有多次查询、握手、请求和响应。

### 误区 3：HTTP 就等于整个上网过程

不对。HTTP 只负责应用层语义，下面还有传输、网络、链路等多层协作。

## 小结

- 浏览器访问网页是多协议协同完成的过程
- DNS 负责找 IP，TCP 负责连接，IP 负责转发，链路层负责本地发送，HTTP 负责业务语义
- 一个包的旅程，本质上是分层模型协作的具体体现
- 如果你能把这一整条路径讲清楚，说明你已经对网络有了系统性的初步理解
