# 第9章：TCP 三次握手与可靠性

## 本章目标

- 理解 TCP 为什么要先建立连接
- 明白三次握手的核心作用不是“形式主义”
- 认识序号、确认应答、重传等可靠性基础机制
- 对 TCP 为什么能提供相对可靠传输形成整体认识

## 先讲结论

TCP 之所以可靠，不是因为它“魔法般不会出错”，而是因为它在通信双方之间建立了一套明确的状态和确认机制。

三次握手的核心价值包括：

- 确认双方都具备收发能力
- 协商连接初始状态
- 为后续可靠传输建立上下文

而 TCP 的可靠性主要来自：

- 序号
- 确认应答
- 超时重传
- 滑动窗口
- 流量控制与拥塞控制

## 核心概念

### 1. 为什么 TCP 要握手

因为 TCP 不是“扔一个包就完事”，它要维护连接状态。

在真正收发业务数据之前，双方需要先搞清楚：

- 你在不在线
- 你能不能接收
- 你的初始序号是什么
- 我们接下来按什么状态开始通信

如果不先做这些确认，后面就很难保证有序和可靠。

### 2. 三次握手的基本过程

最常见的描述是：

1. 客户端发 `SYN`
2. 服务端回 `SYN + ACK`
3. 客户端再回 `ACK`

可以粗略理解成：

- 第一次：我想和你建立连接
- 第二次：我收到了，而且我也愿意和你建立连接
- 第三次：我确认收到了你的回应，我们都准备好了

### 3. 序号与确认应答

TCP 传输的数据不是“没有编号地乱飞”，而是带有序号。

这样做的好处是：

- 接收方知道收到的是哪一段数据
- 可以发现丢失、重复、乱序
- 可以通过 ACK 告诉发送方“我已经收到哪里了”

可靠性不是通过“保证永不出错”实现的，而是通过“出了问题也能检测并补救”实现的。

### 4. 重传与窗口机制

如果发送方在一定时间内没有等到确认，就可能认为包丢了，然后重传。

但如果每发一个字节都等确认，效率会极低，所以 TCP 使用窗口机制：

- 可以连续发送一批数据
- 再根据 ACK 滑动窗口
- 兼顾效率与可靠性

## 过程拆解

### 从三次握手到可靠传输的大致逻辑

假设浏览器要连接网站服务器：

1. 浏览器先发送 `SYN`，请求建立 TCP 连接
2. 服务器回复 `SYN + ACK`，表示同意并确认收到请求
3. 浏览器再发 `ACK`，表示双方连接建立完成
4. 后续浏览器发送 HTTP 请求数据，每段数据都有序号
5. 服务器收到后返回 ACK，表明已经成功接收
6. 如果某段长时间没被确认，发送方就会重传
7. 双方还会根据接收能力和网络状况动态调整发送节奏

这里最关键的是：

- TCP 不只是“先打招呼”
- 它是在建立一个后续可靠传输所依赖的状态机

## 常见误区

### 误区 1：三次握手只是为了防止攻击

不全面。它确实和一些异常场景、攻击防御有关，但更基本的作用是建立连接状态并确认双方通信能力。

### 误区 2：TCP 可靠就意味着绝对不出错

不对。真实网络仍然会丢包、乱序、拥塞。TCP 的价值在于能检测并处理很多常见问题。

### 误区 3：ACK 就是“收到一个包回一个包”

不完全对。真实实现里可能会延迟确认、累计确认，不一定机械地一一对应。

## 小结

- 三次握手用于建立 TCP 连接状态并确认双方可通信
- TCP 可靠性来自序号、ACK、重传、窗口等多种机制配合
- 可靠不是“永不出错”，而是“能发现问题并尽量恢复”
- 理解 TCP 的状态与确认机制，是理解 HTTP/1.1、数据库连接等应用的基础
