# 第8章：TCP 与 UDP

## 本章目标

- 理解 TCP 和 UDP 各自的设计目标
- 知道“可靠”与“快”并不是简单二选一
- 能根据业务场景判断何时更适合 TCP，何时更适合 UDP
- 建立对连接、顺序、重传、时延等问题的基本直觉

## 先讲结论

TCP 和 UDP 都属于传输层协议，但风格很不同：

- **TCP**：面向连接，强调可靠、有序、流量控制、拥塞控制
- **UDP**：无连接，开销更小，更直接，不保证可靠和有序

所以不要把它们简单理解成“TCP 好，UDP 差”，更准确的说法是：

- TCP 更适合对正确性和完整性要求高的场景
- UDP 更适合对实时性和简单性要求更高的场景

## 核心概念

### 1. TCP 在解决什么问题

网络本身并不天然保证：

- 数据不丢
- 数据按顺序到达
- 对方一定能接住
- 网络不会被打爆

TCP 就是在传输层尽量解决这些问题。它通过：

- 序号
- 确认应答
- 重传机制
- 窗口机制
- 拥塞控制

来提供一种“更可靠的字节流传输”。

### 2. UDP 在解决什么问题

UDP 的目标非常克制：

- 提供端口复用
- 封装数据并发出去
- 尽量少做额外保证

它不负责复杂的可靠性和顺序控制，因此协议头更简单、状态更少、延迟路径也可能更短。

### 3. “无连接”是什么意思

UDP 的无连接，不是说网络里没有路径，而是说：

- 发送前不需要像 TCP 那样先建立连接状态
- 每个数据报可以相对独立地发送

而 TCP 要先建立连接，通信双方都维护状态，再在此基础上收发数据。

### 4. 为什么很多实时应用喜欢 UDP

实时音视频、在线游戏等场景有个特点：

- 稍微丢一点数据，用户未必明显感知
- 但如果为了等重传而延迟太久，体验会很差

这类场景通常更在意“尽快拿到当前最新数据”，而不是“百分之百把之前每一个包都补齐”。

## 示例

### 示例 1：下载文件为什么常用 TCP

下载文件时，用户通常要求：

- 不能缺字节
- 顺序不能乱
- 传输中断后最好还能恢复

这天然适合 TCP 的可靠传输能力。

### 示例 2：语音通话为什么常常偏向 UDP 思路

语音通话里，如果一句话晚到 2 秒才补回来，意义已经不大。此时：

- 少量丢包往往比高延迟更能接受
- 保持实时连续感更重要

因此这类业务常基于 UDP，再在应用层补充自己的容错与控制策略。

## 常见误区

### 误区 1：UDP 一定比 TCP 快

不绝对。UDP 协议头简单、状态少，这是事实，但真实应用性能还受网络状况、应用实现、重试策略、拥塞情况等影响。

### 误区 2：TCP 就不会丢包

不对。网络中仍然可能丢包，只是 TCP 会通过确认和重传机制尽量恢复可靠性。

### 误区 3：UDP 完全没有用处

不对。很多实时系统、DNS 查询、流媒体、QUIC 等都高度依赖 UDP 或建立在 UDP 之上。

## 小结

- TCP 强调可靠、有序、受控的传输
- UDP 强调简单、低状态、低额外开销
- 选择哪一个，不是看谁“高级”，而是看业务更在意正确性还是实时性
- 后续理解 HTTP、HTTPS、实时通信时，TCP/UDP 的差异会不断出现
