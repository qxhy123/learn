# 第3章：封装与解封装

## 本章目标

- 理解什么叫“封装”和“解封装”
- 知道为什么同一份数据在不同层会有不同名字
- 认识头部、载荷、尾部在通信中的作用
- 能描述一段应用数据如何逐层打包并被对端逐层拆开

## 先讲结论

封装的本质就是：**上层数据交给下层时，下层会在外面加上本层需要的控制信息**。接收端则反过来一层层拆掉这些信息，这叫解封装。

因此，一条 HTTP 请求在发送过程中，不是“原样飞过去”的，而是会依次变成：

- 传输层的数据单元
- 网络层的数据包
- 链路层的帧

每一层都会加上“自己关心的信息”，例如端口号、IP 地址、MAC 地址等。

## 核心概念

### 1. 头部、载荷、尾部

一层处理数据时，通常会形成：

- **头部（Header）**：说明本层如何处理这份数据
- **载荷（Payload）**：上层交下来的内容
- **尾部（Trailer）**：少数协议会在末尾再附加信息，例如校验信息

可以把它理解为“快递打包”：

- 货物本身是载荷
- 快递单是头部
- 某些封条或校验标记类似尾部

### 2. 为什么要加这些信息

不同层关心的问题不同：

- 传输层关心发给哪个进程，所以要有端口号
- 网络层关心发给哪台主机，所以要有 IP 地址
- 链路层关心当前这一跳发给哪块网卡，所以要有 MAC 地址

因此，封装不是重复劳动，而是**每层补齐自己完成任务所需的信息**。

### 3. 数据单元名称为什么不同

教材里常见这些名字：

- 应用层：消息、报文
- 传输层：TCP 段（segment）/ UDP 数据报
- 网络层：IP 包、IP 数据报、分组（packet）
- 链路层：帧（frame）
- 物理层：比特流（bit stream）

名字不同，不是因为内容彻底变了，而是因为**观察视角变了**。

### 4. 解封装发生在接收端

接收端从网卡收到比特流后，会逐层向上处理：

1. 先看链路层信息
2. 再看网络层信息
3. 再交给传输层
4. 最终交给应用程序

如果某一层发现信息不符合要求，例如目的端口不对、校验失败，就可能直接丢弃。

## 过程拆解

### 以“发送一个 HTTP 请求”为例

假设浏览器要发送这样一句应用数据：

`GET /index.html HTTP/1.1`

发送时大致会这样封装：

1. **应用层**：生成 HTTP 请求报文
2. **传输层**：在前面加 TCP 头，写入源端口、目标端口、序号等信息
3. **网络层**：再加 IP 头，写入源 IP、目标 IP
4. **链路层**：再加帧头，写入源 MAC、目标 MAC；可能还会加帧尾校验
5. **物理层**：把这些内容转换成电信号、光信号或无线信号发出去

接收端则反过来：

1. 网卡收帧并校验
2. 交给 IP 层检查目标 IP
3. 再交给 TCP 层检查目标端口和序号
4. 最终把 HTTP 请求交给 Web 服务器程序

## 常见误区

### 误区 1：封装就是加密

不对。封装只是加上协议控制信息，加密是另外一件事。TLS 可以加密 HTTP 内容，但那是安全层面的机制，不等于封装本身。

### 误区 2：每一层都会修改原始业务内容

不一定。很多时候下层主要是在外面包一层头部，业务数据本身仍作为载荷传递。

### 误区 3：解封装一定要经过所有层

不一定。如果某一层发现目标不是自己，或者校验失败，数据可能在中途就被丢弃，不会继续往上走。

## 小结

- 封装是下层给上层数据增加本层控制信息
- 解封装是接收端逐层拆掉这些信息并向上交付
- 不同层使用不同数据单元名称，是因为关注点不同
- 理解封装/解封装，是理解后续 MAC、IP、TCP、HTTP 如何协作的基础
