# 第15章：Cookie、Session 与 Token

## 本章目标

- 理解 HTTP 无状态为什么会引出登录态问题
- 分清 Cookie、Session、Token 各自的职责
- 知道它们常见的组合方式，而不是把三者混成一个概念
- 能看懂一个最基础的登录态实现思路

## 先讲结论

Cookie、Session、Token 经常一起出现，但它们不是同一层面的东西。

最实用的理解方式是：

- **Cookie**：浏览器帮你保存并自动携带的一小段数据
- **Session**：服务器侧保存的一份会话状态
- **Token**：用于证明身份或权限的一段凭证

它们之间常见的关系是：

- Cookie 可以用来装 Session ID
- Cookie 也可以用来装 Token
- Session 和 Token 都是在解决“HTTP 无状态下，怎么识别用户”的问题

所以不要问“Cookie 和 Session 哪个更高级”，更应该问：这个系统想把状态存在哪里、如何验证身份、如何传递凭证。

## 核心概念

### 1. 为什么 HTTP 需要登录态机制

HTTP 天生偏无状态。

这意味着单独看两次请求时，服务器并不会因为协议本身就自动知道：

- 这两个请求是不是同一个用户发的
- 这个用户之前是否已经登录
- 购物车和会话数据该对应谁

所以系统必须额外设计一套“识别用户”的机制。

### 2. Cookie 是浏览器侧的存储与携带机制

服务器可以在响应里通过 `Set-Cookie` 告诉浏览器保存一段数据。

以后浏览器再次请求同一站点时，通常会自动把对应 Cookie 放进请求头里。

所以 Cookie 的重点是：

- 存在浏览器一侧
- 会被浏览器自动携带
- 常用来承载会话标识、偏好设置等

它本身不等于登录系统，只是一个很常见的“载体”。

### 3. Session 是服务器侧的会话记录

Session 更像是服务器维护的一份“用户会话档案”。

例如服务器内部可能记录：

- Session ID 对应哪个用户
- 登录时间
- 权限信息
- 购物车内容

常见做法是：

1. 用户登录成功后
2. 服务器创建一条 Session 记录
3. 给浏览器一个 Session ID
4. 浏览器后续带上这个 ID
5. 服务器据此找到对应会话

这里浏览器往往通过 Cookie 携带 Session ID。

### 4. Token 更像一种可校验的身份凭证

Token 的核心是：

- 客户端带来一段凭证
- 服务器或其他服务根据规则验证这段凭证是否有效

Token 可以是：

- 随机字符串
- 签名后的结构化数据
- 带过期时间和声明信息的令牌

它常被用于：

- 前后端分离接口认证
- 多服务之间身份传递
- 移动端、单页应用、开放 API 等场景

要注意，Token 不等于一定“无状态”，因为实际系统里仍可能需要额外的黑名单、刷新机制、失效管理等配套设计。

## 示例

### 示例 1：Cookie + Session 的经典流程

1. 用户提交用户名和密码
2. 服务器验证成功
3. 服务器创建一条 Session 记录
4. 服务器通过 `Set-Cookie` 把 Session ID 发给浏览器
5. 浏览器后续请求自动带上这个 Cookie
6. 服务器收到后查 Session，识别出用户身份

这种方案的特点是：

- 浏览器保存的是一个会话标识
- 真正的用户状态主要保存在服务器端

### 示例 2：Token 认证的常见流程

1. 用户登录成功
2. 服务器签发一个 Token
3. 客户端保存这个 Token
4. 后续请求时通过请求头携带，例如 `Authorization`
5. 服务器验证 Token，确认用户身份和权限

这种方式的特点是：

- 客户端显式携带凭证
- 服务端更关注“验证令牌是否有效”

## 常见误区

### 误区 1：Cookie 不安全，Token 就一定安全

不对。安全性不取决于名字，而取决于整体设计。

例如：

- Cookie 可能受 XSS、CSRF 等影响
- Token 也可能被窃取、泄露、滥用

关键在于存放方式、传输方式、过期策略和防护设计。

### 误区 2：Session 和 Cookie 是一回事

不对。Cookie 在浏览器端，Session 在服务器端。很多时候二者配合使用，但概念不同。

### 误区 3：用了 Token 就完全不用保存任何服务器状态

不对。很多真实系统仍会保存刷新令牌、撤销列表、登录设备信息等辅助状态。

## 小结

- HTTP 无状态使得系统需要额外的身份保持机制
- Cookie 是浏览器端的存储与自动携带机制
- Session 是服务器端维护的会话状态
- Token 是用于认证或授权的凭证，常用于接口和分布式场景
- 真正理解三者，要看“状态放哪、凭证怎么传、服务器如何验证”
