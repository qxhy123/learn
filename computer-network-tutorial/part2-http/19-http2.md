# 第19章：HTTP/2

## 本章目标

- 理解 HTTP/2 为什么会出现，以及它主要在改进什么
- 认识二进制分帧、多路复用、头部压缩这些关键特性
- 明白 HTTP/2 改进了传输效率，但没有改变 HTTP 语义本身
- 为理解 HTTP/3 与 QUIC 做好铺垫

## 先讲结论

HTTP/2 的重点不是“换了一套完全不同的 Web 语义”，而是：

- **保留 HTTP 的请求、响应、方法、状态码这些基本语义**
- **把底层传输组织方式改得更高效**

它最关键的改进包括：

- 二进制分帧
- 多路复用
- 头部压缩

你可以把 HTTP/2 理解为：

- 说的话还是 HTTP 那些话
- 但传话方式比 HTTP/1.1 更适合现代网页的大量并发资源请求

## 核心概念

### 1. HTTP/2 没有改掉 HTTP 的核心语义

到了 HTTP/2，下面这些概念仍然存在：

- `GET`、`POST`
- 状态码
- 请求头、响应头
- URL 与资源

也就是说，HTTP/2 不是推翻 HTTP，而是优化“这些消息到底怎么在线路上组织和传输”。

### 2. 二进制分帧让消息传输更规整

HTTP/1.1 更像是基于文本格式组织消息，而 HTTP/2 把消息拆成更底层的二进制帧来传输。

这样做的好处是：

- 机器处理更直接
- 消息边界更清晰
- 更适合高效复用同一连接

初学阶段不必记住所有帧类型，但要知道：

- 在 HTTP/2 里，一个完整请求或响应，不再只是简单的一大段文本
- 它会被拆成多个帧进行传输和组合

### 3. 多路复用提升了同一连接上的并发能力

HTTP/2 最著名的特性之一是多路复用。

它的核心意思是：

- 多个请求和响应可以同时复用同一条连接
- 不必像 HTTP/1.1 那样更容易互相排队等待

这样对网页场景很重要，因为一个页面经常要同时加载：

- HTML
- CSS
- JavaScript
- 图片
- 字体

HTTP/2 能让这些资源在同一连接里更高效地交错传输。

### 4. 头部压缩减少重复开销

浏览器访问同一站点时，很多请求头其实高度重复，例如：

- `Cookie`
- `User-Agent`
- `Accept`

HTTP/2 引入头部压缩机制，目的就是减少这些重复头部反复发送带来的额外开销。

这对高频小请求场景尤其有价值。

## 过程拆解

### 以一个网页加载多个资源为例

假设浏览器访问首页后，还要继续请求 CSS、JS 和多张图片。

在 HTTP/1.1 下，常见做法是：

- 复用少量连接
- 但同一连接中的请求响应组织方式仍较受限制

而在 HTTP/2 下，大致可以这样理解：

1. 浏览器与服务器建立一条 HTTP/2 连接
2. HTML、CSS、JS、图片请求可以映射到不同流
3. 这些流的数据再被拆成很多帧
4. 各个流的帧可以在同一连接中交错发送
5. 接收方再按流把帧重新组装成完整响应

这就是为什么 HTTP/2 往往更适合现代网页的大量并发资源加载。

## 常见误区

### 误区 1：HTTP/2 就是不再使用 HTTP 了

不对。HTTP/2 仍然是 HTTP，只是换了更高效的消息传输方式。

### 误区 2：HTTP/2 一定完全消除了所有阻塞问题

不对。HTTP/2 在应用层连接复用上改进很大，但它通常仍跑在 TCP 之上，因此底层丢包时仍可能受到 TCP 层面的影响。

### 误区 3：HTTP/2 就等于一定不用 HTTPS

不对。协议本身和是否加密不是一个概念。实际浏览器环境里，HTTP/2 通常与 HTTPS 一起使用。

## 小结

- HTTP/2 保留了 HTTP 的核心语义，但改进了底层组织方式
- 二进制分帧、多路复用、头部压缩是它的关键特性
- 它更适合现代网页的高并发资源加载场景
- 但因为通常仍基于 TCP，HTTP/2 也没有彻底解决所有传输层问题
