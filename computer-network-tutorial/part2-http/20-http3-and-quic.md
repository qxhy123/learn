# 第20章：HTTP/3 与 QUIC

## 本章目标

- 理解 HTTP/3 和 QUIC 的关系
- 明白 HTTP/3 为什么不再基于 TCP，而是建立在 QUIC 之上
- 认识 QUIC 虽然跑在 UDP 上，但并不等于“裸 UDP 传网页”
- 建立对 HTTP/1.1、HTTP/2、HTTP/3 演进路径的整体认识

## 先讲结论

HTTP/3 可以先简单理解为：

- **HTTP 语义 + QUIC 传输层**

而 QUIC 又可以先简单理解为：

- **建立在 UDP 之上的、面向现代网络需求设计的可靠传输机制**

HTTP/3 的关键意义在于：

- 不再直接依赖 TCP
- 把多路复用、加密握手、连接迁移等能力更紧密地结合进新的传输方案

所以 HTTP/3 不是“HTTP/2 再小修小补”，而是底层承载方式发生了明显变化。

## 核心概念

### 1. HTTP/3 仍然是 HTTP

即使到了 HTTP/3，下面这些概念依然没变：

- 方法
- 状态码
- 头部
- 资源请求与响应

变化主要不在“业务语义”，而在“这些语义由什么传输机制承载”。

### 2. QUIC 跑在 UDP 上，但不等于不可靠

很多人一看到 UDP 就以为：

- 不可靠
- 不重传
- 不适合网页传输

但 QUIC 不是直接把 HTTP 内容随便塞进裸 UDP 数据报里。

QUIC 自己实现了很多传输层能力，例如：

- 可靠传输
- 流控制
- 拥塞控制
- 多路流
- 加密握手

所以更准确的理解是：

- QUIC 借用了 UDP 作为底座
- 但在其上重新实现了一套更现代的传输机制

### 3. 为什么 HTTP/3 想绕开 TCP

HTTP/2 虽然已经支持多路复用，但通常仍基于 TCP。

这意味着：

- 多个 HTTP 流共享同一条 TCP 连接
- 一旦底层某些数据包丢失，后续交付可能受到 TCP 顺序机制影响

QUIC 试图改进的一个重要点就是：

- 让不同流之间的影响更小
- 减少因底层丢包带来的跨流等待

因此 HTTP/3 可以更好地发挥多路复用的优势。

### 4. QUIC 还带来更现代的连接体验

QUIC 除了多路流，还常被提到这些优点：

- 建连与安全握手整合得更紧
- 更适合减少建立连接时的额外往返
- 支持连接迁移

所谓连接迁移，可以粗略理解为：

- 用户网络环境变化时，例如从 Wi‑Fi 切到蜂窝网络
- 连接不一定要像传统 TCP 那样完全重新来过

这对移动网络场景尤其有价值。

## 过程拆解

### 从浏览器访问一个 HTTP/3 网站来理解

假设浏览器访问一个支持 HTTP/3 的 HTTPS 网站，大致可以这样理解：

1. 浏览器和服务器协商使用 HTTP/3
2. 双方通过 QUIC 在 UDP 之上建立受保护的连接
3. 浏览器把多个资源请求放到不同的流中发送
4. 这些流共享同一条 QUIC 连接，但彼此组织更独立
5. 如果某个流的数据发生丢失，需要等待的是这个流相关的数据，而不是把其他已可处理的流一并卡住

所以 HTTP/3 的核心直觉是：

- 还是那个 Web
- 但底层承载方式更贴合现代高并发、移动化、低时延场景

## 常见误区

### 误区 1：HTTP/3 用了 UDP，所以变得不可靠

不对。HTTP/3 依赖的是 QUIC，而 QUIC 在 UDP 之上补上了可靠传输等关键能力。

### 误区 2：HTTP/3 就是不加密的高速模式

不对。现代 HTTP/3 与安全保护是紧密结合的，不能把它理解成“绕开 TLS”。

### 误区 3：HTTP/3 一定在任何场景下都更快

不对。它解决了很多真实问题，但实际效果仍取决于网络条件、服务器支持、客户端实现等因素。

## 小结

- HTTP/3 是运行在 QUIC 之上的 HTTP
- QUIC 建立在 UDP 上，但提供了可靠传输、多路流和更现代的握手能力
- HTTP/3 试图进一步解决 HTTP/2 在 TCP 之上的一些局限
- 从 HTTP/1.1 到 HTTP/2 再到 HTTP/3，本质上是在不断优化 Web 的承载方式
