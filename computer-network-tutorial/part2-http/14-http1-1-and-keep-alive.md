# 第14章：HTTP/1.1 与 Keep-Alive

## 本章目标

- 理解 HTTP/1.1 相比更早期 HTTP 的一个关键改进是什么
- 明白 Keep-Alive 为什么能减少重复建连开销
- 认识 HTTP/1.1 在性能上的提升与局限
- 为后续理解 HTTP/2、HTTP/3 做铺垫

## 先讲结论

HTTP/1.1 一个非常重要的改进，是默认支持连接复用，也就是常说的 **Keep-Alive**。

它的核心意思是：

- 一次 TCP 连接建立后
- 不必每发一个 HTTP 请求就立刻断开
- 可以在同一条连接上继续发送后续请求和响应

这样做的价值很直接：

- 少做很多次 TCP 建连和断连
- 降低延迟
- 减少服务器和客户端的额外开销

但 HTTP/1.1 虽然比早期方式更高效，仍然存在队头阻塞等限制，这也是后来出现 HTTP/2、HTTP/3 的背景。

## 核心概念

### 1. 为什么短连接代价高

如果每次请求都重新建立 TCP 连接，大致要经历：

1. 三次握手
2. 发送请求
3. 接收响应
4. 再断开连接

如果一个网页要加载：

- HTML
- CSS
- JavaScript
- 图片
- 字体

那就可能重复很多次建连过程。即使单次开销不夸张，叠加起来也会明显拖慢页面加载。

### 2. Keep-Alive 的核心思想

Keep-Alive 不是“永远不断开”，而是：

- 在一段时间内把连接保留下来
- 让后续多个请求复用同一条 TCP 连接

因此它真正优化的是：

- 建连次数
- 往返时延
- 内核和服务器资源消耗

对浏览器访问网页这类高频请求场景，非常有帮助。

### 3. HTTP/1.1 为什么默认更偏向持久连接

在 HTTP/1.0 时代，很多实现更偏向请求完成后就关闭连接。

到了 HTTP/1.1，持久连接成为更常见、也更符合实际 Web 场景的做法，因为：

- 一个页面往往不只一个资源
- 用户和同一站点之间常常会连续发多个请求
- 重复建连浪费太多时间

所以 HTTP/1.1 可以理解为更认真地面向“真实 Web 使用方式”做了优化。

### 4. Keep-Alive 解决不了所有性能问题

虽然连接可以复用，但 HTTP/1.1 仍有明显限制。

最常被提到的是：

- 同一连接上的请求处理顺序会互相影响
- 一个慢响应可能挡住后面的响应返回

这类问题常被概括为队头阻塞。也就是说，Keep-Alive 解决了“反复建连”的问题，但没有从根本上解决“同一连接内并发效率不高”的问题。

## 过程拆解

### 以浏览器加载一个网页为例

假设浏览器访问首页后，还要继续加载 5 个静态资源。

#### 没有持久连接时

可能发生：

1. 建立 TCP 连接
2. 请求 HTML
3. 收到响应后关闭连接
4. 再为 CSS 建一个连接
5. 再为 JS 建一个连接
6. 再为图片建连接

这个过程里，大量时间浪费在重复准备通道。

#### 使用 Keep-Alive 时

可能发生：

1. 先建立一条 TCP 连接
2. 请求 HTML
3. 连接不立即关闭
4. 浏览器继续在这条连接上请求 CSS、JS 等资源
5. 一段时间无请求后，再由某一方关闭连接

这样整体交互会更紧凑，延迟通常更低。

## 常见误区

### 误区 1：Keep-Alive 是 HTTP/1.1 独有的新协议

不对。它不是一个完全独立的新协议，更准确地说，是 HTTP 持久连接的一种工作方式和常见叫法。

### 误区 2：Keep-Alive 就等于可以无限并发、无限快

不对。连接复用能减少建连成本，但 HTTP/1.1 仍有自己的性能瓶颈。

### 误区 3：连接保持越久越好

不对。连接长期占着也会消耗资源，所以实际系统通常会设置空闲超时，不会无限保留。

## 小结

- HTTP/1.1 的重要改进之一是更普遍地使用持久连接
- Keep-Alive 让多个请求可以复用同一条 TCP 连接
- 它主要减少建连开销和额外时延
- 但 HTTP/1.1 仍存在并发效率上的局限，这为 HTTP/2、HTTP/3 的出现埋下了伏笔
