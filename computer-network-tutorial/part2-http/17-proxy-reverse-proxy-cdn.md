# 第17章：正向代理、反向代理与 CDN

## 本章目标

- 分清正向代理、反向代理、CDN 这三个概念
- 明白它们分别更靠近客户端还是服务器端
- 理解代理在访问控制、负载分担、缓存加速中的作用
- 能从一次网页访问中看出它们各自处在什么位置

## 先讲结论

这三个词都和“中间层”有关，但作用并不一样：

- **正向代理**：站在客户端一侧，代替客户端去访问外部资源
- **反向代理**：站在服务器一侧，代替后端服务接收客户端请求
- **CDN**：把内容分布到离用户更近的边缘节点，本质上可以看作一种更大规模的缓存加速体系

最容易混淆的点是：

- 正向代理更像“帮用户出去”
- 反向代理更像“替服务器迎客”

CDN 则常常建立在反向代理和缓存能力之上，用来加速静态资源，甚至部分动态内容。

## 核心概念

### 1. 代理的共同点：中间人代你说话

所谓代理，可以先粗略理解为：

- 客户端和真正的目标之间
- 多了一个中间节点帮忙转发请求或响应

这样做的好处可能包括：

- 隐藏真实地址
- 做访问控制
- 做缓存
- 做负载均衡
- 做安全防护

但不同代理到底在帮谁、部署在哪里，决定了它的角色完全不同。

### 2. 正向代理更靠近客户端

正向代理的典型特点是：

- 客户端知道自己在使用代理
- 服务器未必知道真实客户端是谁

客户端不是直接请求目标站点，而是先请求代理服务器，再由代理去访问目标资源。

它常见于：

- 统一出口访问外网
- 内网访问控制
- 内容过滤
- 隐藏客户端真实来源

所以正向代理的关键不是“加速”，而是“代表客户端访问外部”。

### 3. 反向代理更靠近服务器

反向代理部署在网站或服务入口处。

对客户端来说，它看起来就像目标网站本身；而在它后面，可能挂着很多真正处理业务的后端服务。

它常见承担：

- 请求转发
- 负载均衡
- TLS 终止
- 静态资源分发
- 缓存与限流

因此很多网站的客户端并不是直接连到应用服务器，而是先连到反向代理，再由后者转给后端。

### 4. CDN 是更分布式的加速网络

CDN 的核心思想是：

- 把常用内容提前放到离用户更近的节点
- 让用户优先从附近节点获取内容

这样做的好处是：

- 减少跨地域传输延迟
- 降低源站压力
- 提高静态资源访问速度

CDN 常见加速对象包括：

- 图片
- CSS
- JavaScript
- 视频切片
- 下载文件

可以把 CDN 理解成“带分布式缓存能力的内容分发网络”，很多场景下它和反向代理的思想是连在一起的。

## 过程拆解

### 以浏览器访问一张站点图片为例

假设页面里的图片地址是：

`https://cdn.example.com/logo.png`

大致过程可能是：

1. 浏览器访问 `cdn.example.com`
2. DNS 或调度系统把请求引导到离用户较近的 CDN 边缘节点
3. 边缘节点先看自己有没有这张图片缓存
4. 如果命中缓存，直接把图片返回给浏览器
5. 如果没有命中，边缘节点再去源站或上游节点取回内容
6. 边缘节点把内容缓存下来，再返回给用户

如果没有 CDN，用户可能每次都要跨较远网络回源获取内容，延迟和源站压力都会更高。

再看反向代理场景：

1. 浏览器访问 `https://www.example.com`
2. 先到达站点前面的反向代理
3. 反向代理根据规则把请求转发给某个后端应用实例
4. 后端返回结果后，再由反向代理回给客户端

所以反向代理更像服务入口层，而 CDN 更强调“就近分发与缓存加速”。

## 常见误区

### 误区 1：代理就是 NAT

不对。NAT 主要是在网络层改地址，代理通常工作在更高层，能理解和转发具体应用请求。

### 误区 2：CDN 就是把整个网站复制一份到每个地方

不对。CDN 更常缓存适合分发的内容，尤其是静态资源；并不是所有动态业务逻辑都会完整复制到每个边缘节点。

### 误区 3：正向代理和反向代理只是名字相反

不对。它们服务的对象不同、部署位置不同、典型用途也不同。

## 小结

- 正向代理代表客户端访问外部资源
- 反向代理代表服务器接收并转发请求
- CDN 通过边缘节点和缓存机制把内容更快送到用户附近
- 理解三者的关键，不是背定义，而是看“它站在谁那一边、在帮谁做事”
