# 第18章：同源策略与 CORS

## 本章目标

- 理解什么叫“同源”，以及浏览器为什么要限制跨源读取
- 明白同源策略主要是浏览器安全机制，不是 HTTP 传输故障
- 认识 CORS 的作用，以及它如何让跨源请求在受控条件下发生
- 能看懂一个基础的跨源请求为什么成功或失败

## 先讲结论

同源策略的核心不是“禁止一切跨域通信”，而是：

- **限制一个源的网页随意读取另一个源的数据**

而 CORS 的核心不是“绕过浏览器”，而是：

- **让服务器明确告诉浏览器：哪些源可以访问我**

所以这件事的本质是浏览器安全边界问题。

如果前端页面来自 `https://app.example.com`，却想读取 `https://api.example.com` 的响应内容，这通常就属于跨源场景，需要服务器通过 CORS 头部明确授权。

## 核心概念

### 1. 什么叫同源

通常判断两个地址是否同源，看三件事是否都一样：

- 协议
- 主机名
- 端口

例如：

- `https://example.com:443` 和 `https://example.com:443` 是同源
- `http://example.com` 和 `https://example.com` 不同源
- `https://www.example.com` 和 `https://api.example.com` 不同源
- `https://example.com:443` 和 `https://example.com:8443` 不同源

所以“只要主域名差不多就算同源”是错误的。

### 2. 同源策略是浏览器的安全边界

浏览器之所以有同源策略，是为了防止一个网站页面随便读取另一个网站的敏感数据。

例如如果没有限制，恶意网站可能诱导用户打开页面后，偷偷去读取：

- 用户在其他网站上的个人信息
- 登录后的接口返回
- 账户相关页面内容

因此浏览器会对脚本读取跨源响应做限制。

要注意，这不等于浏览器什么跨源资源都不允许加载。很多资源可以跨源请求，但“能不能被脚本自由读取响应内容”是另一回事。

### 3. CORS 是服务器对跨源访问的显式授权

CORS 全称是 Cross-Origin Resource Sharing。

它的思路是：

- 浏览器发现这是跨源请求
- 服务器在响应头里声明是否允许
- 浏览器再决定是否把响应交给前端脚本

最常见的头部之一是：

- `Access-Control-Allow-Origin`

例如：

`Access-Control-Allow-Origin: https://app.example.com`

可以理解为：服务器允许来自这个源的页面读取当前响应。

### 4. 有些跨源请求会先预检

并不是所有跨源请求都只发一次。

在很多情况下，例如：

- 使用了自定义请求头
- 使用了某些不是“简单类型”的 `Content-Type`
- 使用了 `PUT`、`DELETE` 等方法

浏览器会先发一个 `OPTIONS` 预检请求，确认服务器是否允许这次跨源操作。

只有预检通过，浏览器才会继续发真正的业务请求。

## 示例

### 示例 1：一个简单跨源场景

前端页面来自：

`https://app.example.com`

页面里的脚本要请求：

`https://api.example.com/users`

由于主机名不同，所以这是跨源请求。

如果服务器响应里带有：

```http
Access-Control-Allow-Origin: https://app.example.com
```

浏览器就可能允许前端脚本读取这个响应。

如果没有这个头，或者值不匹配，请求也许已经发出，但浏览器通常不会把响应结果正常开放给页面脚本。

### 示例 2：为什么会出现预检请求

假设前端发出：

```http
POST /orders HTTP/1.1
Host: api.example.com
Content-Type: application/json
Authorization: Bearer xxx
```

这类请求往往不属于最简单的跨源情况，因此浏览器可能先发一个 `OPTIONS` 请求，询问服务器：

- 这个源能不能访问
- 这些方法能不能用
- 这些请求头能不能带

服务器允许后，浏览器才继续真正的 `POST` 请求。

## 常见误区

### 误区 1：CORS 是后端服务器拦截了请求

不对。很多时候真正做限制决定的是浏览器。服务器只是在响应头里表达允许或不允许。

### 误区 2：后端之间调用接口也会被同源策略拦住

不对。同源策略主要约束浏览器环境中的脚本。服务器和服务器之间的调用通常不受这套浏览器规则直接限制。

### 误区 3：同源策略就是完全禁止跨源

不对。它更准确地是在限制跨源读取和某些高风险交互，而不是把所有跨源资源加载都一刀切封死。

## 小结

- 同源通常要求协议、主机名、端口都一致
- 同源策略是浏览器的重要安全机制
- CORS 让服务器可以显式授权特定源访问资源
- 理解跨源问题时，关键要先问：是谁在限制、限制的到底是什么能力
