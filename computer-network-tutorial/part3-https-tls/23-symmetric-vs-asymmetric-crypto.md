# 第23章：对称加密与非对称加密

## 本章目标

- 理解对称加密和非对称加密分别解决什么问题
- 明白为什么 TLS 不会只用其中一种加密方式
- 认识公钥、私钥、共享密钥这些概念之间的关系
- 建立“握手阶段和数据传输阶段关注点不同”的直觉

## 先讲结论

如果只记一句话，这一章最重要的结论是：

- **对称加密适合大量数据传输，因为快**
- **非对称加密适合做身份认证和密钥交换，因为解决了“密钥怎么安全分发”的问题**
- **TLS 的现实做法通常是把两者结合起来，而不是二选一**

很多初学者一看到“公钥、私钥”就觉得非对称加密一定更高级，应该全面替代对称加密。实际上不是这样。

真实系统更看重的是：**谁更适合解决当前这个环节的问题**。

## 核心概念

### 1. 什么是对称加密

对称加密的特点是：

- 加密和解密使用同一把密钥
- 发送方和接收方都必须事先知道这把密钥

可以把它理解成：

- 我们两个人约定了同一个保险箱密码
- 你用这个密码锁上内容
- 我也用同一个密码打开它

这种方式的优点很明显：

- 算法通常更高效
- 适合加密大量数据
- 在持续传输网页、图片、JSON、视频流时更实用

但它有一个非常现实的问题：

- **如果双方还没有安全地共享这把密钥，那第一步怎么完成？**

这就是对称加密最核心的难点：**密钥分发问题**。

### 2. 什么是非对称加密

非对称加密的特点是：

- 有一对相关联的密钥：公钥和私钥
- 公钥可以公开分发
- 私钥必须严格保密

它常见的用法有两类：

1. **用公钥加密，私钥解密**
2. **用私钥签名，公钥验证签名**

你可以先抓住一个直觉：

- 公钥像“可以公开给所有人的锁”
- 私钥像“只有持有者自己掌握的钥匙”

这样一来，别人即使在网上拿到了你的公钥，也不代表他就能伪装成你，因为真正敏感的是私钥。

### 3. 非对称加密不只是“加密”，还承担认证能力

对 TLS 来说，非对称体系真正重要的地方，往往不只是“把一段数据加密起来”，而是：

- 让服务器能够证明自己持有某个私钥
- 让客户端把“这个公钥”与“这个域名身份”联系起来
- 为后续协商共享密钥创造条件

也就是说，非对称加密在 TLS 里更常扮演：

- 身份认证的基础
- 密钥交换或签名验证的基础

而不是拿来直接加密整个网页流量。

### 4. 为什么 TLS 不直接用非对称加密传所有数据

因为代价太高。

和对称加密相比，非对称加密通常：

- 计算开销更大
- 处理大块数据效率更低
- 不适合长时间、高吞吐量的数据传输

一个 HTTPS 会话中，真正占带宽的是：

- HTML
- CSS
- JavaScript
- 图片
- API 响应
- 文件下载内容

这些数据量可能很大。如果都用非对称加密硬顶，性能会非常不划算。

所以更合理的做法是：

- 在握手阶段用非对称机制解决“你是谁”和“我们怎么安全协商出一个共享秘密”
- 在数据传输阶段用对称加密保护后续内容

### 5. 对称加密与非对称加密在 TLS 中的分工

可以先记住这个实用版分工：

- **证书、公钥、签名验证**：更多属于非对称体系的工作
- **会话建立后的应用数据加密**：更多由对称加密承担

这也是为什么很多 TLS 介绍里会说：

- TLS 是一种“混合加密”或“组合式安全协议”

它并不是迷信某一种密码学工具，而是把不同工具放在合适的位置上。

## 过程拆解

### 用 HTTPS 建立连接的过程理解两者如何配合

假设你访问：

`https://example.com`

大致可以这样理解：

1. 浏览器先发起 TLS 握手
2. 服务器把证书发给浏览器，证书里包含公钥等身份信息
3. 浏览器验证证书是否可信、域名是否匹配
4. 双方通过密钥交换机制协商出后续会话使用的共享密钥材料
5. 从这一刻开始，真正的大量 HTTP 数据通常使用对称加密来保护

这里最重要的不是死记某个算法名，而是理解分工：

- 前面几步，重点是“确认身份”和“建立共享秘密”
- 后面几步，重点是“高效、安全地传输大量数据”

所以你可以把 TLS 想成两阶段：

- **建立安全上下文**
- **在这个上下文里传数据**

## 常见误区

### 误区 1：非对称加密比对称加密更高级，所以应该全部替代

不对。它们解决的问题不同。

- 对称加密强在效率
- 非对称加密强在密钥分发和身份认证相关能力

真实协议往往是组合使用，而不是简单替代。

### 误区 2：有了公钥，任何人都能解密服务器内容

不对。公钥通常是公开的，真正需要保密的是私钥。

公开公钥并不会自动泄露会话内容，否则整个公钥体系就失去意义了。

### 误区 3：数字签名等于“把消息加密一次”

不准确。签名的核心目的是：

- 证明消息确实来自私钥持有者
- 证明消息在传输中没有被改动

它和“为了保密而加密内容”关注点不同。

### 误区 4：TLS 只要有公钥就够了，不需要对称密钥

不对。真正持续传输网页内容时，TLS 更依赖会话中的对称密钥来高效保护数据。

## 小结

- 对称加密使用同一把密钥加解密，效率高，适合大量数据传输
- 非对称加密使用公钥和私钥，擅长解决身份认证和密钥分发问题
- TLS 并不是只选一种，而是把两者结合起来
- 理解这两者的分工，后面学习证书、握手、会话恢复时会轻松很多
