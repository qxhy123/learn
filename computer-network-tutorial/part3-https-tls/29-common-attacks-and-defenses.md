# 第29章：常见攻击与防御

## 本章目标

- 认识 HTTPS/TLS 相关的常见攻击面，不把安全想得过于理想化
- 理解中间人、降级、证书错误、配置薄弱等风险是如何出现的
- 明白 TLS 提供保护，但部署与使用方式同样重要
- 建立“安全是协议、实现、配置、运维共同结果”的意识

## 先讲结论

HTTPS 能显著提升 Web 通信安全，但这并不意味着：

- 开了 HTTPS 就万事大吉
- 有证书就不会被攻击
- 用了 TLS 1.3 就自动没有配置问题

真实世界里的风险，常常来自三类来源：

1. **攻击者主动干扰连接**
2. **站点自身配置不当**
3. **用户或系统在验证环节被绕过或误导**

所以学习 HTTPS，不能只学“它能防什么”，也要学“它是怎么被削弱、被绕过、被误用的”。

## 核心概念

### 1. 中间人攻击

中间人攻击是理解 HTTPS 价值和边界时最重要的场景之一。

攻击者试图处在客户端与服务器之间，达到这些目的：

- 偷看通信内容
- 篡改传输数据
- 冒充服务器或客户端

如果是纯 HTTP，这类攻击往往更容易得手。

而 HTTPS 的证书验证、密钥交换和完整性保护，正是在大幅提高中间人攻击成功的门槛。

但如果用户忽略浏览器证书告警，或者系统把不该信任的根证书装进信任库，风险仍然会重新出现。

### 2. 协议降级与弱配置问题

历史上某些攻击会尝试逼迫客户端和服务器使用：

- 更旧的协议版本
- 更弱的密码套件
- 不安全的兼容模式

这类风险的根源是：

- 为了兼容老设备和老客户端，系统有时保留了历史弱选项
- 一旦协商空间过大，攻击者就会寻找把连接往“更差方案”上拖的机会

现代最佳实践通常会通过：

- 禁用老旧协议版本
- 禁用弱加密套件
- 优先使用现代 TLS 配置

来压缩这类攻击面。

### 3. 证书相关攻击与错误

证书问题不一定都来自高深攻击，很多就是部署和运维失误。

例如：

- 证书过期
- 域名不匹配
- 中间证书缺失
- 错把测试证书部署到生产
- 私钥泄露

这些问题带来的结果包括：

- 浏览器直接报错
- 用户被迫点击忽略告警
- 攻击者更容易借机实施冒充

对安全来说，最糟糕的情况之一不是“浏览器偶尔报错”，而是团队和用户开始习惯性忽略告警。

### 4. 私钥泄露为什么严重

服务器证书背后的私钥一旦泄露，攻击者可能获得：

- 冒充该服务器身份的能力
- 在某些条件下劫持会话或部署伪站点的机会

虽然现代 TLS 的前向保密设计能减轻“历史流量全被回溯解密”的风险，但私钥泄露仍然是严重安全事件，通常需要：

- 立即吊销证书
- 重新签发并替换证书
- 追查泄露范围和根因

### 5. 不安全的客户端行为也会破坏 TLS 防线

很多安全体系不是被算法打穿，而是被使用方式打穿。

常见危险行为包括：

- 点击“继续访问”忽略证书错误
- 在代码里关闭证书校验
- 忽略主机名验证
- 在抓包调试后忘记移除临时信任配置

这些做法相当于主动拆掉浏览器和 TLS 帮你建立的护栏。

### 6. HSTS 等机制在防什么

很多网站除了启用 HTTPS，还会配合 HSTS 等机制。

它的直觉作用是：

- 告诉浏览器，这个站点之后只能走 HTTPS
- 尽量减少用户被诱导回 HTTP 明文入口的机会

这有助于抵御一类从“首次明文访问”切入的降级或劫持风险。

## 过程拆解

### 从一次不安全配置看攻击是怎样出现的

假设一个站点存在以下问题：

- 仍支持过旧 TLS 版本
- 配置了弱密码套件
- 用户访问 `http://example.com` 时先明文跳转到 HTTPS
- 某些内部客户端还关闭了证书校验

此时可能发生：

1. 攻击者先在网络中拦截用户的明文 HTTP 起始请求
2. 尝试阻止或篡改跳转，让用户留在不安全连接上
3. 即使进入 HTTPS，若客户端允许弱配置或忽略证书校验，攻击者仍可能继续插手
4. 用户表面上“还能访问网站”，但连接的真实性和完整性已经被破坏

这个例子说明：

- 协议本身的能力只是基础
- 是否真正获得保护，还取决于配置、部署和用户行为

## 常见误区

### 误区 1：HTTPS 能彻底消灭中间人攻击

不准确。更准确的说法是：

- HTTPS 在正确验证证书、正确配置的前提下，能极大降低中间人攻击成功概率

但如果用户绕过验证，或本地信任体系被污染，风险依然存在。

### 误区 2：只要不是大网站，攻击者就不会盯上我

不对。很多攻击并不是“专门点名某个小站”，而是自动化、大范围扫描和利用薄弱配置。

### 误区 3：浏览器报证书错，先点继续再说

这是非常危险的习惯。证书错误恰恰是在提示：

- 当前连接的身份或信任链存在问题

### 误区 4：安全问题都属于运维，开发不用关心

不对。开发如果在代码里关闭证书校验、错误处理重定向、滥用 0-RTT、设计不幂等接口，同样会扩大安全风险。

## 小结

- HTTPS/TLS 能有效抵御很多窃听、篡改和冒充风险，但前提是正确配置和正确使用
- 中间人攻击、降级、证书错误、私钥泄露都是必须重视的现实问题
- 安全不只取决于协议设计，还取决于实现、部署、运维和用户行为
- 真正成熟的安全意识，是既知道 TLS 能保护什么，也知道它怎样被削弱
