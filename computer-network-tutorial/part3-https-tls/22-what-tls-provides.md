# 第22章：TLS 提供了什么

## 本章目标

- 理解 TLS 在 HTTPS 中扮演的角色
- 认识机密性、完整性、身份认证这三个核心价值
- 知道 TLS 不等于“任何安全问题都解决了”
- 对握手、证书、会话保护形成初步整体认知

## 先讲结论

TLS 可以理解为 HTTPS 背后的安全层，它主要提供三类能力：

- **机密性**：尽量让中间人看不懂传输内容
- **完整性**：尽量让中间人不能悄悄改内容而不被发现
- **身份认证**：让客户端更有把握确认对方是不是声称的那个服务器

因此，HTTPS 真正的安全能力，核心并不是来自 HTTP，而是来自 TLS。

## 核心概念

### 1. 机密性

机密性指的是：

- 即使别人截获了传输中的数据
- 也不应该轻易看懂里面的内容

TLS 通过加密来实现这一点。这样用户提交的表单、页面内容、接口返回值等，不再以明文裸奔。

### 2. 完整性

完整性指的是：

- 数据在传输途中没有被偷偷改掉
- 如果被改动，应当能被发现

这能帮助客户端确认：

- 收到的内容是否还是服务器原本发出的内容
- 中间网络设备有没有进行未授权修改

### 3. 身份认证

即使数据被加密，如果你连错了对象，也没有意义。

所以 TLS 还要尽量帮助客户端回答：

- 我现在连接的，真的是这个网站吗？

这通常依赖证书体系来完成。浏览器会根据证书和受信任的证书链，判断当前服务器身份是否可信。

### 4. TLS 握手的大致作用

TLS 在正式传输应用数据前，通常需要先握手。握手的大致目的包括：

- 协商使用哪些安全参数
- 建立后续会话所需的密钥材料
- 让客户端验证服务器身份

你不一定一开始就要记住所有细节，但要知道：

- TLS 不是“突然就开始加密”
- 它需要先建立一套双方都认可的安全上下文

## 过程拆解

### 从浏览器访问 HTTPS 网站的角度理解 TLS

假设你访问：

`https://example.com`

大致会发生：

1. 浏览器先与服务器建立底层传输连接
2. 然后开始 TLS 握手
3. 服务器出示证书等身份材料
4. 浏览器验证证书是否可信、域名是否匹配
5. 双方协商出后续通信所需的安全参数
6. 后续 HTTP 请求和响应在 TLS 保护下传输

因此，当你看到浏览器地址栏里的小锁时，本质上是在说明：

- 当前连接不只是普通 HTTP
- 而是处于 TLS 保护之下的 HTTPS 会话中

## 常见误区

### 误区 1：TLS 只提供加密

不对。TLS 不只是保密，还强调完整性和身份认证。

### 误区 2：有证书就一定绝对可信

不对。证书是身份验证体系的重要部分，但真实安全还受到证书配置、站点本身漏洞、用户是否访问正确域名等因素影响。

### 误区 3：TLS 会让应用层逻辑漏洞自动消失

不对。SQL 注入、XSS、弱口令、越权访问等问题，TLS 都不能直接解决。

## 小结

- TLS 是 HTTPS 背后的安全基础设施
- 它主要提供机密性、完整性和身份认证
- TLS 通过握手、证书和会话保护机制建立安全通道
- 理解 TLS 提供了什么，也要同时知道它没有解决什么
