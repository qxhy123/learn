# 第24章：证书、CA 与 PKI

## 本章目标

- 理解证书为什么存在，以及它试图解决什么问题
- 明白 CA、根证书、中间证书分别扮演什么角色
- 认识 PKI 不只是“发证书”，而是一整套信任基础设施
- 能从浏览器视角理解一次证书验证的大致过程

## 先讲结论

如果只把 HTTPS 理解成“服务器给你一个公钥”，其实还差关键一步：

- **这个公钥到底是不是目标网站的公钥？**

证书存在的意义，就是把：

- 某个身份信息，例如域名
- 某个公钥
- 某个签发者的背书

绑定在一起。

而 CA（Certificate Authority，证书颁发机构）和 PKI（Public Key Infrastructure，公钥基础设施）则是在解决：

- 谁来签发证书
- 谁来决定哪些签发者可信
- 浏览器如何沿着证书链验证这个信任

所以，HTTPS 不是“看到一个公钥就信”，而是“看到一个能被信任体系验证的公钥才继续”。

## 核心概念

### 1. 为什么只有公钥还不够

假设服务器直接说：

- 这是我的公钥，你拿去用吧

问题在于，攻击者也可以说同样的话。

如果没有额外验证机制，中间人完全可以：

1. 截获你的连接
2. 把自己的公钥冒充成服务器公钥发给你
3. 让你误以为已经安全连接到了目标网站

所以，真正的问题不是“有没有公钥”，而是：

- **这个公钥和目标身份之间的绑定是否可信**

证书就是用来回答这个问题的。

### 2. 证书里通常包含什么

一个面向 Web 的服务器证书里，常见信息包括：

- 证书主体信息
- 适用的域名，例如 `example.com`、`www.example.com`
- 公钥
- 签发者是谁
- 有效期起止时间
- 用途限制，例如能否用于服务器认证
- 由上级 CA 产生的数字签名

对浏览器来说，最关键的不是把证书内容背下来，而是知道它会重点检查：

- 域名是否匹配
- 是否还在有效期内
- 签名链是否能验证通过
- 证书用途是否正确

### 3. CA 在做什么

CA 可以先粗略理解为“受信任的签发者”。

它做的不是替你加密网页，而是：

- 按照一定规则核验申请者
- 把申请者身份信息和公钥绑定到证书里
- 用自己的私钥对证书签名

这样浏览器就可以利用 CA 的公钥去验证：

- 这张证书是不是被这个 CA 正式签发的
- 证书内容有没有被改过

在 Web PKI 体系里，常见的层级包括：

- **根 CA**：信任锚点，通常预装在操作系统或浏览器信任库里
- **中间 CA**：由根 CA 签发，再去签发大量终端证书
- **终端证书**：也就是网站服务器实际部署的证书

### 4. 为什么常见的是“根 CA + 中间 CA + 网站证书”

如果根 CA 天天直接给全世界网站签证书，风险会非常大。

所以现实中更常见的做法是：

- 根 CA 尽量离线、极少使用
- 中间 CA 负责日常签发
- 网站拿到的是中间 CA 签出来的终端证书

这样做的好处包括：

- 降低根私钥暴露风险
- 方便分工和撤销
- 某个中间 CA 出问题时，更容易局部处置，而不是全盘推倒

### 5. PKI 不只是证书文件，而是一套体系

PKI 可以理解成“围绕公钥信任建立起来的一整套基础设施”。

它通常包括：

- 证书格式和签名机制
- 根 CA 与中间 CA 体系
- 浏览器或操作系统中的信任库
- 证书签发、更新、吊销相关机制
- 域名验证、身份验证、策略要求

也就是说，证书只是 PKI 中最显眼的一部分，不是全部。

### 6. 浏览器的信任从哪里来

浏览器并不是在访问网站时才临时决定“信谁”，而是本地就维护或依赖一套信任锚。

通常表现为：

- 浏览器内置根证书库
- 或依赖操作系统的受信任根证书库

当服务器发来证书链时，浏览器会尝试把这条链一路追溯到：

- 某个自己已经信任的根证书

如果找不到可信的根，或者链条验证失败，浏览器就会报证书错误。

## 过程拆解

### 浏览器验证证书时，大致在做什么

假设你访问：

`https://example.com`

大致过程可以这样理解：

1. 服务器在 TLS 握手中把自己的证书，通常还包括中间证书，一起发给浏览器
2. 浏览器读取终端证书里的域名信息，检查是否匹配你访问的 `example.com`
3. 浏览器检查证书是否过期、是否尚未生效
4. 浏览器用中间 CA 的公钥验证网站证书签名
5. 再继续验证中间 CA 是否能一路连到某个本地受信任的根 CA
6. 如果整条链成立，浏览器才更有把握相信“当前服务器确实对应这个公钥”
7. 后续 TLS 才会基于这个信任继续完成密钥协商和会话保护

这里要特别注意：

- 服务器一般会发送终端证书和中间证书
- 根证书通常已经在客户端本地信任库里，不需要服务器每次都发

## 常见误区

### 误区 1：证书本身负责加密网页内容

不对。证书的主要作用是把身份和公钥绑定起来，并提供可验证的信任链。

真正保护会话内容的，是 TLS 握手之后协商出来的会话密钥和加密算法。

### 误区 2：有证书就说明这个网站绝对安全、绝对正规

不对。证书主要说明：

- 这个连接的身份绑定和信任链验证大致成立

它不自动保证：

- 网站没有业务漏洞
- 网站不会钓鱼
- 网站运营者一定“值得信任”

### 误区 3：自签名证书完全没价值

不准确。

在公开互联网里，自签名证书通常不会被浏览器默认信任，所以不适合作为公共网站方案。

但在：

- 内网环境
- 开发测试环境
- 明确手动分发信任锚的封闭系统

自签名证书仍然可以有用途。

### 误区 4：服务器每次都会把根证书发给浏览器

通常不是。服务器常发送的是：

- 网站终端证书
- 需要的中间证书

而根证书往往已经在客户端本地信任库中。

## 小结

- 证书的核心作用，是把网站身份与公钥可信地绑定起来
- CA 负责签发和背书，浏览器依靠本地信任锚验证证书链
- PKI 是围绕证书、CA、信任库和策略构成的一整套体系
- 理解证书体系的重点，不是背字段，而是明白浏览器为什么敢信这个公钥
