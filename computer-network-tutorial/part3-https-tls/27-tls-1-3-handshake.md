# 第27章：TLS 1.3 握手

## 本章目标

- 理解 TLS 1.3 相比 TLS 1.2 做了哪些关键简化
- 明白 TLS 1.3 为什么通常能减少握手延迟
- 认识 TLS 1.3 如何更强制地拥抱现代安全设计
- 能从大流程上描述一次 TLS 1.3 握手

## 先讲结论

TLS 1.3 的重要变化，可以先概括成三点：

- **握手更少轮次，通常更快**
- **移除了很多历史包袱和旧式弱设计**
- **更强调以前向保密为代表的现代安全实践**

如果把 TLS 1.2 看作“兼容历史很久的一代通用体系”，那么 TLS 1.3 更像是一次明显的瘦身和现代化升级。

它不是把所有概念推倒重来，而是：

- 保留 TLS 的核心目标
- 重新整理流程和算法选择
- 尽量减少那些容易出错、性能差、历史负担重的部分

## 核心概念

### 1. TLS 1.3 为什么更快

对浏览器访问网站来说，建立 HTTPS 连接前的额外往返次数越多，页面首包越晚到。

TLS 1.3 的一大改进，就是把完整握手在常见场景下压缩得更紧凑。

粗略理解就是：

- 客户端在 `ClientHello` 阶段就尽量把自己后面会用到的关键信息提前带上
- 服务器尽快返回自己选择的参数和证书
- 双方更早进入可保护应用数据的状态

所以很多资料会说 TLS 1.3 在完整握手下通常做到更少的 RTT，这对网页加载体验很重要。

### 2. TLS 1.3 更强调临时密钥交换

TLS 1.3 的设计明确倾向于现代密钥交换方式，例如临时 Diffie-Hellman 思路。

这意味着：

- 会话密钥不应该长期直接依赖服务器私钥本身
- 每次连接都尽量生成新的临时密钥材料

这样做的重要收益之一就是：

- **更好地支持前向保密性**

也就是说，即使未来某个长期私钥泄露，过去抓包保存下来的会话内容也不应因此全部被还原出来。

### 3. TLS 1.3 删除了很多旧选项

TLS 1.3 的另一个特点是“不再什么都兼容”。

它删除或弱化了很多历史机制，目的包括：

- 减少配置复杂度
- 降低错误使用弱算法的概率
- 让安全基线更统一

对于学习者来说，这意味着：

- TLS 1.3 不是单纯在 TLS 1.2 外面又套一层新名字
- 它在协议设计哲学上明显更偏向“少而稳、默认更安全”

### 4. 握手消息虽然还叫 Hello，但内容组织更现代

TLS 1.3 仍然有 `ClientHello` 和服务器响应阶段，但密钥交换和参数协商的组织方式更集中。

你可以先抓住一个工程化理解：

- 客户端更早提供密钥交换所需材料
- 服务器一旦选定参数，就能更快推进密钥派生
- 某些旧时代需要来回折返的步骤被简化掉了

### 5. 握手完成后，应用数据更早进入加密保护

在 TLS 1.3 中，握手后期和应用数据保护之间的切换更紧密。

对开发者和运维来说，这带来的直觉是：

- 更快进入受保护通信状态
- 更少握手历史包袱
- 更清晰的现代安全默认值

## 过程拆解

### 用一条典型 TLS 1.3 连接理解握手

假设浏览器访问：

`https://example.com`

可以把 TLS 1.3 握手大致理解为：

1. 客户端发送 `ClientHello`，其中除了版本能力和密码套件外，还会尽早附带密钥交换相关材料
2. 服务器收到后，选择合适的参数，并返回自己的响应、证书以及必要的证明材料
3. 客户端验证服务器证书、域名和信任链
4. 双方基于前面交换的材料派生出会话密钥
5. 双方发送握手完成校验信息，确认整个握手未被篡改
6. 后续 HTTP 数据在 TLS 保护下传输

如果只和 TLS 1.2 做概念比较，可以这样记：

- TLS 1.2 更像“先谈很多条件，再逐步推进密钥建立”
- TLS 1.3 更像“尽量把关键材料前置，缩短建立安全上下文的路径”

## 常见误区

### 误区 1：TLS 1.3 只是 TLS 1.2 的小版本补丁

不对。虽然名称看起来像小升级，但它在握手流程、算法取舍和安全基线上都有明显变化。

### 误区 2：TLS 1.3 更快，所以一定更不安全

不对。TLS 1.3 的加速主要来自流程精简和历史包袱清理，不是靠偷掉关键安全步骤。

### 误区 3：TLS 1.3 兼容所有旧配置

不对。它刻意移除了很多老旧选项，因此某些只依赖旧算法或旧设备的环境可能需要升级。

### 误区 4：TLS 1.3 就不需要证书验证了

当然不是。无论 TLS 版本如何，服务器身份验证仍然是 HTTPS 安全的核心环节之一。

## 小结

- TLS 1.3 的核心改进是减少握手轮次、清理历史弱设计、强化现代安全默认值
- 它通过更前置的密钥交换材料和更精简的流程，通常带来更低连接延迟
- TLS 1.3 不是牺牲安全换速度，而是在更现代的设计下同时争取两者
- 学懂 TLS 1.3 的重点，不是背所有报文名，而是理解它为什么比 TLS 1.2 更快、更干净
