# 第28章：会话恢复、前向保密与 0-RTT

## 本章目标

- 理解为什么重复连接同一网站时不一定每次都要完整握手
- 明白前向保密性在 TLS 中的重要意义
- 认识会话恢复和 0-RTT 各自带来的收益与代价
- 建立“更快不等于没有风险”的安全直觉

## 先讲结论

现代 TLS 不只是追求“能安全连接上”，还追求：

- 连接建立得更快
- 重复访问时延迟更低
- 即使长期私钥未来泄露，也尽量不影响过去会话的保密性

围绕这些目标，常见的三个关键词是：

- **会话恢复**：让重复连接不必每次都走完整握手
- **前向保密性（PFS）**：长期密钥泄露后，不应让过去会话内容全部失守
- **0-RTT**：在某些恢复场景下，客户端可以更早发送应用数据

它们都很重要，但要一起理解：

- 越追求性能优化，越要清楚边界和风险

## 核心概念

### 1. 什么是会话恢复

浏览器访问一个网站时，往往不会只连一次。

例如：

- 页面刷新
- 再次打开同一站点
- 多个子资源并发访问
- 短时间内频繁 API 请求

如果每次都完整做一轮 TLS 握手，会增加延迟和计算开销。

所以 TLS 提供了会话恢复思路，让客户端和服务器能够利用之前连接留下来的安全上下文信息，更快建立新会话。

你可以先把它理解成：

- 上次我们已经建立过信任和密钥上下文
- 这次如果条件合适，可以基于之前保存的信息更快恢复

### 2. 会话恢复不是“继续用旧连接”，而是“更快建立新连接”

这是一个常见混淆点。

会话恢复通常不是说：

- 原来的 TCP/TLS 连接一直没断

而是说：

- 原连接结束后，双方保留了一些恢复所需信息
- 以后新建连接时，可以减少握手成本

也就是说，它属于“新连接的优化”，不是“旧连接的续命”。

### 3. 什么是前向保密性

前向保密性，英文常写作 PFS（Perfect Forward Secrecy）或更宽泛地称为 Forward Secrecy。

它强调的是：

- 即使服务器长期私钥在未来某天泄露
- 过去已经完成的会话内容，也不应该因此全部被解密还原

为什么这件事重要？

因为现实攻击者可能会：

1. 长期被动抓取大量加密流量
2. 暂时解不开，但先存下来
3. 等未来有机会拿到服务器长期私钥后再尝试统一回溯破解

如果协议没有前向保密性，这种“先存后解”的威胁就非常严重。

而使用临时密钥交换的设计，可以显著降低这种风险。

### 4. 为什么现代 TLS 很重视前向保密

因为 HTTPS 保护的不只是“此刻正在传的数据”，还涉及：

- 过去通信的长期保密性
- 服务端私钥被盗后的损害范围控制

所以现代安全实践通常希望：

- 每次连接尽量使用新的临时密钥材料
- 不让长期身份密钥直接决定所有历史会话内容

这也是为什么现代 TLS 版本和推荐配置都更偏向前向保密友好的方案。

### 5. 什么是 0-RTT

0-RTT 是 TLS 1.3 中一个很容易引起兴趣，也很容易被误解的特性。

它的直觉是：

- 对已经有恢复上下文的客户端
- 在某些条件下，客户端可以在握手完全结束前就先发送一部分应用数据

这样做的收益很直接：

- 更低的首请求延迟
- 对高时延网络更有吸引力

所以它特别适合那些对响应时间很敏感、又能接受一定限制的场景。

### 6. 0-RTT 为什么有风险

0-RTT 的最大学习重点不是“它很快”，而是：

- **它带来了重放风险**

简单理解就是：

- 某些早期数据可能被攻击者复制并在后续重新发送
- 如果服务端对这类请求处理不当，就可能造成重复执行

因此 0-RTT 不适合所有请求，尤其不适合那些：

- 有副作用
- 会改数据
- 会扣费
- 会触发状态变化

更适合 0-RTT 的通常是：

- 幂等或只读请求
- 重放后不会造成业务损失的场景

## 过程拆解

### 从完整握手到恢复连接，再到 0-RTT 的直觉路径

假设你第一次访问：

`https://example.com`

大致过程可能是：

1. 第一次连接时，浏览器和服务器完成一次完整 TLS 握手
2. 握手结束后，双方保留某些恢复所需的信息
3. 过一会儿你再次访问同一站点，浏览器带上恢复相关信息发起新连接
4. 如果服务器接受，会比完整握手更快建立会话
5. 在 TLS 1.3 的特定恢复场景下，客户端甚至可能连同首个请求一起更早发出早期数据
6. 服务器仍然要谨慎决定这些早期数据能否被接受，以及如何防范重放风险

所以你可以把三者的关系记成：

- 会话恢复解决“重复连接太慢”
- 前向保密解决“未来泄露不该拖垮过去”
- 0-RTT 解决“恢复连接时首个请求还想再快一点”

## 常见误区

### 误区 1：会话恢复等于完全不需要握手

不对。它通常是减少握手成本，不是完全没有安全协商。

### 误区 2：前向保密就是“永远无法解密”

不准确。它的重点是降低长期私钥泄露对历史会话的影响，不代表所有其他攻击都失效。

### 误区 3：0-RTT 更快，所以应该默认给所有请求打开

不对。0-RTT 有重放风险，必须结合业务语义谨慎使用。

### 误区 4：只要用了 TLS 1.3，就自动没有任何恢复相关安全边界问题

不对。协议提供了能力，但怎么在服务端和业务层正确使用，仍然非常重要。

## 小结

- 会话恢复让重复连接同一站点时可以减少完整握手开销
- 前向保密性让长期私钥泄露不至于拖垮过去所有会话的保密性
- 0-RTT 可以进一步降低恢复连接的首请求延迟，但存在重放风险
- 学这些机制时，不能只看“快不快”，还要同时看“安全边界在哪里”
