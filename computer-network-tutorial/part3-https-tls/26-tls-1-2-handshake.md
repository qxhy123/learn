# 第26章：TLS 1.2 握手

## 本章目标

- 理解 TLS 1.2 握手的主要阶段和参与方动作
- 明白协商参数、验证证书、建立会话密钥的大致顺序
- 认识为什么 TLS 1.2 的握手往往比 TLS 1.3 更重
- 为后续比较 TLS 1.2 与 TLS 1.3 做准备

## 先讲结论

TLS 1.2 握手可以先抓住三个核心任务：

1. **双方协商使用什么安全参数**
2. **客户端验证服务器身份**
3. **双方建立后续会话使用的对称密钥材料**

如果你不去纠结每一个报文细节，可以把 TLS 1.2 握手理解成：

- 先打招呼并报出各自支持的能力
- 再由服务器出示证书证明身份
- 然后通过密钥交换建立共享秘密
- 最后双方确认“从现在起正式按协商结果加密通信”

它的整体思路并不神秘，但报文轮次相对更多，历史兼容负担也更重。

## 核心概念

### 1. ClientHello 在做什么

客户端发起 TLS 1.2 握手时，通常先发送 `ClientHello`。

它大致会带上这些信息：

- 自己支持的 TLS 版本范围
- 支持的密码套件列表
- 一个客户端随机数
- 可能的扩展能力，例如 SNI

它表达的意思可以粗略翻译成：

- 我想建立 TLS 连接
- 我支持这些算法和能力
- 你从中挑一个我们都能接受的方案

### 2. ServerHello 在做什么

服务器收到后，会返回 `ServerHello`，告诉客户端：

- 我选择使用哪一个 TLS 版本
- 我选择哪一套密码参数
- 这是我的随机数

接下来服务器还会发送：

- 证书
- 与密钥交换有关的必要参数
- 表示自己这边握手消息先发送完毕的信号

所以在 TLS 1.2 里，服务器这半段动作信息量是比较大的。

### 3. 证书验证是握手中的关键安全步骤

客户端收到服务器证书后，不是直接相信，而是要检查：

- 证书域名是否匹配
- 证书是否在有效期内
- 证书链是否可信
- 证书签名是否成立

如果这些验证失败，后续再怎么协商密钥也没有意义，因为你可能正和错误对象通信。

### 4. 密钥交换在做什么

TLS 1.2 可以支持多种密钥交换方式。对学习来说，先抓住目标即可：

- 双方要在网络上建立一个只有彼此能得到的共享秘密
- 这个共享秘密会进一步派生出会话所需的对称密钥

历史上 TLS 1.2 既存在基于 RSA 的旧式思路，也广泛使用基于临时 Diffie-Hellman 的方式。现代更推荐后者，因为它能提供更好的前向保密性。

### 5. Finished 消息意味着什么

当双方都基于握手消息计算出后续密钥后，还会交换 `Finished` 类消息。

它可以粗略理解为：

- 我已经按刚才所有握手内容算出了密钥
- 现在我用协商出的保护方式发一条校验消息给你
- 如果你也能验证通过，说明我们对之前握手的理解是一致的

因此它不只是“礼貌收尾”，而是在确认：

- 握手没有被暗中篡改
- 双方确实建立了同一套安全上下文

## 过程拆解

### 用一条典型 TLS 1.2 连接理解握手顺序

假设浏览器访问：

`https://example.com`

可以把 TLS 1.2 握手大致理解为：

1. 客户端发送 `ClientHello`，声明支持的版本、密码套件和随机数
2. 服务器返回 `ServerHello`，选定具体参数，并发送自己的随机数
3. 服务器发送证书，必要时还会发送与密钥交换相关的参数消息
4. 客户端验证证书链、域名和签名是否可信
5. 客户端生成或参与计算共享秘密所需的材料，并发送相应握手消息
6. 双方根据客户端随机数、服务器随机数和共享秘密派生出会话密钥
7. 客户端发送切换到加密状态后的 `Finished` 消息
8. 服务器也发送自己的 `Finished` 消息，表示双方握手完成
9. 后续 HTTP 数据开始在 TLS 保护下传输

学习时最值得抓住的是：

- TLS 1.2 在真正传输应用数据前，要先完成较完整的一轮参数协商和身份验证
- 这让它具备安全性，但也带来了更高的握手延迟和更复杂的兼容问题

## 常见误区

### 误区 1：TLS 1.2 握手就是“交换一个密钥”这么简单

不对。握手里至少还包含：

- 协商版本和算法
- 验证证书身份
- 校验整个握手过程是否一致

### 误区 2：证书验证通过后，就说明后续密钥自动安全了

不完整。证书验证解决的是“你是谁”，而密钥交换解决的是“我们如何建立共享秘密”。两件事都重要。

### 误区 3：TLS 1.2 一定不安全

不对。TLS 1.2 本身并不是天然不安全，关键在于：

- 是否禁用了弱算法
- 是否使用现代推荐的密钥交换方式
- 是否正确配置证书和密码套件

### 误区 4：握手完成前，HTTP 已经开始传输正文

通常不是。HTTP 应用数据一般要等 TLS 握手完成后，才会在安全上下文中发送。

## 小结

- TLS 1.2 握手的目标是协商参数、验证身份、建立会话密钥
- 它通常经历 ClientHello、ServerHello、证书验证、密钥交换、Finished 等阶段
- TLS 1.2 的安全性来自多个步骤配合，而不是某一个单独消息
- 理解 TLS 1.2 握手，有助于看懂后面 TLS 1.3 为什么能更快、更简洁
