# 附录 A3：握手速查表

## 使用说明

这一附录的目标不是替代章节学习，而是帮助你在复习时快速回忆：

- TCP 三次握手大致在做什么
- TLS 1.2 与 TLS 1.3 握手主线分别是什么
- 哪些地方是在确认身份，哪些地方是在建立密钥，哪些地方是在确认握手未被篡改

如果你发现自己能背表格，却说不清为什么这样设计，建议回到对应章节重新理解。

## 第一部分：TCP 三次握手速查

### TCP 三次握手主线

| 步骤 | 报文方向 | 关键标志 | 可以怎么理解 |
| --- | --- | --- | --- |
| 1 | 客户端 → 服务端 | SYN | 我想和你建立连接 |
| 2 | 服务端 → 客户端 | SYN + ACK | 我收到了，我也同意建立连接 |
| 3 | 客户端 → 服务端 | ACK | 我确认收到你的回应，连接建立 |

### TCP 三次握手在解决什么

- 确认双方都有基本的收发能力
- 建立连接状态
- 同步初始序号相关上下文
- 为后续可靠传输打基础

### 不要把它理解成什么

- 不是纯形式主义“打招呼三次”
- 不是 HTTP 请求本身
- 不是一建立连接就自动说明应用层一定正常

## 第二部分：TLS 1.2 握手速查

### TLS 1.2 大致流程

| 阶段 | 主要动作 | 重点目的 |
| --- | --- | --- |
| 1 | ClientHello | 客户端声明支持的版本、密码套件、随机数、扩展 |
| 2 | ServerHello + 证书等 | 服务器选定参数并出示证书 |
| 3 | 客户端验证证书 | 验证域名、有效期、信任链、签名 |
| 4 | 密钥交换相关消息 | 双方建立共享秘密所需材料 |
| 5 | Finished | 双方确认握手内容一致、后续进入受保护通信 |

### TLS 1.2 学习抓手

- 先协商参数
- 再验证身份
- 再建立会话密钥
- 最后确认整个握手过程未被篡改

### TLS 1.2 常见记忆重点

- 握手轮次相对更多
- 历史兼容包袱更重
- 配置是否现代化很重要

## 第三部分：TLS 1.3 握手速查

### TLS 1.3 大致流程

| 阶段 | 主要动作 | 重点目的 |
| --- | --- | --- |
| 1 | ClientHello | 客户端尽早带上版本能力、密码参数和密钥交换相关材料 |
| 2 | 服务器响应 + 证书 | 服务器选定参数并证明身份 |
| 3 | 客户端验证证书 | 验证域名、有效期、信任链等 |
| 4 | 双方派生会话密钥 | 更快建立后续加密所需上下文 |
| 5 | 握手完成校验 | 双方确认握手一致，开始稳定传输应用数据 |

### TLS 1.3 相比 TLS 1.2 的速记

- 流程更精简
- 关键材料前置得更多
- 默认更偏现代安全实践
- 常见场景下握手延迟更低

## 第四部分：TCP 与 TLS 握手不要混

很多初学者会把 TCP 和 TLS 混成一个“总握手”。这是非常常见的误区。

可以用下面这个表快速区分：

| 类型 | 解决什么问题 | 发生在哪一层 |
| --- | --- | --- |
| TCP 三次握手 | 建立传输连接状态，准备可靠传输 | 传输层 |
| TLS 握手 | 协商安全参数、验证身份、建立会话密钥 | 位于应用数据传输前的安全层 |
| HTTP 请求 | 表达我要什么资源、你给我什么结果 | 应用层 |

一个常见顺序通常是：

1. 先 TCP 建连
2. 如果是 HTTPS，再做 TLS 握手
3. 然后才开始 HTTP 请求与响应

## 第五部分：排障时怎么用这张表

### 如果三次握手都没完成

重点优先查：

- IP 是否可达
- 目标端口是否监听
- 防火墙或安全组是否阻断
- 网络丢包或路由是否异常

### 如果 TCP 成功但 TLS 握手失败

重点优先查：

- 证书是否有效
- 域名是否匹配
- 中间证书是否完整
- TLS 版本或密码参数是否兼容
- SNI 是否正确

### 如果 TLS 成功但页面仍异常

重点优先查：

- HTTP 状态码
- 重定向链路
- 请求头、Cookie、Token
- 缓存、CORS、资源加载顺序

## 小结

- TCP、TLS、HTTP 各有各的“握手”或“交互”目标，不能混为一谈
- TCP 负责把连接建起来，TLS 负责把安全上下文建起来，HTTP 负责传业务语义
- 速查表的价值在于帮助你迅速定位自己忘的是哪一段流程
- 真正掌握的标志，是你能把这张表讲成一个完整的浏览器访问过程
