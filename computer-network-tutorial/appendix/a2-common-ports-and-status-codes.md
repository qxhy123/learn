# 附录 A2：常见端口与状态码

## 使用说明

这一附录不是要你把所有数字都死记硬背，而是帮助你在排障和阅读请求时，对高频端口与高频状态码有一个快速参照。

## 第一部分：常见端口

### 端口到底是什么

可以先用最实用的角度理解：

- IP 地址用于找到哪台主机
- 端口用于找到这台主机上的哪个服务

所以当你看到 `example.com:443` 时，大致意思是：

- 连接到 `example.com` 对应主机上的 `443` 号服务入口

### 高频端口一览

| 端口 | 常见协议 / 服务 | 说明 |
| --- | --- | --- |
| 20 / 21 | FTP | 传统文件传输协议，现代 Web 场景中已不常作为主流公开服务 |
| 22 | SSH | 远程登录与安全管理常用端口 |
| 25 | SMTP | 邮件传输常见端口之一 |
| 53 | DNS | 域名解析常见端口，既可能走 UDP，也可能走 TCP |
| 67 / 68 | DHCP | 动态主机配置协议常见端口 |
| 80 | HTTP | 明文 Web 访问常见端口 |
| 110 | POP3 | 邮件接收协议之一 |
| 123 | NTP | 网络时间同步常见端口 |
| 143 | IMAP | 邮件接收与管理协议之一 |
| 443 | HTTPS | TLS 保护下的 Web 访问常见端口 |
| 3306 | MySQL | 常见数据库服务端口 |
| 5432 | PostgreSQL | 常见数据库服务端口 |
| 6379 | Redis | 常见缓存 / 键值服务端口 |
| 8080 | HTTP 备用端口 | 开发环境、代理或测试服务中常见 |
| 8443 | HTTPS 备用端口 | 测试环境、管理面板或非标准部署中常见 |

### 关于端口的几个实用提醒

#### 1. 端口只是约定，不是铁律

HTTP 常见是 80，HTTPS 常见是 443，但并不是说：

- HTTP 只能跑在 80
- HTTPS 只能跑在 443

协议和端口没有绝对绑定，只是大家通常遵循这些默认约定。

#### 2. 同一个协议可能不只一个端口

例如：

- 开发环境可能把 Web 服务放在 3000、5173、8080
- 管理后台可能放在 8443
- 内部 API 可能跑在自定义端口上

所以排障时不要因为“不是 80/443”就误判协议。

#### 3. 端口通，不代表应用一定正常

例如 `443` 能连上，只说明：

- 某个服务在监听这个端口，或者中间网关把连接接住了

并不自动保证：

- TLS 握手一定成功
- HTTP 请求一定正确
- 应用逻辑一定正常

## 第二部分：常见 HTTP 状态码

### 状态码分类回顾

HTTP 状态码通常按第一位分组：

- `1xx`：信息性状态
- `2xx`：成功
- `3xx`：重定向
- `4xx`：客户端错误
- `5xx`：服务器错误

学习时最重要的不是死背全部，而是抓住：

- 这是哪一类问题
- 它在请求流程中意味着什么

### 高频状态码一览

| 状态码 | 名称 | 常见含义 |
| --- | --- | --- |
| 200 | OK | 请求成功，服务器正常返回结果 |
| 201 | Created | 资源创建成功，常见于创建类接口 |
| 204 | No Content | 请求成功，但响应体为空 |
| 301 | Moved Permanently | 永久重定向 |
| 302 | Found | 临时重定向 |
| 304 | Not Modified | 资源未变化，常见于协商缓存 |
| 307 | Temporary Redirect | 临时重定向，语义上更强调保持原方法 |
| 308 | Permanent Redirect | 永久重定向，语义上更强调保持原方法 |
| 400 | Bad Request | 请求格式或参数有问题 |
| 401 | Unauthorized | 缺少有效认证信息 |
| 403 | Forbidden | 服务器理解请求，但拒绝执行 |
| 404 | Not Found | 目标资源不存在 |
| 405 | Method Not Allowed | 资源存在，但当前方法不被允许 |
| 408 | Request Timeout | 请求超时 |
| 409 | Conflict | 请求与当前资源状态冲突 |
| 429 | Too Many Requests | 请求过于频繁，被限流 |
| 500 | Internal Server Error | 服务器内部错误 |
| 502 | Bad Gateway | 上游服务返回异常，常见于网关 / 反向代理场景 |
| 503 | Service Unavailable | 服务暂时不可用，可能在维护或过载 |
| 504 | Gateway Timeout | 网关等待上游响应超时 |

### 几个特别容易混淆的状态码

#### 401 与 403

- `401` 更强调：你还没有提供有效身份认证
- `403` 更强调：服务器知道你是谁，但不允许你做这件事

#### 404 与 500

- `404` 表示资源没找到
- `500` 表示服务器内部处理过程中出错

所以看到 `404`，不应第一反应就是“后端挂了”。

#### 502、503、504

这三个在网关、反向代理、微服务环境里很常见：

- `502`：上游返回了无效响应或网关处理上游异常
- `503`：服务暂时不可用，可能是过载、维护、限流
- `504`：网关等上游等太久，最终超时

这类状态码往往提示你：

- 问题可能在网关后面的上游链路，不一定是当前入口服务自己直接崩了

## 第三部分：如何在排障中使用这些数字

### 场景 1：看到端口 443 不通

先想到的应是：

- TCP 连通性
- 防火墙 / 安全组
- 服务监听状态
- 入口网关配置

而不是直接先猜 HTTP 接口逻辑。

### 场景 2：看到 `304 Not Modified`

这通常不是报错，而是：

- 浏览器与服务器做了协商缓存
- 服务器告诉浏览器本地缓存副本仍然可用

### 场景 3：看到 `429 Too Many Requests`

这通常说明：

- 服务在做限流或风控
- 问题可能不在“服务器坏了”，而在请求频率超出策略阈值

### 场景 4：看到 `502` 或 `504`

这类状态码常提示：

- 入口代理和上游服务之间出了问题
- 需要看代理层、网关层和上游服务日志，而不只是盯着浏览器

## 小结

- 端口帮助你判断“连接到主机上的哪个服务入口”
- 状态码帮助你判断“HTTP 请求处理到了哪一步、结果属于哪一类”
- 真正有用的不是死背数字，而是看到数字就能联想到它所在的层次和排查方向
- 当端口、TLS、状态码这些信息能在你脑中自动连成路径时，排障效率会明显提升
